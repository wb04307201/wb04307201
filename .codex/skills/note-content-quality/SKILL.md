# note 内容质量评价

## Overview

对 `note/` 中的文章做**内容质量评价**（不是结构审计 —— 结构审计由 `note-audit-and-improvement` 负责）。本 skill 评价的是**内容本身**：准确性、深度、可读性、模块规范符合度。

**与已有 skill 的分工**：

| Skill | 检查什么 | 不检查什么 |
|-------|---------|-----------|
| `note-precipitation-planning` | 位置选择 + 互链结构 | 内容质量 |
| `note-audit-and-improvement` | 链接/数字/索引/孤岛 | 内容准确性、可读性 |
| **`note-content-quality`**（本 skill） | **内容准确性 + 深度 + 可读性 + 模块规范** | 链接有效性、数字一致性 |

## When to Use

**Use when**：
- 用户问"这篇/这些文章质量怎么样？"
- 用户问"评价一下刚沉淀的内容"
- 用户说"帮我 review 一下 note/XX/ 的内容质量"
- 沉淀完成后想做质量验收（配合 `note-precipitation-planning` 的 Step 7）

**Don't use when**：
- 用户问"note 哪里需要优化"（→ 用 `note-audit-and-improvement`）
- 用户问"X 应该沉淀到哪"（→ 用 `note-precipitation-planning`）
- 用户已知具体问题要修复（→ 直接修复）

## 评价流程（4 步）

### Step 1: 确定评价范围

```
用户指定范围？
├─ 单篇文章 → 读全文，逐维度评分
├─ 某个目录 → 读目录内所有 .md，批量评分 + 汇总
└─ 刚沉淀的新文件 → 只评价新增/修改的文件
```

**读取范围**：
- 单篇：Read 全文
- 目录：Glob 找所有 .md，逐个 Read（超过 10 篇时分批，每批 ≤ 5 篇）
- 新沉淀：从 git diff 获取文件列表

### Step 2: 通用维度评分（所有模块共享）

对每篇文章，先过**6 个通用维度**（每项 0-2 分）：

| # | 维度 | 2 分（优秀） | 1 分（及格） | 0 分（不达标） |
|---|------|------------|------------|--------------|
| G1 | **frontmatter 完整性** | 有完整 frontmatter 且字段正确 | 有 frontmatter 但缺字段 | 无 frontmatter |
| G2 | **一句话定位** | H1 后有 ≤80 字的清晰定位 | 有定位但 >80 字或模糊 | 无定位 |
| G3 | **代码块规范** | 所有 `` ``` `` 都声明了语言 | 大部分声明了 | 有裸 `` ``` `` |
| G4 | **回链 + 互链** | 有 footer 回链 + ≥2 个旧章节互链 | 有回链但互链不足 | 无回链 |
| G5 | **内容深度** | 覆盖 ≥4 个层面 + 有实战案例 | 覆盖 2-3 个层面 | 只有定义 + 一句话 |
| G6 | **可读性** | 结构清晰、表格/图/代码辅助、无长段落 | 基本可读但有改进空间 | 大段纯文字、难以阅读 |

### Step 3: 模块专属维度评分

根据文件路径自动选择模块规则集：

```
文件路径 → 模块规则
├─ note/01.java/**        → 技术深度规则（A 类）
├─ note/02.computer-basics/** → 技术深度规则（A 类）
├─ note/03.database/**    → 技术深度规则（A 类）
├─ note/04.system-design/** → 技术深度规则（A 类）
├─ note/05.tools/**       → 工具实践规则（B 类）
├─ note/06.spring/**      → 技术深度规则（A 类）
├─ note/07.workflow/**    → 工具实践规则（B 类）
├─ note/08.application-systems/** → 技术深度规则（A 类）
├─ note/09.front-end/**   → 技术深度规则（A 类）
├─ note/10.big-data/**    → 技术深度规则（A 类）
├─ note/11.ai/**          → AI/研究规则（C 类）
├─ note/12.story/**       → 叙事规则（D 类）
├─ note/13.split-hairs/** → 面试题规则（E 类）
└─ note/14.project-management/** → 工具实践规则（B 类）
```

#### A 类：技术深度规则（01.java / 02-04 / 06 / 08-10）

| # | 维度 | 2 分 | 1 分 | 0 分 |
|---|------|------|------|------|
| A1 | **源码级深度** | 有源码片段 + 注释解释 *为什么* | 有代码但只解释 *是什么* | 无代码示例 |
| A2 | **版本/演进对比** | 有版本对比（如 JDK 7 vs 8）或演进说明 | 提及版本差异但未展开 | 无版本意识 |
| A3 | **反例 vs 正例** | 有 ❌/✅ 对比展示 | 只有正确用法 | 无代码对比 |
| A4 | **参数/配置表** | 有实际参数值 + 调优建议 | 有参数表但无调优建议 | 无参数信息 |

#### B 类：工具实践规则（05.tools / 07.workflow / 14.project-management）

| # | 维度 | 2 分 | 1 分 | 0 分 |
|---|------|------|------|------|
| B1 | **安装步骤可执行** | 命令可直接复制运行 + 有前置条件说明 | 有命令但缺前置条件 | 无安装命令 |
| B2 | **配置示例** | 有完整配置文件示例 + 关键参数注释 | 有配置但无注释 | 无配置示例 |
| B3 | **使用场景** | 有 ≥2 个实际使用场景 + 示例 | 有 1 个场景 | 无使用场景 |
| B4 | **对比/选型** | 有同类工具对比表 + 选型建议 | 提及竞品但无对比 | 无对比信息 |

#### C 类：AI/研究规则（11.ai）

| # | 维度 | 2 分 | 1 分 | 0 分 |
|---|------|------|------|------|
| C1 | **量化严谨性** | 有公式/数字 + 变量定义清晰 | 有数字但缺公式或变量定义 | 纯定性讨论无数字 |
| C2 | **架构对比表** | 有多维对比表（≥3 维度） | 有对比但不完整 | 无对比 |
| C3 | **实战部署指导** | 有场景化推荐（"X 场景用 Y"） | 有泛泛建议 | 无部署指导 |
| C4 | **学术/开源引用** | 有论文链接或开源项目引用 | 提及但无链接 | 无引用 |

#### D 类：叙事规则（12.story）

| # | 维度 | 2 分 | 1 分 | 0 分 |
|---|------|------|------|------|
| D1 | **阿明餐厅叙事** | 以阿明餐厅场景开头 + 全文贯穿 | 提及阿明但不贯穿 | 无阿明餐厅元素 |
| D2 | **技术→隐喻映射** | 每个核心概念都有餐厅隐喻 | 部分概念有隐喻 | 无隐喻映射 |
| D3 | **章节编号规范** | 中文序数（第一章、第二章）+ 一致 | 编号基本一致 | 编号混乱或混用系统 |
| D4 | **叙事弧线** | 有起承转合 + 情节推进 | 有基本结构但平淡 | 无叙事结构 |
| D5 | **系列定位** | 有 `> **系列定位**：` 块 + 正确位置 | 有定位但格式不对 | 无系列定位 |
| D6 | **Mermaid 节制** | ≤4 个图 + `graph TD` 优先 | 图数量合理 | 图过多或格式不对 |

#### E 类：面试题规则（13.split-hairs）

| # | 维度 | 2 分 | 1 分 | 0 分 |
|---|------|------|------|------|
| E1 | **引子（Hook）** | 有 `## 引子：` + 4 类之一（生产事故/反直觉代码/架构困境/性能对比） | 有引子但不够吸引人 | **无引子 = 严重违规** |
| E2 | **陷阱/反直觉** | 有 `### 陷阱 N：` + `**真相**：` 格式 | 有陷阱但格式不对 | 无陷阱部分 |
| E3 | **面试话术** | 有 30 秒版 + 90 秒版（blockquote 格式） | 有 1 个版本 | 无面试话术 |
| E4 | **代码反例 vs 正例** | 有 ❌/✅ 代码对比 | 只有正确代码 | 无代码示例 |
| E5 | **主模块回链** | 有双向链到主模块对应文章 | 有单向链 | 无主模块链接 |
| E6 | **难度 + 频率标注** | frontmatter 有 difficulty ⭐ + frequency | 有其一 | 无标注 |

### Step 4: 输出评价报告

#### 单篇文章报告格式

```markdown
## 📊 内容质量评价：{文章标题}

**文件**：`{路径}`
**模块类型**：{A/B/C/D/E 类}
**总分**：{X}/{满分} = {等级}

### 通用维度（{X}/12）

| # | 维度 | 分数 | 说明 |
|---|------|------|------|
| G1 | frontmatter | {0-2} | {一句话} |
| G2 | 一句话定位 | {0-2} | {一句话} |
| G3 | 代码块规范 | {0-2} | {一句话} |
| G4 | 回链 + 互链 | {0-2} | {一句话} |
| G5 | 内容深度 | {0-2} | {一句话} |
| G6 | 可读性 | {0-2} | {一句话} |

### 模块专属维度（{X}/{满分}）

| # | 维度 | 分数 | 说明 |
|---|------|------|------|
| {X}N | {维度名} | {0-2} | {一句话} |
...

### 🔧 修复建议（按优先级排序）

| # | 优先级 | 问题 | 修复方式 |
|---|--------|------|---------|
| 1 | P0 | {最严重问题} | {具体修复指令} |
| 2 | P1 | {次要问题} | {具体修复指令} |
...

### ✅ 亮点

- {文章做得好的地方，鼓励继续保持}
```

#### 批量评价汇总格式

```markdown
## 📊 内容质量批量评价

**范围**：`{目录}`
**文章数**：{N} 篇
**平均分**：{X}/{满分}

### 评分总表

| 文件 | 模块 | 通用分 | 专属分 | 总分 | 等级 |
|------|------|--------|--------|------|------|
| {path} | {类型} | {X}/12 | {X}/{满分} | {X} | {等级} |
...

### 共性问题（跨文章）

| # | 问题 | 影响文章数 | 修复方式 |
|---|------|-----------|---------|
| 1 | {共性问题} | {N} 篇 | {批量修复方式} |

### 🏆 最佳文章

- {最高分文章} — {为什么好}

### ⚠️ 最需要改进

- {最低分文章} — {主要问题}
```

#### 评分等级

| 总分比例 | 等级 | 含义 |
|---------|------|------|
| ≥ 85% | ⭐⭐⭐⭐⭐ 优秀 | 可直接发布 |
| 70-84% | ⭐⭐⭐⭐ 良好 | 小修后可发布 |
| 55-69% | ⭐⭐⭐ 及格 | 需要补充内容 |
| 40-54% | ⭐⭐ 待改进 | 需要大幅修改 |
| < 40% | ⭐ 不达标 | 需要重写 |

#### 满分计算

- 通用维度：6 项 × 2 分 = **12 分**（所有模块相同）
- 专属维度：按模块不同
  - A 类（技术）：4 项 × 2 分 = 8 分 → 满分 **20**
  - B 类（工具）：4 项 × 2 分 = 8 分 → 满分 **20**
  - C 类（AI）：4 项 × 2 分 = 8 分 → 满分 **20**
  - D 类（叙事）：6 项 × 2 分 = 12 分 → 满分 **24**
  - E 类（面试）：6 项 × 2 分 = 12 分 → 满分 **24**

## 特殊处理

### 引子缺失 = 自动 P0（13.split-hairs 专属）

如果 `13.split-hairs/` 下的文章缺少 `## 引子：` 或 `## 引子` 章节，**无论其他维度得分多高，总分降一个等级**。这是 QUESTION-FORMAT-SPEC.md 的硬性要求。

### 叙事缺失 = 自动 P0（12.story 专属）

如果 `12.story/` 下的文章完全没有阿明餐厅元素（无角色、无场景、无隐喻），**标记为"放错位置"**——可能是技术文章被错误放到了 12.story。

### 新沉淀文章的特殊关注

对刚通过 `note-precipitation-planning` 沉淀的文章，额外检查：
- **事实准确性**：文中提到的数字、版本号、API 名称是否与实际一致（可通过 WebSearch 验证关键声明）
- **安装命令可执行性**：`npm install` / `pip install` 等命令是否是真实的包名
- **引用链接有效性**：GitHub URL / 论文链接是否能访问

### 批量评价的效率

- 单篇：Read 全文 + 逐维度评分（~3 分钟）
- 5 篇以内：逐篇 Read + 逐篇评分（~15 分钟）
- 5-20 篇：break into sequential helper tasks 并行评价（每篇一个 agent），最后汇总
- 20+ 篇：先筛选（< 50 行的可能太短 / > 500 行的可能太冗长），再逐批评价

## 常见反模式速查

| 反模式 | 检测方式 | 修复 |
|--------|---------|------|
| **定义堆砌** | 全文只有 "X 是 Y" 句式，无示例、无对比 | 加代码示例 + 反例对比 |
| **纯翻译** | 内容像英文文档的直译，无本土化 | 加中文场景 + 国内工具对比 |
| **过期内容** | 提到已废弃的 API / 过时的版本号 | 更新到当前版本 + 标注变更 |
| **孤岛文章** | 无任何互链（与 `note-precipitation-planning` 联动） | 加 ≥2 个旧章节互链 |
| **占位符残留** | "TODO" / "待补充" / "TBD" | 补充或删除 |
| **数字无出处** | "性能提升 20 倍" 但无 benchmark 来源 | 加出处或改为 "约 X 倍" |
| **代码无注释** | 大段代码但无解释 | 加行内注释 + 段落说明 |
| **表格无结论** | 有对比表但无"选哪个"的建议 | 加 "推荐" / "适合场景" 列 |

## Output Format

**as the main task**（直接面对用户）：
- 用 Step 4 的报告格式直接输出
- 如有 P0 问题，用 ask the user directly 问是否立即修复

**as a helper task**（被 dispatch 评价单篇）：
- 返回 Step 4 的单篇文章报告格式
- 不要用 ask the user directly（not available in helper tasks）

## Quick Checklist（评价前必过）

- [ ] 确定评价范围（单篇 / 目录 / 新文件）
- [ ] 识别每篇文章的模块类型（A/B/C/D/E）
- [ ] 通用 6 维度 + 模块专属维度都评分
- [ ] 修复建议按 P0 → P2 排序
- [ ] 亮点部分不遗漏（正向反馈很重要）
- [ ] 批量评价时有共性问题汇总
