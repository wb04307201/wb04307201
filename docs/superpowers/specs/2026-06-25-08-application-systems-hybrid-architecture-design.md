# 08.application-systems 混合架构 — 设计文档

> **For agentic workers:** 设计已批准。配套实施计划见 `docs/superpowers/plans/2026-06-25-08-application-systems-hybrid-architecture.md`。

**Goal:** 把 `note/08.application-systems/` 从「单文件速查手册」（531 行）升级为「混合架构」：保留总 README 作为 21 系统的速查手册/索引，同时为 6 个核心系统（PLM / PDM / MES / CRM / ERP / WMS）新增独立深读子目录。

**Architecture:** 总 README 保持「地图」定位（精简、导航、速查），6 个深读子目录作为「放大镜」（每目录 200-500 行，覆盖定义/能力/场景/上下游/选型 + 历史/陷阱/案例）。

**Tech Stack:** Markdown + Mermaid 流程图；继续遵守仓库「零 PNG」约定。

## 1. 背景与动机

### 1.1 现状（2026-06-25）

`note/08.application-systems/README.md` 单一文件 531 行，覆盖 21 个业务系统：
- 6 详讲（PLM / PDM / MES / CRM / ERP / WMS）：每系统约 80-100 行
- 16 简讲：每系统 3-5 行简介

### 1.2 问题

1. **深度受限**：单文件总篇幅 531 行，21 系统分摊，每个详讲系统只能写到 80-100 行，选型/实施/案例讲不透
2. **导航成本高**：业务/产品/需求人员想「先速查后深读」时，必须在同一文件内反复滚动
3. **未来扩展困难**：若要把某一系统再扩深（如 ERP 实施案例库），会进一步膨胀单文件

### 1.3 设计目标

| 目标 | 度量 |
|------|------|
| 速查更轻 | 总 README 从 531 行精简到 380-420 行（-25%） |
| 深读更深 | 6 系统各 200-500 行独立深读，可选 9 节模板按需取用 |
| 跳转双通道 | 价值链章节内「📚 详见深读」 + 速查表「📚 深读」列均可跳 |
| 仓库一致性 | 沿用 Markdown + Mermaid，零 PNG；与 09.front-end / 10.big-data 子目录命名风格统一 |

## 3. 深读子目录 README 标准结构

每个深读 README 采用「6 标准节 + 3 可选节」模板，按系统特性挑选 6-9 节：

```
# 系统名（中文+英文+缩写）

> 一句话定位

## 📌 全景图                （mermaid flowchart，标准节入口）
## 📖 定义                  （标准节 1）
## 🔧 核心能力              （标准节 2）
## 🏭 典型场景              （标准节 3）
## 🔗 上下游关系            （标准节 4）
## ⚖️ 关键考量              （标准节 5）
## 🎯 选型指南              （标准节 6）
## 📜 历史脉络              （可选 1）
## ⚠️ 常见陷阱              （可选 2）
## 📚 代表案例              （可选 3）
## 🔗 关联链接              （返回总 README 对应章节锚点 + 上下游系统深读）
```

### 3.1 各系统的可选节选取建议

| 系统 | 建议选取可选节 | 理由 |
|------|---------------|------|
| PLM | 历史脉络 + 常见陷阱 + 代表案例 | 完整技术演进史、行业实施陷阱丰富、有经典案例 |
| PDM | 历史脉络 + 代表案例 | PDM 是 PLM 的前身，历史脉络独立价值高，案例可与 PLM 互补 |
| MES | 典型场景 + 常见陷阱 + 代表案例 | 离散 vs 流程行业场景差异大，MES 是实施陷阱最多的系统之一 |
| CRM | 典型场景 + 常见陷阱 | 行业 SaaS 化成熟，案例以厂商为主避免主观，陷阱更普适 |
| ERP | 典型场景 + 常见陷阱 + 代表案例 | ERP 是最重、最容易踩坑的系统，案例库丰富 |
| WMS | 典型场景 + 常见陷阱 | WMS 场景相对清晰（仓储作业），不需要大量历史脉络 |

### 3.2 每节内容深度建议

| 节 | 行数范围 | 内容要点 |
|----|---------|---------|
| 📌 全景图 | 5-15 | mermaid flowchart，含输入/输出/上下游 |
| 📖 定义 | 15-30 | 一句话定位 + 完整定义 + 与相邻系统（PDM vs PLM、ERP vs MES）的边界澄清 |
| 🔧 核心能力 | 30-60 | 5-8 个核心能力，每能力 2-3 行描述 |
| 🏭 典型场景 | 30-60 | 3-5 个场景，每场景描述「痛点 + 系统如何解决 + 落地效果」 |
| 🔗 上下游关系 | 20-40 | 上游系统（数据/事件来源）、下游系统（数据/事件去向）、横向系统（同环节） |
| ⚖️ 关键考量 | 30-50 | 选型/实施/运维的关键决策点，每考量 3-5 行 |
| 🎯 选型指南 | 30-50 | 决策维度（功能/规模/行业/集成/成本）+ 选型树/表 |
| 📜 历史脉络 | 20-40 | 时间线（80 年代起源 → 90 年代发展 → 2000 成熟 → 2010+ 云化/AI） |
| ⚠️ 常见陷阱 | 30-50 | 3-5 个陷阱，每陷阱描述「现象 + 根因 + 规避」 |
| 📚 代表案例 | 30-50 | 1-3 个公开案例（行业 + 厂商 + 效果），不写厂商主观对比表 |

总行数范围 200-500 行，6 标准节 + 2-3 可选节平均分布。

## 4. 总 README 调整方案

### 4.1 速查表新增「📚 深读」列

### 4.2 价值链章节详讲精简

对有深读的 5 个章节（01 研发创新 / 02 生产制造 / 03 供应链 / 04 销售服务 / 05 运营管理），将该章节内的 6 处详讲系统从 80-100 行精简到 10-15 行：

- **核心定位**：管理产品从概念到退役的全生命周期数据与流程

## 6. 全局约束（Global Constraints）

继承 `08.application-systems` 既有约定：
- **零 PNG**：所有图必须 mermaid（含流程图、时序图、ER 图、思维导图）
- **Markdown + 中文**：与仓库其他章节风格一致
- **不写厂商主观对比表**：避免倾向性，案例可引用公开资料
- **不重复主 README 已有内容**：深读子目录只放主 README 没空间放的「深读内容」；一句话定位等简述可以重复（便于独立阅读）
- **链接风格**：相对路径（如 `./plm/`），不使用绝对路径
- **Mermaid 兼容性**：避免使用 `mindmap` 等渲染支持有限的语法（沿用 09.front-end 的经验）

## 7. 范围之外（Out-of-Scope）

明确不做：
1. 16 简讲系统不拆深读（保持现有 3-5 行简介，避免工作量爆炸）
2. 集成模式 / 学习路线 / 快速入口 / 全景图四大支撑章节不动
3. 不引入图片（继续 mermaid）
4. 不写厂商对比表
5. 不动 `note/README.md` 的 08 章节索引（链接维持原状）
6. 不动其他 7 个一级章节（01-07、09-13）

## 9. 风险与缓解

| 风险 | 缓解 |
|------|------|
