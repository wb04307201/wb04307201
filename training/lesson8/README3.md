## 🔍 核心问题：长时运行智能体的挑战

随着 AI 智能体（Agent）能力增强，开发者开始让它们承担需要数小时甚至数天的复杂任务。但**跨多个上下文窗口持续工作**仍是开放难题。

### 根本挑战
> 智能体必须在离散的会话中工作，每个新会话开始时都没有之前会话的记忆。就像轮班工作的工程师，每班次的新人都不记得上一班发生了什么。

### 两大典型失败模式
| 失败模式 | 具体表现 | 后果 |
|---------|---------|------|
| **过度尝试（One-shot）** | 智能体试图一次性完成整个应用 | 上下文耗尽，功能实现一半，后续会话需猜测恢复 |
| **过早宣布完成** | 看到部分进展就认为任务已完成 | 遗漏关键功能，交付质量不达标 |

---

## 🛠️ 解决方案：双智能体框架

Anthropic 提出了一套**两阶段架构**，使 Claude Agent SDK 能有效跨越多上下文窗口工作：

### 1️⃣ 初始化智能体（Initializer Agent）
**职责**：首次运行时搭建基础环境
```bash
# 关键产出物
✅ init.sh          # 环境启动脚本
✅ claude-progress.txt  # 进度日志文件
✅ 初始 Git 提交      # 记录文件变更历史
✅ feature_list.json   # 结构化功能需求清单
```

### 2️⃣ 编码智能体（Coding Agent）
**职责**：每次会话专注增量进展，并为下一会话留下清晰交接
```text
核心原则：
• 一次只实现一个功能
• 每次修改后提交清晰的 Git commit
• 更新进度文件，保持代码"可合并"状态
```

> 💡 **关键洞察**：让智能体在新上下文窗口中快速理解工作状态，核心工具是 `claude-progress.txt` + Git 历史记录的组合。

## 📋 环境管理最佳实践

### 🎯 功能清单设计（feature_list.json）
```json
{
  "category": "functional",
  "description": "新建聊天按钮创建新对话",
  "steps": [
    "导航到主界面",
    "点击'新建聊天'按钮",
    "验证新对话已创建",
    "检查聊天区显示欢迎状态",
    "验证对话出现在侧边栏"
  ],
  "passes": false  // 初始标记为未完成
}
```

**设计要点**：
- 基于用户初始需求拆解为 200+ 原子功能项
- 所有功能初始标记为 `"passes": false`
- 编码智能体**仅允许**修改 `passes` 字段，禁止删除/编辑测试项
- 使用 JSON 格式（比 Markdown 更不易被模型误改）

### 🔄 增量开发策略
```text
✅ 每次会话只专注 1 个功能
✅ 代码变更后立即提交 Git + 描述性 commit message
✅ 会话结束时更新进度文件
✅ 保持代码"干净"：无重大 bug、结构清晰、文档完整
```

### 🧪 端到端测试机制
**问题**：智能体常未充分测试就标记功能完成

**解决方案**：
- 明确提示使用浏览器自动化工具（如 Puppeteer MCP）
- 要求像真实用户一样进行端到端验证
- 截图记录测试过程，辅助问题定位

> ⚠️ 当前局限：模型视觉能力 + 浏览器自动化仍有盲区（如无法检测原生 alert 弹窗）

## 🚀 智能体启动流程（每会话标准步骤）

```text
1️⃣ 运行 `pwd` 确认工作目录
2️⃣ 读取 Git 日志 + progress 文件，了解近期进展
3️⃣ 读取 feature_list.json，选择最高优先级的未完成功能
4️⃣ 运行 init.sh 启动开发服务器
5️⃣ 执行基础端到端测试，验证核心功能正常
6️⃣ 开始新功能的实现工作
```

**典型会话开场**：
```text
[助手] 我先了解项目当前状态
[工具] <bash - pwd>
[工具] <read - claude-progress.txt>
[工具] <read - feature_list.json>
[助手] 查看 git log 了解近期工作
[工具] <bash - git log --oneline -20>
[助手] 检查 init.sh 并启动开发服务器
[助手] 验证核心功能是否正常...
[助手] 基础功能正常，现在查看测试清单，开始下一个功能
```

## 📊 问题 - 方案对照表

| 问题           | 初始化智能体方案                | 编码智能体方案                           |
|--------------|-------------------------|-----------------------------------|
| 过早宣布项目完成     | 创建结构化 feature_list.json | 会话开始读取清单，每次只选 1 个功能               |
| 留下 bug 或混乱代码 | 初始化 Git 仓库 + 进度文件       | 会话开始读日志+测试，结束时提交 commit+更新进度      |
| 未测试就标记完成     | 创建带测试步骤的功能清单            | 自我验证所有功能，仅测试通过后才标记 `passes: true` |
| 浪费时间配置环境     | 编写 init.sh 启动脚本         | 会话开始读取并执行 init.sh                 |

## 🔮 未来方向

1. **架构探索**：单通用编码智能体 vs 多智能体协作（测试/QA/代码清理等专用智能体）
2. **领域泛化**：当前优化于全栈 Web 开发，未来可应用于科研、金融建模等长时任务场景
3. **工具增强**：提升模型视觉能力 + 浏览器自动化覆盖范围

## 💡 实践建议总结

```text
✅ 拆解任务：将大目标分解为原子功能清单
✅ 状态持久化：用结构化文件 + Git 记录进度
✅ 增量迭代：每次会话专注小目标，避免上下文过载
✅ 测试驱动：明确要求端到端验证，而非仅代码检查
✅ 环境标准化：提供 init 脚本，降低每会话启动成本
✅ 交接清晰：会话结束必须留下"可继续工作"的代码状态
```

> 📌 **核心思想**：借鉴人类工程师的协作实践——清晰的文档、版本控制、测试流程和交接规范，是让长时运行智能体可靠工作的关键。
