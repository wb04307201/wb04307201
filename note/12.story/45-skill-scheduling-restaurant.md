# 扩招 50 个厨师：Skill 调度的餐厅隐喻

> 一句话定位：当阿明餐厅从 5 个厨师扩到 50 个，**怎么保证前厅喊"做一道宫保鸡丁"时，后厨不会全员出动？** —— 这是 Claude Code 在 50+ Skills 场景下"命中率"问题的餐厅版。

---

## 引子：扩招的烦恼

阿明餐厅火了。从街角小店开到连锁 12 家，**厨师从 5 个扩到 50 个**——川菜、粤菜、湘菜、西餐、日料、甜品……分得越来越细。

但问题来了：

**前厅小李喊"做一道宫保鸡丁"** ——

```text
可能落到：
├─ 川菜厨师 A（正宗做法，花生脆、糊辣荔枝味）
├─ 川菜厨师 B（改良版，加腰果减花椒）
├─ 湘菜厨师 C（带辣度但偏酸）
├─ 鲁菜厨师 D（北方版宫保鸡丁，加黄瓜）
├─ 新来的学徒 E（看菜谱瞎做）
└─ 50 个厨师全员齐动手（灾难）
```

**前端每次喊菜，后厨 50 个人都要看一眼菜单 → 撞名 / 抢活 / 出错。**

阿明这才明白：**扩招不是"招更多厨师"，而是"建立调度系统"**。

---

## 一、问题的本质：50 个候选 = 50 个选择

### 1.1 餐厅版 vs Claude Code 版

| 餐厅场景 | Claude Code 等价 | 失败模式 |
|---------|-----------------|---------|
| 50 个厨师听到"做宫保鸡丁" | 50 个 Skills 注入 Prompt | Token 爆炸 + 注意力分散 |
| 多个厨师抢同一道菜 | 多个 Skill 描述雷同 → Agent 选错 | 描述冲突 |
| 学徒瞎做 | LLM 在长尾 query 上瞎猜 | 选择漂移 |
| 厨师做完不知道效果好不好 | 没有命中率埋点 | 盲飞 |

**核心洞察**：**Skill（厨师）不是越多越好，关键是"调度"——让对的厨师接对的菜。**

### 1.2 阿明的第一反应：写岗位手册

阿明做了第一件事：**给每个厨师写岗位手册**：

```yaml
# 川菜厨师 A
name: 川菜主厨-A
description: |
  做正宗川菜。专注宫保鸡丁、麻婆豆腐、水煮鱼。
  不做粤菜、不做甜品、不做早餐。
when_not_to_use:
  - 粤菜订单（请找粤菜厨师）
  - 甜品订单（请找甜品师）
```

效果：**命中率从 60% 涨到 78%**——但还不够。

### 1.3 阿明的第二反应：建立调度台

光写岗位手册不够，**前厅喊菜也得有规则**：

```text
前厅小李接到订单：
1. 先看订单关键词（"宫保鸡丁" / "叉烧" / "提拉米苏"）
2. 命中关键词 → 直接派单到对应厨师
3. 没命中关键词 → 让资深前厅判断
4. 资深前厅拿不准 → 让顾客稍等，去问店长（兜底）
```

**这就是 Claude Code 的"规则 + LLM 兜底"路由**。

效果：**命中率从 78% 涨到 88%**。

---

## 二、四层调度模型（餐厅版）

### 2.1 L1 岗位说明（描述层）

**每个厨师必须有结构化岗位说明**，不能只写"我是川菜厨师"。

反例：
```yaml
description: 川菜厨师
# 前厅看了还是不知道：能做宫保鸡丁吗？能做水煮鱼吗？
```

正例：
```yaml
description: |
  川菜主厨。专做宫保鸡丁、麻婆豆腐、水煮鱼、回锅肉、鱼香肉丝。
  不做早餐、不做粤菜、不做甜品。
when_to_use:
  - "川菜订单、辣味主菜、家常川菜"
when_not_to_use:
  - "粤菜（找粤菜厨师）"
  - "甜品（找甜品师）"
  - "早餐（找早点组）"
```

**关键：写清 `when_not_to_use`，让前厅能拒识**。

### 2.2 L2 派单规则（路由层）

**前厅不靠"自由判断"，靠规则手册**：

```yaml
# 派单规则表（节选）
rules:
  - keyword: "宫保鸡丁|麻婆|水煮鱼|回锅肉"
    route_to: 川菜主厨-A
  - keyword: "叉烧|白切鸡|肠粉|早茶"
    route_to: 粤菜厨师-F
  - keyword: "提拉米苏|慕斯|蛋糕"
    route_to: 甜品师-K
  # ... 共 30 条
fallback: 资深前厅判断（LLM 兜底）
```

**30 条规则覆盖 80% 订单**，剩下 20% 长尾让资深前厅判断。

**资深前厅的"Prompt"**：
```text
你是餐厅调度员。当前在岗厨师：
- 川菜主厨-A（专做川菜）
- 粤菜厨师-F（专做粤菜）
- 甜品师-K（专做甜品）
- ... 共 50 个

顾客订单：「红烧肉」
请选择最合适的厨师。如果都不匹配，返回 SKIP。
```

### 2.3 L3 上下文加载（加载层）

**前厅每次只给厨师"点单信息"，不是"全部菜单"**：

```text
错误做法：给厨师看完整 50 页菜单（Token 爆炸）
正确做法：给厨师一张小卡片：
  - 顾客订单：「宫保鸡丁，少花生」
  - 顾客备注：「不要太辣」
  - 出餐时间：20 分钟内
```

**对应 Claude Code**：
- 错误：全量 Skill 文件塞进 Prompt（50 个 Skill × 几 KB = 几十 KB）
- 正确：默认只注入 Skill 描述（< 100 tokens），命中后按需 `load_skill_full`

### 2.4 L4 调度评估（评估层）

**阿明建立"调度日志"，每周复盘**：

```text
本周调度日志：
- 总订单 1000 单
- 命中正确厨师：920 单（92%）
- 派错厨师：50 单
- 没有合适厨师：30 单

派错 TOP 3：
1. 鱼香肉丝派给湘菜厨师（8 次）—— 描述里有"鱼"被误判
2. 早餐订单派给川菜厨师（5 次）—— when_not_to_use 没写早餐
3. 甜品订单没人接（3 次）—— 甜品师休息时段没标记
```

**这就是 Claude Code 的"离线评测集 + 回归测试"**——每周跑一次，看命中率退没退化。

## 三、五大反模式（餐厅版）

### 反模式 1：菜名撞车（关键词冲突）

**症状**：川菜的"宫保鸡丁"和鲁菜的"宫保鸡丁"做法完全不同，但都叫"宫保鸡丁" → 前厅随机派。

**修复**：差异化命名。川菜的叫"川式宫保鸡丁"，鲁菜的叫"鲁式宫保鸡丁"。

### 反模式 2：岗位说明雷同（描述雷同）

**症状**：川菜厨师 A 和川菜厨师 B 的说明几乎一样 → 前厅不知道派谁。

**修复**：明确分工。"A 做主菜，B 做小吃凉菜"。

### 反模式 3：前厅喊单太啰嗦（上下文污染）

**症状**：前厅喊"顾客要川菜、微辣、不要香菜、加花生、少油、米饭单独装..." → 厨师被无关信息干扰。

**修复**：精简喊单内容，只保留关键约束。

### 反模式 4：厨师轮换导致派错（LLM 漂移）

**症状**：同一道菜，今天派给 A，明天派给 B → 顾客体验不稳定。

**修复**：**规则路由** + **强制关键词触发**。

### 反模式 5：扩招到 100 个厨师（数量爆炸）

**症状**：50 个厨师已经难调度，扩到 100 个更是灾难。

**修复**：
- **合并岗位**：3 个相似的川菜副厨合并成 1 个
- **分层**：核心 10 个厨师常驻 + 扩展 40 个按需排班
- **退役**：3 个月没出单的厨师转岗或辞退

---

## 四、阿明的最终方案

**扩招 50 个厨师 + 4 层调度模型**：

```text
岗位说明（YAML） + 派单规则（30 条） + 精简喊单 + 每周复盘
= 命中率 92%，顾客投诉下降 60%
```

**对应到 Claude Code**：
```text
Skill frontmatter + 关键词规则路由 + 分层加载 + 离线评测集
= 命中率 92%，Token 成本降 60%
```

**同一个问题，不同的语言。**

---

## 五、给读者的启示

1. **"多" ≠ "好"**。5 个调得动的厨师 > 50 个调不动的厨师。
2. **结构化文档 > 自然语言描述**。YAML / 规则表比"我啥都会做"靠谱。
3. **规则 + 兜底 > 全靠判断**。给前厅手册 + 让资深兜底，比让前厅自由发挥稳定。
4. **量化一切**。没有"命中率"数字，所有优化都是凭感觉。
5. **长期维护**。3 个月没出单的厨师 / Skill，要主动退役，否则整个系统越来越臃肿。
