# 「阿明餐厅」故事格式规范 v1.0

> 本规范定义 `note/12.story/` 目录下所有 `.md` 故事文件的统一写作格式。
> 适用范围：46 个故事段（47 个 `.md` 文件 —— 34 / 35 / 36 各拆 a-b）。
> 适用对象：续集新增、番外补写、孪生篇修订、future 故事的作者与 review 人员。

← [返回系列导读](./index.md)

---

## 0. 为什么需要规范

2026 年系列扩到 46 篇后，写作格式逐渐分散。`README.md`、`index.md`、
`cheatsheet.md`、`glossary.md` 都依赖统一的标题与编号体系。规范不是束缚创作，
而是让 **检索、引用、自动化（cheatsheet 生成、交叉引用、mermaid 拓扑）** 不被
"作者当时的笔兴"拖累。

---

## 1. 文件命名

```
[两位数编号]-[英文主题-slug].md
```

- **编号固定**，前导零（`01`、`02` ... `46`）。
- **孪生篇**：`34a` / `34b`、`35a` / `35b`、`36a` / `36b`。
- 主题 slug 用英文小写 + 短横线，不含中文。
- 编号一旦分配 **永不重用**；废弃稿件不删号、不补号。
- 文件名一旦定稿，**文末 `← [返回系列导读]`、`index.md` 链接、`glossary.md`
  索引** 必须同步更新。

---

## 2. 文件结构（强制骨架）

按下列顺序排版，每节之间用 `---` 空行隔开：

| 区段 | 强制 / 推荐 | 用途 |
|------|-----------|------|
| ① 编号标题 | **强制** | `# 编号 · 标题`（无编号永不通过 review） |
| ② 一句话副标题 | 强制 | `> 从阿明...，看... —— 一句话点题` |
| ③ 系列定位块 | **强制** | `> **系列定位**：本篇是「阿明餐厅」系列的**[位置]**...` |
| ④（可选）兄弟篇块 / 合并说明 / 写给 X 的一句话 | 推荐 | 见 §3 |
| ⑤ 引言 | **强制** | `## 引言：xxx`（**全文唯一**） |
| ⑥ 章节 1...N | 强制 | `## 第一章：...` / `## 0 第零章：...`（见 §4） |
| ⑦（可选）总结表 | 推荐 | 当章节 ≥ 5 时，给一张"目录速查表" |
| ⑧ 延伸阅读 / 跨章节衔接 | 推荐 | 见 §5 |
| ⑨ 结尾回链 | **强制** | `← [返回系列导读](./index.md)`（独占一行，可在前置 "📅" 标注后追加） |

### 完整骨架示例

```markdown
# 47 · 一句话标题

> 一句话副标题 —— 用阿明的场景点题

> **系列定位**：本篇是「阿明餐厅」系列的**[位置]**。...

---

## 引言：一句话引言标题

开篇故事段落。用 1-3 段建立"阿明的某次真实场景"。

---

## 第一章：xxx —— xxx（em dash 副标题）

小节标题：`### 1.1 xxx`

---

## 延伸阅读

- [`02-system-architecture-evolution`](./02-system-architecture-evolution.md) —— 一句话点连接理由

---

← [返回系列导读](./index.md)
```

## 3. 顶部的 4 类元数据块

只有一种 "系列定位" 块是强制的，其余三种是 **互斥的可选项**（最多追加一种）：

### 3.1 系列定位（强制）

```markdown
> **系列定位**：本篇是「阿明餐厅」系列的**续集十八**。在前传 ... 之后，
> 阿明 ...
```

### 3.2 兄弟篇（仅孪生篇 a/b）

```markdown
> **兄弟篇**：35b · A2A 协议 + 协议治理（[← 点击阅读](./35b-a2a-protocol.md)）
```

### 3.3 合并说明（仅合并篇章）

```markdown
> **合并说明**：本篇由原《...》与《...》合并而成。共 N 章。
```

### 3.4 写给 X 的一句话（仅限特定受众文章）

```markdown
> **写给 PM 的一句话**：重构不是"工程师在浪费时间"，而是"在为你下一次
> 大版本交付清空跑道"。
```

## 4. 章节体系

### 4.1 主流结构（强制默认）

- 引言：`## 引言：xxx`（**全文唯一**）
- 章节：`## 第一章：xxx —— xxx` （em dash 副标题，可省略）
  - 子节：`### 1.1 xxx`
- 终章：`## 终章总结：一句心法` 或 `## 一句心法：xxx`

> 使用「第一章」「第二章」的中文数字，与 README、cheatsheet、引述块保持
> 一致。**禁止**"`第 1 章`" "`Chapter 1`" "`一、`" "`第 1 节`" 混用。

### 4.2 戏剧结构变体（仅 `03-refactoring-guide-for-pm.md`）

番外一是 PM 受众，叙事节奏要"用户故事"。允许：

- 取消 `## 第一章` 体系
- 改用 `## 五幕剧：当"加新菜"遇到"旧厨房"` 风格
- 子幕：`### 第一幕：xxx`、`### 第二幕：xxx` ...

> 其他文件 **不允许** 模仿，避免结构泛滥。变体如要复用，需在 PR 里单独
> 标注说明理由。

### 4.3 续集 18-20 戏剧体（44 / 45 / 46）

由于这 3 篇谈"事故 / 复盘 / 职业"，叙事性强，允许"开场 + 第一幕 + 第二幕"结构：

```markdown
### 一、开场：...

### 二、第一幕：...

### 三、第二幕：...

### 四、第四幕：...

### 九、相关章节
```

但 **H1 仍必须** `# NN · 标题` 这种编号 + 间隔点格式；编号不可丢。

### 4.4 第零章特例（仅 `33-ai-fatal-trio.md`）

`33` 因为 "5 分钟快速概览" 是亮点，**第零章合法**：

```markdown
## 引言：3 起看似无关的 AI 事故，背后是同一组漏洞

## 第零章：5 分钟快速概览 —— 三件套是什么，为什么致命

## 第一章：xxx
```

数字 0 是允许的特例（只在"概览 / 前置"用途），其他文件不要用。

### 4.5 引言重复的反例

**全文只允许一个 `## 引言：...`**。常见错误写法（已被修复 6 处）：

```markdown
## 引言：...                                ← 第 1 次 OK

> **阿明的厨房类比（开篇场景）**：...

## 引言：...                                ← 第 2 次 ❌ 重复

<正文>
```

正确写法：

```markdown
## 引言：...                                ← 全文仅一次

> **阿明的厨房类比（开篇场景）**：...     ← 类比块直接跟随

<正文>
```

## 5. 引用 / 衔接章节

- **统一名称**：单数文章用 `## 延伸阅读`（主流做法）。
- **孪生篇**：除延伸阅读外，可在文末追加 `## 跨章节衔接`，专门指向孪生篇 /
  外部引用文章。
- **续集 18-20**（44 / 45 / 46）：用 `### 九、相关章节` 命名。
- 项目内引用一律相对路径：`./02-system-architecture-evolution.md`；
  禁止绝对路径、禁止裸 `02-...` 文本链接。
- 外部路径（跨主模块）：`../11.ai/...`。

## 6. 代码块规范

- 必须用 ` ```text ` / ` ```mermaid ` / ` ```python ` / ` ```sql ` 等语言声明
  包围，**禁止裸 ` ``` `**。
- 任何代码注释不得在 Markdown 中裸写为 `# xxx`，会被 Grep / 渲染器误识别为
  H1。当前在 `11 / 13 / 16 / 18 / 19 / 20 / 21 / 22 / 30` 等多个文件存在该
  历史问题，未来迁移时统一修复。
- 当代码用于表达架构图（如责任金字塔）时，**优先用 `mermaid`**：

    ````markdown
    ```mermaid
    graph TD
      A[顶层：架构师] --> B[中层：SRE] --> C[基层：业务工程师]
    ```
    ````

## 7. 表格规范

- 表头与 `|------|` 对齐可选（Markdown 渲染无要求）。
- 但同一系列内应保持一致 —— **统一使用"冒号对齐"**（左对齐），便于
  Prettier 等工具格式化。
- 表格内字符数较多时，首列做"主语 + 链接"，后续列做"解释"。

示例：

```markdown
| 模块 | 核心问题 | 餐厅类比 | 技术实现 |
|------|----------|----------|----------|
| 感知层 | 听懂用户要什么 | 翻译官 | 多模态 + 意图识别 |
```

---

## 8. mermaid 图规范

- `graph` 方向：`TD`（自顶向下）优先于 `LR`（自左向右）。
- 每个故事 **最多 3-4 张 mermaid**，避免图表通胀。
- 节点 label 加双引号或 `<br/>`，**禁止未引用就 `subgraph`**。

## 9. 反例（明令禁止）

| 反例 | 当前存量 | 修正方案 |
|------|---------|---------|
| 两个 `## 引言：xxx` | 已修复 6 处（34b / 35a / 40 / 41 / 42 / 43） | 合并为一个；类比块作为 `## 引言` 内文 |
| H1 缺编号（如 `# AI 提速 3 倍`） | 已修复 3 处（44 / 45 / 46） | 改 `# NN · 标题` |
| 缺 `← [返回系列导读]` | 已修复 3 处（44 / 45 / 46） | 文末追加一行 |
| 系列定位写成普通段落（非 `>` blockquote） | 散见 | 改 quote 块 + `**系列定位**` 起头 |
| 章内裸 `# xxx` 当代码注释 | 11 / 18 / 19 / 20 / 21 / 22 / 30 等 | 用 ` ```text ` 包围 |
| "第 X 章" 与 "一、" 与 "第零章" 混用 | 33 / 44 / 45 / 46 内局部允许 | 同篇文章内部统一；变体要标注 |
| 章号与目录不一致 | 早期有，README 已修正 | PR 时校对 index.md 编号 |

## 10. frontmatter（可选，推荐）

为了让 `cheatsheet.md` 等自动工具能解析，建议在文末 `← [返回系列导读]`
之前追加 HTML 注释 frontmatter：

```markdown
<!--
story:
  number: 44
  title: AI 替你写代码，谁替系统负责？
  type: 续集
  position: 续集十八
  sibling: null
  audience: CTO / 技术管理者 / AI 工程师
  core_concepts: [系统责任金字塔, AI协作工程师, AI SRE, AI 安全审计师]
-->
```

不强制，但加上后便于静态扫描脚本产出 cheatsheet。
