# 你接手的烂代码，正在决定你哪天被淘汰

## —— 阿明餐厅里那个"救火 3 年"的工程师小张

> 阿明餐厅·续集 20

---

### 一、开场：3 年救火，简历空空

2026 年春，阿明的研发团队来了个新工程师小张，28 岁，面试时聊得头头是道——微服务、K8s、云原生、AI Agent。

3 年后，小张 31 岁，准备跳槽。打开简历，却发现——

> **技术栈：Spring Boot 2.3 + MyBatis + JSP + Dubbo 2.7 + Oracle 11g**

全是 2018 年的东西，最近 3 年的项目全是"维护老系统"。

面试官的问题让小张心里一凉：
- "讲讲你们最近用的 K8s 实践？"——"我们没用过"
- "你们微服务怎么做链路追踪的？"——"我们没拆微服务"
- "AI 编程工具怎么用的？"——"我们项目太老，接不进来"

**小张这才意识到：他以为自己在"积累项目经验"，实际上他被困在 2018 年的代码里 3 年，技能完全不增长。**

---

### 二、第一幕：3 年救火实录

#### 第 1 年：激情救火

小张入职后接手的第一个项目是"会员系统重构"。

```
第 1 个月：通读 5 万行老代码（Spring 2.0 + JSP）
第 3 个月：发现核心交易逻辑藏在第 3 层 if 嵌套里，没人能看懂
第 6 个月：改了 30 个 bug，每天修 1-2 个
第 12 个月：发现改了 A 炸 B，技术债利息比本金还高
```

**第 1 年的产出**：bug 数 -20%，但技术债指数 +15%。

#### 第 2 年：绝望救火

- 新需求不敢接，怕动到老代码
- 老 bug 反复出现
- 每天 50% 时间在"考古"——找 5 年前为什么这样写
- 加班时长从 35 小时涨到 55 小时

**第 2 年的产出**：2 个新功能上线，但每次都延期 2 倍。

#### 第 3 年：麻木救火

- 提了 5 次"重写会员系统"，都被驳回（业务方说"先做新需求"）
- 学会了"绕着走"——遇到老代码就加 if 判断，不改原逻辑
- 代码 review 时只看自己写的部分，老代码无人敢动

**第 3 年的产出**：5 个新功能，但每个都在老代码上加更多 if。

---

### 三、第二幕：4 个判断标准——你的岗位在让你增值还是掏空你？

技术总监老张看到小张的状态后，给了 4 个判断标准：

#### 标准 1：技能是"累积"还是"折旧"？

```
✅ 增值的信号：
  - 学到了新工具/新架构/新方法论
  - 能讲清楚"为什么这样设计"
  - 简历每 6 个月能加新东西

❌ 掏空的信号：
  - 技能树原地不动 3 年
  - 简历拿不出像样的项目
  - 同行在聊的东西你完全没听过
```

**小张的现状**：技能树原地不动 3 年。

#### 标准 2：是"解决问题"还是"重复救火"？

```
✅ 增值的信号：
  - 同一类问题出现 2 次以上 → 抽象成工具/平台
  - 救火时顺便消灭一类问题，不只是这一次的

❌ 掏空的信号：
  - 同一个 bug 每月修一次（没根因分析）
  - 救火后下次同样问题再次出现
  - 每天在"修昨天的问题"而不是"做明天的事"
```

**小张的现状**：3 年修了 200+ bug，但同样的"用户登录失败"修了 15 次。

#### 标准 3：项目有"复利"还是"损耗"？

```
✅ 复利项目：
  - 写一次工具，能用 3 年
  - 抽象一次架构，能复用 10 个项目
  - 沉淀一次文档，新人 1 周上手

❌ 损耗项目：
  - 写一次代码，6 周后被 AI 重写（参见 45 续集）
  - 加一个 if 判断，下次维护更难
  - 救火时只"修当前"，不"建防御"
```

**小张的现状**：100% 损耗项目，0 个复利项目。

#### 标准 4：团队在"成长"还是"原地打转"？

```
✅ 成长型团队：
  - 季度 OKR 包含"技术债偿还率"
  - 每月有技术分享 / Code Review
  - 工程师晋升看"解决了什么系统性问题"

❌ 原地打转型团队：
  - 永远是"赶需求"模式
  - 没有技术债预算
  - 救火英雄反而受表彰，建防御的工程师没奖励
```

**小张的现状**：团队连续 4 个季度"技术债偿还率 = 0%"。

### 四、第三幕：4 个增值动作——"在屎山上"也能让自己值钱

老张给小张的 4 个增值建议：

#### 动作 1：20% 时间投资"复利项目"

即使团队只给救火时间，也要每天挤 1-2 小时做：
- 抽象救火工具（一个 Python 脚本自动定位同类 bug）
- 写团队 wiki（5 年前那段逻辑的来龙去脉）
- 抽象内部库（封装老系统的"破 API"）

**原则**：救火是债务，复利项目才是资产。

#### 动作 2：每年至少学一个"新技术栈"

不一定要在工作中用，但要保持简历竞争力：

| 阶段 | 推荐学 | 为什么 |
|------|--------|--------|
| 1 年内 | Spring Boot 3 + Java 21 | 主流升级 |
| 2 年内 | K8s + 微服务 + 链路追踪 | 行业标准 |
| 3 年内 | AI 编程工具 + 大模型应用 | 未来趋势 |

**注意**：学完要在简历上写出来，否则"学了 = 没学"。

#### 动作 3：每 6 个月做一次"简历盘点"

问自己：
- 我的简历上**这 6 个月新增了什么**？
- 如果现在面试，我能**讲清楚 3 个有深度的项目**吗？
- 我**在行业会议上能讲 30 分钟**吗？

如果 3 个问题都答"否"，说明这半年在"原地打转"。

#### 动作 4：每年评估一次"该不该跑"

```
值得留下的信号：
  ✅ 团队在变好（OKR 有技术债偿还率）
  ✅ 救火时顺带建防御
  ✅ 学到新东西

应该跑的信号：
  ❌ 3 年没变化
  ❌ 救火只是救火，不建防御
  ❌ 简历越写越空
```

**老张的金句**："**一份救火 3 年的工作，不如一份建设 1 年的工作值钱。**"

### 五、第四幕：阿明的反思

小张的故事让阿明重新审视团队——

#### 阿明的 4 个反思

1. **救火英雄 ≠ 优秀工程师**
   - 救火只是"填坑"，建防御才是"挖坑少"
   - 团队激励应该奖励"消灭一类问题"，不是"修一个 bug"

2. **技术债是"复利"，但反向的**
   - 1 元技术债，第 1 年还 1.1 元，第 2 年还 1.3 元，第 3 年还 1.6 元
   - **越晚还越贵，而且越晚越没人敢还**

3. **工程师是资产，不是救火队员**
   - 工程师被技术债困住 = 公司损失资产
   - 应该每月给 20% 时间做"复利项目"

4. **招聘 JD 要写"成长性"**
   - 不只说"维护老系统"
   - 要说"老系统现代化 + 新技术栈 + 团队成长"

**阿明在内部信写道：**

> "我曾经以为招个工程师来救火是'划算的'——救火工程师工资不高，能解决问题。
>
> 3 年后我意识到：我招的不是工程师，是'救火队员'。他们的技能在折旧，简历在掏空，公司在他们眼里只是'跳板'。
>
> **真正划算的不是招人救火，是让工程师在解决问题的同时增值。**"

---

### 六、给所有工程师的 4 个忠告

1. **技术债是"团队问题"，但掏空的是"个人"**
   - 团队不还债，个体被掏空
   - 你不能改变团队，但能选择自己的增值路径

2. **救火 3 年，不如建设 1 年**
   - 简历上写的应该是"做了什么"，不是"修了多少 bug"
   - 同样 3 年，选"建设型"项目

3. **每年评估一次"该不该跑"**
   - 留下 = 团队在变好 + 你在学新东西
   - 跑 = 原地打转 + 简历越写越空

4. **AI 时代，加速了分化**
   - 用 AI 处理重复工作，腾出时间学新东西
   - 不要让 AI 替你救火——那只会让你更快贬值
