# CPU 调度

> CPU 调度器决定哪个就绪进程/线程获得 CPU 时间片。好的调度算法在公平性、吞吐量和响应时间之间取得平衡。

---

## 一、调度基本概念

| 指标 | 定义 | 优化目标 |
|------|------|---------|
| **周转时间** | 进程从提交到完成的总时间 | 越小越好 |
| **等待时间** | 进程在就绪队列中等待的总时间 | 越小越好 |
| **响应时间** | 从提交到首次获得 CPU 的时间 | 越小越好（交互式系统） |
| **吞吐量** | 单位时间内完成的进程数 | 越大越好 |
| **CPU 利用率** | CPU 忙碌时间占比 | 越大越好 |

```
时间线：
提交    首次执行        完成
 │        │              │
 ├──等待──►├──执行──►暂停──├──►
 │←─ 响应时间 ──►│
 │←──────── 周转时间 ────────►│
```

## 二、经典调度算法

### 2.1 算法对比总览

| 算法 | 类型 | 抢占 | 公平性 | 饥饿问题 | 适用场景 |
|------|------|------|--------|---------|---------|
| **FCFS** | 非抢占 | 否 | 先到先得 | 无 | 批处理 |
| **SJF** | 非抢占 | 否 | 短作业优先 | 长作业饥饿 | 批处理 |
| **SRTF** | 抢占 | 是 | 最短剩余时间优先 | 长作业饥饿 | 理论最优 |
| **优先级** | 两者 | 可选 | 高优先级优先 | 低优先级饥饿 | 实时系统 |
| **轮转 (RR)** | 抢占 | 是 | 时间片轮转 | 无 | 交互式系统 |
| **多级反馈队列** | 抢占 | 是 | 动态优先级 | 无（老化机制） | 通用系统 |

### 2.2 FCFS (First Come First Served)

```
到达顺序: P1(0ms, 需24ms), P2(1ms, 需3ms), P3(2ms, 需3ms)

时间线: P1(0-24) → P2(24-27) → P3(27-30)

         周转时间  等待时间
P1:       24        0
P2:       26        23
P3:       28        25
平均等待: (0+23+25)/3 = 16 ms

护卫舰效应 (Convoy Effect)：
短作业 P2/P3 被长作业 P1 阻塞 → 平均等待时间飙升
```

### 2.3 SJF (Shortest Job First)

```
进程: P1(需6ms), P2(需8ms), P3(需7ms), P4(需3ms)，同时到达

调度顺序: P4(3) → P1(6) → P3(7) → P2(8)

         周转时间  等待时间
P4:       3         0
P1:       9         3
P3:       16        9
P2:       24        16
平均等待: (0+3+9+16)/4 = 7 ms  ← 理论最优

问题：需要预知运行时间（实际中只能估算）
```

### 2.4 轮转调度 (Round Robin)

```
时间片 q = 4ms
进程: P1(需24ms), P2(需3ms), P3(需3ms)，同时到达

时间线:
P1(0-4) → P2(4-7) → P3(7-10) → P1(10-14) → P1(14-18) → ... → P1(26-30)

         周转时间  等待时间
P2:       7         4
P3:       10        7
P1:       30        6
平均等待: (4+7+6)/3 = 5.67 ms

时间片选择：
  q 太大 → 退化为 FCFS
  q 太小 → 上下文切换开销过大
  经验值: 10~100ms（Linux 默认 ~6ms 基准）
```

### 2.5 多级反馈队列 (MLFQ)

```
                    ┌─────────────┐
 新进程 ──────────► │ 队列 0       │ ← 最高优先级
                    │ q = 8ms     │    时间片小、响应快
                    └──────┬──────┘
                           │ 未完成（CPU 密集型）
                           ▼
                    ┌─────────────┐
                    │ 队列 1       │ ← 中优先级
                    │ q = 16ms    │    时间片中
                    └──────┬──────┘
                           │ 未完成
                           ▼
                    ┌─────────────┐
                    │ 队列 2       │ ← 最低优先级
                    │ FCFS        │    时间片大、批量处理
                    └─────────────┘

规则：
1. 新进程进入最高优先级队列
2. 用完时间片未完成 → 降一级（惩罚 CPU 密集型）
3. 等待过久 → 升一级（老化机制，防饥饿）
4. I/O 完成 → 回到高优先级队列（奖励 I/O 密集型）
```

> **MLFQ 是现代通用调度算法的基础**。Linux CFS 虽不直接使用多级队列，但其虚拟运行时间 (vruntime) 机制实现了类似的公平性目标。

## 三、Linux CFS 调度器

### 3.1 CFS 核心思想

**Completely Fair Scheduler (CFS)** 是 Linux 2.6.23+ 的默认调度器（2023 年 Linux 6.6 引入 **EEVDF** 替代）：

| 概念 | 说明 |
|------|------|
| **虚拟运行时间 (vruntime)** | 每个进程的"逻辑时钟"，记录实际运行时间（按优先级加权） |
| **红黑树** | 以 vruntime 为 key 维护就绪队列，O(log n) 插入/查找 |
| **调度延迟** | 一个调度周期内，每个进程至少运行一次 |
| **最小粒度** | 进程每次至少运行 `sched_min_granularity`（默认 ~0.75ms） |

```
CFS 红黑树（就绪队列）:
         vruntime=15 (P3)
        /              \
   vruntime=10 (P1)    vruntime=20 (P5)
        \              /
    vruntime=12 (P2)  vruntime=18 (P4)

调度器总是选择 vruntime 最小的进程（最左节点）
→ P1 被调度运行 → vruntime 增加 → 树重新平衡
```

### 3.2 nice 值与权重

```
nice 值范围: -20（最高优先级）到 +19（最低优先级）
默认 nice = 0

权重计算: weight = 1024 * 1.25^(-nice)

nice  -20  -10   0   +10  +19
权重  88761  8876 1024  110  15

vruntime 增量 = (实际运行时间 × 1024) / weight

→ nice=-20 的进程 vruntime 增长极慢，获得更多 CPU 时间
→ nice=+19 的进程 vruntime 增长很快，获得较少 CPU 时间
```

### 3.3 CFS vs 实时调度

| 调度类 | 策略 | 优先级 | 使用场景 |
|---------|------|--------|---------|
| **CFS (SCHED_NORMAL)** | 公平调度 | 0 | 普通进程（绝大多数） |
| **SCHED_FIFO** | 先进先出，无时间片 | 1~99 | 实时任务（不主动让出 CPU） |
| **SCHED_RR** | 轮转，有时间片 | 1~99 | 实时任务（时间片用完让出） |
| **SCHED_IDLE** | 极低优先级 | — | 后台批处理 |
| **SCHED_DEADLINE** | EDF 截止期限调度 | — | 硬实时任务（Linux 3.14+） |

## 四、上下文切换成本

### 4.1 切换过程

```
上下文切换步骤：
1. 保存当前进程的 CPU 寄存器 → PCB (进程控制块)
2. 保存当前进程的栈指针、程序计数器
3. 切换页表（CR3 寄存器）  ← 最昂贵的操作
4. 刷新 TLB（部分或全部）
5. 加载新进程的寄存器、栈指针
6. 恢复新进程的程序计数器，继续执行

直接成本: ~1~10 μs（取决于是否跨进程）
间接成本: TLB 刷新 → 缓存失效 → 流水线清空
```

### 4.2 线程切换 vs 进程切换

| 类型 | 切换内容 | 典型耗时 |
|------|---------|---------|
| **线程切换**（同进程内） | 寄存器 + 栈（共享页表） | ~1~3 μs |
| **进程切换** | 寄存器 + 栈 + 页表 + TLB 刷新 | ~5~20 μs |

### 4.3 实际影响

```java
// 线程过多导致性能下降的示例
int cpuCores = Runtime.getRuntime().availableProcessors();

// 好的做法：线程数 ≈ CPU 核心数（CPU 密集型）
ExecutorService cpuPool = Executors.newFixedThreadPool(cpuCores);

// 好的做法：线程数 > CPU 核心数（I/O 密集型）
// 因为线程在 I/O 等待时不占用 CPU
ExecutorService ioPool = Executors.newFixedThreadPool(cpuCores * 4);

// 坏的做法：创建过多线程
// 1000 个线程 → 大量上下文切换 → 性能反而下降
ExecutorService badPool = Executors.newFixedThreadPool(1000);
```

> **经验法则**：CPU 密集型任务线程数 ≈ CPU 核心数；I/O 密集型任务线程数 ≈ CPU 核心数 × (1 + I/O 等待时间 / CPU 计算时间)。

## 五、CPU 亲和性 (CPU Affinity)

### 5.1 原理

将进程/线程绑定到特定 CPU 核心，避免跨核迁移带来的缓存失效开销：

```
无亲和性绑定:
  线程 T1: Core 0 → Core 3 → Core 1 → Core 2  (频繁迁移)
  每次迁移: L1/L2 cache 失效 → 重新加载 → ~10μs 开销

绑定 CPU 亲和性:
  线程 T1: Core 0 (固定)  ← cache 热，无迁移开销
  线程 T2: Core 1 (固定)
  线程 T3: Core 2 (固定)
```

### 5.2 Linux 中的设置

```bash
# 查看进程当前 CPU 亲和性
taskset -p <PID>

# 绑定到 CPU 0 和 1
taskset -c 0,1 ./my_program

# 绑定到 CPU 0（仅一个核心）
taskset -c 0 ./my_program

# 在 Java 中通过 JNI 调用 sched_setaffinity()
# 或使用 JNA / Panama FFI
```

### 5.3 NUMA 架构的影响

```
NUMA (Non-Uniform Memory Access):
┌─────────────────────┐   ┌─────────────────────┐
│  Node 0             │   │  Node 1             │
│  ┌─────┐ ┌───────┐ │   │  ┌─────┐ ┌───────┐ │
│  │CPU 0│ │ Local │ │   │  │CPU 2│ │ Local │ │
│  │CPU 1│ │Memory │ │   │  │CPU 3│ │Memory │ │
│  └─────┘ └───────┘ │   │  └─────┘ └───────┘ │
└────────┬────────────┘   └────────┬────────────┘
         │          QPI/UPI        │
         └────────────────────────┘

本地内存访问: ~70ns
远端内存访问: ~140ns（2 倍延迟）

→ NUMA 感知调度：将线程绑定到同一 Node 的 CPU + 内存
```

## 六、与 Java 线程调度的关系

### 6.1 JVM 线程调度模型

JVM 依赖操作系统内核的线程调度器（1:1 模型），但可以通过以下方式影响调度行为：

| 手段 | 机制 | 效果 |
|------|------|------|
| **线程优先级** | `Thread.setPriority(1~10)` | 映射到 OS nice 值（不保证精确对应） |
| **线程池** | `ThreadPoolExecutor` | 控制并发线程数，减少上下文切换 |
| **Fork/Join** | `ForkJoinPool` + work-stealing | 自适应负载均衡，减少空闲核心 |
| **Virtual Thread** | Project Loom (JDK 21+) | 用户态调度，I/O 阻塞时自动让出载体线程 |

### 6.2 Fork/Join 的 work-stealing

```
ForkJoinPool (parallelism = 4):
┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
│ Worker 0 │ │ Worker 1 │ │ Worker 2 │ │ Worker 3 │
│ 双端队列  │ │ 双端队列  │ │ 双端队列  │ │ 双端队列  │
│ [A][B][C]│ │ [D]      │ │   空     │ │ [E][F]   │
└──────────┘ └──────────┘ └────┬─────┘ └──────────┘
                               │
                          队列空！
                               │ steal
                               ▼
                         从 Worker 0 偷取任务 A
                         （从队列尾部偷取）

优势：自动负载均衡，无需手动分配
```

## 七、常见面试题

| 问题 | 核心要点 |
|------|---------|
| FCFS 的护卫舰效应？ | 短作业被长作业阻塞，平均等待时间飙升 |
| 轮转调度时间片如何选择？ | 太大退化为 FCFS，太小切换开销大；经验值 10~100ms |
| Linux CFS 如何保证公平？ | vruntime 红黑树，始终调度 vruntime 最小的进程 |
| 上下文切换的成本在哪？ | 寄存器保存、页表切换、TLB 刷新、缓存失效 |
| NUMA 对性能的影响？ | 远端内存访问延迟 2 倍；需 NUMA 感知调度 + 内存分配 |
