# 内存管理

> 内存管理是操作系统的核心职责之一：通过虚拟内存抽象，让每个进程认为自己独占整个地址空间，同时实现内存隔离与高效利用。

---

## 一、虚拟内存 vs 物理内存

### 1.1 为什么需要虚拟内存？

| 问题 | 没有虚拟内存 | 有虚拟内存 |
|------|------------|-----------|
| **内存隔离** | 进程可直接访问物理地址，互相干扰 | 每个进程有独立虚拟地址空间 |
| **内存不足** | 程序大于物理内存则无法运行 | 按需加载 + 页面置换（swap） |
| **内存共享** | 需手动管理物理地址 | `mmap` 共享同一物理页 |
| **地址冲突** | 多个程序编译时地址冲突 | 编译时使用虚拟地址，运行时动态映射 |

### 1.2 地址空间布局（64 位 Linux）

```text
虚拟地址空间（48 位有效 = 256 TB）

高地址 ┌───────────────────────┐ 0xFFFFFFFFFFFF
       │     内核空间           │ (高 128 TB)
       │   (所有进程共享)       │
       │   内核代码/数据/页表   │
       ├───────────────────────┤ 0x800000000000
       │     用户空间           │ (低 128 TB)
       │  ┌─────────────────┐  │
       │  │ 栈 (Stack)  ↓   │  │ ← 函数调用/局部变量
       │  ├─────────────────┤  │
       │  │ mmap 区域       │  │ ← 共享库 / mmap 文件
       │  ├─────────────────┤  │
       │  │ 堆 (Heap)   ↑   │  │ ← malloc / new
       │  ├─────────────────┤  │
       │  │ BSS (未初始化)   │  │ ← 全局未初始化变量
       │  ├─────────────────┤  │
       │  │ Data (已初始化)  │  │ ← 全局已初始化变量
       │  ├─────────────────┤  │
       │  │ Text (代码段)    │  │ ← 可执行指令
       │  └─────────────────┘  │
低地址 └───────────────────────┘ 0x000000000000
```

## 二、分页 (Paging) 与分段 (Segmentation)

### 2.1 对比

| 维度 | 分页 (Paging) | 分段 (Segmentation) |
|------|-------------|-------------------|
| **划分方式** | 固定大小（通常 4 KB） | 按逻辑段（代码段/数据段/堆/栈） |
| **大小** | 固定 | 可变 |
| **用户可见性** | 对用户透明 | 用户可感知（段号 + 偏移） |
| **内部碎片** | 有（最后一页可能不满） | 无 |
| **外部碎片** | 无 | 有（段间空隙） |
| **现代系统** | 主流方案（x86/ARM 均使用） | 早期系统 / 与分页结合使用 |

### 2.2 分页原理

```text
虚拟地址（32 位）:
┌─────────────────┬──────────────┐
│  虚拟页号 (VPN)  │  页内偏移     │
│   (20 bit)      │  (12 bit)    │
└────────┬────────┴──────────────┘
         │
         ▼
    ┌─────────┐
    │  页 表   │  ← 存储在内存中
    │ VPN→PPN │
    └────┬────┘
         │ 物理页号 (PPN)
         ▼
┌─────────────────┬──────────────┐
│  物理页号 (PPN)  │  页内偏移     │
│   (20 bit)      │  (12 bit)    │
└─────────────────┴──────────────┘
         │
         ▼
    物理内存 (RAM)
```

## 三、页表与 TLB

### 3.1 多级页表

32 位系统使用 2 级页表，64 位系统使用 4 级页表（x86-64）：

```text
64 位虚拟地址 (48 位有效):
┌──────┬──────┬──────┬──────┬──────────────┐
│ PML4 │ PDPT │ PD   │ PT   │  页内偏移     │
│9 bit │9 bit │9 bit │9 bit │  12 bit      │
└──┬───┴──┬───┴──┬───┴──┬───┴──────────────┘
   │      │      │      │
   ▼      ▼      ▼      ▼
 CR3 → [PML4E] → [PDPTE] → [PDE] → [PTE] → 物理页
       (512项)   (512项)   (512项)  (512项)

每级 512 项 × 8 字节 = 4 KB（正好一页）
```

### 3.2 TLB (Translation Lookaside Buffer)

TLB 是页表的硬件缓存，加速虚拟→物理地址转换：

```text
虚拟地址 (VPN)
     │
     ▼
  ┌──────┐    命中    ┌──────────┐
  │ TLB  │───────────►│ 物理地址  │
  │(缓存) │           └──────────┘
  └──┬───┘
     │ 未命中
     ▼
  ┌──────┐           ┌──────────┐
  │ 页表  │──────────►│ 物理地址  │
  │(内存) │           │ + 更新TLB │
  └──────┘           └──────────┘
```

| 特性 | 值 |
|------|-----|
| TLB 条目数 | 通常 64~1024 条 |
| TLB 命中延迟 | ~1 个时钟周期 |
| TLB 未命中延迟 | 4 级页表 = 4 次内存访问（~100+ 周期） |
| TLB 命中率 | 典型 99%+（利用空间/时间局部性） |
| TLB 刷新 | 进程切换时清空（或通过 ASID/PCID 避免） |

> **大页 (Huge Page)**：使用 2 MB 或 1 GB 页代替 4 KB 页，减少 TLB 未命中。Linux 中通过 `hugetlbfs` 或 `madvise(MADV_HUGEPAGE)` 启用。

## 四、页面置换算法

当发生**缺页中断 (Page Fault)** 且物理内存已满时，OS 需要选择一个页面换出：

| 算法 | 策略 | 优点 | 缺点 | 实现复杂度 |
|------|------|------|------|-----------|
| **FIFO** | 先进先出，淘汰最早进入的页 | 实现简单 | 可能淘汰常用页（Belady 异常） | O(1) |
| **LRU** | 最近最少使用，淘汰最久未访问的页 | 效果接近最优 | 硬件开销大（需记录每次访问时间） | O(1) 近似 |
| **Clock (时钟)** | LRU 的近似实现，循环扫描 + 访问位 | 开销小 | 不如精确 LRU | O(n) 扫描 |
| **LFU** | 最不经常使用，淘汰访问次数最少的页 | 保留高频页 | 历史访问可能不代表未来 | O(log n) |

### 4.1 Clock 算法示意

```text
物理页帧（环形链表）：
    ┌───┬───┬───┬───┬───┐
    │ P0│ P1│ P2│ P3│ P4│
    │ 1 │ 0 │ 1 │ 1 │ 0 │  ← 访问位 (reference bit)
    └───┴───┴─┬─┴───┴───┘
              ↑
          Clock 指针

扫描规则：
1. 指针指向的页访问位 = 0 → 淘汰该页
2. 指针指向的页访问位 = 1 → 清零，指针前移
3. 循环直到找到访问位 = 0 的页
```

### 4.2 Belady 异常

FIFO 算法存在反直觉现象：**增加物理页帧数反而增加缺页次数**。

```text
引用序列: 1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5

3 个页帧: 9 次缺页
4 个页帧: 10 次缺页  ← 多了页帧反而更多缺页！

LRU 不存在此异常（LRU 是栈算法）
```

## 五、内存映射 (mmap) 与零拷贝

### 5.1 mmap 原理

`mmap()` 将文件或设备映射到进程的虚拟地址空间，通过内存访问代替 I/O 系统调用：

```text
传统 read():                     mmap:
用户空间 ┌────────┐            用户空间 ┌────────┐
         │ buffer │                     │映射区域 │
         └───┬────┘                     └───┬────┘
       copy  │                         直接访问
         ┌───┴────┐                     ┌───┴────┐
内核空间 │page    │            内核空间 │page    │
         │cache   │                     │cache   │
         │  ┌──┐  │                     │  ┌──┐  │
         └──┤文├──┘                     └──┤文├──┘
            │件│                            │件│
            └──┘                            └──┘

read: 2 次拷贝（磁盘→page cache→用户 buffer）
mmap: 0 次拷贝（用户直接访问 page cache）
```

### 5.2 零拷贝技术对比

| 技术 | 系统调用 | 拷贝次数 | 上下文切换 | 适用场景 |
|------|---------|---------|-----------|---------|
| 传统 I/O | `read()` + `write()` | 4 次 | 4 次 | 通用 |
| mmap + write | `mmap()` + `write()` | 3 次 | 4 次 | 大文件随机访问 |
| sendfile | `sendfile()` | 2 次 | 2 次 | 文件→Socket 传输 |
| splice | `splice()` | 0~2 次 | 2 次 | pipe 中转 |

> **Kafka / RocketMQ** 高性能的核心之一：利用 `mmap`（小文件元数据）+ `sendfile`（大数据传输）实现零拷贝。

## 六、堆 vs 栈内存分配

| 维度 | 堆 (Heap) | 栈 (Stack) |
|------|----------|-----------|
| **分配方式** | 手动 (`malloc`/`new`) 或 GC | 自动（编译器管理） |
| **大小** | 受限于虚拟地址空间（GB 级） | 受限于线程栈大小（1~8 MB） |
| **速度** | 较慢（需搜索空闲链表） | 极快（移动栈指针） |
| **碎片** | 频繁分配/释放产生碎片 | 无碎片（LIFO 顺序） |
| **生命周期** | 手动管理或 GC 回收 | 函数返回自动释放 |
| **线程共享** | 进程内所有线程共享 | 每个线程独立栈 |
| **增长方向** | 向高地址增长 (↑) | 向低地址增长 (↓) |

```c
int global_var = 42;       // 数据段（Data Segment）

void example() {
    int stack_var = 10;    // 栈上分配（函数返回自动释放）
    int *heap_var = (int*)malloc(sizeof(int));  // 堆上分配
    *heap_var = 20;
    // ... 使用 heap_var ...
    free(heap_var);        // 必须手动释放（否则内存泄漏）
}
```

## 七、Java 堆外内存 (DirectByteBuffer)

### 7.1 JVM 内存与 OS 内存的关系

```text
JVM 进程地址空间:
┌────────────────────────────────────┐
│  JVM Heap (-Xmx)                  │ ← 受 GC 管理
│  ┌──────────────────────────────┐ │
│  │  Eden / Survivor / Old Gen  │ │
│  └──────────────────────────────┘ │
├────────────────────────────────────┤
│  Metaspace                        │ ← 类元数据
├────────────────────────────────────┤
│  Direct Memory (堆外)             │ ← ByteBuffer.allocateDirect()
│  ┌──────────────────────────────┐ │
│  │  直接映射到 OS 虚拟内存       │ │ ← 不受 GC 直接管理
│  │  绕过 JVM Heap 的拷贝        │ │
│  └──────────────────────────────┘ │
├────────────────────────────────────┤
│  Thread Stacks (-Xss)             │ ← 每个线程 1 MB
├────────────────────────────────────┤
│  JVM Code / GC / JIT              │
└────────────────────────────────────┘
```

### 7.2 DirectByteBuffer 使用场景

```java
// 堆内内存（HeapByteBuffer）
ByteBuffer heapBuf = ByteBuffer.allocate(1024);
// 数据在 JVM Heap 中，GC 可回收

// 堆外内存（DirectByteBuffer）
ByteBuffer directBuf = ByteBuffer.allocateDirect(1024);
// 数据在 JVM Heap 外，通过 Unsafe 分配
// 优势：I/O 操作无需 JVM Heap ↔ OS 之间的拷贝

// 典型用法：Netty 的 ByteBuf
// NIO 文件通道传输
FileChannel fc = FileChannel.open(Path.of("data.bin"));
fc.read(directBuf);  // OS page cache → DirectByteBuffer（零拷贝潜力）
```

### 7.3 堆内 vs 堆外对比

| 维度 | HeapByteBuffer | DirectByteBuffer |
|------|---------------|-----------------|
| **内存位置** | JVM 堆内 | JVM 堆外（OS 虚拟内存） |
| **GC 管理** | 是 | 否（依赖 Cleaner / 引用队列） |
| **分配速度** | 快 | 慢（需系统调用） |
| **I/O 性能** | 需拷贝到 native 内存 | 直接传递地址给 native |
| **适用场景** | 短生命周期、非 I/O 场景 | 长生命周期、大 Buffer、NIO |
| **内存限制** | `-Xmx` | `-XX:MaxDirectMemorySize` |

## 八、常见面试题

| 问题 | 核心要点 |
|------|---------|
| 虚拟内存的作用？ | 隔离进程地址空间、按需加载、支持共享内存 |
| 什么是缺页中断？ | 访问的虚拟页不在物理内存中 → OS 从磁盘加载该页 |
| LRU 和 Clock 算法的关系？ | Clock 是 LRU 的近似实现（用访问位代替精确时间戳） |
| mmap 为什么快？ | 减少用户态↔内核态的拷贝次数（零拷贝） |
| Java 堆外内存的优势？ | I/O 场景避免 JVM Heap ↔ native 内存的额外拷贝 |
