# 核心网络协议

> TCP/IP 协议族由多个协议协同工作，各司其职。

---
---

## 协议速览

| 协议 | 层级 | 连接类型 | 可靠性 | 速度 | 典型场景 |
|------|------|---------|--------|------|---------|
| **IP** | 网络层 | 无连接 | 不可靠 | — | 所有网络通信基础 |
| **ICMP** | 网络层 | 无连接 | 不可靠 | — | Ping/Traceroute |
| **TCP** | 传输层 | 面向连接 | 可靠 | 较慢 | 网页/邮件/文件 |
| **UDP** | 传输层 | 无连接 | 不可靠 | 快 | 视频/游戏/DNS |
| **HTTP** | 应用层 | 依赖 TCP | 依赖 TCP | — | 非加密网页 |
| **HTTPS** | 应用层+安全 | 依赖 TCP | 依赖 TCP | 略慢 | 网银/登录/支付 |

---

## 核心协议详解

### IP 协议
- **作用** — 设备 IP 地址分配，数据包路由
- **特性** — 无连接、不保证送达
- **IPv4 vs IPv6** — IPv4 约 43 亿地址已枯竭；IPv6 地址近乎无限

### ICMP 协议
- **用途** — 网络诊断（`ping` 发送 ICMP 回显请求）
- **错误反馈** — 路由失败时返回"目标不可达"等信息

### TCP 协议
- **三大保障机制：**
  - 三次握手 — 建立连接
  - 滑动窗口 — 动态调整传输速度
  - 超时重传 — 丢包自动重发
- **代价** — 延迟较高（握手/确认），头部开销大（20 字节+）

### UDP 协议
- **特点** — 无连接、直接发送、无重传
- **优势** — 头部仅 8 字节，极低延迟
- **场景** — 视频会议、游戏、DNS 查询（宁可丢包也不能卡顿）

---

## 常见问题

**Q: DNS 用 TCP 还是 UDP？** → 主要用 **UDP**（53 端口），响应 >512 字节时切换到 TCP。

**Q: QUIC 会取代 TCP 吗？** → QUIC（基于 UDP）是 HTTP/3 的底层，解决 TCP 队头阻塞问题，已在 Google/Cloudflare 大规模部署。

**Q: 为什么先有 TCP 再有 HTTP？** → TCP 解决"可靠传输"基础问题，HTTP 在其之上定义"如何传网页"。
