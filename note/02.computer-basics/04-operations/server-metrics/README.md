# 服务器性能指标

> 确保业务稳定运行的关键，需结合硬件、软件及网络等多维度指标进行系统性监控。

---
## 引言：性能对比

服务器性能指标 的关键不是'快'——是**什么时候慢、慢多少、为什么**。

本篇用'常见 vs 极端'两组数字切入，把排查思路和优化边界讲清。

## 一、CPU性能指标
1. **CPU使用率**
    - **定义**：CPU在单位时间内处理任务的占比。
    - **标准**：
        - 持续高于80%可能引发性能瓶颈，需关注进程负载分布。
        - 突发峰值（如90%以上）需结合业务场景判断是否合理（如批处理任务）。
    - **优化建议**：
        - 升级多核CPU或优化多线程编程。
        - 使用负载均衡分散请求。
    - **查看命令**：

    ```bash
    # 实时 CPU 使用率（按进程排序）
    top -c -d 1

    # 1 秒粒度，CPU 队列 / 上下文切换
    vmstat 1

    # 每个 CPU 核心使用率
    mpstat -P ALL 1
    ```

2. **CPU负载（Load Average）**
    - **定义**：系统平均负载，反映等待CPU资源的进程数（含 R + D 状态）。
    - **标准**：
        - 理想值：负载 ≤ CPU核心数 × 0.7（如4核CPU，负载≤2.8为健康状态）。
        - 持续高于核心数需警惕性能下降——可能是 CPU 瓶颈，也可能是磁盘/网络 I/O 阻塞（D 状态）。
    - **查看命令**：

    ```bash
    uptime                    # 1/5/15 分钟负载
    htop                      # 图形化：按核心分布看进程
    cat /proc/loadavg         # 原始数值
    ```

## 二、内存性能指标
1. **内存使用率**
    - **定义**：已用内存占总内存的比例。
    - **标准**：
        - 持续高于80%可能触发OOM（Out of Memory）错误。
        - 需区分"实际使用"与"缓存/缓冲区"——Linux 会把空闲内存当页缓存，`available` 才是应用真正可用的内存。
    - **优化建议**：
        - 增加物理内存或优化内存分配策略。
        - 使用内存数据库（如Redis）缓存高频数据。
    - **查看命令**：

    ```bash
    free -m                   # 看 total / used / free / available / buff/cache
    cat /proc/meminfo         # 详细各项
    ```

2. **Swap使用率**
    - **定义**：虚拟内存（Swap）的使用情况。
    - **标准**：
        - Swap使用率应长期为0，频繁使用表明物理内存不足或内存泄漏。
    - **风险**：Swap依赖磁盘I/O，性能远低于物理内存（千倍差距），可能导致系统卡顿。
    - **查看命令**：

    ```bash
    free -m                   # 看 Swap 行的 used
    swapon --show             # 各 Swap 设备的优先级
    ```

## 三、磁盘I/O性能指标
1. **IOPS（Input/Output Operations Per Second）**
    - **定义**：每秒磁盘读写次数。
    - **标准**（量级，非承诺值）：
        - SSD：通常数万至数十万 IOPS。
        - HDD：约 100-200 IOPS（随机读写）。
    - **场景**：数据库、日志写入等高I/O场景需重点关注。
    - **查看命令**：

    ```bash
    iostat -x 1               # 关键指标：r/s, w/s, await, %util
    iotop -o                  # 按 I/O 排序的进程
    ```

2. **吞吐量（Throughput）**
    - **定义**：单位时间内磁盘传输的数据量（MB/s 或 GB/s）。
    - **标准**（量级）：
        - SSD：通常可达数百 MB/s 以上。
        - HDD：约 100-200 MB/s（顺序读写）。
    - **查看命令**：

    ```bash
    iostat -x 1               # 看 rMB/s, wMB/s
    sar -d 1 5                # 历史回放
    dd if=/dev/zero of=tmp bs=1M count=1024 oflag=direct   # 顺序写基准
    ```

3. **磁盘利用率（Utilization）**
    - **定义**：磁盘I/O请求的繁忙程度（`%util`，接近 100% 表明已饱和）。
    - **标准**：
        - 持续高于70%可能引发队列堆积，需优化存储或升级硬件。
    - **配套指标**：与 `await`（平均 I/O 等待时间）一起看，%util 高但 await 低 ≠ 真瓶颈。

## 四、网络性能指标
1. **带宽利用率**
    - **定义**：网络接口实际传输数据占最大带宽的比例。
    - **标准**：
        - 持续高于80%需考虑扩容或优化网络架构。
    - **查看命令**：

    ```bash
    iftop -i eth0             # 实时流量 + 连接来源
    nload                     # 图形化带宽
    sar -n DEV 1              # 历史网卡收发
    ```

2. **延迟（Latency）**
    - **定义**：数据包从发送到接收的时间（ms级）。
    - **标准**：
        - 局域网：<1ms。
        - 跨机房：<10ms。
        - 跨地域：<100ms（视业务容忍度）。
    - **影响**：高延迟会显著降低Web应用、数据库查询等响应速度。
    - **查看命令**：

    ```bash
    ping <host>                # 平均延迟
    mtr -rwbzc 100 <host>      # 丢包率 + 各跳延迟
    traceroute <host>          # 路径
    ```

3. **丢包率（Packet Loss）**
    - **定义**：传输过程中丢失的数据包比例。
    - **标准**：
        - 理想值：0%。
        - 容忍阈值：<1%（关键业务需<0.1%）。
    - **查看命令**：

    ```bash
    ping -c 100 <host>          # 看末尾丢包统计
    mtr -rwbzc 100 <host>       # 各跳丢包定位
    ```

## 五、综合性能指标
1. **QPS/TPS（Queries/Transactions Per Second）**
    - **定义**：每秒处理请求或事务的数量。
    - **标准**：
        - 需结合业务场景设定基准（如电商系统TPS需达数千级）。
    - **优化方向**：缓存、异步处理、数据库优化。

2. **响应时间（Response Time）**
    - **定义**：从请求发出到收到响应的时间。
    - **标准**：
        - Web应用：<200ms（优秀），<500ms（可接受）。
        - 数据库查询：<10ms（索引优化后）。
    - **工具**：`Prometheus`、`Grafana`、`New Relic`。

3. **并发连接数（Concurrent Connections）**
    - **定义**：服务器同时处理的连接数量。
    - **标准**：
        - Web服务器：需支持数千至数万并发（如Nginx默认1024，可调优）。
    - **风险**：过高可能导致资源耗尽，需配置连接池或限流。

## 六、性能评估标准与工具
1. **基准测试工具**
    - **CPU**：`sysbench`、`UnixBench`。
    - **内存**：`memtester`、`stress`。
    - **磁盘**：`fio`、`iozone`。
    - **网络**：`iperf`、`netperf`。

2. **监控工具**
    - **开源**：`Prometheus` + `Grafana`、`Zabbix`、`Nagios`。
    - **商业**：`Datadog`、`New Relic`、`SolarWinds`。

3. **行业标准参考**
    - **TPC基准**：如TPC-C（数据库事务处理）、TPC-H（决策支持）。
    - **SPEC基准**：如SPECcpu（CPU性能）、SPECweb（Web服务器性能）。

## 七、性能优化建议
1. **硬件升级**：根据瓶颈选择升级CPU、内存、SSD或网络带宽。
2. **软件调优**：优化数据库索引、配置缓存策略（如Redis）、使用CDN加速静态资源。
3. **架构优化**：引入负载均衡、微服务拆分、读写分离。
4. **资源隔离**：使用容器（Docker）或虚拟化（KVM）隔离资源密集型应用。

通过系统性监控这些指标，并结合业务场景设定合理阈值，可提前发现性能隐患，确保服务器稳定高效运行。
