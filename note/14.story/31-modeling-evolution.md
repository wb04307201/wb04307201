# 看世界的方式决定了你造什么系统

> 从"管对象"到"管动作"到"管组织"，看数字化认知的三次跃迁

> **系列定位**：本篇是「阿明餐厅」系列的**番外八**。如果说[前传](./02-system-architecture-evolution.md)讲的是阿明如何一步步把小面馆做成大平台，那这篇讲的是另一个问题：**为什么我们会在不同的阶段，选择完全不同的架构方法论？** OOD、DDD、TOGAF 不是"谁替代谁"，而是数字化认知深化的三次跃迁。

> 写给新架构师的一句话：你选择的不是"技术方案"，而是"你看世界的方式"。

---

## 引言：三个架构师，三种菜单

阿明的平台做大了，要招架构师。来了三个面试者，阿明给了同一个考题："你来设计我们餐厅的点餐系统。"

第一个人画了一张类图：`User`、`Order`、`Dish`、`Payment`，属性和方法清清楚楚。阿明觉得挺完整。

第二个人画了一张流程图：顾客进店 → 浏览菜单 → 下单 → 支付 → 出餐 → 评价，每个环节一个限界上下文。阿明觉得比第一个人多了业务视角。

第三个人先问："你们现在有几个团队？团队之间怎么沟通？组织架构图能给我看看吗？"阿明愣了一下，但给了。第三个人看完说："你们的系统架构已经是你们组织结构的翻版了，而且有几个地方因为团队沟通不畅，导致系统边界画错了。"

阿明最后录用了第三个人。但他私下琢磨了很久：三个人都没有错，只是**看世界的方式不同**。

## 第一章：第一次跃迁 —— 从"管对象"到"管动作"

### 阿明的起点：把餐厅看成"东西"

刚开店时，阿明的世界观是这样的：

```
餐厅 = 菜品 + 厨具 + 食材 + 员工 + 顾客
```

他把每样东西做成一张卡片：

```
┌─────────────────────────┐
│   菜品卡（对象/Entity）  │
├─────────────────────────┤
│ 名称：牛肉面              │
│ 价格：28 元               │
│ 配料：牛肉、面、葱花       │
│ 做法：煮 3 分钟            │
└─────────────────────────┘
```

这就是**面向对象设计（OOD）** 的思维起点：把世界拆成一个个对象，封装属性和方法，对象之间通过方法调用协作。

> **OOD 的核心假设**：业务 = 实体集合。只要把实体建模对了，系统就能运转。

### 问题出在哪？

阿明开了三家店后发现：同样的"牛肉面"对象，在不同店里行为完全不同。

城东店的牛肉面要加辣，城西店的要清淡；堂食的出餐流程和外卖的出餐流程不一样；同一个"订单"，正常流程走 5 步，退款流程走 12 步。

阿明恍然大悟：**餐厅的本质不是"有什么"，而是"做什么"。**

同一碗面，在"采购域"的动作是"选供应商→验货→入库"，在"烹饪域"的动作是"配料→翻炒→装盘"，在"出餐域"的动作是"质检→打包→叫号"。每个域对同一碗面的理解和规则完全不同。

这就是从 OOD 到 **DDD（领域驱动设计）** 的认知跃迁：

| 维度 | OOD 思维 | DDD 思维 |
|------|---------|---------|
| 看餐厅 | "有菜品、厨具、员工" | "有采购、烹饪、出餐、服务" |
| 看系统 | 实体 + 方法 | 限界上下文 + 领域行为 |
| 拆分依据 | 技术模块（数据层/逻辑层/展现层） | 业务能力（采购域/烹饪域/出餐域） |
| 核心问题 | "这个类应该怎么设计？" | "业务边界在哪里？" |
| 统一语言 | 只有工程师懂 | 业务 + 技术共享同一套术语 |

> **DDD 的核心洞察**：同一个对象在不同业务域中的规则完全不同。必须先划清业务边界（限界上下文），再在每个域内设计模型。

（关于 DDD 在厨房的具体落地，详见[前传第八章](./02-system-architecture-evolution.md)的"从管菜品到管能力"。）

## 第二章：第二次跃迁 —— 从"管动作"到"管组织"

### 阿明的第二次困惑

阿明按 DDD 把系统拆成了五大域，每个域独立开发，理论上应该很高效。但实际运行半年后，问题全来了：

- 采购域和烹饪域之间，数据格式对不上，每周开三次对齐会
- 出餐域改了接口，烹饪域的服务全部报错，因为两边团队没沟通
- 每个域都说"我的设计是最优的"，但拼在一起，用户体验割裂

阿明去找那个面试时先看组织架构图的架构师。对方听完说了一句话：

> "你按业务能力拆了系统，但你有没有按业务能力拆团队？"

阿明一看组织架构图：采购域的开发团队和烹饪域的团队坐在不同的楼层，每周只开一次同步会，用的还是不同的项目管理工具。

**康威定律**（Conway's Law）精准描述了这种现象：

> 任何组织在设计系统时，最终产生的设计在结构上都等同于该组织的沟通结构。
>
> —— Melvin Conway, 1967

阿明终于明白了：系统拆得对不对，不取决于技术功底，而取决于**组织结构是否和业务边界对齐**。

### 从产品架构到企业架构

这就是从 DDD 到 **TOGAF（企业架构框架）** 的认知跃迁：

```
产品架构（DDD）回答的问题：
  "这个系统应该拆成几个域？每个域的规则是什么？"

企业架构（TOGAF）回答的问题：
  "需要几个团队？每个团队的边界在哪？
   团队之间怎么沟通？治理机制是什么？"
```

| 维度 | DDD（产品架构） | TOGAF（企业架构） |
|------|---------------|-----------------|
| 视角 | 产品和系统 | 整个企业 |
| 核心问题 | "业务边界在哪？" | "组织架构怎么对齐？" |
| 关键概念 | 限界上下文、聚合、领域事件 | 业务能力地图、ADM 流程、架构治理 |
| 产出物 | 领域模型、上下文映射 | 业务能力地图、价值流图、技术架构蓝图 |
| 适用规模 | 一个产品/一个平台 | 多个产品/整个企业 |

### 业务能力地图：TOGAF 的"上帝视角"

阿明用 TOGAF 的方法，画出了第一张业务能力地图：

```
                    ┌─ 供应链管理 ─┐  ┌─ 门店运营 ──┐  ┌─ 顾客体验 ──┐
    核心能力：      │  采购验收    │  │  烹饪出餐   │  │  点餐支付   │
                    │  库存管理    │  │  排班调度   │  │  会员积分   │
                    └──────┬──────┘  └──────┬─────┘  └──────┬─────┘
                           │               │               │
                    ┌──────▼───────────────▼───────────────▼──────┐
    支撑能力：      │         数据中台 · 技术平台 · 财务管理        │
                    └─────────────────────────────────────────────┘
```

每个核心能力对应一个团队，每个团队对应一个限界上下文。**先调整组织，再调整系统** —— 这就是逆康威定律。

> **逆康威定律（Inverse Conway Maneuver）**：想要什么样的系统架构，就先建立什么样的组织结构。让组织架构匹配期望的架构。

## 第三章：三次跃迁的完整图谱

### 认知深化的本质

回头看这三个阶段，你会发现一个规律：

```
数字化认知深化的方向 →
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  阶段1：管对象 │ →  │  阶段2：管动作 │ →  │  阶段3：管组织 │
│  "有什么"      │    │  "做什么"      │    │  "谁来做"      │
│              │    │              │    │              │
│  面向对象设计  │    │  领域驱动设计  │    │  企业架构      │
│  OOD          │    │  DDD          │    │  TOGAF        │
│              │    │              │    │              │
│  类比：认识    │    │  类比：理解    │    │  类比：组织    │
│  食材和厨具    │    │  做菜流程      │    │  厨房团队      │
└─────────────┘    └─────────────┘    └─────────────┘
       │                   │                   │
       ▼                   ▼                   ▼
   封装与复用          统一语言            康威定律
   类/模块             限界上下文          企业架构域
   单体/模块化         微服务              产品架构→企业架构
```

每一次跃迁，都不是否定前一阶段，而是**站到了更高的认知维度**：

- OOD 教会我们**如何写出高内聚、低耦合的代码** —— 这是战术层的基本功，永远需要
- DDD 教会我们**如何按业务边界拆分系统** —— 这是中观层的战略眼光
- TOGAF 教会我们**如何让组织结构和系统结构对齐** —— 这是全局层的治理思维

### 三层递进，不是三种选择

```
战略层：TOGAF（企业架构）
  决定"做什么系统、由谁做、怎么治理"
        ↓
中观层：DDD（领域驱动设计）
  决定"系统边界在哪、业务是什么"
        ↓
战术层：OOD（面向对象设计）
  决定"类如何组织、方法如何分配"
        ↓
编码层：设计模式
  决定"常见问题如何优雅解决"
```

很多人纠结"选 OOD 还是选 DDD"，就像厨师纠结"用菜刀还是用炒锅"。菜刀和炒锅不是互斥的，它们分别解决不同层面的问题。OOD 是菜刀（切菜的基本功），DDD 是菜谱（做菜的流程），TOGAF 是厨房布局（决定几个灶台、怎么排布动线）。

### 三个认知阶段对应的阿明餐厅阶段

| 阶段 | 阿明的餐厅规模 | 认知焦点 | 架构方法论 | 典型问题 |
|------|--------------|---------|-----------|---------|
| 阶段1：管对象 | 1-3 家店 | "有什么" | OOD + UML | 类怎么设计？模块怎么组织？ |
| 阶段2：管动作 | 3-10 家店 | "做什么" | DDD + 微服务 | 系统怎么拆？边界在哪？ |
| 阶段3：管组织 | 10-100+ 家店 | "谁来做" | TOGAF + 康威定律 | 团队怎么建？组织怎么对齐？ |

阿明在每个阶段被"逼出来"的方法论，恰好对应了数字化认知的深化路径。不是他选择了方法论，而是**认知深度决定了方法论的选型**。

## 第四章：认知跃迁的陷阱

### 陷阱一：越级跳跃

刚开一家店就画业务能力地图、搞企业架构治理 —— 这不是前瞻性，这是**过度设计**。OOD 在 MVP 阶段完全够用，不需要提前引入 DDD。TOGAF 在只有一个产品时用不上，不需要提前搞企业架构。

**判断标准**：当你的痛点不再是"类怎么设计"，而是"系统怎么拆"时，才需要从 OOD 跃迁到 DDD。当你的痛点不再是"系统怎么拆"，而是"团队怎么组织"时，才需要从 DDD 跃迁到 TOGAF。

### 陷阱二：认为"越新越好"

有人觉得 DDD 比 OOD 高级，TOGAF 比 DDD 高级。这是错的。它们是**不同层面的工具**，不是版本迭代。TOGAF 的应用架构层需要 DDD 来做领域建模，DDD 的聚合内部仍然使用 OOD 来组织代码。OOD 的设计模式在 DDD 的战术设计中仍然适用。

### 陷阱三：只升不降

认知可以升维，但执行必须降维。用 TOGAF 的视角思考，用 DDD 的方法拆分，用 OOD 的技术编码。如果反过来 —— 用 OOD 的视角思考全局、用 TOGAF 的细节来写类 —— 就会变成大泥球架构。

> **升维思考，降维执行**：站得越高，看得越远；写得越低，做得越好。

## 第五章：给新架构师的一张检查清单

当你在纠结"该用什么方法论"时，问自己三个问题：

| 问题 | 如果答案是… | 你需要的认知维度 |
|------|------------|----------------|
| 你的核心痛点是什么？ | "代码乱、类之间耦合严重" | OOD（战术层） |
| 你的核心痛点是什么？ | "系统太大、改不动、边界不清" | DDD（中观层） |
| 你的核心痛点是什么？ | "团队多、沟通成本高、系统对不上组织" | TOGAF（战略层） |
| 你的团队多大？ | < 10 人 | OOD 足够 |
| 你的团队多大？ | 10-50 人 | DDD 是标配 |
| 你的团队多大？ | > 50 人，多个产品 | TOGAF 视角 |
| 你的系统几个？ | 1 个单体应用 | OOD |
| 你的系统几个？ | 3-10 个服务 | DDD |
| 你的系统几个？ | 10+ 个系统、跨部门 | TOGAF |

## 总结：看世界的方式，决定了你造什么系统

阿明面试的三个架构师都没有错。第一个人看到了餐厅的**静态实体**（OOD），第二个人看到了餐厅的**动态行为**（DDD），第三个人看到了餐厅的**组织结构**（TOGAF）。

区别不在于谁更高级，而在于**谁看到了当前阶段最需要看到的东西**。

- 小团队、单系统：OOD 足够，别被方法论绑架
- 中等规模、多服务：DDD 帮你画对边界
- 大组织、多产品：TOGAF 帮你让组织和系统对齐

**认知可以升维，但执行必须降维。** 用 TOGAF 的眼光看全局，用 DDD 的方法划边界，用 OOD 的技术写代码。三层递进，各司其职。

```
                    看世界的方式
                  ┌───────┐
                  │ TOGAF │  ← 站在最高处，看到组织
                  ├───────┤
                  │  DDD  │  ← 站在中间，看到业务
                  ├───────┤
                  │  OOD  │  ← 站在最近，看到代码
                  └───────┘
                    造什么系统
```
