# 从接单到出餐

> 从阿明的"手写菜单"到自动化流水线，看 CI/CD 与 DevOps 的完整旅程

> **系列定位**：本篇是「阿明餐厅」系列的**正传 5**。在[前传](./02-system-architecture-evolution.md)中，阿明完成了架构演进；在[厨房质检员](./08-qa-testing-strategy.md)中建立了质量保障体系。但代码写完了、测试通过了，**怎么安全、快速地交付到生产环境？** 这就是 CI/CD 与 DevOps 的价值。

---

## 引言：新菜上线，手忙脚乱

阿明的厨师长研发了一道新菜"松露牛肉面"，想上线试卖。

传统流程：厨师长写好配方（代码写完了）→ 找阿明审批（人工审核）→ 手动更新菜单系统（手动部署）→ 更新错了，系统报错（出问题难回滚）→ 新菜上线，顾客反馈"太贵了"（没有灰度验证）。

整个过程花了 3 天，还出了两次事故。

阿明意识到：**从"代码写完"到"用户用上"之间，需要一条自动化流水线**。这条流水线要解决三个问题：自动化（减少人工干预）、安全（出问题能回滚）、快速（缩短交付周期）。

---

## 第一章：持续集成（CI）—— 代码合并的自动化

阿明的技术团队有 5 个人，各自开发不同功能。传统流程：每个人在自己的分支上开发 2 周，合并时冲突不断，花 3 天解决冲突，合并后才发现代码不兼容。

**持续集成（Continuous Integration, CI）** 的核心是：**频繁地（每天多次）将代码合并到主分支，并自动运行测试**。

### CI 流水线

```text
开发者提交代码（git push）
    ↓
自动触发 CI 流水线：
  1. 代码检查（Lint）：语法规范、代码风格
  2. 单元测试：运行所有单元测试
  3. 集成测试：运行模块间交互测试
  4. 构建（Build）：编译代码，生成 Docker 镜像
  5. 代码覆盖率：检查测试覆盖率是否达标
    ↓
全部通过 → 合并成功 | 任一失败 → 合并失败，通知开发者
```

### CI 的价值与反模式

| 价值 | 说明 |
|------|------|
| 尽早发现冲突 | 每天合并，冲突少，容易解决 |
| 自动化测试 | 每次合并都跑测试，保证代码质量 |
| 快速反馈 | 5 分钟内知道代码是否有问题 |

**反模式**：CI 流水线太慢 — 阿明最初的 CI 流水线跑一次要 30 分钟，优化后（并行执行、缓存依赖、增量测试）缩短到 5 分钟。

CI 的核心是**自动化 + 快速反馈**。没有 CI，持续交付就是空谈。

## 第二章：持续交付（CD）—— 从代码到生产的自动化

CI 保证了"代码合并后测试通过"，但**怎么把代码部署到生产环境？**

**持续交付（Continuous Delivery, CD）** 的核心是：**将部署过程自动化，随时可以一键部署到生产环境**。

### CD 流水线

```text
CI 通过（代码测试通过，Docker 镜像构建完成）
    ↓
  1. 部署到测试环境（Staging）：自动部署，运行 E2E 测试
  2. 部署到预生产环境（Pre-prod）：自动部署，运行冒烟测试
  3. 部署到生产环境（Production）：人工审批后，自动部署
    ↓
部署完成 → 监控告警（观察 15 分钟，确认无异常）
```

### 部署和发布分离

很多团队认为"部署 = 发布"。正确做法：**部署和发布分离**。

- **部署（Deploy）**：代码部署到生产环境，但用户还看不到
- **发布（Release）**：通过特性开关（Feature Toggle）或灰度策略，逐步对用户开放

好处：部署后先观察 15 分钟，确认无异常再发布；出问题可以快速回滚，影响范围小。这里的特性开关和[高峰保卫战的降级策略](./04-peak-traffic-defense.md)使用的是同一套 Feature Toggle 基础设施。

CD 的核心是**自动化 + 可回滚**。有了 CD，部署应该像按电梯按钮一样平常，而不是像拆炸弹一样紧张。

## 第三章：灰度发布 —— 新菜先给 1% 顾客试吃

还记得引言里那道让阿明手忙脚乱的"松露牛肉面"吗？定价 188 元，如果直接全量上线，可能面临：顾客觉得太贵差评如潮、系统出问题所有用户受影响、回滚成本高。

**灰度发布（Gray Release）** 的核心是：**新功能先对少数用户开放，验证无问题后再全量开放**。

### 灰度发布的策略

```text
阶段 1：内部测试（0%）→ 只对内部员工开放
阶段 2：小流量灰度（1%）→ 观察错误率、用户反馈、系统性能
阶段 3：中流量灰度（10%）→ 继续观察
阶段 4：大流量灰度（50%）→ 确认无问题
阶段 5：全量发布（100%）→ 对所有用户开放
```

**反模式**：灰度比例跳跃太大 — 阿明最初的策略是 1% → 50% → 100%，结果跳到 50% 时系统扛不住；正确做法是逐步放量（1% → 5% → 10% → 25% → 50% → 100%），每阶段观察 15-30 分钟。

灰度发布的核心是**小步快跑 + 快速回滚**。阿明再也不想经历"全量上线后双手合十祈祷"的十分钟了。

## 第四章：蓝绿部署 —— 新旧系统并行，一键切换

灰度发布是"逐步放量"，但**如果新版本有严重 Bug，怎么快速回滚？** 传统做法回滚要 10-15 分钟，期间用户无法使用。

**蓝绿部署（Blue-Green Deployment）** 的核心是：**新老版本同时运行，通过路由切换流量，实现秒级回滚**。

```text
初始状态：
  蓝环境（Blue）：运行 v1 版本，承接 100% 流量
  绿环境（Green）：空闲

部署新版本：
  绿环境：部署 v2 版本，运行冒烟测试
  
切换流量：
  蓝环境 → 绿环境：承接 100% 流量
  
观察 15 分钟：
  无问题 → 绿环境（v2）继续承接流量，下一次部署时将蓝环境更新为 v2
  有问题 → 流量切回蓝环境（秒级回滚，蓝环境仍为 v1）
```

| 价值 | 说明 |
|------|------|
| 秒级回滚 | 切换 Service selector 即可回滚 |
| 零停机 | 新老版本并行，切换时无感知 |
| 验证充分 | 绿环境部署后先内部测试，确认无问题再切换 |

**反模式**：蓝绿环境资源配置不一致 — 蓝绿部署需要两套环境，阿明的策略是环境资源配置一致，切换后观察 15 分钟确认无问题再释放旧环境资源。

蓝绿部署的核心是**并行运行 + 秒级切换**。回滚不再是深夜的紧急操作，而是一次鼠标点击。

## 第五章：金丝雀发布 —— 用真实用户验证

灰度发布和蓝绿部署都是"部署策略"，但**怎么知道新版本是否真的更好？**

**金丝雀发布（Canary Deployment）** 的核心是：**新版本先对少数真实用户开放，通过监控数据判断是否全量发布**。

阿明这次学乖了 —— 松露牛肉面上次灰度发布只看了"有没有差评"，太粗糙。这回他用金丝雀策略：先对 1% 的用户开放松露牛肉面的订购，同时盯着三组数据 —— 下单转化率、出餐超时率、用户评分。只有三项指标都不劣于老菜品，才逐步放量。

### 金丝雀发布的判断标准

```text
金丝雀阶段（1% 流量到新版本）：

监控指标（不劣于老版本即为通过）：
  - 错误率：新版本 ≤ 0.5%（老版本 0.5%）✅
  - P99 延迟：新版本 ≤ 450ms（老版本 450ms）✅
  - 用户评分：新版本 ≥ 4.3（老版本 4.3）✅

判断：
  所有指标优于或等于老版本 → 进入下一阶段（10%）
  任一指标劣于老版本 → 回滚到老版本
```

金丝雀发布依赖**实时监控指标**（如 P99 延迟、错误率），这些指标的设计和告警策略详见[《厨房装监控》](./05-observability.md)。没有好的 Metrics，金丝雀发布就是盲人开车。

**反模式**：金丝雀阶段太短 — 阿明最初的策略是 1% 流量跑 5 分钟就全量，样本量太小且某些问题（如内存泄漏）需长时间才暴露；正确做法是至少 15-30 分钟、积累足够样本（实践中通常需 5000-10000 次请求才能获得统计显著性）。

金丝雀发布的核心是**数据驱动 + 小步快跑**。没有金丝雀，发布就是"盲飞"。

## 第六章：GitOps —— 基础设施即代码

阿明的运维团队有一个问题：**配置变更难以追踪**。运维工程师手动修改 Kubernetes 配置（如副本数从 3 改为 5），修改后没有记录，出问题不知道谁改的、改了什么。

**GitOps** 的核心是：**把基础设施配置存储在 Git 仓库中，通过 Git 提交触发自动化部署**。

用餐厅的比喻：GitOps 就像把配方库上了锁，所有配方变更必须走审批流程（Pull Request），审批通过后厨房自动照本执行 —— 不存在"某个厨师偷偷改了配方但没人知道"的情况。

### GitOps 的流程

```text
1. 开发者修改 Git 仓库中的配置文件（如 deployment.yaml）
2. 提交 Pull Request，团队 Review
3. Review 通过后合并到主分支
4. GitOps 工具（如 Argo CD）检测到 Git 变更
5. 自动将配置同步到 Kubernetes 集群
6. 监控配置变更，确认部署成功
```

| 价值 | 说明 |
|------|------|
| 版本控制 | 所有配置变更都有 Git 记录，可追溯 |
| 审计合规 | 谁改了什么、什么时候改的，一目了然（详见[安全架构的审计日志](./06-security-architecture.md)） |
| 自动化同步 | Git 变更后自动同步到集群，减少人工操作 |

**反模式**：GitOps 和手动操作混用 — 既用 GitOps 自动同步又手动修改集群配置会导致配置漂移；阿明的策略是所有配置变更都通过 GitOps，紧急情况先手动修改但事后必须同步到 Git。

GitOps 的核心是**配置即代码 + 自动化同步**。阿明的经验是：凡是手动改过的配置，三天后一定没人记得改了什么。

## 核心总结：CI/CD 与 DevOps 的完整旅程

```mermaid
graph TD
    A[代码提交] --> B[CI 流水线<br/>Lint + 测试 + 构建]
    B --> C[CD 流水线<br/>部署到测试/预生产]
    C --> D[部署策略<br/>灰度/蓝绿/金丝雀]
    D --> E[生产环境]
    E --> F[监控告警<br/>观察 15 分钟]
    F -->|无问题| G[全量发布]
    F -->|有问题| H[回滚]
    H --> B
    G --> I[GitOps<br/>配置即代码]
```

| 策略 | 核心问题 | 餐厅类比 | 技术实现 |
|------|----------|----------|----------|
| CI（持续集成） | 代码合并后测试通过吗？ | 新配方先小批量试做 | GitHub Actions / Jenkins |
| CD（持续交付） | 怎么自动化部署？ | 自动化出餐流水线 | Argo CD / Spinnaker |
| 灰度发布 | 怎么逐步放量？ | 新菜先给 1% 顾客试吃 | Nginx / Istio |
| 蓝绿部署 | 怎么秒级回滚？ | 新旧菜单并行，一键切换 | K8s Service + Deployment |
| 金丝雀发布 | 怎么数据驱动决策？ | 用真实用户反馈判断 | Istio / Prometheus |
| GitOps | 怎么管理配置变更？ | 菜单变更走审批流 | Argo CD / Flux |

## 完整 CI/CD 流水线图

一个成熟的 CI/CD 流水线包含以下 10 个阶段：

```mermaid
graph LR
    A[代码提交] --> B[代码检查]
    B --> C[单元测试]
    C --> D[构建打包]
    D --> E[集成测试]
    E --> F[安全扫描]
    F --> G[镜像推送]
    G --> H[预发布环境]
    H --> I[金丝雀发布]
    I --> J{监控正常?}
    J -->|是| K[全量发布]
    J -->|否| L[自动回滚]
    K --> M[生产环境]
```

### 10 个阶段的关键指标

| 阶段 | 目标时长 | 关键指标 | 工具 |
|------|---------|---------|------|
| 代码检查 | < 30 秒 | 静态分析通过率 | ESLint / SonarQube |
| 单元测试 | < 2 分钟 | 覆盖率 > 70% | Jest / pytest |
| 构建打包 | < 3 分钟 | 缓存命中率 | Docker BuildKit |
| 集成测试 | < 5 分钟 | 接口契约通过 | Pact / Hoverfly |
| 安全扫描 | < 2 分钟 | 高危漏洞 = 0 | Trivy / Snyk |
| 镜像推送 | < 1 分钟 | 镜像大小 < 500MB | Harbor / ECR |
| 预发布 | < 3 分钟 | 端到端测试通过 | Playwright / Cypress |
| 金丝雀 | 30 分钟 | 错误率 < 阈值 | Istio / Argo Rollouts |
| 全量发布 | < 2 分钟 | 流量切换完成 | K8s Service |
| 回滚 | < 1 分钟 | 服务恢复正常 | 自动 / 一键 |

**端到端目标**：从代码提交到全量发布 < 30 分钟。**超过 1 小时 = 流水线太慢**。

## 5 大发布策略对比

不同发布策略适合不同场景：

| 策略 | 风险 | 回滚速度 | 适用场景 | 不适用场景 |
|------|------|---------|---------|----------|
| 蓝绿部署 | 中（旧环境占用） | 秒级 | 高风险变更 | 资源受限 |
| 金丝雀发布 | 低 | 1-2 分钟 | 数据敏感场景 | 实时性要求高 |
| 灰度发布 | 低 | 1-2 分钟 | A/B 测试 | 强一致性 |
| 滚动发布 | 中 | 5-10 分钟 | 通用场景 | 数据库变更 |
| 红黑部署 | 中 | 分钟级 | 云原生 | 旧版必须保留 |

**阿明的实战选择**：
- **核心订单/支付**：金丝雀 + 蓝绿组合（数据敏感，回滚要求高）
- **会员/营销系统**：灰度发布（需要 A/B 数据）
- **后台管理工具**：滚动发布（影响小，速度优先）

## 3 个真实 CI/CD 翻车案例

### 案例 1：CI 跑 2 小时，团队放弃测试

阿明最初 CI 流水线跑 2 小时，团队反馈"太慢了"，于是开始"挑着跑测试"，覆盖率从 80% 降到 40%。半年后线上事故率翻倍。

**教训**：**CI 慢 = 测试形同虚设**。CI 必须快到"开发者愿意每次都跑"，超过 10 分钟 = 失败。

**优化手段**：
- 并行化（单测 70% 时间从 30 分钟降到 5 分钟）
- 缓存依赖（Docker 层缓存 + npm 缓存）
- 增量测试（只跑变更影响的测试）
- 测试分层（快速测试必跑，慢测试夜间跑）

### 案例 2：灰度跳跃太大，1% → 100% 翻车

2024 年 9 月，研发"图省事"直接从 1% 灰度到 100%，中间跳过了 5% / 10% / 25% / 50% 阶段。结果 100% 上线后才发现接口兼容性问题，回滚导致 30 分钟故障。

**教训**：**灰度必须阶梯式放量**，1% → 5% → 10% → 25% → 50% → 100%，每阶段至少观察 15 分钟。跳级 = 赌博。

### 案例 3：数据库变更没纳入发布流程

2025 年初，一个新功能上线，代码走金丝雀没问题，但数据库 DDL 没回滚方案。结果 100% 流量上来后，新字段类型不兼容，老数据查询超时。紧急修复 1 小时，损失 200 万 GMV。

**教训**：**CI/CD 必须是端到端**，包括代码、配置、数据库变更、缓存更新。**数据库变更是 CI/CD 的盲区**。

**数据库变更安全三件套**：
- 灰度 DDL：先在影子表加字段，验证后切换
- 双写：旧字段新字段都写，对账一致后再废弃
- 一键回滚：DBA 准备回滚 SQL，5 分钟内执行

## CI/CD 文化的 5 个反模式

| # | 反模式 | 后果 |
|---|--------|------|
| 1 | "CI 跑通了就发布" | 缺少 staging / 预发布环境验证 | 
| 2 | "测试失败就跳过" | 测试形同虚设 |
| 3 | "发布不在工作时间" | 半夜 3 点发版，救火人少 |
| 4 | "回滚 = 失败" | 出问题不敢回滚，导致更大损失 |
| 5 | "CI/CD = 工具" | 忽视文化，只追工具，团队不配合 |

### 一句心法

**CI/CD 不是工具，而是文化。核心六个字：自动化、可回滚、快速反馈。**

## 结语

阿明从"手写菜单"到"自动化流水线"的旅程，回答了一个所有工程团队都在追问的问题：**怎么让代码安全、快速地到达用户手中 —— 靠的是流水线，不是运气。**

答案是 CI/CD + 部署策略 + GitOps：持续集成保证代码质量，持续交付自动化部署，灰度/蓝绿/金丝雀降低发布风险，GitOps 管理配置变更。

下次当你发布代码时，不妨问自己：

- 我有 CI 流水线吗？每次合并都自动跑测试吗？
- 我用了灰度/蓝绿/金丝雀吗？还是直接全量发布？
- 出问题能快速回滚吗？回滚流程是自动化的吗？
- 配置变更有版本控制吗？还是手动修改，难以追踪？

> 好的 CI/CD，不是"让发布变快"，而是"让发布变安全"。自动化、可回滚、快速反馈，这三点是 CI/CD 的核心。
