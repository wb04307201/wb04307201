## 关键决策速查

### 架构选型

| 场景 | 选择 | 原因 |
|------|------|------|
| 读多写少，缓存一致性 | Cache Aside | 菜单更新频率低，简单够用 |
| 分布式事务 | Saga | 比 2PC 性能好，比 TCC 实现成本低 |
| 存储选型 | 按场景匹配 | 没有银弹数据库，只有场景匹配 |

### 流量治理

| 场景 | 选择 | 原因 |
|------|------|------|
| 限流算法 | 令牌桶 | 允许突发，长期速率可控 |
| 服务超时率高 | 熔断器 | 快速失败，保护调用方 |
| 资源不足 | 降级预案表 | 提前设计，不是临时拍脑袋 |

### 部署策略

| 策略 | 核心优势 | 适用场景 |
|------|----------|----------|
| 灰度发布 | 逐步放量 | 新功能验证 |
| 蓝绿部署 | 秒级回滚 | 高风险变更 |
| 金丝雀发布 | 数据驱动 | 需要 A/B 对比 |
| GitOps | 配置版本化 | 基础设施管理 |

### API 风格

| 场景 | 选择 | 原因 |
|------|------|------|
| 对外 API | REST | 生态成熟，文档友好 |
| 前端 API | GraphQL | 灵活查询，减少请求 |
| 微服务间 | gRPC | 高性能，强类型 |

### 测试比例

| 类型 | 占比 | 速度 | 成本 |
|------|------|------|------|
| 单元测试 | 70% | 毫秒 | 低 |
| 集成测试 | 20% | 秒 | 中 |
| E2E 测试 | 10% | 分钟 | 高 |

### 数据架构选型

| 场景 | 选择 | 原因 |
|------|------|------|
| BI 报表和分析 | 数据仓库（Star Schema） | 结构化查询性能好，分析师友好 |
| 机器学习和探索性分析 | 数据湖 | 原始数据存储，灵活度高 |
| 两者兼顾 | 数据湖仓（Lakehouse） | 兼具结构化和灵活性 |

### 性能优化优先级

| 阶段 | 选择 | 原因 |
|------|------|------|
| 数据库慢查询 | 索引优化 + SQL 改写 | 80% 性能问题出在 20% 查询上 |
| 重复查询 | 多级缓存 | 空间换时间，命中率高时效果显著 |
| 网络延迟 | CDN + 压缩 | 减少传输距离和体积 |
| 串行瓶颈 | 异步 + 消息队列 | 解耦 + 削峰 |

### 云成本优化

| 场景 | 选择 | 原因 |
|------|------|------|
| 稳定核心服务 | 预留实例（Reserved） | 锁定 1 年省 30-60% |
| 弹性负载 | 按需付费 + 自动伸缩 | 灵活，按需付费 |
| 可中断批处理 | 竞价实例（Spot） | 省 60-90%，中断可重试 |

### 消息队列选型

| 场景 | 选择 | 原因 |
|------|------|------|
| 高吞吐日志流 | Kafka | 百万级 TPS，持久化存储 |
| 企业级业务消息 | RabbitMQ | 协议丰富，路由灵活 |
| 金融级事务消息 | RocketMQ | 事务消息原生支持，可靠性高 |

### 多端框架选型

| 场景 | 选择 | 原因 |
|------|------|------|
| 快速多端覆盖 | React Native / Flutter | 一套代码两端运行，开发成本低 |
| 极致体验 | 原生开发 | 性能上限最高，平台能力全覆盖 |
| 统一后端聚合 | BFF 模式 | 每端独立适配层，后端保持统一 |

### 多租户隔离策略

| 场景 | 选择 | 原因 |
|------|------|------|
| 租户少、数据敏感 | 独立部署 | 物理隔离，最安全 |
| 大量中小租户 | 行级隔离（Row-Level） | 共享数据库，成本最低 |
| 中等规模、需要定制 | Schema 隔离 | 逻辑隔离，兼顾安全与成本 |

### 搜索排序策略

| 场景 | 选择 | 原因 |
|------|------|------|
| 基础文本搜索 | 倒排索引 + TF-IDF | 简单高效，覆盖 80% 场景 |
| 综合搜索 | 混合搜索（文本 + 语义） | 兼顾精确匹配和语义理解 |
| 个性化推荐 | 协同过滤 + 内容推荐 | 挖掘用户隐性偏好 |

## 演进路线图

```text
初创期 ─── 单机 + 缓存 + 读写分离（基础设施线）
   │         简单就是最高级的设计
   ▼
成长期 ─── 垂直拆分 + 水平分片 + 多存储（基础设施线）
   │         + 按实体→按能力→按边界（业务建模线）
   ▼
成熟期 ─── 云原生 + 微服务 + 多活容灾
   │         配套：可观测性 + 安全架构 + CI/CD
   ▼
智能化 ─── AI Agent（感知→记忆→规划→工具→协同→反馈→安全）
   ▼
AI 转型 ─── 用人悖论 → 成本转移 → 岗位重塑 → 人机协同 → AI Native 组织
   ▼
AI 创业 ─── 构思验证 → MVP → PMF 辨别 → 智能体工作流 → 创始人编排
   ▼
AI 自进化 ─── Agent Loop → 五层循环 → Burn Tokens → AI 可读性 → 边界守护
   ▼
AI 信任 ─── 幻觉分类 → 信任校准 → 三层护栏 → HITL → 毕业制
   ▼
数据化 ─── 数据孤岛 → 数据仓库 → 数据建模 → 数据质量 → 数据应用 → 数据合规
   ▼
体验化 ─── 8 秒加载 → 性能优化 → UX 设计 → 组件化 → 工程化 → 体验度量
   ▼
韧性化 ─── 故障被动 → 故障分级 → 应急预案 → 混沌工程 → 韧性文化
   ▼
异步化 ─── 同步阻塞 → 消息队列 → 事件驱动 → 事件溯源 → 流处理
   ▼
规模化 ─── 单店 → 十店 → CAP 取舍 → 分布式锁 → 多租户 SaaS → 国际化
   ▼
平台化 ─── 多端覆盖 → BFF → 低代码 → 知识工程 → 平台工程
```

## 安全六大防线

```text
请求 → 身份认证 → 权限控制 → 数据加密 → 零信任 → 审计日志 → 数据脱敏 → 响应
```

| 防线 | 解决什么 | 技术实现 |
|------|----------|----------|
| 身份认证 | 你是谁？ | OAuth / SSO / MFA |
| 权限控制 | 你能做什么？ | RBAC / ABAC |
| 数据加密 | 数据怎么保护？ | TLS / AES / KMS |
| 零信任 | 内网也可信吗？ | mTLS / 微隔离 |
| 审计日志 | 出问题能追溯吗？ | 不可删改的日志 |
| 数据脱敏 | 敏感信息隐藏？ | 掩码 / 哈希 / 令牌化 |

| 模块 | 核心问题 | 关键技术 |

## 三个不变原则

> **业务驱动** — 所有架构升级都源于真实痛点，不是为了用新技术而用新技术。
>
> **渐进演进** — 没有"一步到位"的完美架构，只有"小步快跑"的持续迭代。
>
> **权衡艺术** — 性能/成本/一致性/可维护性，永远在做选择题。选 A 就得接受 A 的代价。
