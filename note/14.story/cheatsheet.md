## 关键决策速查

### 架构选型

| 场景 | 选择 | 原因 |
|------|------|------|
| 读多写少，缓存一致性 | Cache Aside | 菜单更新频率低，简单够用 |
| 分布式事务 | Saga | 比 2PC 性能好，比 TCC 实现成本低 |
| 存储选型 | 按场景匹配 | 没有银弹数据库，只有场景匹配 |

### 流量治理

| 场景 | 选择 | 原因 |
|------|------|------|
| 限流算法 | 令牌桶 | 允许突发，长期速率可控 |
| 服务超时率高 | 熔断器 | 快速失败，保护调用方 |
| 资源不足 | 降级预案表 | 提前设计，不是临时拍脑袋 |

### 部署策略

| 策略 | 核心优势 | 适用场景 |
|------|----------|----------|
| 灰度发布 | 逐步放量 | 新功能验证 |
| 蓝绿部署 | 秒级回滚 | 高风险变更 |
| 金丝雀发布 | 数据驱动 | 需要 A/B 对比 |
| GitOps | 配置版本化 | 基础设施管理 |

### API 风格

| 场景 | 选择 | 原因 |
|------|------|------|
| 对外 API | REST | 生态成熟，文档友好 |
| 前端 API | GraphQL | 灵活查询，减少请求 |
| 微服务间 | gRPC | 高性能，强类型 |

### 测试比例

| 类型 | 占比 | 速度 | 成本 |
|------|------|------|------|
| 单元测试 | 70% | 毫秒 | 低 |
| 集成测试 | 20% | 秒 | 中 |
| E2E 测试 | 10% | 分钟 | 高 |

## 演进路线图

```
初创期 ─── 单机 + 缓存 + 读写分离
   │         简单就是最高级的设计
   ▼
成长期 ─── 垂直拆分 + 水平分片 + 多存储
   │         业务驱动，渐进演进
   ▼
成熟期 ─── 云原生 + 微服务 + 多活容灾
   │         配套：可观测性 + 安全架构 + CI/CD
   ▼
智能化 ─── AI Agent（感知→记忆→规划→工具→协同→反馈→安全）
```

## 安全六大防线

```
请求 → 身份认证 → 权限控制 → 数据加密 → 零信任 → 审计日志 → 数据脱敏 → 响应
```

| 防线 | 解决什么 | 技术实现 |
|------|----------|----------|
| 身份认证 | 你是谁？ | OAuth / SSO / MFA |
| 权限控制 | 你能做什么？ | RBAC / ABAC |
| 数据加密 | 数据怎么保护？ | TLS / AES / KMS |
| 零信任 | 内网也可信吗？ | mTLS / 微隔离 |
| 审计日志 | 出问题能追溯吗？ | 不可删改的日志 |
| 数据脱敏 | 敏感信息隐藏？ | 掩码 / 哈希 / 令牌化 |

## AI Agent 七大模块

| 模块 | 核心问题 | 关键技术 |
|------|----------|----------|
| 感知层 | 听懂用户要什么 | 多模态 + 意图识别 |
| 记忆层 | 越用越懂你 | 工作/情节/语义/程序记忆 + RAG |
| 规划层 | 拆解复杂需求 | ReAct / ToT / GoT |
| 工具层 | 把想法变现实 | Function Calling + JSON Schema |
| 协同层 | 多核协作 | Orchestrator + 消息总线 |
| 反馈层 | 自我纠错 | RLHF / DPO + 三层进化管道 |
| 安全层 | 不翻车守底线 | Prompt 注入防护 + 权限沙箱 |

## 三个不变原则

> **业务驱动** — 所有架构升级都源于真实痛点，不是为了用新技术而用新技术。
>
> **渐进演进** — 没有"一步到位"的完美架构，只有"小步快跑"的持续迭代。
>
> **权衡艺术** — 性能/成本/一致性/可维护性，永远在做选择题。选 A 就得接受 A 的代价。
