# App 技术栈选型：原生 vs 跨端 vs H5 vs 小程序

> 50 万 App 报价中"前端开发"那一项的成本来源。考察的是 **技术决策能力 + 真实成本对比能力**，而不是"什么流行用什么"。

## 一、核心结论（TL;DR）

| 方案 | 开发成本 | 性能 | 体验 | 维护成本 | 适用场景 |
|------|---------|------|------|---------|---------|
| **原生（Swift/Kotlin）** | 💰💰💰💰 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 💰💰💰 | 大型商业 App |
| **跨端（Flutter）** | 💰💰💰 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | 💰💰 | 跨平台 + 高性能 |
| **跨端（React Native）** | 💰💰💰 | ⭐⭐⭐ | ⭐⭐⭐ | 💰💰💰 | JS 团队跨端 |
| **H5 套壳（Hybrid）** | 💰 | ⭐⭐ | ⭐⭐ | 💰 | MVP / 内部工具 |
| **小程序（微信/支付宝）** | 💰 | ⭐⭐⭐ | ⭐⭐⭐ | 💰 | 工具类 / 电商 |

> 一句话：**没有"最好"的技术栈，只有"最匹配场景"的技术栈**。大型 App 必选原生，中型 App 选 Flutter，MVP 选 H5/小程序。

## 二、5 种方案详细分析

### 1. 原生开发（Native）

**技术**：Swift（iOS）+ Kotlin（Android）
**代表 App**：支付宝、抖音、淘宝

| 维度 | 评价 |
|------|------|
| 性能 | ⭐⭐⭐⭐⭐（直接调用系统 API） |
| 体验 | ⭐⭐⭐⭐⭐（最接近系统） |
| 开发成本 | 💰💰💰💰（双端各 1 团队） |
| 维护成本 | 💰💰💰（双端独立维护） |
| 包大小 | 小（无运行时） |

**优点**：
- 性能最佳，复杂动画、滑动无卡顿
- 系统新特性第一时间支持（如 iOS 新版 API）
- 用户体验最接近系统

**缺点**：
- 双端团队，开发成本翻倍
- 双端代码不同步，新功能要开发两次
- 小团队负担重

### 2. Flutter（Google 跨端）

**技术**：Dart + 自绘引擎（Skia）
**代表 App**：闲鱼、贝壳、Now in Android

| 维度 | 评价 |
|------|------|
| 性能 | ⭐⭐⭐⭐（AOT 编译后接近原生） |
| 体验 | ⭐⭐⭐⭐（自绘引擎，一致性好） |
| 开发成本 | 💰💰💰（单端代码 1 团队） |
| 维护成本 | 💰💰（一套代码两端运行） |
| 包大小 | 大（自带 Flutter 引擎 ~5MB） |

**优点**：
- 单端代码同时跑 iOS/Android（节省 30-50% 成本）
- 自绘引擎，跨端 UI 一致性好
- 性能接近原生

**缺点**：
- 包体积大（影响下载转化率）
- 第三方库不如 RN 丰富
- 学习曲线（Dart 语言）

### 3. React Native（Facebook 跨端）

**技术**：JavaScript + Bridge + 原生组件
**代表 App**：Facebook、Instagram、京东

| 维度 | 评价 |
|------|------|
| 性能 | ⭐⭐⭐（Bridge 通信有损耗） |
| 体验 | ⭐⭐⭐（使用原生组件，体验较好） |
| 开发成本 | 💰💰💰（JS 团队 + 原生插件） |
| 维护成本 | 💰💰💰（Bridge 升级、版本兼容） |
| 包大小 | 中 |

**优点**：
- JS 团队可快速上手
- 第三方库丰富
- 热更新支持（CodePush）

**缺点**：
- Bridge 通信有性能损耗（新版用 JSI 缓解）
- 复杂动画卡顿
- 原生模块需要懂 Java/Obj-C

### 4. H5 套壳（Hybrid）

**技术**：HTML/CSS/JS + WebView 容器
**代表 App**：早期 Hybrid App、电商活动页

| 维度 | 评价 |
|------|------|
| 性能 | ⭐⭐（WebView 性能差） |
| 体验 | ⭐⭐（滑动卡顿） |
| 开发成本 | 💰（Web 前端团队即可） |
| 维护成本 | 💰（一套代码） |
| 包大小 | 极小 |

**优点**：
- 开发成本最低
- Web 前端团队即可上手
- 一套代码全平台

**缺点**：
- 性能差（滑动、长列表、动画）
- 用户体验差（C 端用户能明显感知）
- 无法使用系统高级 API（如相机高级功能）

### 5. 小程序（微信/支付宝/抖音）

**技术**：各平台自定义 DSL（WXML/AXML）
**代表 App**：所有微信小程序、抖音小程序

| 维度 | 评价 |
|------|------|
| 性能 | ⭐⭐⭐（平台优化后接近原生） |
| 体验 | ⭐⭐⭐（接近原生体验） |
| 开发成本 | 💰（前端团队即可） |
| 维护成本 | 💰（无 App Store 上架审核） |
| 用户获取 | 🚀 流量红利 |

**优点**：
- 无需下载，扫码即用
- 微信/支付宝生态流量
- 审核快，迭代灵活

**缺点**：
- 受平台限制（不能调用某些系统 API）
- 包大小限制（微信主包 2MB）
- 多平台需多套代码（微信/支付宝/抖音）

## 三、选型决策树

```
Q1: 用户量预期？
├── DAU < 1000 → 小程序 / H5 套壳（5 万档）
├── DAU 1000-10 万 → Flutter（20 万档）
└── DAU > 10 万 → 原生或 Flutter（50 万档）

Q2: 团队技术栈？
├── JS 为主 → React Native
├── 愿意学新语言 → Flutter
└── 双端原生团队 → 原生

Q3: 业务复杂度？
├── 简单 CRUD + 展示 → 小程序 / H5
├── 中等（含支付、IM） → Flutter / RN
└── 复杂（音视频、AI、AR） → 原生
```

## 四、真实案例

| App | 技术栈 | 选择理由 |
|-----|--------|---------|
| **抖音** | 原生 | 复杂动画 + 性能极致要求 |
| **闲鱼** | Flutter | 跨端 + 高性能（复杂业务） |
| **京东** | React Native | JS 团队庞大 + 快速迭代 |
| **美团** | 原生 + Hybrid 混合 | 核心页面原生，活动页 H5 |
| **小程序** | 微信原生 | 工具类，无下载需求 |

## 五、面试陷阱

### 陷阱 1：以为 Flutter 能完全替代原生

- **真相**：复杂动画、底层 API 调用（如相机高级功能）仍需原生实现。Flutter 是"折中"方案，不是"完美"方案

### 陷阱 2：以为 React Native 性能等同 Flutter

- **真相**：RN 用 Bridge 通信，性能损耗明显；Flutter 用 AOT 编译 + 自绘引擎，性能接近原生

### 陷阱 3：以为 H5 套壳性能可以优化到接近原生

- **真相**：WebView 的天花板就在那里，复杂场景（如长列表滚动）永远无法达到原生体验

### 陷阱 4：以为小程序可以替代 App

- **真相**：小程序受平台限制（包大小、API 限制），不能做复杂商业产品

---

## 六、架构师话术模板

> App 技术栈选型要考虑 4 个维度：用户量、性能要求、团队技术栈、业务复杂度。
>
> - **DAU < 1000 + 简单业务**：选 H5/小程序，5万档搞定
> - **DAU 1-10 万 + 中等业务**：选 Flutter（性能接近原生，开发成本节省 30-50%）
> - **DAU > 10 万 + 复杂业务**（音视频/AI/复杂动画）：选原生双端，50万+ 档
>
> 我们团队 JS 为主 + 业务中等复杂 → 推荐 React Native；如果愿意投入学习 + 跨端 + 性能要求高 → 推荐 Flutter。
>
> 不推荐 H5 套壳做正式商业产品（C 端用户体验差），不推荐小程序做复杂商业产品（平台限制多）。
