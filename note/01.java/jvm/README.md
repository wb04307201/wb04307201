# JVM — 讲明白 Java 虚拟机

> 目标：读完这篇，理解 JVM 是怎么让 Java 程序跑起来的。

---
## 引言：性能对比

JVM — 讲明白 Java 虚拟机 的关键不是'快'——是**什么时候慢、慢多少、为什么**。

本篇用'常见 vs 极端'两组数字切入，把排查思路和优化边界讲清。

---

## 一、JVM 是什么

Java 程序不直接跑在硬件上，而是跑在 JVM（Java Virtual Machine）这个"虚拟计算机"上。

```
你的代码 (.java)
    ↓ javac 编译
字节码 (.class)
    ↓ JVM 加载
JVM 解释/编译为机器码
    ↓
操作系统 → 硬件
```

**"一次编写，到处运行"的本质**：字节码是统一的，JVM 是平台相关的。Windows 上跑 Windows 版 JVM，Linux 上跑 Linux 版 JVM，它们都能理解同一份字节码。

当前最主流的 JVM 实现是 **HotSpot**（Oracle JDK 和 OpenJDK 都使用它），下面讲的都是 HotSpot 的行为。

---

## 二、一个 Java 程序的完整生命周期

用一行代码串起整个 JVM：

```java
Person p = new Person("Alice");
```

当这行代码执行时，JVM 做了这些事：

```
1. 类加载     检查 Person.class 是否已加载 → 没有则加载到方法区
2. 内存分配   在堆的 Eden 区分配一块内存
3. 零值初始化 name=null, age=0（字段默认值）
4. 设置对象头 Mark Word（锁状态、GC 年龄）+ 类指针
5. 执行构造   调用 Person("Alice")，设置 this.name = "Alice"
6. 返回引用   引用 p 存入当前线程栈帧的局部变量表
```

这个流程贯穿了类加载、内存分配、对象创建、栈帧管理 — 覆盖了 JVM 的大部分核心机制。

### 2.1 对象在内存中长什么样

一个对象在堆里不是随便丢一块内存，而是有严格的结构：

```
┌─────────────────── 对象内存布局 ───────────────────┐
│                                                     │
│  ┌─────────────── 对象头（Object Header）──────────┐  │
│  │                                                  │  │
│  │  Mark Word（8 字节 / 64位）                      │  │
│  │  ┌─────────────────────────────────────────┐    │  │
│  │  │ hashCode | 分代年龄 | 锁状态 | 其他标记   │    │  │
│  │  └─────────────────────────────────────────┘    │  │
│  │                                                  │  │
│  │  Klass Pointer（4 字节压缩 / 8 字节未压缩）       │  │
│  │  → 指向方法区中的类元数据                         │  │
│  │                                                  │  │
│  │  数组长度（仅数组对象，4 字节）                    │  │
│  └──────────────────────────────────────────────────┘  │
│                                                     │
│  ┌─────────────── 实例数据（Instance Data）────────┐  │
│  │  各个字段的值（按类型排列）                        │  │
│  │  int age = 25 → 4 字节                           │  │
│  │  String name → 4 字节（压缩指针）或 8 字节        │  │
│  └──────────────────────────────────────────────────┘  │
│                                                     │
│  ┌─────────────── 对齐填充（Padding）─────────────┐  │
│  │  补齐到 8 字节的整数倍（HotSpot 要求）             │  │
│  └──────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

**算一个对象占多少内存**：

```java
public class Person {
    int age;        // 4 字节
    String name;    // 4 字节（压缩指针）
}
```

```
对象头：12 字节（Mark Word 8 + 压缩 Klass Pointer 4）
age：    4 字节
name：   4 字节（引用，压缩指针）
──────────────
合计：  20 字节
对齐：  → 24 字节（向上取到 8 的倍数）
```

**最小对象**：即使是一个空对象 `new Object()`，也要占 16 字节（12 字节头 + 4 字节对齐填充）。

**压缩指针（Compressed Oops）**：

| | 未压缩 | 压缩（默认） |
|---|---|---|
| 对象引用 | 8 字节 | 4 字节 |
| Klass Pointer | 8 字节 | 4 字节 |
| 对象头大小 | 16 字节 | 12 字节 |
| 可寻址堆大小 | 无限制 | **约 32GB** |

压缩指针把 8 字节的引用压缩到 4 字节，省下接近一半的引用内存。但 4 字节只能表示 2^32 个地址，配合 8 字节对齐（地址末尾 3 位必为 0），实际能寻址 **2^35 ≈ 32GB**。这就是为什么堆不要超过 32GB — 超过了压缩指针失效，每个引用都变成 8 字节，内存利用率反而下降。

可以用 `-XX:-UseCompressedOops` 关闭压缩指针来验证，但**永远不要在生产环境关**。

## 三、内存布局（运行时数据区）

JVM 把内存分成几大块，每块用途不同：

```
┌─────────────────────────────────────────────────────────┐
│                    JVM 内存全景                           │
│                                                         │
│  ┌─────────────────────┐  ┌──────────────────────────┐  │
│  │   线程私有（每线程一份）  │  │     线程共享（所有线程共用）   │  │
│  │                     │  │                          │  │
│  │  ┌───────────────┐  │  │  ┌────────────────────┐  │  │
│  │  │  程序计数器(PC) │  │  │  │        堆 (Heap)    │  │  │
│  │  └───────────────┘  │  │  │                    │  │  │
│  │  ┌───────────────┐  │  │  │  ┌──────────────┐  │  │  │
│  │  │  虚拟机栈       │  │  │  │  │   新生代      │  │  │  │
│  │  │  ┌──────────┐ │  │  │  │  │ Eden│S0│S1   │  │  │  │
│  │  │  │ 栈帧 A   │ │  │  │  │  │  8  │ 1 │ 1  │  │  │  │
│  │  │  │ 栈帧 B   │ │  │  │  │  └──────────────┘  │  │  │
│  │  │  │ ...      │ │  │  │  │  ┌──────────────┐  │  │  │
│  │  │  └──────────┘ │  │  │  │  │   老年代      │  │  │  │
│  │  └───────────────┘  │  │  │  └──────────────┘  │  │  │
│  │  ┌───────────────┐  │  │  └────────────────────┘  │  │
│  │  │  本地方法栈     │  │  │  ┌────────────────────┐  │  │
│  │  └───────────────┘  │  │  │  方法区 / 元空间     │  │  │
│  └─────────────────────┘  │  └────────────────────┘  │  │
│                           └──────────────────────────┘  │
│  ┌──────────────────────────────────────────────────┐   │
│  │           直接内存（堆外，NIO/New I/O 使用）          │   │
│  └──────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

### 3.1 程序计数器（PC Register）

**存什么**：当前线程正在执行的字节码指令地址（行号）。

**为什么需要**：CPU 同一时刻只能跑一个线程，线程切换后，靠 PC 恢复到上次执行的位置。

**特点**：
- 唯一不会发生 `OutOfMemoryError` 的区域
- 执行 Native 方法时，PC 值为 undefined

### 3.2 虚拟机栈（JVM Stack）

**存什么**：每个方法调用对应一个**栈帧**，栈帧里装着：

| 组成 | 内容 | 举例 |
|------|------|------|
| 局部变量表 | 方法参数 + 局部变量 | `int age = 25`、`Person p`（引用） |
| 操作数栈 | 运算的临时数据 | `1 + 2` → 先压 1，再压 2，弹出相加，压入 3 |
| 动态链接 | 指向方法区中该方法的引用 | 运行时确定方法的实际地址 |
| 返回地址 | 方法结束后回到哪里 | 正常 return 或异常抛出的位置 |

**什么时候出问题**：
- **StackOverflowError**：方法递归太深，栈帧太多撑爆了
- **OutOfMemoryError**：创建太多线程，每个线程都要分配栈内存，总内存不够

**参数**：`-Xss` 控制每个线程的栈大小（默认 1MB），设成 256k~512k 通常够用。

### 3.3 本地方法栈（Native Method Stack）

和虚拟机栈基本一样，区别是它服务的是 Native 方法（通过 JNI（Java Native Interface）调用的 C/C++ 代码）。HotSpot 直接把两个栈合并了。

### 3.4 堆（Heap）— GC 的主战场

**存什么**：几乎所有对象实例和数组。`new` 出来的东西都在这里。

**特点**：线程共享，所有线程都能访问。GC 主要就在堆里干活。

**分代结构**：

```
堆 (Heap)
├── 新生代 (Young Generation)  ← 新对象在这里分配
│   ├── Eden 区 (80%)          ← 新对象的首选分配地
│   ├── Survivor 0 (10%)       ← Minor GC 后存活对象的暂存区
│   └── Survivor 1 (10%)       ← 与 S0 交替使用
│
└── 老年代 (Old Generation)    ← 长期存活的对象
```

**对象在堆里的生命旅程**：

```
new 对象 → Eden 区
    ↓ Eden 满了，触发 Minor GC
存活对象 → Survivor 区（年龄 +1）
    ↓ 每次 Minor GC 存活，年龄 +1
年龄达到阈值（默认 15）→ 晋升到老年代
    ↓ 老年代也满了
触发 Full GC（全堆回收，代价大）
```

**特殊路径**：
- 大对象直接进老年代（Serial/ParNew 下由 `-XX:PretenureSizeThreshold` 控制；G1 下超过 Region 一半的对象进入 Humongous Region）
- Survivor 放不下的对象直接晋升老年代

### 3.5 方法区 / 元空间（Metaspace）

**存什么**：
- 类的元数据（结构、方法、字段定义）
- 常量池（字符串常量、编译期常量）
- 静态变量
- JIT 编译后的代码

**JDK 版本变化**：

| 版本 | 实现方式 | 存储位置 | 问题 |
|------|----------|----------|------|
| JDK ≤ 7 | 永久代（PermGen） | 堆内存的一部分 | 大小固定，容易 OOM |
| JDK ≥ 8 | 元空间（Metaspace） | 本地内存（Native Memory） | 默认无上限，按需增长 |

JDK 8 把方法区搬到堆外，解决了永久代 OOM 的老大难问题。但如果动态加载大量类（如 Spring + 热部署），仍需通过 `-XX:MaxMetaspaceSize` 限制上限。

### 3.6 直接内存（Direct Memory）

不在 JVM 堆里，通过 `ByteBuffer.allocateDirect()` 在堆外分配。

**为什么用它**：避免数据在 JVM 堆和操作系统之间来回拷贝，NIO 场景下性能更好。

**参数**：`-XX:MaxDirectMemorySize`，不设置则默认等于 `-Xmx`。

### 内存区域总结

| 区域 | 线程 | 存什么 | 会报什么错 |
|------|------|--------|-----------|
| 程序计数器 | 私有 | 字节码行号 | 无 |
| 虚拟机栈 | 私有 | 方法调用的栈帧 | StackOverflowError / OOM |
| 本地方法栈 | 私有 | Native 方法栈帧 | StackOverflowError / OOM |
| 堆 | 共享 | 对象实例、数组 | OOM: Java heap space |
| 方法区(元空间) | 共享 | 类元数据、常量、静态变量 | OOM: Metaspace |
| 直接内存 | 共享 | NIO 堆外缓冲区 | OOM: Direct buffer memory |

## 四、垃圾回收（GC）

GC 是 JVM 自动内存管理的核心 — 你只管 `new`，JVM 负责回收。

### 4.1 怎么判断对象是"垃圾"

Java 用的是**可达性分析**（不是引用计数）：

```
GC Roots（起点）
    │
    ├── 线程栈中的引用（局部变量）
    ├── 静态变量引用的对象
    ├── 常量引用的对象
    ├── JNI 引用
    └── synchronized 持有的对象
          │
          ↓ 从 Roots 往下找
    能找到的对象 → 存活
    找不到的对象 → 垃圾，可以回收
```

**为什么不用引用计数**：循环引用问题。A 引用 B，B 引用 A，计数都不为 0，但实际都没人用了 — 回收不了。

### 4.2 分代假说 — GC 设计的基础

> 绝大多数对象都是朝生夕死的，只有少数能长期存活。

基于这个假设，JVM 把堆分成新生代和老年代，用不同策略对待：
- **新生代**：大部分对象很快死去 → 用复制算法（快速回收存活少的）
- **老年代**：大部分对象长期存活 → 用标记-整理（避免移动大量对象）

### 4.3 三种核心 GC 算法

**标记-清除**
```
回收前：[A][ ][B][ ][C][ ][D]     ← 空格是已死对象
         ↓ 标记 → 清除
回收后：[A][ ][B][ ][ ][ ][D]     ← 清除了 C，但碎片更严重了
```
- 优点：简单
- 缺点：**内存碎片** — 零散的空位拼不出连续大块内存

**复制算法**
```
回收前：                          回收后：
┌─────────────┐  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│ [A][B][ ][C] │  │ [ ][ ][ ][ ] │  │ [A][B][C][ ] │  │ [ ][ ][ ][ ] │
└─────────────┘  └─────────────┘  └─────────────┘  └─────────────┘
    From              To               From              To
  (使用中)          (空闲)          (清空，变空闲)     (存活的都在这)
```
- 优点：无碎片，分配快（指针碰撞）
- 缺点：浪费一半内存 → 实际用 Eden:S0:S1 = 8:1:1 缓解（只浪费 10%）

**标记-整理**
```
回收前：[A][ ][B][ ][C][ ][D]
         ↓ 标记 → 压缩整理
回收后：[A][B][C][D][ ][ ][ ]    ← 存活对象挤到一端，空闲空间连续
```
- 优点：无碎片
- 缺点：移动对象要更新所有引用，STW 时间长

### 4.4 Minor GC 和 Full GC

| | Minor GC（新生代回收） | Full GC（全堆回收） |
|---|---|---|
| **触发条件** | Eden 区满 | 老年代空间不足 / 元空间不足 / 手动 `System.gc()` |
| **回收范围** | 只回收新生代 | 新生代 + 老年代 + 元空间 |
| **速度** | 快（新生代小，存活少） | 慢（通常是 Minor GC 的 10 倍以上） |
| **影响** | 短暂 STW | 较长 STW，影响响应时间 |
| **频率** | 高（应用正常运行的主要 GC） | 低（频繁 Full GC = 有问题） |

### 4.5 常用垃圾收集器

| 收集器 | 世代 | 算法 | 线程 | 特点 | 适用场景 |
|--------|------|------|------|------|----------|
| **Serial** | 新生代 | 复制 | 单线程 | 简单高效，无线程切换开销 | 客户端、嵌入式 |
| **ParNew** | 新生代 | 复制 | 多线程 | Serial 的多线程版 | 配合 CMS 使用 |
| **Parallel Scavenge** | 新生代 | 复制 | 多线程 | 自适应调节，吞吐优先 | JDK 8 默认 |
| **Parallel Old** | 老年代 | 标记-整理 | 多线程 | 配合 Parallel Scavenge | 高吞吐场景 |
| **CMS** | 老年代 | 标记-清除 | 并发 | 低停顿，但有碎片 | JDK 8 低延迟（已废弃） |
| **G1** | 全堆 | Region + 分代 | 并发+并行 | 可预测停顿，Region 化 | **JDK 9+ 默认，通用首选** |
| **ZGC** | 全堆 | 着色指针 | 并发 | **亚毫秒级停顿** | 大堆低延迟（JDK 15+ 正式发布，JDK 17+ 生产推荐） |
| **Shenandoah** | 全堆 | 并发压缩 | 并发 | 类似 ZGC | OpenJDK 低延迟替代 |

**G1 的 Region 模型**：
```
传统分代：
┌──────────┬─────────────────────┐
│  新生代   │       老年代         │
└──────────┴─────────────────────┘

G1 的 Region：
┌───┬───┬───┬───┬───┬───┬───┬───┬───┐
│ E │ S │ O │ O │ E │ H │ O │ E │ O │  ← 每个 Region 1~32MB
└───┴───┴───┴───┴───┴───┴───┴───┴───┘
  E = Eden  S = Survivor  O = Old  H = Humongous(大对象)
```

G1 把堆切成很多小 Region，每个 Region 可以动态充当不同角色。优先回收垃圾最多的 Region（Garbage-First 的由来）。

## 五、类加载

### 5.1 加载过程

```
         .class 文件
              ↓
  ┌─────── 加载 ───────┐
  │ 读取字节流           │
  │ 生成 Class 对象      │
  │ 放入方法区           │
  └─────────┬──────────┘
            ↓
  ┌─────── 链接 ──────────────────────────────┐
  │ 验证：字节码格式合法吗？有没有安全问题？        │
  │ 准备：给静态变量分配内存，赋默认零值             │
  │ 解析：符号引用 → 直接引用（名字 → 内存地址）     │
  └─────────┬────────────────────────────────┘
            ↓
  ┌─────── 初始化 ──────┐
  │ 执行 static {} 块    │
  │ 执行静态变量显式赋值   │
  └─────────────────────┘
```

### 5.2 双亲委派模型

```
Bootstrap ClassLoader（启动类加载器）
    ↑ 委派         加载 rt.jar（java.lang.*, java.util.* 等）
    │
Extension ClassLoader（扩展类加载器）
    ↑ 委派         加载 ext/*.jar
    │
Application ClassLoader（应用类加载器）
                   加载 classpath 下的类（你的代码）
```

**规则**：收到加载请求 → 先问爸爸能不能加载 → 爸爸加载不了 → 自己加载。

**为什么这么设计**：
- **安全**：你写一个 `java.lang.String` 也加载不了，因为 Bootstrap 会先加载 JDK 自带的
- **避免重复**：同一个类只会被加载一次

## 六、执行引擎

JVM 执行字节码有两种方式：

```
字节码
  │
  ├─→ 解释器：逐行翻译执行
  │     优点：启动快，不需要预热
  │     缺点：每条指令都要翻译，长期运行慢
  │
  └─→ JIT 编译器：把"热点代码"直接编译成机器码
        优点：编译后执行速度和 C/C++ 接近
        缺点：编译本身要花时间，需要预热
```

**HotSpot 的做法 — 分层编译**：

| 层级 | 编译器 | 优化程度 | 触发条件 |
|------|--------|----------|----------|
| 0 | 解释器 | 无 | 冷启动，所有代码从这里开始 |
| 1 | C1 | 轻度优化 | 方法被调用多次（简单方法到此为止） |
| 2 | C1 | 中度优化 | 收集更多运行时信息 |
| 3 | C1 | 全量 profile | 收集类型信息，为 C2 提供优化依据 |
| 4 | C2 | 激进优化 | 热点方法，内联、逃逸分析、锁消除等 |

**简单方法**调用几次就直接到 C1 编译结束；**复杂热点方法**走完全部分层，最终由 C2 深度优化。

**JIT 的核心优化手段**：
- **方法内联**：把小方法直接嵌入调用处，省掉方法调用开销
- **逃逸分析**：发现对象没有逃出方法 → 分配在栈上而非堆上 → 减少 GC 压力
- **锁消除**：发现某个锁不可能被其他线程竞争 → 直接去掉加锁操作
- **标量替换**：对象不需要整体存在 → 拆解成基本类型直接用

## 七、Java 内存模型（JMM）

前面讲的是"JVM 怎么管理内存"，这一节讲的是"多个线程怎么共享内存" — 这是理解 `volatile`、`synchronized`、甚至 GC 安全点的基础。

### 7.1 主内存 vs 工作内存

```
┌─────────────────────────────────────────┐
│            主内存（共享）                  │
│     所有变量的"正式值"都存在这里             │
│     （对应堆中的对象字段）                  │
└──────┬──────────────┬───────────────────┘
       │              │
  读/写 │         读/写 │
       ↓              ↓
┌──────────┐    ┌──────────┐
│ 线程 1    │    │ 线程 2    │
│ 工作内存   │    │ 工作内存   │  ← 每个线程自己的"副本"
│ (CPU缓存/  │    │ (CPU缓存/  │    （对应栈帧中的局部变量）
│  寄存器)   │    │  寄存器)   │
└──────────┘    └──────────┘
```

**关键问题**：线程 1 修改了变量 `x`，是先改自己的工作内存，不一定立刻写回主内存。线程 2 可能读到旧值。这就是**可见性问题**。

### 7.2 三大问题

| 问题 | 含义 | 例子 |
|------|------|------|
| **可见性** | 一个线程的修改，其他线程能否立刻看到 | 线程 A 设 `stop = true`，线程 B 可能永远看不到 |
| **有序性** | 代码是否按你写的顺序执行 | 编译器/CPU 可能把 `a = 1; b = 2` 重排成 `b = 2; a = 1` |
| **原子性** | 一个操作是否不可分割 | `i++` 实际是"读→加→写"三步，不是原子的 |

### 7.3 volatile — 轻量级的可见性保证

```java
volatile boolean stop = false;

// 线程 A
stop = true;    // 立刻写回主内存，其他线程可见

// 线程 B
while (!stop) { // 每次都从主内存读，不会读到旧值
    // ...
}
```

**volatile 做了什么**：
- **写操作**：写完立刻 flush 到主内存
- **读操作**：每次从主内存重新读，不用工作内存的缓存
- **禁止重排序**：volatile 前后的指令不会被重排

**volatile 不能做的**：
- 不保证原子性。`volatile int count; count++;` 仍然不是线程安全的

### 7.4 happens-before — JVM 的"顺序承诺"

JMM 定义了一组 happens-before 规则，如果 A happens-before B，那么 A 的结果对 B **一定可见**：

| 规则 | 含义 |
|------|------|
| **程序顺序** | 同一线程内，前面的操作 happens-before 后面的 |
| **监视器锁** | `unlock` happens-before 同一个锁的 `lock` |
| **volatile** | `volatile` 写 happens-before 后续的 `volatile` 读 |
| **线程启动** | `Thread.start()` happens-before 线程内的操作 |
| **线程终止** | 线程内的操作 happens-before `Thread.join()` 返回 |
| **传递性** | A hb B，B hb C → A hb C |

**为什么这很重要**：只要你的代码满足 happens-before 规则，JVM 就保证可见性。不满足的话，你就需要加 `volatile` 或 `synchronized`。

### 7.5 安全点（Safepoint）— GC 和 JMM 的交汇

```
应用线程执行
  │
  │  ... 执行字节码 ...
  │
  ▼ 到达安全点（方法调用、循环回跳、异常跳转等位置）
  ┌─────────────────────────┐
  │ 此时线程状态是已知的：     │
  │ - 所有引用都在确定的位置   │
  │ - 可以安全地暂停线程       │
  └───────────┬─────────────┘
              │
              ▼ GC 请求所有线程到达安全点
         STW（Stop-The-World）
              │
              ▼ GC 执行回收
              │
              ▼ 恢复应用线程
```

**为什么需要安全点**：GC 要遍历所有引用（GC Roots），如果线程正在执行中，引用的位置随时在变。只有在安全点暂停，GC 才能安全地扫描。

**这也是为什么 STW 不可避免**（即使是 ZGC，也有极短的 STW 来扫描线程栈）：JVM 必须在一个"已知状态"下暂停世界。

**JVM 的同步屏障**：
- **LoadLoad**：确保 Load1 在 Load2 之前完成
- **StoreStore**：确保 Store1 在 Store2 之前对其他处理器可见
- **LoadStore / StoreLoad**：混合屏障

`volatile` 和 `synchronized` 的底层实现就是插入这些内存屏障。

## 总结

```
Java 源码
  ↓ javac
字节码 (.class)
  ↓ 类加载器（双亲委派）
方法区（类元数据）
  ↓ 执行引擎
  ├── 解释器（冷代码）
  └── JIT 编译器（热代码 → 机器码）
      ↓ 运行时
  ┌───────────────────────────────┐
  │ 堆：对象分配 → GC 自动回收       │
  │ 栈：方法调用 → 栈帧管理          │
  │ PC：线程切换 → 恢复执行位置       │
  │ JMM：volatile/锁 → 线程间可见性  │
  └───────────────────────────────┘
```

**一句话总结**：JVM 就是一个虚拟计算机，它加载你的字节码，管理内存（堆 + 栈 + 方法区），自动回收垃圾，通过 JIT 把热点代码编译成机器码来获得接近原生的性能，并通过 JMM 定义了一套内存可见性规则来保证多线程程序的正确性。
