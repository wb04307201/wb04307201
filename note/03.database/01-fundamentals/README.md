## 二、关系模型基础

### 1. 元组与码

| 术语 | 说明 |
|------|------|
| **元组** | 表中的一行记录 |
| **属性** | 表中的一列 |
| **码（超码）** | 能唯一标识元组的属性或属性组 |
| **候选码** | 最小的超码（去掉任一属性就不能唯一标识） |
| **主码（主键）** | 从候选码中选出的一个，用于唯一标识元组 |
| **外码（外键）** | 引用另一张表主键的属性 |
| **主属性** | 出现在任一候选码中的属性 |
| **非主属性** | 不属于任何候选码的属性 |

### 2. 主键与外键

| 特性 | 主键 | 外键 |
|------|------|------|
| 唯一性 | 值必须唯一 | 可以重复 |
| 空值 | 不允许 NULL | 允许 NULL |
| 数量 | 每表只能有一个 | 可以有多个 |
| 作用 | 唯一标识一行 | 建立表间关联 |

> **不推荐在生产环境使用外键约束**：增加开发复杂度、对分库分表不友好、影响写入性能。建议在应用层维护数据一致性。

## 三、ER 图（实体联系图）

ER 图（Entity Relationship Diagram）用于描述现实世界中实体及其关系。

**三大要素**：

| 要素 | 图形 | 说明 |
|------|------|------|
| 实体 | 矩形 | 业务对象（如学生、课程） |
| 属性 | 椭圆 | 实体的特征（如姓名、学号） |
| 联系 | 菱形 | 实体间的关系 |

**联系类型**：

| 类型 | 说明 | 示例 |
|------|------|------|
| 1:1 | 一对一 | 一个人对应一个身份证 |
| 1:N | 一对多 | 一个班级有多个学生 |
| M:N | 多对多 | 一个学生选多门课，一门课被多个学生选 |

## 四、数据库范式

范式（Normal Form）是关系数据库设计的规范标准，目标是减少数据冗余、消除异常。

### 1. 三大范式

| 范式 | 要求 | 解决的问题 |
|------|------|-----------|
| **1NF** | 每个属性不可再分（原子性） | 所有关系型数据库的基本要求 |
| **2NF** | 在 1NF 基础上，消除非主属性对码的**部分函数依赖** | 确保每个非主属性完全依赖于整个主键 |
| **3NF** | 在 2NF 基础上，消除非主属性对码的**传递函数依赖** | 消除非主属性之间的间接依赖 |

### 2. 函数依赖

| 类型 | 说明 | 示例 |
|------|------|------|
| **函数依赖** X→Y | X 确定时 Y 唯一确定 | 学号 → 姓名 |
| **部分函数依赖** | Y 依赖于 X 的真子集 | (学号, 身份证号) → 姓名，实际学号 → 姓名 |
| **完全函数依赖** | Y 依赖于完整的 X | (学号, 课程号) → 成绩 |
| **传递函数依赖** | X→Y, Y→Z, 则 X→Z | 学号 → 系名, 系名 → 系主任 |

### 3. 规范化 vs 反规范化

| 方面 | 规范化 | 反规范化 |
|------|--------|---------|
| 做法 | 拆分为多个关联表 | 合并为一个宽表 |
| 数据冗余 | 低 | 高 |
| 数据一致性 | 好 | 可能出现不一致 |
| 查询性能 | 需要 JOIN，可能较慢 | 减少 JOIN，查询快 |
| 适用场景 | OLTP（增删改频繁） | OLAP / 读密集型报表 |

> **实践建议**：先按 3NF 设计，出现性能瓶颈时再针对性反规范化。

## 六、常用操作对比

### drop、truncate 与 delete

| 特性 | DROP | TRUNCATE | DELETE |
|------|------|----------|--------|
| 作用 | 删除整张表（结构+数据） | 清空数据，保留表结构 | 删除行数据 |
| 语言类型 | DDL | DDL | DML |
| 能否回滚 | 不能 | 不能 | 能（事务提交前） |
| 自增 ID | 表不存在了 | 重置为 1 | 不重置 |
| WHERE 条件 | 不支持 | 不支持 | 支持 |
| 触发器 | 不触发 | 不触发 | 触发 |
| 速度 | 最快 | 快 | 最慢 |

> **速度**：`DROP > TRUNCATE > DELETE`

存储过程是预编译的 SQL 语句集合，可包含逻辑控制（IF/LOOP 等）。

## 九、数据类型(MySQL 为例)

### 1. 整数类型

| 类型 | 字节 | 范围(有符号) | 用途 |
|------|------|------------|------|
| `TINYINT` | 1 | -128 ~ 127 | 状态、布尔 |
| `SMALLINT` | 2 | -32768 ~ 32767 | 小计数器 |
| `INT` | 4 | -2^31 ~ 2^31-1 | **最常用** |
| `BIGINT` | 8 | -2^63 ~ 2^63-1 | 大数据量、自增主键 |

> 阿里规约:**主键建议使用 `BIGINT UNSIGNED` 自增**,避免达到上限。

### 2. 字符串类型

| 类型 | 说明 | 适用场景 |
|------|------|---------|
| `CHAR(n)` | 定长字符串,最多 255 字节 | 长度固定(身份证、MD5) |
| `VARCHAR(n)` | 变长字符串,最多 65535 字节 | **最常用**,长度可变 |
| `TEXT` | 长文本,最多 65535 字节 | 文章、评论 |
| `MEDIUMTEXT` | 中等文本,最多 16MB | 长文 |
| `LONGTEXT` | 超大文本,最多 4GB | 极长内容 |

> `VARCHAR(n)` 中 `n` 表示**字符数**(非字节),实际存储空间由字符集决定。

### 3. 时间类型

| 类型 | 格式 | 范围 | 特点 |
|------|------|------|------|
| `DATE` | YYYY-MM-DD | 1000-01-01 ~ 9999-12-31 | 仅日期 |
| `TIME` | HH:MM:SS | -838:59:59 ~ 838:59:59 | 仅时间 |
| `DATETIME` | YYYY-MM-DD HH:MM:SS | 1000-01-01 ~ 9999-12-31 | **不受时区影响** |
| `TIMESTAMP` | YYYY-MM-DD HH:MM:SS | 1970-01-01 ~ 2038-01-19 | **UTC 存储、自动时区转换** |

> 阿里规约:**时间字段统一使用 `DATETIME`,避免 2038 年问题**。`TIMESTAMP` 范围只到 2038 年。

### 4. 数值与精度类型

| 类型 | 用途 |
|------|------|
| `DECIMAL(M,D)` | **精确小数**,金额、汇率(必须用) |
| `FLOAT` / `DOUBLE` | 近似浮点,科学计算(有精度损失) |

> 阿里规约:**金额字段必须用 `DECIMAL`,禁止使用 `FLOAT` 和 `DOUBLE`**。

## 十、字符集与排序规则

### 1. 常见字符集

| 字符集 | 字节 | 支持字符 | 说明 |
|--------|------|---------|------|
| `latin1` | 1 | 西欧字符 | 单字节,空间省 |
| `utf8` | 1~3 | 基本平面 BMP | **MySQL 的 utf8 不完整,不支持 emoji** |
| `utf8mb4` | 1~4 | 完整 Unicode | **推荐,支持 emoji 和所有中文** |

> **MySQL 的 `utf8` 实际上是 `utf8mb3`,只支持 1~3 字节,不能存 emoji。生产环境必须用 `utf8mb4`。**

### 2. 排序规则(Collation)

| 排序规则 | 特点 |
|---------|------|
| `utf8mb4_general_ci` | 速度快,排序不精确 |
| `utf8mb4_unicode_ci` | 基于 Unicode 标准,排序准确 |
| `utf8mb4_bin` | 区分大小写、二进制比较 |

> `ci` = Case Insensitive(大小写不敏感),`bin` = Binary。

### 3. 推荐配置(MySQL 8.0)

```ini
[mysqld]
character-set-server = utf8mb4
collation-server = utf8mb4_unicode_ci

[client]
default-character-set = utf8mb4
```

## 十一、数据库三大完整性约束

| 完整性 | 含义 | 实现方式 |
|--------|------|---------|
| **实体完整性** | 每行唯一可标识 | 主键约束、唯一索引 |
| **参照完整性** | 外键引用关系一致 | 外键约束(生产慎用) |
| **用户定义完整性** | 业务规则 | CHECK 约束、触发器、NOT NULL |

> **CHECK 约束**(MySQL 8.0.16+ 原生支持):

```sql
CREATE TABLE employees (
    id INT PRIMARY KEY,
    salary DECIMAL(10,2),
    age INT,
    CONSTRAINT chk_salary CHECK (salary > 0),
    CONSTRAINT chk_age CHECK (age BETWEEN 18 AND 65)
);
```

## 十二、主键策略对比

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **自增 INT/BIGINT** | 简单、聚簇索引顺序写、性能高 | 单调递增可猜测、分布式 ID 难 | 单库应用 |
| **UUID(128位)** | 唯一性强、可分布式生成 | 存储大(36 字符)、无序、索引碎片 | 分布式主键 |
| **UUID v7(2023+) | 唯一 + 时间有序 | 新标准、客户端支持有限 | 分布式主键(新项目) |
| **雪花 ID (Snowflake)** | 64 位、时间有序、趋势递增 | 依赖时钟、需独立发号器 | 大型分布式系统 |
| **Redis INCR** | 简单、高性能 | 依赖 Redis 持久化、需重置策略 | 中小规模分布式 |

> 阿里规约:**禁止使用 UUID 作为主键**(索引碎片、聚簇索引无序写导致页分裂)。分布式场景推荐**雪花算法**或**号段模式**。

## 十三、反规范化实战案例

### 场景:电商订单查询性能优化

**规范化设计(3NF)**:

```sql
-- 用户表
users(id, name, email)
-- 订单表
orders(id, user_id, total_amount, created_at)
-- 订单明细
order_items(id, order_id, product_id, quantity, price)
```

**问题**:查询"订单详情"需要 3 张表 JOIN,大表时性能差。

**反规范化方案**:**在订单表中冗余 `user_name`**。

```sql
ALTER TABLE orders ADD COLUMN user_name VARCHAR(50);

-- 写入时同步
INSERT INTO orders (user_id, user_name, total_amount)
SELECT id, name, 100 FROM users WHERE id = 100;
```

**收益**:省去 1 次 JOIN,查询从 ~20ms 降至 ~3ms。
**代价**:`users.name` 变更时需同步更新 `orders.user_name`,通过**事务 + 异步消息**保证最终一致。

> **核心原则**:**先按 3NF 设计,出现性能瓶颈时再针对性反规范化,不做全盘反规范化**。

## 十四、高阶范式(BCNF / 4NF / 5NF)

| 范式 | 要求 | 解决的问题 |
|------|------|----------|
| **BCNF** | 3NF 基础上,每个决定因素都是候选码 | 消除主属性对候选码的部分/传递依赖 |
| **4NF** | BCNF 基础上,消除非平凡多值依赖 | 多值依赖(一个属性决定多个独立多值属性) |
| **5NF** | 4NF 基础上,消除连接依赖 | 由连接生成的关系可分解 |

> **实践**:实际工程中 **3NF 已足够**,BCNF 仅在严格学术或金融领域使用,4NF/5NF 极少涉及。
