## 二、关系模型基础

### 1. 元组与码

| 术语 | 说明 |
|------|------|
| **元组** | 表中的一行记录 |
| **属性** | 表中的一列 |
| **码（超码）** | 能唯一标识元组的属性或属性组 |
| **候选码** | 最小的超码（去掉任一属性就不能唯一标识） |
| **主码（主键）** | 从候选码中选出的一个，用于唯一标识元组 |
| **外码（外键）** | 引用另一张表主键的属性 |
| **主属性** | 出现在任一候选码中的属性 |
| **非主属性** | 不属于任何候选码的属性 |

### 2. 主键与外键

| 特性 | 主键 | 外键 |
|------|------|------|
| 唯一性 | 值必须唯一 | 可以重复 |
| 空值 | 不允许 NULL | 允许 NULL |
| 数量 | 每表只能有一个 | 可以有多个 |
| 作用 | 唯一标识一行 | 建立表间关联 |

> **不推荐在生产环境使用外键约束**：增加开发复杂度、对分库分表不友好、影响写入性能。建议在应用层维护数据一致性。

## 三、ER 图（实体联系图）

ER 图（Entity Relationship Diagram）用于描述现实世界中实体及其关系。

**三大要素**：

| 要素 | 图形 | 说明 |
|------|------|------|
| 实体 | 矩形 | 业务对象（如学生、课程） |
| 属性 | 椭圆 | 实体的特征（如姓名、学号） |
| 联系 | 菱形 | 实体间的关系 |

**联系类型**：

| 类型 | 说明 | 示例 |
|------|------|------|
| 1:1 | 一对一 | 一个人对应一个身份证 |
| 1:N | 一对多 | 一个班级有多个学生 |
| M:N | 多对多 | 一个学生选多门课，一门课被多个学生选 |

## 四、数据库范式

范式（Normal Form）是关系数据库设计的规范标准，目标是减少数据冗余、消除异常。

### 1. 三大范式

| 范式 | 要求 | 解决的问题 |
|------|------|-----------|
| **1NF** | 每个属性不可再分（原子性） | 所有关系型数据库的基本要求 |
| **2NF** | 在 1NF 基础上，消除非主属性对码的**部分函数依赖** | 确保每个非主属性完全依赖于整个主键 |
| **3NF** | 在 2NF 基础上，消除非主属性对码的**传递函数依赖** | 消除非主属性之间的间接依赖 |

### 2. 函数依赖

| 类型 | 说明 | 示例 |
|------|------|------|
| **函数依赖** X→Y | X 确定时 Y 唯一确定 | 学号 → 姓名 |
| **部分函数依赖** | Y 依赖于 X 的真子集 | (学号, 身份证号) → 姓名，实际学号 → 姓名 |
| **完全函数依赖** | Y 依赖于完整的 X | (学号, 课程号) → 成绩 |
| **传递函数依赖** | X→Y, Y→Z, 则 X→Z | 学号 → 系名, 系名 → 系主任 |

### 3. 规范化 vs 反规范化

| 方面 | 规范化 | 反规范化 |
|------|--------|---------|
| 做法 | 拆分为多个关联表 | 合并为一个宽表 |
| 数据冗余 | 低 | 高 |
| 数据一致性 | 好 | 可能出现不一致 |
| 查询性能 | 需要 JOIN，可能较慢 | 减少 JOIN，查询快 |
| 适用场景 | OLTP（增删改频繁） | OLAP / 读密集型报表 |

> **实践建议**：先按 3NF 设计，出现性能瓶颈时再针对性反规范化。

## 五、数据库设计步骤

| 阶段 | 任务 | 产出 |
|------|------|------|
| 1. 需求分析 | 分析数据、功能和性能需求 | 需求规格说明书 |
| 2. 概念结构设计 | 画 ER 图，定义实体和联系 | E-R 图 |
| 3. 逻辑结构设计 | ER 图转换为关系表 | 表结构定义 |
| 4. 物理结构设计 | 选择存储引擎、索引策略、分区方案 | 物理存储方案 |
| 5. 实施 | 建表、编程、测试、试运行 | 可运行的数据库 |
| 6. 运行维护 | 监控、优化、备份、扩容 | 运维方案 |

## 六、常用操作对比

### drop、truncate 与 delete

| 特性 | DROP | TRUNCATE | DELETE |
|------|------|----------|--------|
| 作用 | 删除整张表（结构+数据） | 清空数据，保留表结构 | 删除行数据 |
| 语言类型 | DDL | DDL | DML |
| 能否回滚 | 不能 | 不能 | 能（事务提交前） |
| 自增 ID | 表不存在了 | 重置为 1 | 不重置 |
| WHERE 条件 | 不支持 | 不支持 | 支持 |
| 触发器 | 不触发 | 不触发 | 触发 |
| 速度 | 最快 | 快 | 最慢 |

> **速度**：`DROP > TRUNCATE > DELETE`

### 存储过程

存储过程是预编译的 SQL 语句集合，可包含逻辑控制（IF/LOOP 等）。

> **阿里巴巴 Java 开发手册明确禁止使用存储过程**：难以调试、缺乏移植性、消耗数据库资源、不利于版本管理。

## 七、数据库索引概览

索引是提升查询性能的核心数据结构。详细内容包括 B+ 树原理、聚簇/非聚簇索引、覆盖索引等，参见：[索引](../04-index/README.md)。

| 索引类型 | 数据结构 | 说明 |
|---------|---------|------|
| B+ 树索引 | B+ Tree | MySQL InnoDB 默认索引结构 |
| Hash 索引 | 哈希表 | 等值查询 O(1)，不支持范围查询 |
| 全文索引 | 倒排索引 | 用于文本搜索（MySQL 5.6+ 支持） |

## 八、数据库管理系统分类

| 类型 | 代表产品 | 特点 |
|------|---------|------|
| 关系型 (RDBMS) | MySQL、PostgreSQL、Oracle | 结构化存储、ACID 事务、SQL 查询 |
| 文档型 | MongoDB、CouchDB | JSON 文档、灵活 Schema |
| 键值型 | Redis、Memcached | 高性能缓存、简单数据结构 |
| 列存储 | Cassandra、HBase | 大数据量、高写入吞吐 |
| 图数据库 | Neo4j、Amazon Neptune | 关系网络分析 |
| 时序数据库 | InfluxDB、TimescaleDB | 时间序列数据、物联网监控 |
