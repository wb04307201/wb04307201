> 数据库通常是系统性能瓶颈所在。本章从 SQL 优化、读写分离、分库分表、冷热分离四个维度,系统性地讲解数据库性能优化的方法论与实践。

## 为什么需要数据库优化？

在高并发系统中,数据库往往是 **最先达到瓶颈的组件**。一次未经优化的 SQL 可能导致:

- 慢查询占用数据库连接,影响其他请求
- 锁竞争加剧,吞吐量急剧下降
- 主从延迟增加,读写分离出现数据不一致
- 单表数据膨胀,备份恢复时间过长

> 经验法则:优化顺序应该是 **SQL 优化 → 索引优化 → 读写分离 → 分库分表**,而不是一上来就分库分表。

## 1. SQL 优化

[→ 常见 SQL 优化手段总结](sql/README.md)

**核心要点**:
- 避免 `SELECT *`,使用覆盖索引
- 深度分页优化(子查询/延迟关联/覆盖索引)
- 控制 JOIN 表数量,优先单表查询 + 内存关联
- 批量操作替代单条循环
- 使用 `EXPLAIN` 分析执行计划

**适用场景**: 所有系统都应作为基础优化项。

## 2. 读写分离

[→ 数据库读写分离](read-write-splitting/README.md)

**核心要点**:
- 一主多从,主库写、从库读
- 通过 Binlog 进行主从复制
- 代理方式 (MySQL Router/MyCat) 或组件方式 (Sharding-JDBC)
- 注意主从延迟导致的读不一致

**适用场景**: 读多写少 (如电商商品浏览、新闻阅读)。

## 3. 分库分表

[→ 数据库分库分表](db-sharding/README.md) | [→ ShardingSphere 实战](db-sharding/sharding-sphere/README.md)

**核心要点**:
- 垂直分库:按业务拆分(用户库、订单库、商品库)
- 水平分库:按数据特征拆分(同一张表按 user_id 取模)
- 垂直分表:按字段访问频率拆分
- 水平分表:按数据行拆分
- 选好分片键,使用一致性哈希、范围分片等策略

**适用场景**: 单表数据量千万级以上,QPS 持续高位。

**注意事项**: 分库分表会引入分布式 ID、跨库 JOIN、分布式事务等复杂度,**慎用**。

## 4. 冷热数据分离

[→ 数据冷热分离](cold-hot-data-separation/README.md)

**核心要点**:
- 按时间维度区分(订单 1 年内为热,1 年前为冷)
- 按访问频率区分(浏览量高为热,低为冷)
- 业务层判断 / 任务调度扫描 / 监听 binlog
- 热数据放 SSD,冷数据放 HDD 或低成本存储(HBase/TiDB)

**适用场景**: 数据有明显时间/频率分布的场景(订单、日志、内容)。

## 5. 综合优化策略

### 5.1 优化顺序优先级

| 优先级 | 优化手段 | 投入产出比 | 复杂度 |
|--------|---------|-----------|--------|
| ⭐⭐⭐⭐⭐ | 索引设计 / SQL 优化 | 极高 | 低 |
| ⭐⭐⭐⭐ | 慢查询治理 | 极高 | 低 |
| ⭐⭐⭐ | 读写分离 | 高 | 中 |
| ⭐⭐⭐ | 缓存前置 (Redis) | 高 | 中 |
| ⭐⭐ | 冷热数据分离 | 中 | 中 |
| ⭐ | 分库分表 | 视情况 | 高 |

### 5.2 容量评估

```
单库承载建议:
- MySQL 单库建议:数据量 < 2TB,QPS < 5万
- 单表建议:行数 < 5000万,字段数 < 50
- 超出后优先考虑:归档/分区/分表,再考虑分库
```

### 5.3 监控指标

- **QPS / TPS**: 每秒查询/事务数
- **慢查询数量**: `slow_query_log` 中的条目
- **主从延迟**: `Seconds_Behind_Master`
- **连接数使用率**: `Threads_connected / max_connections`
- **缓冲池命中率**: `Innodb_buffer_pool_read_hits`
