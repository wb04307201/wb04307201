# SSO 概念与原理 · 一次登录多系统访问

> **一句话**：SSO 不是单一技术——是"用户在**多套应用系统**中**只登录一次**，就能访问所有互信应用"的设计目标。核心是**信任传递**——首次登录后发"信任票据"给子系统。

## 1. 什么是单点登录（SSO）？

### 1.1 形式定义

**Single Sign-On（SSO）**：用户**一次登录**，就能在**多个互信系统间**无缝访问，**无需重复登录**。

### 1.2 反面例子（没有 SSO）

```
公司有 5 个内部系统：A / B / C / D / E

员工张三今天访问：
1. 打开系统 A → 登录（账号密码）
2. 打开系统 B → 又要登录（账号密码）
3. 打开系统 C → 又要登录（账号密码）
4. 打开系统 D → 又要登录（账号密码）
5. 打开系统 E → 又要登录（账号密码）

每天上班 5 次输入密码（密码管理器也救不了）→ 用户体验崩盘
```

### 1.3 正面例子（有 SSO）

```
公司有 5 个内部系统：A / B / C / D / E（有 SSO IdP）

员工张三今天访问：
1. 打开系统 A → SSO 登录（账号密码）
2. 打开系统 B → 自动登录（SSO 信任票据）
3. 打开系统 C → 自动登录（SSO 信任票据）
4. 打开系统 D → 自动登录（SSO 信任票据）
5. 打开系统 E → 自动登录（SSO 信任票据）

每天上班 1 次输入密码 → 体验 + 安全双赢
```

## 2. SSO 解决的 3 大问题

```
┌──────────────────────────────────────┐
│ 用户视角                              │
│ - 一次登录多系统访问（体验）          │
│ - 减少密码疲劳（安全）                │
│ - 第三方登录（如微信 / Google）        │
└──────────────────────────────────────┘
                    +
┌──────────────────────────────────────┐
│ 管理员视角                            │
│ - 集中管理账号（效率）                │
│ - 集中失效（员工离职立即失效所有）    │
│ - 审计日志集中（合规）                │
└──────────────────────────────────────┘
                    +
┌──────────────────────────────────────┐
│ 安全视角                              │
│ - 减少密码泄露面（每个系统一个密码 →  多个系统一个）│
│ - 集中安全策略（MFA / 风控）           │
│ - 减少弱密码（用户复用密码风险）       │
└──────────────────────────────────────┘
```

## 3. SSO 核心原理

### 3.1 3 大角色

```
┌──────────────────────────────────────────────────┐
│  角色 1: Principal（主体）                          │
│  - 用户 / 服务 / 设备                               │
│  - 需要被识别的"谁"                                │
└──────────────────────────────────────────────────┘
                    ↓ 身份认证
┌──────────────────────────────────────────────────┐
│  角色 2: Identity Provider (IdP / 身份提供者)       │
│  - 验证用户凭证（账号密码 / MFA / OAuth）           │
│  - 颁发信任票据（Ticket / Token）                  │
│  - 例子：Keycloak / Okta / Authing / 自建         │
└──────────────────────────────────────────────────┘
                    ↓ 颁发票据
┌──────────────────────────────────────────────────┐
│  角色 3: Service Provider (SP / 服务提供者)        │
│  - 业务系统（OA / CRM / ERP / GitLab 等）           │
│  - 信任 IdP 颁发的票据                              │
│  - 给用户授权访问                                   │
└──────────────────────────────────────────────────┘
```

### 3.2 通用流程（4 步）

```
Step 1：用户访问 SP-A 资源
        → SP-A 发现用户未登录
        → 重定向到 IdP（含回调地址）

Step 2：用户在 IdP 登录
        → IdP 验证账号密码（+ MFA）
        → IdP 生成"信任票据"（Ticket / Token）

Step 3：IdP 重定向回 SP-A（带票据）
        → SP-A 验证票据（签名 / 时效）
        → SP-A 创建本地会话
        → 用户访问 SP-A 资源

Step 4（其他系统）：用户访问 SP-B
        → SP-B 发现用户未登录
        → 重定向到 IdP
        → IdP 已有会话（Cookie 共享）
        → 直接颁发信任票据（无需输入密码）
        → SP-B 创建本地会话
        → 用户访问 SP-B 资源
```

**关键洞察**：Step 4 是 SSO 的"魔法"——IdP 已有会话，**无需重新输入密码**。

### 3.3 SSO 核心协议矩阵

| 协议 | 时代 | 票据格式 | 适用 |
|------|------|---------|------|
| **CAS 1.0/2.0/3.0** | 2002-2010 | Service Ticket | 老牌企业内部 |
| **OAuth2 2.0** | 2012 | Access Token | 现代互联网（授权）|
| **OpenID Connect 1.0** | 2014 | ID Token + Access Token | 现代互联网（身份）|
| **SAML 2.0** | 2005 | XML 断言 | 企业 / 跨国 / 政府 |
| **JWT (RFC 7519)** | 2015 | JSON Token | 内部 API / 微服务 |

## 4. SSO 6 大实现方案对比

| # | 方案 | 票据类型 | 跨域 | 复杂度 | 主流 |
|---|------|---------|------|--------|------|
| 1 | **Cookie 共享（同根域）** | Cookie | ❌ 仅同根域 | 低 | 小公司 |
| 2 | **CAS** | Service Ticket | ✅ | 中 | 老企业内部 |
| 3 | **OAuth2** | Access Token | ✅ | 中 | 现代互联网 |
| 4 | **OpenID Connect** | ID Token | ✅ | 中 | 现代互联网 SSO（推荐 ✅）|
| 5 | **SAML 2.0** | XML Assertion | ✅ | 高 | 企业合规 |
| 6 | **JWT 自建** | JWT | ✅ | 低 | 微服务内部 |

详细对比见 [02-six-schemes-comparison.md](02-six-schemes-comparison.md)。

## 5. SSO 6 大反模式

### ⚠️ 反模式 1：SSO = OAuth2

- **错**："我们要 SSO，用 OAuth2"
- **真**：SSO 是设计目标，OAuth2 是实现方案之一
- **建议**：用 OAuth2 + OIDC（IdP + 颁发 ID Token）才能完整 SSO

### ⚠️ 反模式 2：JWT 自签自验 = 安全

- **错**："我用 JWT 所有系统都能验证，无需 IdP"
- **真**：JWT 失去 IdP 控制，**吊销困难**（黑名单复杂）
- **建议**：JWT 配合 Redis 黑名单（详见 04 章节）

### ⚠️ 反模式 3：同域 Cookie 共享跨主域

- **错**："主域 a.com + 子域 app.a.com 共享 SSO"
- **真**：主域 a.com 设置的 Cookie **不能**被 b.com 访问（浏览器同源策略）
- **建议**：跨主域用 OAuth2 / CAS

### ⚠️ 反模式 4：Token 永不过期

- **错**："Token 设 30 天有效期，用户登录一次管一个月"
- **真**：Token 一旦泄露，30 天都能用
- **建议**：Access Token 短（1h）+ Refresh Token + 后端 Rotate（OAuth2 模式）

### ⚠️ 反模式 5：SSO IdP 单点故障

- **错**："IdP 挂了就让用户登不上"
- **真**：IdP 挂了 = 所有 SP 都登不上
- **建议**：IdP 高可用集群 + 多 AZ 部署

### ⚠️ 反模式 6：忽略登出

- **错**：SSO 只有登录，没有统一登出
- **真**：单点登出才能"员工离职立即禁用所有"
- **建议**：SAML SingleLogout / OIDC backchannel logout

---

## 6. 一句话总结

> **SSO = 一次登录多系统访问 = 设计目标 = 信任传递。6 大方案各有适用：互联网首选 OAuth2+OIDC，老企业内部用 CAS，跨国用 SAML，微服务自建 JWT+Redis。**
