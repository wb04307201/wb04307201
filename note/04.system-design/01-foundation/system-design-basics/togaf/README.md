# 企业架构（TOGAF）

> 企业架构（Enterprise Architecture）是从全局视角规划和治理组织 IT 结构的系统性方法。TOGAF 是目前最主流的企业架构框架，它帮助组织将业务战略转化为可执行的 IT 蓝图。

## 一、核心思想：从业务战略到技术实现的全链路对齐

当系统规模从"一个应用"扩展到"一个企业的数百个系统"时，单纯的技术设计已不够。企业架构回答的不是"这个类怎么设计"，而是：

```
业务战略 → 需要什么业务能力？ → 需要哪些数据支撑？
    ↓           ↓                    ↓
  价值链      业务能力地图         数据资产规划
    ↓           ↓                    ↓
  需要哪些应用？ → 用什么技术实现？ → 团队怎么组织？
    ↓              ↓                    ↓
  应用架构         技术架构            治理体系
```

TOGAF 的核心贡献是提供了一套**标准化的流程**（架构开发方法 ADM），确保上述每个环节都被系统化地思考和处理。

## 二、TOGAF 架构开发方法（ADM）

ADM 是一个循环迭代的过程模型，包含 9 个核心阶段：

```
                    ┌──────────┐
                    │  愿景管理  │
                    │ (Prelim) │
                    └────┬─────┘
                         │
              ┌──────────▼──────────┐
              │                     │
         ┌────▼────┐          ┌────▼────┐
         │ A.架构   │◄────────►│ 需求管理 │
         │  愿景   │          │         │
         └────┬────┘          └─────────┘
              │
    ┌─────────▼─────────┐
    │  B.业务架构        │ ← 业务能力、价值流、组织映射
    │  C.信息系统架构    │ ← 数据架构 + 应用架构
    │  D.技术架构        │ ← 基础设施、中间件、技术栈
    └─────────┬─────────┘
              │
    ┌─────────▼─────────┐
    │ E.机会与解决方案   │ ← 迁移规划、项目组合
    │ F.迁移规划         │
    │ G.实施治理         │ ← 项目执行监督
    └─────────┬─────────┘
              │
    ┌─────────▼─────────┐
    │ H.架构变更管理     │ ← 持续监控、反馈循环
    └───────────────────┘
```

### 四个架构层次（BCAT）

| 层次 | 关注点 | 核心产出物 | 与 DDD/OOD 的关系 |
|------|-------|-----------|------------------|
| **B 业务架构** | 战略目标、业务能力、价值流、组织结构 | 业务能力地图、价值流图 | 为 DDD 的领域划分提供业务视角 |
| **C 信息系统架构** | 数据实体、应用系统、接口关系 | 数据模型、应用目录、接口规范 | DDD 的限界上下文在此层落地 |
| **A 应用架构** | 应用间交互、服务边界、集成模式 | 服务依赖图、集成架构 | 微服务拆分、事件驱动设计 |
| **T 技术架构** | 服务器、网络、数据库、中间件选型 | 技术栈清单、部署拓扑 | OOD 设计模式在此层编码实现 |

## 三、业务能力建模：TOGAF 的"领域建模"

业务能力是 TOGAF 中连接业务战略与 IT 实现的核心概念，定义为一个组织**做什么**（而非怎么做）。

### 业务能力地图示例（电商企业）

```
战略层：     ┌─ 商品管理 ─┐  ┌─ 订单履约 ─┐  ┌─ 客户服务 ─┐
            │  商品目录   │  │  订单处理   │  │  售后支持   │
核心能力：   │  价格管理   │  │  库存管理   │  │  投诉处理   │
            │  类目管理   │  │  物流调度   │  │  用户反馈   │
            └─────────────┘  └─────────────┘  └─────────────┘
                                    ↓
支撑层：     ┌─ 财务管理 ─┐  ┌─ 人力资源 ─┐  ┌─ IT 基础 ─┐
            │  预算控制   │  │  人员管理   │  │  运维管理   │
            │  成本核算   │  │  绩效考核   │  │  安全合规   │
            └─────────────┘  └─────────────┘  └────────────┘
```

### 业务能力与 DDD 限界上下文的映射

```
业务能力地图（TOGAF 业务架构层）
    ↓ 每个能力 = 一个限界上下文
限界上下文划分（DDD 战略设计）
    ↓ 每个上下文 = 一组聚合
聚合与实体设计（DDD 战术设计 / OOD）
    ↓ 每个聚合 = 一组协作的类
类与方法设计（OOD + 设计模式）
```

## 四、康威定律与组织架构对齐

TOGAF 强调架构必须与组织结构协同，这正是康威定律在企业架构中的体现：

> 任何组织在设计系统时，最终产生的设计在结构上都等同于该组织的沟通结构。

### 逆康威定律实践

```
期望的微服务架构：     对应的组织调整：
┌─────────┐          ┌──────────┐
│ 用户服务 │  ←对应→  │ 用户团队  │  3-5人，独立部署
├─────────┤          ├──────────┤
│ 订单服务 │  ←对应→  │ 订单团队  │  3-5人，独立部署
├─────────┤          ├──────────┤
│ 支付服务 │  ←对应→  │ 支付团队  │  3-5人，独立部署
└─────────┘          └──────────┘
   接口边界              沟通协议
```

### 团队拓扑（Team Topology）

现代企业架构推荐四种团队类型：

| 团队类型 | 职责 | 示例 |
|---------|------|------|
| **流式团队** | 端到端交付业务价值 | 订单团队（从下单到履约） |
| **赋能团队** | 提升其他团队的能力 | 平台工程团队（提供 CI/CD 工具） |
| **复杂子系统团队** | 处理高技术领域难题 | 推荐算法团队 |
| **平台团队** | 提供自服务基础设施 | 云平台团队（K8s、中间件） |

## 五、架构治理：确保设计落地不走样

TOGAF 的架构治理包含六个维度：

| 维度 | 内容 | 实践方法 |
|------|------|---------|
| **合规** | 确保项目符合架构标准 | 架构评审委员会（ARB） |
| **合同** | 外部供应商的架构约束 | 架构条款写入采购合同 |
| **决策** | 架构决策的记录与追踪 | ADR（Architecture Decision Records） |
| **沟通** | 架构信息的分发与理解 | 架构看板、架构Wiki |
| **能力** | 架构人员的技能发展 | 架构师认证、培训体系 |
| **控制** | 架构变更的管理 | 变更请求流程、影响评估 |

## 六、优势与局限

**优势**：
1. **全局视角**：从业务到技术的全链路思考，避免"技术驱动、业务脱节"
2. **标准化流程**：ADM 提供可复用的架构开发方法论
3. **治理体系**：确保架构不仅被设计出来，而且被持续执行
4. **跨团队协作**：为大型组织提供统一的架构语言和流程

**局限**：
1. **重量级**：完整实施 TOGAF 需要大量文档和流程，不适合敏捷小团队
2. **学习成本高**：概念体系庞大，需要专职的架构师角色
3. **敏捷适配难**：传统 TOGAF 偏瀑布式，与现代敏捷开发需要融合

## 七、实践建议：TOGAF 的轻量级落地

1. **从业务架构入手**：不要跳入技术选型，先画出业务能力地图
2. **裁剪 ADM 流程**：小团队只需关注 A（愿景）、B（业务）、C（信息系统）三个阶段
3. **与 DDD 结合**：业务能力地图 → 限界上下文 → 微服务拆分，形成闭环
4. **引入 ADR**：用轻量级架构决策记录替代厚重的架构文档
5. **关注逆康威定律**：先调整团队结构，再调整系统架构

## 总结：TOGAF 在系统设计中的位置

TOGAF 是**战略层面的治理框架**——它不教你如何设计一个类，而是教你如何让数百个系统、数十个团队在统一的方向下协同工作。

```
战略层：TOGAF（企业架构） → 决定"做什么系统、由谁做、怎么治理"
        ↓
中观层：DDD（领域驱动设计）→ 决定"系统边界在哪、业务是什么"
        ↓
战术层：OOD（面向对象设计） → 决定"类如何组织、方法如何分配"
        ↓
编码层：设计模式 + 编码规范  → 决定"常见问题如何优雅解决"
```

TOGAF + DDD + OOD 不是竞争关系，而是**从战略到战术到编码的三层递进**，分别解决"做什么 → 怎么拆 → 怎么写"的问题。
