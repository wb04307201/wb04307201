# 架构认知的演进

> 系统设计的演进本质是数字化认知的不断深化：从"对象"到"动作"到"组织"，每一次认知的升级都带来架构方法论的革命。

## 一、认知三阶段模型

```
认知深化方向 →
┌──────────┐    ┌──────────┐    ┌──────────┐
│  阶段1    │ →  │  阶段2    │ →  │  阶段3    │
│ 技术视角  │    │ 业务视角  │    │ 组织视角  │
│ 拆分对象  │    │ 拆分行为  │    │ 拆分上下文│
│ 面向对象  │    │ DDD      │    │ TOGAF    │
└──────────┘    └──────────┘    └──────────┘
     │               │               │
     ▼               ▼               ▼
  封装与复用      统一语言        康威定律
  类/模块        限界上下文      企业架构域
  单体/模块化     微服务         产品架构→企业架构
```

## 二、阶段一：面向对象设计（OOD）— 认知"是什么"

### 核心思想

数字化认知初期，我们只能将业务世界拆分为**静态的实体对象**，通过封装属性和方法来模拟现实。

```
业务世界：用户、订单、商品、支付...
         ↓
代码世界：User、Order、Product、Payment...
```

### 代表工具

- **UML 建模**：用类图、时序图可视化对象关系
- **GoF 设计模式**：23 种对象间协作的最佳实践
- **SOLID 原则**：面向对象设计的五大准则

### 局限性

- 以技术实现为导向，容易陷入"如何设计类"而非"业务是什么"
- 对象之间的边界是技术边界（包、模块），不是业务边界
- 复杂业务场景下，对象模型容易膨胀为"大泥球"架构

## 三、阶段二：领域驱动设计（DDD）— 认知"做什么"

### 认知升级

当数字化认知增强后，我们发现**业务本质是一系列动作和规则**，而非静态实体。同一个"订单"对象，在"下单"、"支付"、"发货"、"退款"等场景中的行为和规则完全不同。

```
旧认知：订单是一个对象，有状态和方法
新认知：下单是一个领域行为，产生订单创建事件
```

### 核心转变

| 维度 | 面向对象 | DDD |
|------|---------|-----|
| **建模起点** | 实体对象 | 业务行为（用例/故事） |
| **边界划分** | 包/模块（技术边界） | 限界上下文（业务边界） |
| **通用语言** | 开发团队内部 | 业务+技术团队共享 |
| **一致性** | 事务（数据库层面） | 聚合根（领域层面） |
| **关注点** | 如何组织代码 | 如何映射业务 |

### DDD 的关键概念

- **限界上下文（Bounded Context）**：业务能力的天然边界
- **聚合根（Aggregate Root）**：一致性边界内的入口对象
- **领域事件（Domain Event）**：业务中发生的重要动作的记录
- **统一语言（Ubiquitous Language）**：业务与技术共享的术语体系

## 四、阶段三：企业架构（TOGAF）— 认知"谁来做"

### 认知升级

当认知进一步深化，我们发现**架构不是纯技术问题，而是组织问题的镜像**。康威定律揭示了这个本质：

> 设计系统的组织，其产生的设计等同于组织之内、之间的沟通结构。
> —— Melvin Conway, 1967

这意味着：
- 微服务的拆分边界，应该对应团队的职责边界
- 系统间的接口复杂度，反映了团队间的沟通成本
- **架构演进的本质是组织演进**

### 逆康威定律（Inverse Conway Maneuver）

先调整组织结构，再调整系统架构。让组织架构匹配期望的系统架构。

```
期望的系统架构：独立的微服务 A、B、C
对应的组织调整：独立的团队 A、B、C，各自负责一个服务
```

### TOGAF 架构框架

TOGAF（The Open Group Architecture Framework）将架构分为四个层次：

| 层次 | 关注点 | 示例 |
|------|-------|------|
| **业务架构** | 战略、治理、组织、关键业务流程 | 业务能力地图、价值流 |
| **数据架构** | 数据资产的结构与管理 | 数据模型、数据流 |
| **应用架构** | 应用系统的蓝图与交互 | 应用目录、接口规范 |
| **技术架构** | 基础设施与中间件 | 部署拓扑、技术栈 |

四个层次的关系：
```
业务架构（做什么 + 为什么做）
    ↓ 驱动
数据架构（需要什么数据）
    ↓ 驱动
应用架构（用什么应用处理）
    ↓ 驱动
技术架构（用什么技术实现）
```

## 五、三阶段的对比与选择

### 对比总览

| 维度 | 阶段1：OOD | 阶段2：DDD | 阶段3：TOGAF |
|------|-----------|-----------|-------------|
| **认知焦点** | 实体（是什么） | 行为（做什么） | 组织（谁来做） |
| **拆分依据** | 对象属性与方法 | 业务领域与用例 | 业务能力与团队 |
| **架构产物** | 单体/模块化应用 | 微服务/事件驱动系统 | 企业级架构 |
| **适用规模** | 小团队、单系统 | 中等团队、多系统 | 大型组织、企业级 |
| **沟通成本** | 团队内部 | 跨团队协作 | 全企业协同 |
| **核心方法论** | UML、GoF 模式 | 事件风暴、上下文映射 | ADM 流程、架构制品 |

### 何时用哪个？

- **个人/小团队项目** → OOD 足够。关注代码质量、设计模式。
- **中等规模业务系统** → DDD 是标配。关注领域建模、限界上下文划分。
- **企业级系统/数字化转型** → 需要 TOGAF 视角。关注业务架构先行、组织架构对齐。

### 演进不是替代

三个阶段不是互斥的，而是**层层递进、相互包含**的：
- TOGAF 的应用架构层需要 DDD 来做领域建模
- DDD 的聚合根内部仍然使用 OOD 来组织代码
- OOD 的设计模式在 DDD 的战术设计中仍然适用

```
TOGAF（企业架构）
├── 业务架构：业务能力地图、价值流
├── 数据架构：数据资产规划
├── 应用架构
│   └── DDD（领域驱动设计）
│       ├── 战略设计：限界上下文划分
│       └── 战术设计
│           └── OOD（面向对象设计）
│               ├── 聚合根、实体、值对象
│               └── 设计模式、SOLID 原则
└── 技术架构：基础设施与中间件
```

## 六、实践建议：认知升级路线图

1. **先掌握 OOD**：理解封装、继承、多态，熟悉 GoF 设计模式
2. **学习 DDD 战略设计**：用事件风暴识别限界上下文和领域事件
3. **理解康威定律**：在拆分服务时，同时考虑团队组织结构
4. **引入 TOGAF 视角**：在大型企业项目中，从业务架构出发，而非直接跳入技术选型
