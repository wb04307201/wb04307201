# 架构认知的演进

## 引言：架构困境

架构认知的演进 的关键不是'选型'——是**选完之后怎么在 5 个 trade-off 里活下来**。

本篇用'决策困境'切入，比较几种主流路径并讲清取舍。

---

> 系统设计的演进本质是数字化认知的不断深化：从"对象"到"动作"到"组织"，每一次认知的升级都带来架构方法论的革命。同时，系统的成熟度也在从 Level 1（烟道式）走向 Level 7（自适应服务），**认知深度**和**成熟度等级**共同构成架构演进的双螺旋。

## 一、认知三阶段模型

```
认知深化方向 →
┌──────────┐    ┌──────────┐    ┌──────────┐
│  阶段1    │ →  │  阶段2    │ →  │  阶段3    │
│ 技术视角  │    │ 业务视角  │    │ 组织视角  │
│ 拆分对象  │    │ 拆分行为  │    │ 拆分上下文│
│ 面向对象  │    │ DDD      │    │ TOGAF    │
└──────────┘    └──────────┘    └──────────┘
     │               │               │
     ▼               ▼               ▼
  封装与复用      统一语言        康威定律
  类/模块        限界上下文      企业架构域
  单体/模块化     微服务         产品架构→企业架构
     │               │               │
     ▼               ▼               ▼
  L1-L3 主战场    L4-L5 主战场    L6-L7 主战场
```

## 二、阶段一：面向对象设计（OOD）— 认知"是什么"

### 核心思想

数字化认知初期，我们只能将业务世界拆分为**静态的实体对象**，通过封装属性和方法来模拟现实。

```
业务世界：用户、订单、商品、支付...
         ↓
代码世界：User、Order、Product、Payment...
```

### 代表工具

- **UML 建模**：用类图、时序图可视化对象关系
- **GoF 设计模式**：23 种对象间协作的最佳实践
- **SOLID 原则**：面向对象设计的五大准则

### 局限性

- 以技术实现为导向，容易陷入"如何设计类"而非"业务是什么"
- 对象之间的边界是技术边界（包、模块），不是业务边界
- 复杂业务场景下，对象模型容易膨胀为"大泥球"架构

## 三、阶段二：领域驱动设计（DDD）— 认知"做什么"

### 认知升级

当数字化认知增强后，我们发现**业务本质是一系列动作和规则**，而非静态实体。同一个"订单"对象，在"下单"、"支付"、"发货"、"退款"等场景中的行为和规则完全不同。

```
旧认知：订单是一个对象，有状态和方法
新认知：下单是一个领域行为，产生订单创建事件
```

### 核心转变

| 维度 | 面向对象 | DDD |
|------|---------|-----|
| **建模起点** | 实体对象 | 业务行为（用例/故事） |
| **边界划分** | 包/模块（技术边界） | 限界上下文（业务边界） |
| **通用语言** | 开发团队内部 | 业务+技术团队共享 |
| **一致性** | 事务（数据库层面） | 聚合根（领域层面） |
| **关注点** | 如何组织代码 | 如何映射业务 |

### DDD 的关键概念

- **限界上下文（Bounded Context）**：业务能力的天然边界
- **聚合根（Aggregate Root）**：一致性边界内的入口对象
- **领域事件（Domain Event）**：业务中发生的重要动作的记录
- **统一语言（Ubiquitous Language）**：业务与技术共享的术语体系

## 四、阶段三：企业架构（TOGAF）— 认知"谁来做"

### 认知升级

当认知进一步深化，我们发现**架构不是纯技术问题，而是组织问题的镜像**。康威定律揭示了这个本质：

> 设计系统的组织，其产生的设计等同于组织之内、之间的沟通结构。
> —— Melvin Conway, 1967

这意味着：
- 微服务的拆分边界，应该对应团队的职责边界
- 系统间的接口复杂度，反映了团队间的沟通成本
- **架构演进的本质是组织演进**

### 逆康威定律（Inverse Conway Maneuver）

先调整组织结构，再调整系统架构。让组织架构匹配期望的系统架构。

```
期望的系统架构：独立的微服务 A、B、C
对应的组织调整：独立的团队 A、B、C，各自负责一个服务
```

### TOGAF 架构框架

TOGAF（The Open Group Architecture Framework）将架构分为四个层次：

| 层次 | 关注点 | 示例 |
|------|-------|------|
| **业务架构** | 战略、治理、组织、关键业务流程 | 业务能力地图、价值流 |
| **数据架构** | 数据资产的结构与管理 | 数据模型、数据流 |
| **应用架构** | 应用系统的蓝图与交互 | 应用目录、接口规范 |
| **技术架构** | 基础设施与中间件 | 部署拓扑、技术栈 |

四个层次的关系：
```
业务架构（做什么 + 为什么做）
    ↓ 驱动
数据架构（需要什么数据）
    ↓ 驱动
应用架构（用什么应用处理）
    ↓ 驱动
技术架构（用什么技术实现）
```

## 五、三阶段的对比与选择

### 对比总览

| 维度 | 阶段1：OOD | 阶段2：DDD | 阶段3：TOGAF |
|------|-----------|-----------|-------------|
| **认知焦点** | 实体（是什么） | 行为（做什么） | 组织（谁来做） |
| **拆分依据** | 对象属性与方法 | 业务领域与用例 | 业务能力与团队 |
| **架构产物** | 单体/模块化应用 | 微服务/事件驱动系统 | 企业级架构 |
| **适用规模** | 小团队、单系统 | 中等团队、多系统 | 大型组织、企业级 |
| **沟通成本** | 团队内部 | 跨团队协作 | 全企业协同 |
| **核心方法论** | UML、GoF 模式 | 事件风暴、上下文映射 | ADM 流程、架构制品 |

### 何时用哪个？

- **个人/小团队项目（Level 1-2）** → OOD 足够。关注代码质量、设计模式。
- **中等规模业务系统（Level 3-5）** → DDD 是标配。关注领域建模、限界上下文划分。
- **企业级系统/数字化转型（Level 5-7）** → 需要 TOGAF 视角。关注业务架构先行、组织架构对齐。

### 演进不是替代

三个阶段不是互斥的，而是**层层递进、相互包含**的：
- TOGAF 的应用架构层需要 DDD 来做领域建模
- DDD 的聚合根内部仍然使用 OOD 来组织代码
- OOD 的设计模式在 DDD 的战术设计中仍然适用

```
TOGAF（企业架构）
├── 业务架构：业务能力地图、价值流
├── 数据架构：数据资产规划
├── 应用架构
│   └── DDD（领域驱动设计）
│       ├── 战略设计：限界上下文划分
│       └── 战术设计
│           └── OOD（面向对象设计）
│               ├── 聚合根、实体、值对象
│               └── 设计模式、SOLID 原则
└── 技术架构：基础设施与中间件
```

## 六、实践建议：认知升级路线图

1. **先掌握 OOD**：理解封装、继承、多态，熟悉 GoF 设计模式
2. **学习 DDD 战略设计**：用事件风暴识别限界上下文和领域事件
3. **理解康威定律**：在拆分服务时，同时考虑团队组织结构
4. **引入 TOGAF 视角**：在大型企业项目中，从业务架构出发，而非直接跳入技术选型

## 七、架构成熟度模型 — Level 1 至 Level 7 全景评估

### 7.1 为什么需要成熟度模型？

认知三阶段（OOD → DDD → TOGAF）解决的是 **"工程师该怎么思考"** 的问题。但一个企业或团队还需要回答另一个问题：

> **我们现在的系统处于什么水平？下一步该往哪里走？**

这就是架构成熟度模型的价值 —— 它是一个**评估框架**，帮助你对标当前的架构能力，规划合理的演进路径。

### 7.2 Level 1 → Level 7 全景表

下表从六个维度刻画了 IT 系统开发方式从 Level 1（烟道式）到 Level 7（自适应服务）的完整阶梯：

| 维度 / 阶段 | Level 1（烟道式） | Level 2（集成式） | Level 3（组件化） | Level 4（服务） | Level 5（集成服务） | Level 6（虚拟服务） | Level 7（自适应服务） |
|:---|:---|:---|:---|:---|:---|:---|:---|
| **IT 对业务支持** | 面向功能 | 面向功能 | 面向功能 | 面向服务 | 面向服务 | 面向服务 | 面向服务 |
| **设计方法** | 结构化设计 | 面向对象设计 | 组件化设计 | 面向服务设计 | 面向服务设计 | 面向服务设计 | 语言式设计 |
| **应用系统** | 模块 | 对象 | 组件 | 服务 | 流程驱动服务 | 流程驱动服务 | 动态组合 |
| **系统架构** | 单层架构 | 分层架构 | 组件架构 | 初步 SOA | 面向对象 SOA | 网络型 SOA | 动态调整机构 |
| **IT 平台** | 特定平台 | 特定平台 | 特定平台 | 特定平台 | 特定平台 | 多平台 | 动态自适应 |

### 7.3 三大阶段解读

#### 传统阶段（Level 1 → Level 3）：面向功能

| Level | 核心特征 | 典型表现 |
|-------|---------|---------|
| **L1 烟道式** | 结构化设计，单层架构 | 所有功能写在一个系统里，没有分层、没有复用 |
| **L2 集成式** | 面向对象设计，分层架构 | 引入了 MVC、ORM 等分层思想，对象封装，但仍是单体 |
| **L3 组件化** | 组件化设计，组件架构 | 按功能模块拆分为可复用组件，但组件间耦合紧密，平台依赖特定技术 |

**关键趋势：** 设计方法从结构化向面向对象、组件化演进；架构从单层向分层、组件架构发展；但全部依赖"特定平台"。

**大多数中国企业现在处于 Level 2 到 Level 3 之间。**

#### 服务化阶段（Level 4 → Level 5）：面向服务

| Level | 核心特征 | 典型表现 |
|-------|---------|---------|
| **L4 服务** | 面向服务设计，初步 SOA | 按业务能力拆分为独立服务，服务间通过标准接口通信 |
| **L5 集成服务** | 面向服务设计，流程驱动 SOA | 服务编排为业务流程，服务间形成协作链路，SOA 完整落地 |

**关键趋势：** IT 从"提供功能"转向"提供服务"；应用从单一服务发展为流程驱动的服务链。

#### 未来阶段（Level 6 → Level 7）：自适应

| Level | 核心特征 | 典型表现 |
|-------|---------|---------|
| **L6 虚拟服务** | 面向服务设计，网络型 SOA | 服务形成服务网格（Service Mesh），IT 平台支持多平台异构部署 |
| **L7 自适应服务** | 语言式设计，动态组合 | 系统根据业务需求动态组合服务，IT 平台动态自适应（AI 驱动） |

**关键趋势：** 系统架构演进为网络型和动态调整机构；IT 平台从特定平台走向多平台，最终实现动态自适应。

### 7.4 与认知三阶段的映射关系

认知三阶段（OOD → DDD → TOGAF）与 Level 1-7 成熟度模型是**互补关系**，而非替代关系：

```
┌─────────────────────────────────────────────────────────────────┐
│                    认知维度 × 成熟度等级                         │
├─────────────┬───────────────────────────────────────────────────┤
│             │                                                   │
│  认知维度    │  Level 1    Level 2    Level 3    Level 4-5       │
│  三阶段      │  ─────────────────────────────────────            │
│              │                                                   │
│  阶段1 OOD   │  ★主战场     ▲延伸                                  │
│  (是什么)    │  结构化 → 面向对象 → 组件化                        │
│              │                                                   │
│              │                              ★主战场               │
│  阶段2 DDD   │                              ▲延伸                │
│  (做什么)    │                              服务化 → 流程驱动     │
│              │                                                   │
│              │                                                  ★ │
│  阶段3 TOGAF │                                         主战场      │
│  (谁来做)    │                                         网络型SOA  │
│              │                                         → 动态自适应│
│              │                                                   │
└─────────────┴───────────────────────────────────────────────────┘

解读：
- OOD 的成熟过程贯穿 L1-L3，在 L2-L3 达到高峰
- DDD 从 L3 末开始发力，在 L4-L5 成为核心方法论
- TOGAF 从 L5 末开始引入，在 L6-L7 成为主导思维
- 三个认知阶段不是互斥的，而是层层递进、相互包含的
```

| 成熟度等级 | 主导认知方法 | 辅助认知方法 | 说明 |
|-----------|-------------|-------------|------|
| Level 1-2 | OOD | — | 学习面向对象设计，建立封装和复用意识 |
| Level 3 | OOD + DDD 入门 | — | 组件化需要 DDD 的限界上下文思想来划边界 |
| Level 4-5 | DDD | OOD（战术层） | 服务拆分用 DDD，聚合根内部用 OOD |
| Level 6-7 | TOGAF | DDD（应用架构层） | 企业级治理用 TOGAF，微服务设计用 DDD |

### 7.5 两个关键分水岭

**分水岭一：Level 3 → Level 4（功能 → 服务）。**

这是从"我能做什么功能"到"我能提供什么服务"的思维转变。很多团队号称"微服务"，实际上只是把单体拆成了"分布式单体"——接口仍是 CRUD，思维仍是面向功能。真正的服务化要求每个服务是一个**业务能力**的独立表达，对应 DDD 的限界上下文。

**分水岭二：Level 5 → Level 6（SOA → 网络型 SOA）。**

当服务不再是点对点调用，而是形成服务网络；当 IT 平台不再绑定特定技术栈，而是支持多平台异构部署（容器化、K8s 编排）——你就跨过了这个分水岭，对应 TOGAF 的企业架构治理。

### 7.6 企业自评指南

用以下六个问题快速判断你所在的 Level：

| # | 问题 | L1 | L2 | L3 | L4 | L5 | L6 | L7 |
|---|------|----|----|----|----|----|----|----|
| 1 | 系统按什么组织？ | 功能 | 对象 | 组件 | 服务 | 流程服务 | 服务网络 | 动态组合 |
| 2 | 设计方法是什么？ | 结构化 | OO | 组件化 | SOA | SOA | SOA | 语言式 |
| 3 | 应用系统形态？ | 模块 | 对象 | 组件 | 服务 | 流程驱动 | 流程驱动 | 动态 |
| 4 | 架构形态？ | 单层 | 分层 | 组件 | 初步 SOA | 完整 SOA | 网络 SOA | 动态 |
| 5 | IT 平台状态？ | 特定 | 特定 | 特定 | 特定 | 特定 | 多平台 | 自适应 |
| 6 | 业务支持方式？ | 功能 | 功能 | 功能 | 服务 | 服务 | 服务 | 服务 |

**自测方法：** 对每个问题，选择最符合你当前状态的选项，6 个问题中**出现最多的 Level 就是你当前的大致成熟度**。

### 7.7 演进路线图建议

```
Level 1 → Level 2 → Level 3 → Level 4 → Level 5 → Level 6 → Level 7
  │         │         │         │         │         │         │
  ▼         ▼         ▼         ▼         ▼         ▼         ▼
 学习      引入      建立      服务      流程      容器      AI驱动
 结构化    OOD       组件化    拆分      编排      化        自适应
 设计      分层      复用      DDD       SOA       多平台    动态组合
           MVC       模块化    限界      服务      服务      语言式
           封装      组件库    上下文    网格      网格      设计
```

**核心原则：永远不要跳过 Level 直接奔着 Level N+1 去。** 很多企业在 Level 2 就硬拆微服务，结果不是架构升级，而是复杂度爆炸。
