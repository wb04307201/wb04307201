# 面向对象设计

---

> 面向对象设计（Object-Oriented Design, OOD）是将业务世界建模为相互协作的对象的系统化方法，是软件工程中最基础、最广泛使用的设计范式。

## 一、核心思想：用对象模拟现实世界

OOD 的核心是将复杂的业务系统拆分为一组**高内聚、低耦合的对象**，每个对象封装自己的数据和行为，通过消息传递（方法调用）完成协作。

```text
业务世界：用户下单购买商品，系统扣减库存、生成订单、发送通知
         ↓
对象世界：User  →  Order  →  Inventory  →  Notification
          调用      调用       调用          调用
```

### 三大基石

1. **封装**：将数据和操作数据的方法绑定在一起，隐藏内部实现细节
2. **继承**：子类复用父类的属性和行为，建立层次化的类型关系
3. **多态**：同一接口在不同对象上有不同的行为，提高代码的灵活性

## 二、设计原则：SOLID 与 GRASP

### SOLID 五大原则

| 原则 | 含义 | 违反的表现 | 正确做法 |
|------|------|-----------|---------|
| **单一职责（SRP）** | 一个类只有一个变化的原因 | 上帝类（God Class）包含几十个不相关的方法 | 按职责拆分：订单类管订单，支付类管支付 |
| **开闭原则（OCP）** | 对扩展开放，对修改关闭 | 每加一种支付方式都要改原有代码 | 用接口抽象支付方式，新增实现类即可 |
| **里氏替换（LSP）** | 子类必须能替换父类 | 子类重写方法后行为异常或抛出异常 | 子类不削弱父类契约，不违反前置/后置条件 |
| **接口隔离（ISP）** | 接口尽量小而专 | 胖接口（Fat Interface）要求实现大量不需要的方法 | 拆分接口：UserRepository 拆分为 ReadRepository 和 WriteRepository |
| **依赖倒置（DIP）** | 高层模块不应依赖低层模块，都应依赖抽象 | 业务层直接 new 具体 DAO 类 | 业务层依赖 DAO 接口，由容器注入具体实现 |

### GRASP 九大职责分配原则

| 原则 | 用途 | 示例 |
|------|------|------|
| **信息专家** | 将职责分配给拥有完成它所需信息的类 | 计算订单总价 → 放在 Order 类而非 OrderService |
| **创建者** | 谁负责创建对象 | Order 创建 OrderItem（聚合关系） |
| **控制器** | 处理系统事件的第一个非UI对象 | OrderController 接收用户下单请求 |
| **低耦合** | 减少类之间的依赖 | 通过接口而非具体类通信 |
| **高内聚** | 类的职责应聚焦且相关 | Payment 类只管支付逻辑，不管日志 |
| **多态** | 利用多态处理变化 | Strategy 模式处理不同折扣规则 |
| **纯虚构** | 引入不存在于业务中的类来简化设计 | PriceCalculator 纯粹为计算而存在的类 |
| **间接性** | 通过中间对象解耦 | 用 Facade 封装子系统接口 |
| **稳定性** | 将职责分配给不易变化的类 | 基础工具类而非业务类 |

## 三、设计流程：从需求到代码

1. **识别对象**
   - 从需求文档中提取名词（实体）和动词（行为）
   - 例：「用户提交订单」→ User、Order、submit()

2. **建立关系**
   - 用 UML 类图描述对象间的关系（关联、聚合、组合、继承）
   - 聚合：订单包含多个订单项（可独立存在）
   - 组合：订单包含支付信息（不可独立存在）

3. **分配职责**
   - 应用 GRASP 原则决定方法归属
   - 避免"贫血模型"（只有 getter/setter 没有行为）

4. **应用设计模式**
   - 在职责分配后，用 GoF 设计模式解决常见结构问题
   - 工厂模式：对象的创建逻辑复杂时
   - 策略模式：同一行为有多种实现时
   - 观察者模式：一个对象变化需通知多个对象时

5. **迭代重构**
   - 随着理解加深，持续重构类的设计
   - 提取方法、提取类、引入接口、消除重复

## 四、优势与局限

**优势**：
1. **认知门槛低**：对象是现实世界的自然映射，初学者容易上手
2. **代码复用性强**：继承和多态机制有效减少重复代码
3. **生态成熟**：Java、C++、Python、C# 等主流语言均深度支持
4. **与设计模式无缝衔接**：GoF 23 种模式建立在 OOD 基础之上

**局限**：
1. **以技术为中心**：容易陷入"如何设计类"而非"业务是什么"
2. **大泥球风险**：缺乏业务边界指导，复杂系统中对象关系混乱
3. **贫血模型泛滥**：数据（Entity）和行为（Service）分离，退化为"面向过程"
4. **不适合分布式场景**：对象间的同步调用在微服务中会变成网络调用，导致性能问题

## 五、实践建议：OOD 的正确打开方式

1. **先理解业务，再设计对象**：不要跳过需求分析直接画类图
2. **组合优于继承**：优先用"has-a"而非"is-a"关系，降低耦合
3. **避免过度设计**：YAGNI 原则——不需要的设计不要提前做
4. **持续重构**：第一版设计通常不是最优的，随着理解加深要敢于重构
5. **贫血模型是陷阱**：让对象拥有行为，不要让 Entity 退化为数据容器

## 总结：OOD 在系统设计中的位置

OOD 是**战术层面的基础技能**——它教会我们如何写出高内聚、低耦合的代码。但 OOD 本身不提供**战略层面的指导**：它不告诉你系统应该拆成几个模块、边界在哪里、如何与业务对齐。这正是 DDD 和 TOGAF 需要解决的更高层次的问题。

```text
战略层：TOGAF（企业架构） → 决定"做什么系统、由谁做"
        DDD（领域建模）   → 决定"系统边界在哪、业务是什么"
战术层：OOD（对象设计）   → 决定"类如何组织、方法如何分配"
        设计模式         → 决定"常见问题如何优雅解决"
```
