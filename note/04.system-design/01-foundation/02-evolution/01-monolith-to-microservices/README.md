# 单体到微服务的演进：5 大策略与 6 大陷阱

> 一份按层次梳理的单体拆分速查手册：从策略选择到陷阱规避的完整实战。

---
## 引言：架构困境

单体到微服务的演进：5 大策略与 6 大陷阱 的关键不是'选型'——是**选完之后怎么在 5 个 trade-off 里活下来**。

本篇用'决策困境'切入，比较几种主流路径并讲清取舍。

---

## 一、为什么要拆分单体？

```
单体应用的痛点（用户 100 万 + 后）：
  - 部署 1 小时（重启整个应用）
  - 1 个 bug → 全站挂
  - 100+ 工程师在同一个代码库冲突
  - 任何新功能都要全量回归
```

**微服务的价值**：
- ✅ 独立部署（10 分钟 → 1 分钟）
- ✅ 故障隔离
- ✅ 团队自治（每个团队 5-10 人）
- ✅ 技术栈多样化

---

## 二、5 大拆分策略

### 策略 1：绞杀者模式（Strangler Fig Pattern）⭐ 推荐

```
老单体应用
   ↓
  路由（API Gateway）
   ↓
新服务逐步替换老功能
   ↓
  ↓  ↓  ↓
  v1  v2  v3  ← 新服务逐步上线
   ↓  ↓  ↓
老功能逐步退役
```

**特点**：渐进式迁移（1-2 年），风险最低。

### 策略 2：绞杀者 + 事件驱动

```
老单体 → 抽事件 → 写新服务
   ↓
老服务发出 Domain Event
   ↓
新服务订阅事件
   ↓
新老并存
```

### 策略 3：按业务能力拆分（推荐起点）

```
老单体（电商）
   ↓ 拆分为：
   ├── 用户服务
   ├── 商品服务
   ├── 订单服务
   ├── 支付服务
   ├── 库存服务
   └── 物流服务
```

**原则**：每个服务 = 一个 bounded context（DDD 概念）

### 策略 4：按团队拆分（Conway 定律）

```
康威定律：组织结构决定系统架构

10 个团队（10 人/队）
   ↓
10 个微服务
   ↓
每个团队负责自己的服务
   ↓
自治（独立开发 / 部署 / 运维）
```

### 策略 5：按性能特征拆分

```
读写分离：
  ├── 读服务（缓存优化）
  └── 写服务（事务优化）

冷热分离：
  ├── 热点服务（Redis + 集群）
  └── 冷数据服务（数据库 + 缓存）
```

## 三、6 大拆分陷阱

### 陷阱 1：拆分过细（Microservice Envy）

```
❌ 错误：1000 行代码拆成 10 个服务（10 行/服务）
   ↓
问题：RPC 调用比函数调用慢 1000 倍
   ↓
结果：性能暴跌 / 团队维护崩溃
```

**原则**：一个服务至少 200-500 行代码。

### 陷阱 2：分布式事务

```
单体：数据库事务（ACID）
   ↓
微服务：分布式事务（2PC / Saga / TCC）
   ↓
复杂度 × 10，性能 ÷ 5
```

**解决**：避免跨服务事务（最终一致性 / Saga 模式）

### 陷阱 3：分布式调用链

```
单体：本地方法调用（debug 简单）
   ↓
微服务：A → B → C → D（30 个服务）
   ↓
一个 bug 怎么查？（链路追踪 / 日志聚合）
```

**解决**：OpenTelemetry / Jaeger / SkyWalking

### 陷阱 4：数据一致性

```
单体：单数据库（强一致）
   ↓
微服务：每个服务独立数据库（最终一致）
   ↓
"我下单了，但库存没扣" → 用户体验差
```

**解决**：事件驱动 + 幂等性 + 补偿

### 陷阱 5：运维成本

```
单体：1 个应用 / 1 个数据库
   ↓
微服务：50 个应用 / 50 个数据库 / 50 个 CI / 50 个监控
   ↓
运维人力 × 10
```

**解决**：K8s + 平台工程 + IDP

### 陷阱 6：组织未准备好

```
单体 → 微服务 ≠ 单纯技术升级
   ↓
需要：
  - DevOps 文化
  - 团队重构（按业务能力划分）
  - 监控 / 日志 / 告警体系
  - 自动化测试 + CI/CD
```

## 四、6 大拆分步骤

### 步骤 1：识别 bounded context（DDD）

```
业务分析：
  电商 = 6 个 bounded context
    ├── 用户（账号 / 会员）
    ├── 商品（SKU / 库存）
    ├── 订单（购物车 / 订单）
    ├── 支付（钱包 / 交易）
    ├── 物流（订单跟踪）
    └── 售后（退款 / 评价）
```

### 步骤 2：优先级排序

```
拆分优先级 = 业务价值 × 改动风险
   ↓
高价值低风险 → 先拆（用户 / 商品）
高价值高风险 → 后拆（订单 / 支付）
低价值 → 不拆（先放着）
```

### 步骤 3：建立基础设施

```
微服务前置条件：
  ✅ CI/CD（Jenkins / GitLab）
  ✅ 服务发现（Nacos / Consul）
  ✅ 配置中心（Apollo / Nacos）
  ✅ 链路追踪（SkyWalking / Jaeger）
  ✅ 日志聚合（ELK / Loki）
  ✅ 监控告警（Prometheus + Grafana）
```

### 步骤 4：绞杀者模式逐步迁移

```
阶段 1（3 个月）：
  拆分 1 个非核心服务（如"用户评价"）
  ↓
  验证基础设施 / 团队协作模式
  ↓
阶段 2（6 个月）：
  拆分 3-5 个核心服务
  ↓
阶段 3（12 个月）：
  拆分剩余服务
  ↓
阶段 4（最后）：
  老单体退役
```

### 步骤 5：建立团队能力

```
微服务要求团队具备：
  - 容器化（Docker + K8s）
  - DevOps（CI/CD + 监控）
  - 分布式（CAP / Saga / 最终一致性）
  - 可观测性（Metrics / Logs / Traces）
  - 故障演练（Chaos Engineering）
```

### 步骤 6：持续优化

```
- 监控服务数量（不要无限拆分）
- 定期合并过细的服务
- 优化调用链
- 性能测试（定期压测）
```

## 五、典型案例：电商单体拆分

### 阶段 1（3 个月）

```
单体电商应用
   ↓ 抽出
   - 用户服务（Spring Boot + MySQL）
   - 评价服务（Spring Boot + MongoDB）
   ↓
   路由层：API Gateway（Nginx / Spring Cloud Gateway）
   ↓
   公共组件：注册中心（Nacos）
```

### 阶段 2（6 个月）

```
新增：
   - 商品服务
   - 库存服务
   ↓
   公共组件：配置中心 / 链路追踪
   ↓
   通信：HTTP（同步）+ Kafka（异步事件）
```

### 阶段 3（12 个月）

```
新增：
   - 订单服务
   - 支付服务
   - 物流服务
   ↓
   高级特性：
     - 分布式事务（Seata / Saga）
     - 链路追踪（SkyWalking）
     - 监控告警（Prometheus）
     - 熔断限流（Sentinel）
```

### 阶段 4（最后）

```
老单体 → 退役
所有服务 → K8s 部署
Service Mesh → Istio / Linkerd
可观测 → Prometheus + Grafana + Loki
```

## 六、拆分后 7 大常见问题

| 问题 | 解决 |
|------|------|
| **服务调用链太长** | BFF 聚合 / CQRS |
| **跨服务查询慢** | 数据冗余 / 宽表 |
| **分布式事务** | Saga / 本地消息表 |
| **服务雪崩** | 熔断 / 限流 / 隔离 |
| **配置不一致** | 配置中心（Nacos）|
| **调用关系混乱** | 服务网格（Istio）|
| **测试困难** | 契约测试 / 服务虚拟化 |

## 七、什么时候不要拆？

| 信号 | 建议 |
|------|------|
| 团队 < 5 人 | ❌ 不要拆（单体更好）|
| 用户 < 10 万 | ❌ 不要拆 |
| 业务频繁变化 | ❌ 不要拆（服务边界未定型）|
| 团队不具备 K8s / DevOps | ❌ 不要拆 |
| 一个服务每周都要改 | ⚠️ 考虑合并（拆错了）|

---

## 八、最佳实践

1. **绞杀者模式 + 渐进式迁移**：不要"一次性重写"
2. **先建基础设施**：监控 / CI / 服务发现
3. **DDD 驱动拆分**：按 bounded context
4. **团队对齐 Conway 定律**：组织结构 = 架构
5. **数据迁移谨慎**：双写 / 灰度
6. **持续合并**：发现拆错立即合并
7. **小步快跑**：每 2 周一个迭代
8. **可观测性优先**：出问题能快速定位
