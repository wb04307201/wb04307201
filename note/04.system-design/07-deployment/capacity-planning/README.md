## 为什么需要容量规划

### 不规划容量的后果

- **容量不足**：系统崩溃、响应变慢、用户流失（典型的"上线即崩溃"）
- **容量过剩**：资源浪费、成本飙升（过度配置）
- **无预警**：流量突增时没有缓冲时间扩容

### 容量规划的核心问题

1. **当前容量**：系统现在能承载多少流量？
2. **增长预期**：未来 3 个月/6 个月/1 年流量会增长多少？
3. **瓶颈在哪**：CPU、内存、数据库、网络，哪个先到达极限？
4. **如何扩容**：垂直扩容（升配）还是水平扩容（加机器）？

### 容量规划流程

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 业务预测  │───▶│ 单机压测  │───▶│ 全链路   │───▶│ 容量估算  │───▶│ 弹性策略  │
│ QPS目标   │    │ 基准数据  │    │ 压测验证  │    │ 机器数量  │    │ 自动扩缩容│
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
```

## 压测方法论

### 全链路压测 vs 单机压测

| 维度 | 单机压测 | 全链路压测 |
|------|----------|------------|
| **范围** | 单个服务/组件 | 完整的请求链路 |
| **目的** | 评估单点性能上限 | 评估系统整体容量 |
| **准确性** | 高（变量少） | 中（变量多） |
| **复杂度** | 低 | 高 |
| **发现** | 服务本身瓶颈 | 网络、依赖、数据库瓶颈 |

**推荐策略**: 先做单机压测建立基准，再做全链路压测验证整体容量。

### 压测工具对比

| 工具 | 语言 | 特点 | 适用场景 |
|------|------|------|----------|
| **JMeter** | Java | GUI + CLI，功能最丰富，插件生态好 | 复杂场景、脚本化压测 |
| **wrk** | C | 极简，单机高并发，HTTP 压测首选 | 快速 HTTP 基准测试 |
| **vegeta** | Go | 恒定速率攻击，结果可视化 | 持续压力测试 |
| **Gatling** | Scala | 代码驱动，报告精美 | CI/CD 集成 |
| **Apache AB** | C | 简单快速，功能有限 | 快速验证 |
| **k6** | JavaScript | 开发者友好，云原生 | 现代 CI/CD 流水线 |

### wrk 快速示例

```bash
# 基本用法: wrk -t线程数 -c连接数 -d持续时间 URL
wrk -t12 -c400 -d30s http://api.example.com/orders

# 输出示例:
# Running 30s test @ http://api.example.com/orders
#   12 threads and 400 connections
#   Thread Stats   Avg      Stdev     Max   +/- Stdev
#     Latency    45.23ms   12.45ms  180.32ms   89.12%
#     Req/Sec   742.34     56.78     890.00     85.20%
#   267,890 requests in 30.01s, 45.67MB read
# Requests/sec:   8928.45
# Transfer/sec:    1.52MB
```

### JMeter 压测流程

```
1. 设计测试计划
   ├── Thread Group (模拟用户数、 ramp-up 时间)
   ├── HTTP Request Defaults (服务器配置)
   ├── CSV Data Set Config (参数化数据)
   └── HTTP Request (接口定义)

2. 添加监听器
   ├── Summary Report (汇总报告)
   ├── Response Time Graph (响应时间图)
   └── Aggregate Report (聚合报告)

3. 执行压测
   ├── 命令行模式: jmeter -n -t test.jmx -l result.jtl
   └── GUI 模式仅用于调试，不用于正式压测

4. 分析报告
   ├── TPS 曲线
   ├── RT 分布 (P50/P90/P95/P99)
   └── 错误率
```

### 压测注意事项

1. **数据隔离**: 使用测试数据库，不影响生产数据
2. **数据预热**: 数据库缓存预热，模拟真实状态
3. **网络环境**: 压测机与被测服务在同一网络，排除网络干扰
4. **逐步加压**: 从低负载逐步增加，找到性能拐点
5. **持续时间**: 稳态压测至少持续 30 分钟，观察是否有性能衰减
6. **监控同步**: 压测同时收集系统指标（CPU、内存、GC、DB）

## 容量估算模型

### 核心公式

```
所需机器数 = (目标 QPS × 峰值系数) / 单机 QPS
```

| 参数 | 说明 | 计算方式 |
|------|------|----------|
| 目标 QPS | 业务预期的平均 QPS | 日请求量 / 86400 |
| 峰值系数 | 峰值 QPS 相对于平均的倍数 | 通常取 3~10 |
| 单机 QPS | 单台机器能处理的 QPS | 压测得到 |

### 示例计算

假设某电商系统：
- 日均订单量：100 万单
- 每单 API 调用：5 次（下单 + 支付 + 库存 + 通知 + 日志）
- 日 API 请求量：500 万
- 平均 QPS：5,000,000 / 86,400 ≈ **58 QPS**
- 峰值系数：大促期间峰值约为平均的 **8 倍**
- 峰值 QPS：58 × 8 ≈ **464 QPS**
- 单机压测 QPS：每个服务实例能处理 **200 QPS**

```
所需实例数 = 464 / 200 = 2.32 → 向上取整 = 3 台
考虑冗余（安全系数 1.5）= 3 × 1.5 = 4.5 → 向上取整 = 5 台
```

### Little's Law（利特尔法则）

```
并发请求数 = QPS × 平均响应时间

示例:
  QPS = 500
  平均 RT = 100ms = 0.1s
  并发请求数 = 500 × 0.1 = 50
```

这意味着在任何时刻，系统中平均有 50 个请求正在被处理。

### 数据库容量估算

```
DB QPS = 总 QPS × DB 访问比例 / (1 - 缓存命中率)

示例:
  总 QPS = 5000
  DB 访问比例 = 30%（70% 被缓存拦截）
  缓存命中率 = 90%
  DB QPS = 5000 × 0.3 / (1 - 0.9) = 1500 / 0.1 = 15,000 QPS
```

> 缓存命中率极高时，DB 实际 QPS 远高于直觉值。

### 存储容量估算

```
日新增数据量 = 日均记录数 × 单条记录大小
年存储需求 = 日新增数据量 × 365 × 冗余系数(通常 1.5~2)

示例:
  日均订单: 100 万
  单条订单: ~2KB
  日新增: 100万 × 2KB ≈ 2GB
  年需求: 2GB × 365 × 2 ≈ 1.46TB
```

## 压测关键指标

### 性能指标

| 指标 | 说明 | 正常范围 | 告警阈值 |
|------|------|----------|----------|
| **TPS** | 每秒事务数 | 越高越好 | 低于预期 80% |
| **RT (P50)** | 50% 请求的响应时间 | < 200ms | > 500ms |
| **RT (P95)** | 95% 请求的响应时间 | < 500ms | > 1000ms |
| **RT (P99)** | 99% 请求的响应时间 | < 1000ms | > 2000ms |
| **错误率** | 失败请求占比 | < 0.1% | > 1% |
| **成功率** | 成功请求占比 | > 99.9% | < 99% |

### 资源利用率指标

| 指标 | 说明 | 正常范围 | 告警阈值 |
|------|------|----------|----------|
| **CPU 使用率** | CPU 繁忙程度 | < 70% | > 85% |
| **内存使用率** | 堆内存使用比例 | < 80% | > 90% |
| **GC 频率** | 垃圾回收次数 | 取决于业务 | Young GC > 50次/分钟 |
| **GC 耗时** | 垃圾回收总耗时 | Young < 50ms | Full GC > 1s |
| **磁盘 I/O** | 磁盘读写延迟 | < 10ms | > 50ms |
| **网络带宽** | 网络使用率 | < 70% | > 85% |
| **连接数** | 数据库/连接池 | < 80% 上限 | > 90% |
| **线程数** | 应用线程池使用 | < 80% 上限 | > 95% |

### 压测结果分析模板

```
压测报告模板:
─────────────────────────────────────────────
场景: 下单接口压测
时间: {{压测日期}}
环境: 8C16G × 3 实例

压测配置:
  并发用户数: 500
  持续时间: 30 分钟
  数据量: 100 万条预热数据

结果:
  平均 TPS:  1,250
  P50 RT:    45ms
  P95 RT:    120ms
  P99 RT:    350ms
  错误率:    0.02%

资源使用:
  CPU:       65%
  内存:      72%
  GC:        Young 15次/分, 平均 25ms
  DB 连接:   45/100

瓶颈分析:
  1. DB 慢查询 (SELECT * FROM orders WHERE user_id = ?)
     缺少索引，平均耗时 80ms
  2. 线程池配置过小 (core=50, max=100)
     峰值时排队严重

优化建议:
  1. 添加索引: CREATE INDEX idx_user_id ON orders(user_id)
  2. 调整线程池: core=200, max=400
  3. 优化后预期 TPS: 2,500+
─────────────────────────────────────────────
```

## 弹性伸缩策略

### HPA（Horizontal Pod Autoscaler）— 水平扩缩容

根据指标自动增加或减少 Pod 数量。

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: order-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: order-service
  minReplicas: 3
  maxReplicas: 20
  metrics:
    # 基于 CPU 使用率
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
    # 基于自定义指标（QPS）
    - type: Pods
      pods:
        metric:
          name: http_requests_per_second
        target:
          type: AverageValue
          averageValue: "500"
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60   # 扩容稳定窗口
      policies:
        - type: Pods
          value: 5                      # 每次最多扩 5 个 Pod
          periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300  # 缩容稳定窗口（更长，防抖动）
      policies:
        - type: Pods
          value: 2                      # 每次最多缩 2 个 Pod
          periodSeconds: 60
```

### VPA（Vertical Pod Autoscaler）— 垂直扩缩容

自动调整 Pod 的 CPU 和内存资源配置。

```yaml
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: order-service-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: order-service
  updatePolicy:
    updateMode: "Auto"  # Auto / Initial / "Off"
  resourcePolicy:
    containerPolicies:
      - containerName: order-service
        minAllowed:
          cpu: 250m
          memory: 256Mi
        maxAllowed:
          cpu: 4000m
          memory: 4Gi
```

### 扩缩容触发条件

| 触发条件 | 扩缩容 | 说明 |
|----------|--------|------|
| CPU 使用率 > 80% | 扩容 | 最常见触发条件 |
| 内存使用率 > 85% | 扩容 | 防止 OOM |
| QPS > 阈值 | 扩容 | 基于业务量 |
| 响应时间 > 阈值 | 扩容 | 基于用户体验 |
| 消息堆积量 > 阈值 | 扩容 | 消费者扩容 |
| CPU 使用率 < 30%（持续 5 分钟） | 缩容 | 节省成本 |
| 定时扩缩容 | 扩容/缩容 | 按业务规律（工作日/节假日） |

### 扩缩容策略最佳实践

1. **设置最小实例数**：始终保留 2~3 个实例，保障高可用
2. **缩容比扩容慢**：缩容窗口应大于扩容窗口，防止频繁抖动
3. **预热机制**：新实例启动后需要预热（加载配置、建立连接池）才能接收流量
4. **优雅下线**：缩容时等待活跃请求处理完成后再终止实例
5. **监控扩缩容事件**：记录每次扩缩容的时间和原因，持续优化阈值

## 状态服务容量规划

前面的容量模型主要针对**无状态服务**（QPS × RT 即可估算）。本节补充**有状态服务**（Session、消息队列、缓存）的容量规划要点。

### Session 服务容量模型

Session 服务的容量由"**新建速率**"和"**平均在线时长**"共同决定，本质是稳态在线会话数与并发处理能力之间的平衡。

```
稳态在线会话数 = 新建速率（QPS） × 平均在线时长（s）
```

**核心公式**

```
所需实例数 = ceil(在线会话数 / 单机承载会话数)
         = ceil(QPS_new × AvgSessionDuration / MaxSessionsPerInstance)
```

**示例**

```
新建 Session 速率:  1,000 QPS
平均在线时长:        30 分钟 = 1,800 s
稳态在线会话数:      1,000 × 1,800 = 1,800,000 个

单机承载:           100,000 个会话（受限于堆内存、JVM 会话对象大小）
所需实例数:          1,800,000 / 100,000 = 18 台
```

**关键监控指标**

| 指标 | 计算方式 | 告警阈值 |
|------|----------|----------|
| 新建速率 | `session_create_total` 速率 | 高于基线 2x |
| 平均在线时长 | `histogram_quantile(0.5, session_duration_seconds)` | 偏离基线 ±30% |
| 活跃会话数 | Gauge 指标 | 接近单机承载上限 80% |
| Session GC 频率 | 单位时间内失效 Session 数 | 突发性上涨 |

**容量扩展注意点**

- 内存是 Session 服务的主要瓶颈，需监控堆内存中 Session 对象占用
- 启用 Session 过期淘汰（LRU/TTL）防止内存泄漏
- 分布式 Session 共享（如 Redis）可降低单机内存压力
- 注意**会话粘性**与水平扩容的冲突

### 消息队列容量规划

消息队列的核心容量指标是**消息堆积量**和**消费延迟**。

```
稳态消息量 = 生产速率（msg/s） × 允许积压时间窗口（s）
峰值积压 = max(生产速率, 消费速率) × 时间窗口
```

**关键监控指标**

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 消息堆积量（Lag） | 生产者已发送但消费者未确认的总数 | 持续 > 阈值 5 分钟 |
| 生产 TPS | 单位时间消息生产数 | 偏离基线 |
| 消费 TPS | 单位时间消息消费数 | 持续 < 生产 TPS |
| 消费 P99 延迟 | 消息从生产到消费的时间 | > 业务容忍值 |
| 消费者实例数 | 活跃消费者数 | < 最小副本数 |

**背压（Backpressure）策略**

当消费能力不足时，必须主动背压防止 OOM：

1. **生产者限流**：监控到堆积告警时降级生产速率
2. **消费者扩容**：基于堆积量的 HPA（不同于基于 CPU 的 HPA）
3. **队列容量上限**：超过上限后直接拒绝（快速失败）
4. **降级策略**：非关键消息直接丢弃，仅保留核心链路

**示例：基于 Lag 的 HPA**

```yaml
# KEDA + Prometheus 触发器示例
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: order-consumer-hpa
spec:
  scaleTargetRef:
    name: order-consumer
  triggers:
    - type: prometheus
      metadata:
        serverAddress: http://prometheus:9090
        metricName: kafka_consumergroup_lag
        query: |
          sum(kafka_consumergroup_lag{consumergroup="order-consumer"})
        threshold: "10000"   # 超过 1 万条积压即扩容
```

### 缓存热 Key 检测

热 Key（Hot Key）是指被高频访问的少量 Key，会导致：

- 单个缓存节点压力过大
- 缓存击穿后压垮数据库
- 响应延迟飙升

**热 Key 检测方式**

| 方式 | 工具 | 特点 |
|------|------|------|
| 客户端统计 | 在访问层埋点计数 | 准确但侵入大 |
| Proxy 统计 | Redis Proxy（如 Codis、Twemproxy） | 集中统计 |
| Redis 内置 | `redis-cli --hotkeys`（仅 Redis 7+） | 简单轻量 |
| 流式监控 | Redis Exporter + Prometheus + 告警 | 实时性高 |

**处理策略**

1. **Key 分散**：将单一 Key 拆分为多个子 Key（`key_1`, `key_2` ... `key_n`）
2. **本地缓存**：在应用内存中加 L1 缓存，缓解 Redis 压力
3. **限流保护**：对热 Key 访问做单 Key 限流
4. **预热加载**：启动时主动加载热 Key 到缓存

### 缓存击穿/雪崩的容量预留

**击穿（Breakdown）**：热 Key 失效瞬间，大量请求穿透到数据库。

**雪崩（Stampede）**：大量 Key 同时过期，请求集中打到数据库。

**容量预留策略**

- **冗余容量**：数据库必须预留 **2~3 倍**正常容量的 CPU/连接数/IOPS
- **熔断降级**：缓存故障时快速降级到兜底数据（空对象、默认值）
- **互斥重建**：单 Key 重建使用分布式锁，防止击穿
- **过期时间分散**：基础过期时间 + 随机抖动（±10%），避免同时过期
- **永不过期 + 后台刷新**：对核心数据使用逻辑过期 + 异步刷新

**容量规划示意**

```
正常情况:  缓存承担 95% 流量，DB 承担 5%
击穿场景:  缓存瞬间降为 0%，DB 流量激增 20 倍
预留要求:  DB 容量 >= 5% × 20 = 100% 正常流量 + 安全余量
```
