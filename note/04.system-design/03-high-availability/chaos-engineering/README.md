## 1. 什么是混沌工程

混沌工程由 Netflix 于 2011 年（Simian Army 项目发布，2012 年正式命名为 Chaos Monkey）提出，标志性工具是 **Chaos Monkey**。其核心理念是：**与其在半夜被告警叫醒，不如主动在生产环境中制造故障，提前发现问题。**

### 混沌工程 vs 故障测试

| 维度 | 传统故障测试 | 混沌工程 |
|------|-------------|----------|
| 思维方式 | 验证已知问题 | 发现未知问题 |
| 执行方式 | 手动/预定义场景 | 自动化持续实验 |
| 范围 | 有限场景覆盖 | 生产环境随机注入 |
| 目标 | 确认系统能恢复 | 建立对系统的信心 |

### 核心原则

> "混沌工程不是制造混乱，而是通过受控实验揭示系统中的脆弱性。"

## 2. 核心原则

### 2.1 稳态假设(Steady State Hypothesis)

实验开始前定义系统的"正常"状态指标：
- **可用性指标**: 请求成功率 > 99.9%
- **延迟指标**: P99 延迟 < 200ms
- **吞吐量指标**: QPS > 10000

实验期间持续监控这些指标，验证系统是否偏离稳态。

### 2.2 爆炸半径(Blast Radius)

控制故障的影响范围，从小到大逐步扩大：

1. 开发环境
2. 测试环境
3. 预发环境
4. 生产环境（小流量）
5. 生产环境（全量）

**原则**: 始终确保爆炸半径在可控范围内，避免引发真正的线上事故。

### 2.3 最小化影响

- 选择低峰时段进行实验
- 优先影响无状态服务
- 准备好快速回滚/停止的机制(Kill Switch)
- 实验团队随时可中断实验

### 2.4 自动化与持续运行

手动注入故障不可持续，需要自动化：
- 定时/随机触发实验
- 自动收集实验数据
- 自动生成实验报告
- 与 CI/CD 集成

## 4. 故障注入类型

### 4.1 网络故障注入

| 故障类型 | 描述 | 模拟场景 |
|---------|------|---------|
| **网络延迟** | 增加请求延迟 100ms~5s | 网络抖动、跨地域调用 |
| **网络丢包** | 随机丢弃 1%~50% 数据包 | 弱网环境、拥塞 |
| **网络分区** | 切断两个服务间的通信 | 机房断网、DNS 故障 |
| **DNS 故障** | 篡改或延迟 DNS 解析 | DNS 服务异常 |

```bash
# 模拟网络延迟: 对 8080 端口注入 500ms 延迟
blade create network delay --time 500 --local-port 8080

# 模拟网络丢包: 30% 丢包率
blade create network loss --percent 30
```

### 4.2 容器故障注入

| 故障类型 | 描述 | 模拟场景 |
|---------|------|---------|
| **Pod Kill** | 随机删除 Pod 实例 | 容器异常退出、OOM |
| **Node 宕机** | 模拟节点不可用 | 物理机故障、云实例中断 |
| **容器 OOM** | 限制内存导致 OOM | 内存泄漏、突发流量 |

### 4.3 系统资源故障注入

| 故障类型 | 描述 | 模拟场景 |
|---------|------|---------|
| **CPU 压力** | 将 CPU 使用率提升至 80%~100% | CPU 密集型任务、死循环 |
| **磁盘满** | 消耗磁盘空间至 80%~95% | 日志暴涨、数据未清理 |
| **磁盘 I/O** | 增加磁盘读写延迟 | 磁盘老化、共享存储争抢 |
| **内存耗尽** | 消耗系统内存 | 内存泄漏、大对象堆积 |

### 4.4 应用层故障注入

| 故障类型 | 描述 | 模拟场景 |
|---------|------|---------|
| **方法延迟** | 在指定 Java 方法中注入延迟 | 下游依赖慢 |
| **方法异常** | 在指定方法中抛出异常 | 下游服务不可用 |
| **返回值篡改** | 修改方法返回值 | 数据不一致场景 |

## 5. 容灾演练实践流程

### 5.1 演练前准备

```
1. 确定演练目标
   ├── 验证主从切换是否自动
   ├── 验证服务降级是否生效
   └── 验证数据一致性是否保证

2. 评估影响范围
   ├── 涉及哪些服务
   ├── 影响哪些用户
   └── 是否在业务低峰期

3. 准备回滚方案
   ├── 一键停止实验
   ├── 手动回滚步骤
   └── 紧急联系人列表
```

### 5.2 演练执行步骤

```
Step 1: 建立基线
   └── 记录系统正常运行时的指标(延迟/QPS/错误率)

Step 2: 注入故障
   └── 按计划注入预设故障(如: 切断主数据库)

Step 3: 观察系统行为
   ├── 监控告警是否正常触发
   ├── 自动切换是否执行
   ├── 服务降级是否生效
   └── 错误率是否在预期范围内

Step 4: 恢复系统
   └── 停止故障注入，恢复正常运行

Step 5: 验证恢复结果
   └── 确认所有指标回到基线水平

Step 6: 总结复盘
   ├── 哪些预期内？
   ├── 哪些意料外？
   └── 需要改进什么？
```

### 5.3 典型演练场景

**场景一: 数据库主节点故障**
```
注入: 主数据库节点网络分区
预期: 
  ✅ 30s 内自动切换到从节点
  ✅ 数据不丢失(RPO < 1s)
  ✅ 服务中断时间 < 1min(RTO < 1min)
```

**场景二: 缓存集群大规模故障**
```
注入: 30% 缓存节点同时宕机
预期:
  ✅ 请求自动降级到数据库
  ✅ 数据库不被击穿(限流保护)
  ✅ 缓存恢复后自动回切
```

**场景三: 跨机房网络分区**
```
注入: 机房 A 与机房 B 之间的网络中断
预期:
  ✅ 双写一致性保证
  ✅ 流量自动切换到健康机房
  ✅ 网络恢复后数据自动同步
```

## 6. 适用边界

### 什么时候应该使用混沌工程

- 系统已具备完善的监控和告警
- 系统有自动化部署和快速回滚能力
- 团队对系统架构有清晰认知
- 核心业务链路已经经过压力测试

### 什么时候不应该使用混沌工程

| 情况 | 原因 |
|------|------|
| **没有监控体系** | 无法判断实验结果，实验失去意义 |
| **没有回滚机制** | 故障注入后可能无法恢复，风险极高 |
| **生产环境初期** | 系统本身不稳定，无法区分是混沌导致还是系统本身问题 |
| **关键业务高峰期** | 双 11/大促期间严禁任何混沌实验 |
| **合规/审计限制** | 金融/医疗等受监管行业需先获得审批 |
| **团队准备不足** | 没有应急流程和值班机制 |

### 成熟度模型

| 级别 | 实践 | 适用环境 |
|------|------|---------|
| **Level 0** | 无混沌工程 — 先建设监控告警 | — |
| **Level 1** | 手动故障注入 | 测试 / 预发环境 |
| **Level 2** | 自动化实验 | 预发 / 生产灰度 |
| **Level 3** | 持续随机注入 | 生产环境常态化 |
| **Level 4** | 游戏日（GameDay）文化 — 全员参与定期演练 | 全公司推广 |

> **关键提醒**: 混沌工程的最终目标不是证明系统有多脆弱，而是通过持续实验来**建立对系统的信心**。每一次实验都应该让系统变得更加可靠。
