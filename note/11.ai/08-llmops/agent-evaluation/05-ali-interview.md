# 阿里一面原题深度回答 · Agent Evaluation 设计

> **一句话答案**：阿里一面问"如何设计 Agent Evaluation"时，按 **6 大维度（40% 任务 + 20% 步骤 + 10% 工具 + 10% 成本 + 15% 满意度 + 5% 稳定性）+ 5 种方法组合 + 4 阶段 Pipeline（黄金集→CI→A/B→在线监控）+ 7 反模式** 答。

## 1. 原题

```
阿里一面：如果让你设计一个 Agent Evaluation（评测系统），
         你会如何设计评测指标与评测流程？
```

**面试官考点**：
- 是否理解 Agent vs LLM 评测的区别
- 是否有完整的评测指标体系
- 是否有评测流程的全链路思考（不只是单点）
- 是否有实战经验（知道反模式）

---

## 2. 满分回答模板（4 层递进）

### 2.1 第一层：场景区分（15 秒）

```
"Agent 评测 ≠ LLM 评测 —— Agent 是多步任务 + 工具调用 + 状态维护，
LLM 评测那套'GPT-judge 输出文本质量'不够，
必须有任务级别的成功判官。"
```

**关键**：展示对 Agent / LLM 区别有认知。

### 2.2 第二层：6 大指标（30 秒）

```
"我会设计 6 大评测指标：

1. 任务完成率（40%）：任务是否完成（全完成 / 部分 / 失败三级）
2. 步骤效率（20%）：实际步骤数 / 最优步骤数（≤ 1 即最优）
3. 工具使用准确率（10%）：选对工具 + 参数正确 + 调用顺序
4. 成本（10%）：单任务 token / 美元
5. 用户满意度（15%）：CSAT / NPS / 行为推断
6. 稳定性（5%）：同任务 10 次成功率方差

评分函数 = 加权求和（生产阈值 ≥ 0.80）"
```

**关键**：6 个不是凭空想——3 是用户满意度（15%）、5 是稳定性（5%）需口述理由。

### 2.3 第三层：5 种方法组合（30 秒）

```
"评测方法不是单一 —— 我会组合 5 种：

1. 自动化指标（80% 任务）：Schema 验证 + 字段匹配 + 工具断言
2. 黄金集（500 条核心场景）：离线核心测试
3. LLM-as-Judge（开放式任务）：GPT-4 / Claude / Gemini 多模型投票
4. A/B Test（生产环境）：5% 流量 → 7 天显著性检验
5. 用户模拟（LLM 模拟用户）：1000 并发压测

关键：避免单一方法偏差 —— LLM-as-Judge 必须多模型投票 + 截尾均值"
```

**关键**：不仅说 LLM-as-Judge，还要说**多模型投票**（避偏见）。

### 2.4 第四层：4 阶段 Pipeline（30 秒）

```
"评测不是发版前一次 —— 是 4 阶段 Pipeline：

阶段 1：黄金集构建（持续维护）→ 真实场景 500 条 + 难度分级
阶段 2：离线 CI 评估（PR 触发）→ 阈值 0.85 阻塞合并
阶段 3：A/B Test 灰度（5% 流量 7 天）→ 显著性检验 + 回滚机制
阶段 4：在线持续监控（漂移检测 + 告警 + 自动回滚）

关键：
- 黄金集每周补充失败用例
- CI 阈值 80-85%（不是 100%）
- 漂移检测：每日快测 + KS 检验

7 大反模式：
1. 只看单一指标
2. 上线前跑一次
3. A/B 时间不足
4. 黄金集不更新
5. 阈值不合理（100%）
6. 漂移无告警
7. 评测集太干净（无对抗）"
```

## 3. 评分话术（90 秒版本）

### 完整 90 秒回答

```
1. 【场景区分 · 15 秒】
"Agent ≠ LLM：多步 + 工具 + 状态，LLM 评测不够"

2. 【6 大指标 · 30 秒】
"40% 任务完成率 + 20% 步骤效率 + 10% 工具准确率 + 10% 成本 + 15% 满意度 + 5% 稳定性"

3. 【5 种方法 · 30 秒】
"自动化 + 黄金集 + LLM-as-Judge（多模型投票）+ A/B Test + 用户模拟"

4. 【4 阶段 Pipeline · 30 秒】
"黄金集 → CI 阈值 0.85 → A/B 7 天 → 漂移检测 + 自动回滚"

5. 【反模式 · 15 秒】
"7 反模式：单一指标 / 一次性 / A/B 太短 / 黄金集不更新 / 阈值 100% / 无告警 / 测试集太干净"
```

## 4. 面试官追问预案

### Q1：为什么任务完成率权重 40%？

```
任务完成率是终极目标 —— 用户要的是任务做完，不是过程高效。
其他指标（效率 / 成本）是约束，不是目标。
```

### Q2：单一 GPT-4 judge 不行吗？

```
不行，GPT-4 Judge 有 3 大偏见：
- 位置偏好（喜欢第一个）
- 长度偏好（喜欢长答）
- 同源偏见（judge 自己 output）

正解：3-5 个不同模型投票 + 截尾均值
```

### Q3：怎么防止评测过拟合？

```
3 大要点：
1. 黄金集持续更新（每周加失败用例）
2. 注入对抗样本（乱序字段 / 模糊 query）
3. 多维度指标 + LLM-judge（不是单一指标）
```

### Q4：A/B Test 用多少流量？

```
先 5%，无异常就 25%，然后 50%，最后 100%
最少 7 天（覆盖工作日 + 周末 + 高峰期）
显著性 p-value < 0.05 才算成功
```

### Q5：漂移检测怎么做？

```
每日快测 50 条 → 与历史 30 天数据 KS 检验
p < 0.05 = 显著漂移
自动告警 + 可选自动回滚
```

## 5. 反模式 · 5 个常见错

### ⚠️ 反模式 1：任务完成率 100% 是目标

- 错：努力让所有任务都完成
- 对：80-90% 完成率 + 0.85+ 综合分就行

### ⚠️ 反模式 2：单一 GPT-4 judge

- 错：图省事只用一个 judge
- 对：3-5 个 judge 投票

### ⚠️ 反模式 3：A/B Test 时间太短

- 错：跑 1-2 天就全量
- 对：≥ 7 天 + 显著性检验

### ⚠️ 反模式 4：黄金集从不更新

- 错：1 年前标的数据
- 对：每月 review + 每周补充

### ⚠️ 反模式 5：评测和业务脱钩

- 错：A/B 看 score 上升就上线
- 对：业务指标（CSAT / 留存） + Agent 分数 双指标

---

## 6. 一句话总结（阿里一面）

> **答：6 大指标（任务 40% + 步骤 20% + 工具 10% + 成本 10% + 满意 15% + 稳定 5%）+ 5 种方法（自动化 / 黄金集 / LLM-judge 多模型 / A/B / 用户模拟）+ 4 阶段 pipeline（黄金集 → CI 0.85 → A/B 7 天 → 漂移检测）。7 反模式必避。**
