# Salesforce Agentforce：让销售 Agent 自己挖线索、约会议、跟进客户

#销售 #CRM #办公 #客服

## 一、从 Copilot 到 Agentforce：Salesforce 的"换挡"

2024 年 9 月 Dreamforce 大会上，Salesforce 甩出 Agentforce 平台。Marc Benioff 给它的定义相当直接：这是"25 年来最令人兴奋的创新"，一个"数万亿美金"的数字劳务市场正在被打开。**这并不是 Salesforce 第一次在 AI 上喊口号，但确实是第一次把 Agent 从附加功能升级成产品主轴。**

Copilot 是"坐在 CRM 旁边的助手"：用户问，它答，再帮忙点开某个按钮。剧本由人写，它负责执行。Agentforce 把这个关系翻了过来：剧本由 Agent 自己写，把"理解→决策→行动"完整闭环在自己手里。Benioff 把这种状态称作 **"hard swap"**——把过去由人工坐席、销售代表、客服工程师承担的重复性工作，完整移交给 Agent。

## 二、销售 Agent 的五段式自主工作流

Agentforce 把销售流程拆成了几段 Agent 化的标准动作。**其中最关键的，是 Salesforce 直接提供了四个预构建 Agent，覆盖销售链条上 80% 的高频场景。** 五段动作包括：线索挖掘（Agent 自动调用联系人的历史互动与已有商机，输出"线索评分 + 跟进优先级"）；邮件与会议预约（Agent 根据联系人职位、邮件主题、时区动态调整话术，对方回复"有空"后直接调用日历工具发起邀请）；会议前情报准备与商机更新（会议前自动汇总公司动态、财报亮点、过去沟通摘要生成"一页纸情报卡"，会议结束后自动转写纪要、更新商机阶段、发跟进邮件）；跨系统协同（通过 MuleSoft 与 Slack 直接派单给法务、财务、客户成功团队）。

把五段连起来看，Agent 真正在做的不再是"替人跑流程"，而是"自己决定流程该怎么跑"。这是 Agent 与 RPA、Copilot 的本质区别。

## 三、Agentforce 的工程底座

Agentforce 能在销售场景里跑得起来，并不是只靠一个大模型在背后"猜"。四个工程机制构成它的底座：

**Atlas 推理引擎。** Agentforce 2.0 起引入，把企业数据、业务流程、业务逻辑连成上下文图谱。Agent 做决策时，先在图谱里查"这个客户在我们体系里处于什么位置、过去有过哪些承诺、合规上有哪些红线"。

**无代码 Agent Builder。** 销售运营人员不需要写 Python，只要用自然语言描述业务需求，系统就能自动生成对应的 Agent 配置。**这条机制把 Agent 的迭代周期从季度发版压缩到小时级调整，** 是 Agentforce 在客户内部大规模铺开的真正加速器。

**Data Cloud 加权限继承。** Agent 所有动作的数据源都在 Salesforce 自家的 Data Cloud 里，不外发到第三方模型。Agent 自动继承底层 CRM 的权限体系：一个销售代表下的 Agent 只能看到该代表名下的客户和商机。

**预构建加自定义 Agent 双轨。** Salesforce 提供销售、服务、营销、商业四个预构建 Agent；同时允许企业基于自身流程构建自定义 Agent。

四个机制叠在一起，构成的是一个可被部署、可被治理、可被审计的 Agent 运行时。

## 四、自家先吃自己的狗粮：真实数据说话

Salesforce 习惯用自己验证自己的产品。Agentforce 发布后，第一个被改造的就是它自家的客户自助平台 help.salesforce.com。公开数据显示（来源：scrm.com 2026 年 5 月深度评测）：每周处理约 32,000 次客户交互；人工介入次数从每周 10,000 次降至 5,000 次；83% 的问题由 Agent 独立解决。**这组数字验证了一件事：Agent 可以在没有人类逐句审批的情况下，独立跑完一个完整的客户交互闭环。** 外部客户层面，全球人力资源服务巨头 Adecco Group 用 Agentforce 构建了招聘匹配 Agent：根据候选人教育背景、技能证书等维度，自动与职位要求做多维度匹配，并生成面试安排建议。

## 五、启发：从"按剧本跑"到"自己判断下一步"

Agentforce 这个案例最大的启发，是它让我们看清了一件事：**RPA 和 Agent 之间，差的不是"更聪明"，而是"有没有决策权"。**

过去十年的企业自动化几乎都是 RPA 式的：把人工流程录制成"如果 A 触发，就执行 B"，机器人机械地按剧本跑。剧本一旦写死，遇到剧本里没写的情况，要么报错停下，要么强行推进留下烂尾。Agent 改变的是这一层的权力结构——它不再按剧本跑，而是先判断"现在是什么情况"，再决定"下一步该怎么走"。这是从"流程被代码固定"到"流程被模型动态生成"的范式跃迁。

落到企业 IT 团队的实践里，这条范式跃迁带来三个判断标准：能不能在没有剧本的情况下处理异常；决策是否可被审计（Agentforce 把这层审计做成了产品内置能力）；能否跨系统串起端到端流程（单点 Agent 价值有限，真正的杠杆点在系统串联）。

如果把视角再拉远一点，Agentforce 真正的信号是：**企业软件下一个十年的竞争点，已经从"功能多不多"切换到"Agent 能不能替我干活"。** 谁先把"判断下一步"的权力交给模型，谁就能在数字劳务的新市场上拿到定价权。
