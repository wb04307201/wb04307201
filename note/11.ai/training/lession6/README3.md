## 🔑 核心观点一句话总结

> 多智能体系统今天最有效的使用方式是：**让写作行为保持单线程，让额外 Agent 贡献智能而不是动作**。

---

## 📌 背景回顾

10 个月前，作者曾发文《别构建多智能体系统》，指出并行 Agent 会在风格、边界处理、代码模式上产生隐性冲突，导致系统脆弱。  
10 个月后，Cognition 已部署真正能跑在生产环境的多智能体系统，并总结出**真正有效的窄模式**。

---

## 🧭 上下文工程的核心原则

| 原则           | 说明                                                       |
|--------------|----------------------------------------------------------|
| **共享上下文**    | Agent 之间应尽可能看同一份资料、任务清单、计划文件，对目标有相同先验                    |
| **动作携带隐性决策** | 一个 Agent 的修改可能与其他并行 Agent 产生风格/逻辑冲突，因此多数系统实际是「只读子 Agent」 |

---

## 🚀 10 个月的关键变化

1. **模型更「本能 agentic」**：自然理解工具使用、上下文限制、协作者上下文提炼
2. **Agent 用量爆炸**：即使在保守企业客户中，6 个月内增长约 8 倍
3. **推力与拉力并存**：
    - 推力：用户自然尝试更多多智能体设置
    - 拉力：用量爆炸带来成本压力，催生「低成本实现前沿能力」的需求

---

## ✅ 三种真正有效的模式

### 模式 1️⃣：代码审查循环（简单到不该有效）

```
编码 Agent → 生成代码 → 审查 Agent（干净上下文）→ 发现 Bug → 迭代
```

🔹 **反直觉发现**：编码 Agent 和审查 Agent **不要事先共享上下文**，效果反而最好  
🔹 **原因**：
- 注意力数学：长上下文降低决策质量，审查 Agent 用短上下文更敏锐
- 无自我意识：两个 Agent 用同一模型但无 ego，能客观审查

🔹 **关键成功因素**：编码与审查 Agent 之间的**通信桥**，确保正确过滤反馈、防止循环、遵守用户意图

> 💡 经验：干净上下文在生成器-验证器循环中带来能力跃升，但与整体上下文的清晰沟通同样重要

---

### 模式 2️⃣：Smart Friend（大模型回来了）

```
主模型（小/快/便宜） ↔ 遇到难题时调用 ↔ Smart Friend（大/强/贵）
```

🔹 **核心挑战**：

| 挑战                 | 解决方案思路                       |
|--------------------|------------------------------|
| 弱模型如何知道自己到极限？      | 鼓励至少调用一次评估；领域特定预设指导          |
| 主模型分享什么上下文？        | 80/20 方案：直接 fork 完整上下文；问宽泛问题 |
| Smart Friend 如何回馈？ | 不编造答案，而是指导主模型获取必要信息；主动提供轨迹洞察 |

🔹 **实际表现**：
- 弱主模型 + 强 Friend：SWE 1.5 效果有限，1.6 开始有价值，本质是**训练问题**
- 前沿模型之间：确实有效，核心是**能力路由**而非难度升级

> 💡 经验：Smart Friend 在两个模型都足够强时有效；让弱主模型也能工作是开放问题

---

### 模式 3️⃣：Manager + Child Agent（规模化扩展）

```
Manager Devin → 分解大任务 → 多个 Child Devin 执行 → MCP 协调 → 综合报告
```

🔹 **工程挑战**：
- Manager 默认过于规范，缺乏深层上下文时适得其反
- Agent 假定共享状态，实际没有
- 跨 Agent 通信不会自然发生（模型未在此类环境训练）

🔹 **有效形态**：**Map-Reduce-and-Manage**
```
Manager 分裂工作 → Children 并行执行 → Manager 综合并报告
```
> ❌ 无结构的 Agent swarm 互相协商，大多是干扰而非增益

---

## 🎯 结论与开放问题

### ✅ 已验证的主线
> 多智能体系统最佳实践 = **单线程写作 + 多智能体贡献智能**

- 干净上下文的审查者 → 发现编码者忽略的 bug
- 前沿级 Smart Friend → 弥补弱主模型的微妙遗漏
- Manager 协调范围 → 避免决策碎片化

### ❓ 开放问题（本质都是通信问题）
1. 更弱的模型如何学会**何时升级**到更强模型？
2. 子 Agent 如何呈现一个**应该改变兄弟姐妹工作**的发现？
3. 如何在不同 Agent 间传递上下文，**既不丢失信息也不淹没接收者**？
