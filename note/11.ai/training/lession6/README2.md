# 🤖 多智能体协调模式：五种方法及应用指南

> 来源：[Claude Blog - Multi-agent coordination patterns](https://claude.com/blog/multi-agent-coordination-patterns)  
> 发布日期：2026年4月10日 | 阅读时间：5分钟

---

## 📋 核心原则

> **"从最简单的可行模式开始，观察其瓶颈，再逐步演进"**

选择协调模式时应基于**问题本身的结构**，而非追求技术上的"高级感"。关键考量：
- 任务如何分解？
- 智能体之间需要怎样的信息共享？
- 工作流是预定义的还是事件驱动的？

---

## 🔧 五种协调模式详解

### 1️⃣ Generator-Verifier（生成器-验证器）

**📐 工作原理**
```text
任务 → 生成器 → 初步输出 → 验证器 → [通过✓ / 反馈✗] → 循环迭代
```

**✅ 适用场景**
- 输出质量要求高，且有明确的评估标准
- 代码生成（一智能体写代码，另一智能体写测试并运行）
- 事实核查、合规验证、基于评分标准的批改
- 客服邮件回复生成与质量检查

**⚠️ 局限与风险**
- 验证器依赖明确的评判标准，否则会"走过场"
- 生成与验证能力需可分离（创意类任务可能不适用）
- 迭代可能陷入死循环 → 需设置最大迭代次数+降级策略

---

### 2️⃣ Orchestrator-Subagent（编排器-子智能体）

**📐 工作原理**
```text
主智能体（编排器）
├─ 接收任务 → 分解规划
├─ 部分任务自行处理
├─ 其他任务分发给子智能体
└─ 汇总子智能体结果 → 输出最终答案
```

**✅ 适用场景**
- 任务可清晰分解，子任务间依赖较少
- 自动化代码审查：安全检查、测试覆盖、代码风格、架构评估并行执行
- Claude Code 采用此模式：主智能体聚焦核心任务，子智能体并行探索

**⚠️ 局限与风险**
- 编排器成为信息瓶颈：子智能体间的关联信息需经编排器中转，易丢失细节
- 默认串行执行影响吞吐量 → 需显式并行化设计
- 编排器逻辑复杂度随场景增加而膨胀

---

### 3️⃣ Agent Teams（智能体团队）

**📐 工作原理**
```text
协调器
├─ 创建多个持久化工作智能体
├─ 任务放入共享队列 → 智能体自主认领
├─ 智能体多步骤独立工作，积累领域上下文
└─ 协调器收集结果，执行集成验证
```

**✅ 适用场景**
- 子任务独立、可并行、需长期上下文积累
- 大型代码库框架迁移：每个智能体负责一个服务，积累依赖关系/测试模式知识
- 需要"领域专家"持续优化的场景

**⚠️ 局限与风险**
- 智能体间难以共享中间发现 → 工作可能冲突
- 完成时间不一致 → 协调器需处理异步完成
- 共享资源（代码库/数据库）需防冲突机制（锁、版本控制、分区）

---

### 4️⃣ Message Bus（消息总线）

**📐 工作原理**
```text
智能体 ←发布/订阅→ 消息路由器 ←事件→ 外部系统
```
- 智能体订阅感兴趣的主题
- 路由器按规则/语义分发事件
- 新智能体可无缝接入，无需重构现有连接

**✅ 适用场景**
- 事件驱动流水线，工作流由事件触发而非预定义
- 安全运营自动化：告警→分类→分派调查→上下文增强→响应决策
- 智能体生态持续扩展的场景

**⚠️ 局限与风险**
- 事件链路复杂 → 调试和追踪困难，需完善日志与关联分析
- 路由准确性关键：错误分类/丢事件会导致"静默失败"
- LLM路由提供语义灵活性，但引入新的不确定性

---

### 5️⃣ Shared-State（共享状态）

**📐 工作原理**
```text
所有智能体 ↔ 读写 ↔ 共享存储（数据库/文档/文件系统）
```
- 无中心协调器，智能体完全自主
- 通过读取共享状态感知他人工作，写入发现供他人使用
- 终止条件：时间限制/收敛阈值/专用判断智能体

**✅ 适用场景**
- 协作型研究：多智能体从不同维度探索同一问题，发现需实时互通
- 需要构建演进式知识库的场景
- 要求去中心化、避免单点故障的系统

**⚠️ 局限与风险**
- 无显式协调 → 可能重复工作或方向冲突
- 系统行为由交互涌现，结果可预测性降低
- **反应式循环风险**：智能体A写→B读写→A再响应→无限消耗 → 需设计一级终止条件

## 🔄 模式选择决策指南

| 对比维度                      | 选择建议                                     |
|---------------------------|------------------------------------------|
| **编排器-子智能体** vs **智能体团队** | 子任务是否需要**跨调用保留上下文**？需要→选团队模式             |
| **编排器-子智能体** vs **消息总线**  | 工作流是否**预定义**？预定义→编排器；事件驱动/可变→消息总线        |
| **智能体团队** vs **共享状态**     | 智能体是否需要**实时共享中间发现**？需要→共享状态              |
| **消息总线** vs **共享状态**      | 信息流动是**离散事件**还是**累积知识**？累积→共享状态；需路由→消息总线 |
| **高可用性要求**                | 避免单点故障→优先考虑共享状态                          |

## 🎯 快速选型参考表

| 业务场景特征               | 推荐模式                  |
|----------------------|-----------------------|
| 🔍 质量关键输出 + 明确评估标准   | Generator-Verifier    |
| 📋 任务可清晰分解 + 子任务边界清晰 | Orchestrator-Subagent |
| ⚡ 并行负载 + 独立长周期子任务    | Agent Teams           |
| 📡 事件驱动流水线 + 智能体生态扩展 | Message Bus           |
| 🤝 协作研究 + 智能体实时共享发现  | Shared-State          |
| 🛡️ 要求无单点故障          | Shared-State          |

## 💡 实践建议

1. **起步推荐**：大多数场景从 **Orchestrator-Subagent** 开始，平衡能力与复杂度
2. **混合使用**：生产系统常组合多种模式
   - 例：主流程用编排器，协作子任务用共享状态
   - 例：消息总线路由事件，团队模式处理每类事件
3. **演进思维**：模式是构建模块，非互斥选择；根据瓶颈逐步升级架构
4. **终止设计**：尤其对迭代/共享状态模式，**提前设计终止条件**避免资源浪费
