# 📝 LLM Wiki：用大语言模型构建持久化个人知识库

> **作者**：Andrej Karpathy
> **来源**：[llm-wiki.md (GitHub Gist)](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)
> **主题**：LLM Wiki、知识管理、持久化知识库

---

## 🔑 核心观点

> **LLM Wiki 的本质：把"每次查询都重新检索"的临时模式，升级为"知识编译一次、持续复利"的持久模式。人类负责战略思考，LLM 负责战术执行，共同构建一个会"生长"的个人知识库。**

---

## 🔍 一、从"临时检索"到"知识积累"

### 当前 RAG 模式的问题

当前大多数人使用 LLM 处理文档的方式是 **RAG（检索增强生成）**：上传文件 → 提问时检索相关片段 → 生成答案。

| 问题 | 说明 |
|------|------|
| ❌ 每次从零开始 | 每次提问，LLM 都要重新"发现"知识 |
| ❌ 没有知识积累 | 无法形成系统性理解 |
| ❌ 复杂问题效果差 | 需要综合多份文档时，效果大打折扣 |

### LLM Wiki 的创新

| 特点 | 说明 |
|------|------|
| ✅ 增量构建 | LLM 不是临时检索，而是增量构建并维护一个持久的 Wiki |
| ✅ 结构化链接 | 这是一个结构化的、相互链接的 Markdown 文件集合 |
| ✅ 持续整合 | 新资料加入时，LLM 阅读、提取关键信息，并整合到现有 Wiki 中 |
| ✅ 持久化资产 | 知识被"编译"一次后持续更新，而非每次查询都重新推导 |

> 🎯 **你几乎不用亲手写 Wiki** —— LLM 负责所有撰写和维护工作。你负责：筛选资料、探索方向、提出好问题。

## 🏗️ 二、三层架构设计

```
┌─────────────────────────┐
│   1️⃣ 原始资料层 (Raw)    │
│   • 你策划的文档集合      │
│   • 文章/论文/图片/数据   │
│   • 不可变，只读          │
└────────┬────────────────┘
         │ 读取
         ▼
┌─────────────────────────┐
│   2️⃣ Wiki 层 (核心)      │
│   • LLM 生成的 Markdown  │
│   • 摘要/实体页/概念页    │
│   • 交叉引用/对比/综合    │
│   • LLM 完全负责维护      │
└────────┬────────────────┘
         │ 遵循
         ▼
┌─────────────────────────┐
│   3️⃣ 模式层 (Schema)     │
│   • CLAUDE.md / AGENTS.md│
│   • 定义 Wiki 结构规范    │
│   • 指导 LLM 的工作流程   │
│   • 你与 LLM 共同演进     │
└─────────────────────────┘
```

### 各层职责

| 层级 | 维护者 | 内容 | 特点 |
|------|--------|------|------|
| **原始资料层** | 你 | 文章、论文、图片、数据 | 不可变，只读 |
| **Wiki 层** | LLM | 摘要、实体页、概念页、交叉引用 | LLM 完全负责 |
| **模式层** | 你 + LLM | 结构规范、工作流程 | 共同演进 |

## ⚙️ 三、三大核心操作

### 1️⃣ 摄入（Ingest）：添加新知识

```text
你：放入新资料 → 告诉 LLM 处理
↓
LLM：
  • 阅读资料，与你讨论要点
  • 撰写摘要页并加入 Wiki
  • 更新索引和相关内容页
  • 在日志中记录变更
↓
结果：单个资料可能更新 10-15 个 Wiki 页面
```

> 💡 **建议**：初期逐个摄入并参与审核，熟练后可批量处理

### 2️⃣ 查询（Query）：获取答案

```text
你：提出问题
↓
LLM：
  • 搜索相关 Wiki 页面
  • 综合信息生成带引用的答案
  • 答案形式：文本/表格/幻灯片/图表
↓
关键：优质答案可反向存入 Wiki，成为新知识资产
```

### 3️⃣ 检查（Lint）：维护健康

定期让 LLM 执行"健康检查"：

| 检查项 | 说明 |
|--------|------|
| 🔍 矛盾检测 | 发现页面间的矛盾 |
| 🔄 过时标记 | 标记被新资料推翻的旧观点 |
| 🔗 链接完整性 | 识别孤立页面或缺失的交叉引用 |
| ❓ 探索建议 | 建议可深入探索的新问题 |
| 🌐 资料补充 | 推荐可补充的外部资料 |

## 🗂️ 四、索引与日志：导航系统

### 📑 index.md（内容导向）

- Wiki 内容的目录清单
- 每页包含：链接 + 一句话摘要 + 元数据（日期/来源数等）
- 按类别组织：实体、概念、来源等
- LLM 每次摄入后自动更新
- 查询时先读索引定位，再深入阅读

### 📋 log.md（时间导向）

- 追加式操作日志：摄入/查询/检查记录
- 统一前缀格式便于解析：`## [2026-04-02] ingest | 文章标题`
- 提供 Wiki 演进的时间线
- 帮助 LLM 理解近期工作内容

## 💡 六、为什么这个模式有效？

### 人类放弃 Wiki 的根本原因

> 维护知识库最累的不是阅读和思考，而是**琐碎的文书工作**：更新引用、保持摘要最新、标记矛盾、维护一致性……随着页面增多，维护成本呈指数级增长。

### LLM 的优势

| 能力 | 说明 |
|------|------|
| ✅ 不会厌倦 | 重复性工作无影响 |
| ✅ 不会忘记更新 | 交叉引用自动维护 |
| ✅ 一次修改多文件 | 一次操作可修改 15 个文件 |
| ✅ 维护成本趋零 | Wiki 能持续保持健康 |

### 分工明确

```text
👤 人类：策划资料 → 引导分析 → 提出好问题 → 思考意义
🤖 LLM： 总结 → 引用 → 归档 → 维护一致性 → 执行琐事
```

## 🧭 八、哲学渊源：从 Memex 到 LLM Wiki

这个理念与 **Vannevar Bush 1945 年提出的 Memex** 一脉相承：

| Memex 特征 | LLM Wiki 对应 |
|-----------|-------------|
| 个人策划的知识库 | 你筛选的原始资料 |
| "联想路径" | 交叉引用和链接 |
| 私有、主动维护 | 本地 Markdown + Git |

**Bush 当年无法解决的问题**：谁来承担繁琐的维护工作？
**LLM 给出的答案**：让我来。

## ⚠️ 九、注意事项

1. **本文档是抽象设计，非具体实现** — 目录结构、命名规范、页面格式、工具链都需根据你的领域定制

2. **模块化选择，按需采用**
   - 纯文本资料？无需图片处理
   - Wiki 规模小？index.md 足够，无需搜索引擎
   - 不需要幻灯片？忽略 Marp 即可

3. **与你的 LLM 协作迭代** — 将此模式分享给你的 Agent，共同定制适合你的版本

4. **警惕幻觉污染** — LLM 可能将错误信息写入 Wiki 并传播。关键事实保留来源引用，定期执行"检查"操作
