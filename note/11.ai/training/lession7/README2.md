## 📌 研究背景与动机

随着大语言模型（LLM）驱动的AI代理（如Claude Code、Codex、OpenHands）越来越多地被部署为**长期运行的系统**，它们需要能够自主构建并持续演化定制化的软件，以在动态环境中有效交互。

然而，**现有基准测试存在关键缺陷**：
- 仅评估孤立的、一次性的编码任务（如函数补全、单点问题修复）
- 忽略了真实软件开发中的**时间依赖性**和**技术债务累积**
- 无法衡量代理在长期维护、错误传播控制等方面的能力

> 💡 **核心问题**：如何让代理像人类开发者一样，在持续演化的代码库中"边开发、边维护、边迭代"？

## 🔧 核心贡献一：DeepCommit 自动化流水线

为解决上述问题，作者提出了 **DeepCommit**——一个全自动的智能代理流水线，用于从嘈杂的Git提交历史中重建**可验证的软件演化路径**。

### 关键创新：里程碑（Milestone）粒度
| 粒度级别 | 优点 | 缺点 |
|---------|------|------|
| Release级 | 语义完整 | 过于粗粒度，丢失演化细节 |
| Commit级 | 细粒度 | 噪声多、依赖关系混乱 |
| **Milestone级（本文采用）** | ✅ 平衡功能连贯性与演化感知 | 需要智能重组提交 |

### DeepCommit 三阶段流程：

```
📥 阶段1：提交历史预处理
   ├─ 过滤非源代码文件（文档、CI配置等）
   ├─ 构建提交级DAG（基于git blame追踪行级依赖）
   ├─ 提取符号级变更与文件共现统计

🧠 阶段2：里程碑DAG构建（LLM代理驱动）
   ├─ 种子发现：识别具有高影响力的基础提交
   ├─ 里程碑聚合：将语义/结构相关的提交聚类
   ├─ 依赖推断：基于代码依赖+代理推理确定先后关系
   ├─ 粒度优化：拆分过大里程碑，合并过小里程碑

🧪 阶段3：运行时环境解析
   ├─ 按拓扑顺序重放提交，重建可执行代码状态
   ├─ 自动生成Docker环境与测试配置
   ├─ 过滤不稳定测试，确保评估可靠性
```

✅ **质量保障**：测试收集成功率 >85%，构建失败率 <0.1%，测试稳定性 >99.9%

## 🎯 核心贡献二：EvoClaw 基准测试

基于DeepCommit构建的 **EvoClaw** 是首个面向**持续软件演化**的AI代理评估基准。

### 基准特点：
- 📦 **98个** 人工验证的里程碑任务
- 🔗 **7个** 真实开源项目的演化路径（Milestone DAG）
- 🌐 覆盖 **5种编程语言**：Go、Rust、Java、TypeScript、Python
- 🔀 **124条** 跨里程碑依赖关系
- 💰 单次完整评估成本约 **$500**（使用前沿模型）

### 评估框架设计：
```
🔄 持续任务评估流程：
1️⃣ 外部规划器根据DAG动态解锁可执行任务
2️⃣ 代理在持久化代码库中连续实现功能
3️⃣ 每完成一个里程碑，快照隔离评估（不影响代理工作环境）
4️⃣ 依赖满足后自动解锁下游任务
```

### 评估指标（创新设计）：
| 指标 | 含义 | 计算公式 |
|-----|------|---------|
| **Recall** | 新功能实现完整度 | 已修复的F2P测试数 / 总要求测试数 |
| **Precision** | 修改安全性（防回归） | 修复数 / (修复数 + 引入的回归数) |
| **Score** | 综合得分（F1调和平均） | 2×P×R / (P+R) |
| **Resolve Rate** | 完全通过率 | 所有测试均通过的里程碑比例 |

> 🎯 F2P（Fail-to-Pass）：验证新功能；P2P（Pass-to-Pass）：防止回归

## 📊 实验结果与关键发现

### 实验设置：
- 🔍 评估 **12个前沿LLM** + **4个代理框架**（Claude Code、Codex CLI、Gemini CLI、OpenHands）
- ⚖️ 对比两种模式：**独立任务评估**（类似SWE-bench）vs **持续演化评估**（EvoClaw）

### 🔥 核心发现 #1：性能断崖式下降
| 评估模式 | 最佳模型得分 | 完全解决率 |
|---------|-------------|-----------|
| 独立任务 | >80% | ~40% |
| **持续演化** | **≤38.03%** | **≤13.37%** |

> 📉 Claude Opus 4.6在持续模式下得分最高（38.03%），但仍不足独立模式的一半

### 🔥 核心发现 #2：Recall线性增长，Precision快速饱和
```
📈 Recall（实现能力）：随任务推进近似线性增长
📉 Precision（防退化能力）：早期即饱和，后期几乎不提升

💡 结论：代理"会写新功能"，但"不会保护旧代码"
```

### 🔥 核心发现 #3：错误雪崩效应
- 早期未修复的bug会通过依赖链传播
- 下游任务在"带病代码"上开发，难度指数级上升
- 后期里程碑解决率趋近于0

### 🔥 核心发现 #4：成功代理的行为特征
✅ 有效策略：
- 主动探索代码库结构
- 严格执行测试验证
- 谨慎处理依赖关系

❌ 失败模式：
- 盲目试错修改
- 跳过测试直接提交
- 忽视架构一致性

## 🧩 案例洞察：代码质量差距

通过定性分析，发现代理生成的补丁存在三类典型质量问题：

1. **职责边界错位**：逻辑正确但放在错误的抽象层级
2. **散弹式修复**：用全局抑制替代精准修复，破坏接口契约
3. **API签名退化**：为通过测试牺牲类型安全，积累技术债务

> ⚠️ 这些质量缺陷在自动化测试中"不可见"，但会严重影响长期可维护性

## 🌟 研究意义与影响

### 学术价值：
- 首次系统性定义"持续软件演化"评估范式
- 提出Milestone DAG作为演化建模的新抽象
- 揭示当前AI代理在长期任务中的根本性局限

### 实践启示：
- 部署长期运行的代理系统时，需特别关注**错误累积控制**
- 代理设计应增强**架构感知**与**回归预防**能力
- 评估标准应从"能否完成任务"转向"能否可持续演化"

## 🔚 总结

> EvoClaw揭示了一个关键现实：**当前最先进的AI代理在孤立任务上表现优异，但在持续软件演化场景中仍面临严峻挑战**。真正的"自主软件工程师"不仅需要编码能力，更需要系统思维、长期规划和技术债务管理意识。这一工作为下一代代理系统的研发指明了方向，也为评估框架的设计树立了新标准。

*论文核心贡献可概括为：提出新问题（持续演化评估）、新方法（DeepCommit流水线）、新基准（EvoClaw）、新发现（能力缺口与失败模式），推动AI for Software Engineering领域向更真实、更长期的场景迈进。*
