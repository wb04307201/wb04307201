# 📝 Tony Kipkemboi 推文整理：《区分 Agent Frameworks 与 Agent Harnesses》

> **作者**：Tony Kipkemboi (@tonykipkemboi)  
> **来源**：X (Twitter) [[1]]  
> **主题**：AI Agent 开发中的两个关键概念辨析

---

## 🔍 核心观点

如果你经常听到 **"agent framework"（代理框架）** 和 **"agent harness"（代理 harness/托管平台）** 这两个术语却分不清区别，那么这篇内容就是为你准备的。这两个词听起来可以互换，但**实际上有本质差异**。

作者曾就职于 CrewAI（一个典型的 agent framework），因此对两者的边界有切身体会。

---

## 📊 概念谱系：从自由到约束

如果把 Agent 开发想象成一条光谱，不同工具位于不同位置：

### 🔹 最左端：原始代码（Raw Code）
- 直接调用 LLM API
- 手动管理状态、记忆、工具调用
- **优点**：完全灵活，零抽象损耗
- **缺点**：完全责任，开发成本高

### 🔹 中间层：Agent Framework（代理框架）
> 代表：CrewAI、LangChain、AutoGen 等

- 提供结构化抽象，但保留决策空间
- 你需要：
   - 选择记忆系统（memory system）
   - 配置工具（tools）
   - 定义编排逻辑（orchestration logic）
- 框架对"组件如何连接"有预设意见，但**模块化设计**允许你替换组件
- **适合**：需要定制能力、理解底层逻辑的开发者

### 🔹 最右端：Agent Harness（代理托管平台）
> 代表：Anthropic Cowork、OpenAI Frontier 等一体化平台

- **高度预设、开箱即用**
- 你只需：
   - 填入 API Key
   -  optionally 连接几个外部工具
- 其余全部由平台决定：
   - 记忆管理 ✅
   - 上下文控制 ✅
   - Agent 执行循环 ✅
   - 安全检查机制 ✅
- **适合**：追求快速交付、不想处理基础设施的团队

---

## 💡 关键对比总结

| 维度 | Agent Framework | Agent Harness |
|------|----------------|---------------|
| **抽象程度** | 中等 | 极高 |
| **灵活性** | 高（可替换组件） | 低（黑盒预设） |
| **开发责任** | 你负责架构设计 | 平台负责大部分 |
| **学习曲线** | 较陡 | 平缓 |
| **适用场景** | 定制化 Agent 系统 | 快速业务集成 |

---

## 🎯 作者建议

> 理解你使用的工具在"意见化光谱"上的位置，至关重要——因为它直接决定了：
> - **你需要负责什么**
> - **工具替你处理什么**

选择框架还是 harness，没有绝对优劣，关键在于：
1. 你的团队技术能力
2. 项目对定制化的需求
3. 交付速度与长期维护的权衡

---

## 🔗 延伸思考

随着 2025-2026 年 AI 基础设施逐渐成熟，Agent 开发工具正在经历分化：
- 早期"框架热"填补了模型能力与工程落地之间的鸿沟
- 如今企业更关注**可控性、可维护性与合规性**，harness 类工具需求上升

> 正如作者所言：*"The gap is closing."*（鸿沟正在弥合）
