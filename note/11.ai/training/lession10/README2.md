# 📝 文章整理：Demystifying Evals for AI Agents
*Anthropic 工程博客 | 2026年1月*

---

## 🔑 核心观点摘要

> **好的评估（evals）能让团队更自信地发布 AI Agent**。没有评估，团队容易陷入"生产环境救火"的被动循环；有了评估，问题能在影响用户前被提前发现，价值随 Agent 生命周期持续复利增长。

---

## 📚 一、评估的基本结构

### 核心概念定义
| 术语                         | 含义                            |
|----------------------------|-------------------------------|
| **Task/Problem**           | 单个测试用例，有明确输入和成功标准             |
| **Trial**                  | 对同一任务的一次执行尝试（因模型输出有随机性，需多次运行） |
| **Grader**                 | 评分逻辑，可包含多个断言/检查项              |
| **Transcript/Trace**       | 完整执行记录：输出、工具调用、推理过程、中间结果等     |
| **Outcome**                | 环境中的最终状态（如数据库是否真正创建了订单）       |
| **Evaluation Harness**     | 端到端运行评估的基础设施                  |
| **Agent Harness/Scaffold** | 让模型作为 Agent 运行的系统框架           |
| **Evaluation Suite**       | 一组相关任务的集合，用于测试特定能力            |

### 评估类型演进
```
单轮评估 → 多轮评估 → Agent 评估（最复杂）
```
- Agent 评估的难点：多轮工具调用、状态修改、错误累积、模型可能找到"创意解法"超越静态评估预期

---

## 🎯 二、为什么要构建评估体系？

### 常见问题（无评估时）
- 🔁 用户反馈"变差了"但无法验证
- 🔍 调试靠猜，修复一个 bug 可能引入新问题
- 📉 无法区分真实回归和随机波动
- 🚫 无法自动化测试变更、衡量改进效果

### 评估的核心价值
| 阶段       | 价值               |
|----------|------------------|
| **早期开发** | 明确"成功"的定义，消除需求歧义 |
| **迭代优化** | 快速验证变更效果，防止回归    |
| **模型升级** | 快速评估新模型能力，缩短适配周期 |
| **长期运营** | 自动追踪延迟、成本、错误率等指标 |

> 💡 **案例**：Claude Code 从手动测试 → 窄领域评估（简洁性/文件编辑）→ 复杂行为评估（过度工程），评估体系支撑了规模化迭代。

## 🛠️ 三、如何评估 AI Agent？

### 1️⃣ 三类 Grader 对比

| 类型             | 方法示例                   | 优势                   | 局限             |
|----------------|------------------------|----------------------|----------------|
| **代码型 Grader** | 字符串匹配、单元测试、静态分析、结果验证   | 快速/便宜/客观/可复现/易调试     | 对合法变体敏感、缺乏细微判断 |
| **模型型 Grader** | 评分标准、自然语言断言、对比评估、多评委共识 | 灵活/可扩展/捕捉细微差别/处理开放任务 | 非确定性/成本高/需人工校准 |
| **人工 Grader**  | 专家评审、众包、抽样检查、A/B 测试    | 金标准质量/匹配专家判断/用于校准模型  | 昂贵/缓慢/需专家资源    |

> ✅ **最佳实践**：组合使用，按任务选择；评分可采用加权、二元或混合方式

### 2️⃣ 能力评估 vs 回归评估
| 类型                    | 目标                 | 预期通过率       | 用途     |
|-----------------------|--------------------|-------------|--------|
| **能力评估 **(Capability) | "Agent 能做好什么？"     | 初始较低，留出提升空间 | 指引优化方向 |
| **回归评估 **(Regression) | "Agent 还能做好以前的事吗？" | 接近 100%     | 防止能力倒退 |

> 🔄 能力评估通过率高后，可"毕业"为回归评估套件

### 3️⃣ 不同类型 Agent 的评估策略

#### 🔧 编码 Agent
- **核心**：明确任务 + 稳定测试环境 + 代码验证
- **基准参考**：SWE-bench Verified（GitHub issue 修复）、Terminal-Bench（端到端技术任务）
- **Grader 组合**：单元测试（正确性）+ LLM 评分（代码质量）+ 工具调用验证

#### 💬 对话 Agent
- **挑战**：交互质量本身是评估对象
- **方法**：用第二个 LLM 模拟用户 + 多维度评分（任务完成 + 交互质量 + 语气）
- **基准参考**：τ-Bench / τ2-Bench（零售/航空等场景的多轮模拟）

#### 🔍 研究 Agent
- **难点**：答案主观性强、事实动态变化、输出开放
- **策略**：组合 Grader（来源可信度 + 覆盖度 + 事实核查 + LLM 综合评分）
- **关键**：定期用专家判断校准 LLM 评分器

#### 🖥️ 计算机使用 Agent
- **特点**：通过 GUI 操作软件，需真实/沙箱环境
- **验证**：URL/页面状态检查 + 后端数据验证 + 文件系统/数据库状态确认
- **优化**：平衡 DOM 操作（快但 token 多）与截图操作（慢但节省 token）

### 4️⃣ 处理非确定性：pass@k 与 pass^k

| 指标 | 含义 | 适用场景 |
|------|------|----------|
| **pass@k** | k 次尝试中**至少一次**成功的概率 | 只需一个可行解的场景（如代码生成） |
| **pass^k** | k 次尝试中**全部成功**的概率 | 要求稳定可靠的场景（如客服 Agent） |

> 📊 示例：单次成功率 75%，则 pass@3 ≈ 98%，pass^3 ≈ 42% — 指标选择取决于产品需求

## 🗺️ 四、从零到一：评估体系建设路线图

### 📋 阶段 1：收集初始任务集
1. **尽早开始**：20-50 个真实失败案例即可起步，效果显著
2. **从手动测试转化**：开发时的手动检查、用户反馈的 bug、支持工单
3. **编写无歧义任务**：
    - 两个专家应能独立得出相同判定
    - 每个任务配参考解决方案（验证任务可解 + Grader 正确）
    - 0% 通过率通常说明任务/评分器有问题，而非模型不行
4. **平衡问题集**：同时测试"应该触发"和"不应触发"的场景，避免优化偏差

### ⚙️ 阶段 2：设计评估框架与 Grader
5. **构建稳定环境**：
    - 评估 Agent 应与生产环境一致
    - 每次 Trial 从干净环境开始，避免状态污染
6. **精心设计 Grader**：
    - 优先确定性 Grader，必要时用 LLM，人工用于校准
    - **避免过度约束执行路径**：评估"产出"而非"过程"，保留模型创意空间
    - 支持**部分得分**：区分"部分成功"与"完全失败"
    - LLM 评分器需与专家校准，提供"未知"选项避免幻觉
    - 防作弊设计：确保通过评估需真正解决问题

### 🔁 阶段 3：长期维护与使用
7. **定期阅读 Transcript**：
    - 验证 Grader 是否公平、任务是否清晰
    - 区分"模型错误"与"评估缺陷"
8. **监控评估饱和**：
    - 100% 通过率 = 失去改进信号
    - 接近饱和时需设计更难任务或新评估维度
9. **持续维护评估套件**：
    - 设立专门团队维护基础设施
    - 产品/领域专家贡献任务、运行评估
    - 实践"评估驱动开发"：先定义评估，再迭代实现

## 🔍 五、评估与其他方法的协同

| 方法                  | 优势                    | 局限                   | 最佳使用时机         |
|---------------------|-----------------------|----------------------|----------------|
| **自动化评估**           | 快速迭代/可复现/无用户影响/每提交可运行 | 前期投入大/需持续维护/可能脱离真实场景 | 预发布、CI/CD、模型升级 |
| **生产监控**            | 反映真实用户行为/发现评估遗漏问题     | 被动响应/信号噪声大/缺乏标准答案    | 上线后持续监控        |
| **A/B 测试**          | 衡量真实用户结果/控制混杂因素       | 周期长/需足够流量/难分析根因      | 重大变更验证         |
| **用户反馈**            | 发现意外问题/带真实案例/关联产品目标   | 稀疏/偏向严重问题/缺乏解释/非自动化  | 持续收集+定期分析      |
| **人工阅读 Transcript** | 建立失败模式直觉/捕捉细微问题/校准标准  | 耗时/难扩展/覆盖不均/主观性      | 每周抽样+深度排查      |
| **系统性人工研究**         | 金标准判断/处理主观任务/校准模型评分   | 昂贵/周期长/需专家/需处理评分者差异  | 校准评分器/评估主观输出   |

> 🧀 **瑞士奶酪模型**：没有单一方法能捕获所有问题，多层方法组合才能构建可靠保障

## 💡 关键行动建议（可直接执行）

```
✅ 立即行动
• 从 20-50 个真实失败案例开始构建评估集
• 为每个任务编写无歧义描述 + 参考解决方案
• 优先使用确定性 Grader，LLM Grader 需人工校准

✅ 中期建设  
• 构建隔离、稳定的评估环境
• 设计多维度评分（任务完成 + 过程质量 + 效率）
• 同时建设能力评估和回归评估套件

✅ 长期运营
• 定期阅读 Transcript 验证评估有效性
• 监控评估饱和，及时更新任务难度
• 让产品/客户团队参与评估任务贡献
• 组合自动化评估 + 生产监控 + 人工校准
```

## 📌 一句话总结

> **评估不是开发的负担，而是加速剂**：早期投入构建评估体系，能将"感觉变差了"转化为可测量的改进目标，让 Agent 开发从"救火模式"转向"数据驱动迭代"。
