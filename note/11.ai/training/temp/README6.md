# RAG 不再是默认答案：2025 年大模型知识接入的 5 条演进路线与决策指南

## 引言：你可能在解决一个已经不存在的问题

当接到一个“给 LLM 接入私有知识”的需求时，你的第一反应是什么？

直觉答案往往是：**上 RAG！** 建向量库、切 chunk、做 embedding、搞检索……

但在 2025 年的今天，行业里一个心照不宣的事实是：**RAG 已经不再是默认答案了。**

从 2023 年的“接入知识 = 建向量库”，到 2025 年的“RAG = 工具箱里的一个工具”，RAG 范式正在被重新划定边界。它没有死，而是**退位**了——从“默认答案”变成了“备选方案之一”。

很多时候，**你可能在解决一个已经不存在的问题。** 技术进步让某些问题自然消解，而不是被解决。在动手写代码之前，请先想清楚问题，再选工具。

---

## 第一章：决策前置——选方案前先问 2 个问题

很多团队接到需求后的操作流程是典型的“本末倒置”：先开调研向量库（Pinecone/Weaviate），再定 embedding 模型（OpenAI/BGE），最后才回头看用户问什么。

正确的做法是，在选任何方案之前，**先问 2 个问题。问完，答案出来一半。**

### Q1：数据能不能塞进 context？（看数据规模）
注意，是“能不能”，不是“想不想”。很多人直觉觉得“肯定塞不下”，担心成本、token、性能，从而绕开长上下文。

**算账才知道：**
*   **Claude Sonnet 4.5**：1M tokens ≈ 75 万中文字
*   **Gemini 1.5 Pro**：2M tokens ≈ 150 万中文字

**实际换算**：200 份文档 × 平均 3,000 字 = 60 万字 → **全部装下**。
只要总字数在 150 万以内，现在的模型完全有能力“一口吞下”。

### Q2：查询是精确匹配还是语义模糊？（看查询模糊度）
这一题决定要不要 embedding。
*   **精确匹配（EXACT）**：找关键词（代码、配置、日志、API）。关键词查询是天然准的，向量化这玩意绕一大圈，不如 `grep` 一把梭。
*   **语义模糊（SEMANTIC）**：找意思（描述 ≠ 命名）。比如用户问“用户功能在哪实现的？”，代码里叫 `UserProfileController.handleQuery()`。这时候 embedding 才有用武之地。

### 决策矩阵：数据规模 × 查询模糊度
把这两个问题画成一张图，所有方案都挂在这张图上：
*   **小数据 + 语义**：长 Context + Caching
*   **大数据 + 语义**：重型 RAG (Hybrid + Rerank)
*   **小数据 + 精确**：直接问 LLM / grep
*   **大数据 + 精确**：Agent / SQL / grep

## 第二章：5 条技术路线深度解析

### 路线一：长 Context + Prompt Caching（2025 甜蜜区）
**定位**：最朴素，也最被低估。
**适用场景**：< 200 份文档，相对稳定的知识库。

**核心优势：成本与工程的降维打击**
*   **月成本**：$120（vs RAG 全套 $840，**-86%**）
*   **工程量**：0.5 人天（vs RAG 25 人天，**50x 提速**）
*   **二次命中**：Prompt Caching 让第二次调用成本 **-90%**（首次 $1.00，命中 $0.10）。

**陷阱：针在草堆（Lost in the middle）**
长 context ≠ 等同短 context。关键信息如果在中间地带，召回率会显著掉。
**解法**：重要文档放两头；或者控制在 200 份文档的“甜蜜区”内。

**结论**：几百份文档以内，直接塞 context → **别建 RAG**。

### 路线二：生产级 RAG（塞不下了，才到主场）
**定位**：重型武器，三条件缺一不可（大数据量、语义模糊、没法塞 context）。
**适用场景**：10K+ 份文档量级。

**反共识：Baseline RAG 别上线**
网上教程教的那套（切块→Embedding→向量库→检索→生成），做 demo 可以，做产品就是给自己挖坑。用户稍微绕一点的问题，立刻翻车。

**生产级 RAG 的三大必做升级**：
1.  **Hybrid Search（混合检索）**：单向量检索是 2023 的玩法。BM25（关键词命中） + Vector（语义近似）两套都召回，短板补齐。
2.  **Reranker（重排序）**：性价比最高的一次升级，没有之一。召回 100 个 chunk，精选 Top 5。**加 reranker 比换 embedding 模型提升大 10x**（+25pp vs +2pp）。
3.  **Contextual Retrieval（上下文检索）**：Anthropic 2024 提案。为每个 chunk 生成上下文摘要再 embed，召回失败率降为**原来的 1/3**。

### 路线三：Agentic Retrieval（2024-2025 最大范式转变）
**定位**：重头戏。从“retrieve → generate”的静态流水线，变为“Agent loop”（模型自己决定查不查、查什么、查够了没）。

**代码场景全面碾压**：
内部实测代码任务完成率：传统代码 RAG **42%** vs Agent (grep + read) **89%**。Cursor、Claude Code 都走 agent 路线。
**原因**：代码有结构，关键词天然准；长 context 让读完整文件变可行；多跳推理几条 bash 就出来。

**陷阱**：
1.  **贵**：每一步 = 一次完整 LLM 调用。10步搜索 = $0.40/次（vs RAG $0.02）。在线交互场景延迟和成本都顶不住。
2.  **弱模型灾难**：弱模型 agent 化基本是灾难（不会判断“够了”、不会换策略、死循环）。门槛 ≥ Sonnet / GPT-4o 级别。

### 路线四：结构化数据走 SQL
**定位**：道理简单，但太多团队搞反方向。
**反模式**：业务表 dump 成文档 → 进向量库（南辕北辙，精度爆死）。
**正解**：Text-to-SQL。让模型生成 SQL，调 API，查内部系统。
**工具链**：Vanna、LlamaIndex SQL、或自己拼 schema + few-shot。

### 路线五：Deep Research（最重一档）
**定位**：多轮检索，给复杂研究类问题用，一次查询 = 一篇报告。
**架构**：Planner（拆解问题） → Search（独立检索） → Reader（理解文档） → Aggregator（综合写报告）。
**成本**：一次查询 $5 - $20（40+ LLM调用，30+ Web Search）。延迟 5-15 分钟。
**适用场景**：尽调报告、学术综述（用户付得起 + 问题足够复杂）。OpenAI、Google、Perplexity 的深研产品，**没一个是 RAG**。

## 第三章：避坑指南——80% 的翻车不是架构问题

架构升级到顶配（Hybrid + Rerank + Contextual + Agent loop + 长 context），效果还是稀烂？问题出在两件事：

### 1. 没建 Eval（评估体系）
**案例**：改了一个 prompt，测试集（5个query）+18%，全员庆祝；下周用户报 bug，真实日志整体召回率 -15%。没人知道，因为没人测。
**铁律**：**Eval 比换 embedding 模型重要 10x。没 eval，任何升级都是赌。**
**行动**：第一天就开始建。用 RAGAS/Phoenix/TruLens，从真实日志挖 100 个 query 标预期，每次改东西都跑一遍。

### 2. 数据质量 < 算法
文档脏、表格碎、版本乱、过时数据没清，架构再先进也救不回。
**Chunking 之前的“脏活”**：
*   **格式归一**：md/html/pdf → 统一 markdown。
*   **表格单独处理**：不切块，单独建索引或走 text-to-SQL。
*   **Metadata 设计**：来源、时间、部门、版本（**比 embedding 还重要**）。
*   **冲突/过时机制**：新版覆盖旧版，过期归档。

**残酷真相**：三个月的项目，算法工作（embedding/rerank）只需 ~2 周，**数据洗涮需要 ~10 周**。这不是工程师水平差，这是 RAG 本身的特性。

**逃生舱**：如果不想洗数据，回去看**路线一（长 Context + Caching）**。脏活躲不掉，要么你干（洗数据喂给 RAG），要么模型干（直接塞原始文档让长 Context 模型自己消化）。

## 第四章：终极决策清单

5 条决策，对应 5 种场景：

1.  **小数据 + 稳定**（几百份文档，知识库不常变）
    **选：长 Context + Prompt Caching**（别折腾）
2.  **大数据 + 语义模糊**（几万份文档，用户问得绕）
    👉 **选：Hybrid RAG + Rerank + Contextual**（别用 baseline）
3.  **代码 / 结构化 / 多跳**（代码库、数据库、多步推理）
    👉 **选：Agent + grep / SQL / API**（别建向量索引）
4.  **复杂研究 + 用户付得起**（尽调、学术综述、长报告）
    **选：Deep Research 架构**（Planner → Search → Reader → Aggregator）
5.  **以上都不确定**（需求不清、数据脏、没数据）
    👉 **选：先建 eval · 先洗数据**（再选架构）

---

## 结语：未来展望与核心

**接下来一两年的三个判断**：
1.  **Prompt Caching 继续挤压**：Context 窗口继续涨（1M/2M/5M），caching 折扣继续降，中小规模 RAG 越来越没必要建。
2.  **所有严肃系统 agentic 化**：从 `retrieve-then-generate` 静态流水线，走向 `Agent loop`。Vector search、SQL、grep、web search 都是工具箱里一视同仁的工具。
3.  **Eval 工具被卷**：架构趋同之后，比的是谁的评估闭环转得快。

**RAG 没死，只是不再是默认答案了。**
从默认到备选，从必建到看情况，从单方案到 5 条路。先理解问题，再选择工具，**省下一笔不该花的工程预算**。
