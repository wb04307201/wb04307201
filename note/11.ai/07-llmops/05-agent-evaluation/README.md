## 一、Agent 评估 vs LLM 评估：本质区别

### 1.1 层次区分

| 维度 | LLM 评估 | Agent 评估 |
|------|---------|-----------|
| **评估对象** | 单次 LLM 调用 | 多步任务执行（可能 5-20 次 LLM 调用） |
| **核心问题** | "这个答案对不对？" | "任务完成了没？花了多少步？成本多少？" |
| **指标** | 准确率 / BLEU / ROUGE / Faithfulness | 任务完成率 / 步骤效率 / 工具成功率 / 成本效益 |
| **时间维度** | 无（单次） | 有（多步循环 + 重试） |
| **成本维度** | 单次 Token 消耗 | 多步累计成本 + 重试成本 + 工具调用成本 |
| **失败模式** | 幻觉 / 不相关 / 有害 | 卡死 / 循环 / 工具调用失败 / 成本爆炸 / 中途放弃 |

### 1.2 为什么需要单独讨论 Agent 评估？

**关键洞察**：Agent 可能每一步 LLM 调用都"对"，但整体任务失败。

**案例**：
```
用户任务：帮我订一张明天从北京到上海的机票

Agent 执行过程：
Step 1: 调用 search_flights(from="北京", to="上海", date="明天")  ✅ LLM 输出正确
Step 2: 调用 book_flight(flight_id="CA1234", passenger="张三")  ✅ LLM 输出正确
Step 3: 调用 confirm_booking()  ✅ LLM 输出正确

但任务失败：
- Step 2 调用时，API 返回"航班已满"
- Agent 没有重试逻辑，直接返回"订票失败"
- 用户不满意（任务未完成）
```

**结论**：LLM 每一步都"对"（准确率 100%），但任务完成率 0%。

**另一个案例**：
```
用户任务：帮我总结一下这篇论文

Agent 执行过程（ReAct 模式）：
Step 1: 读取论文（调用 read_paper）  ✅
Step 2: 总结第一段（调用 summarize）  ✅
Step 3: 总结第二段（调用 summarize）  ✅
Step 4: 总结第三段（调用 summarize）  ✅
Step 5: 总结第四段（调用 summarize）  ✅
Step 6: 总结第五段（调用 summarize）  ✅
Step 7: 总结第六段（调用 summarize）  ✅
Step 8: 合并所有总结（调用 merge）  ✅

任务完成，但：
- 用了 8 步（最优可能是 3 步：读取 → 总结 → 合并）
- 消耗 5000 Token（最优可能是 2000 Token）
- 成本 $0.15（最优可能是 $0.05）
```

**结论**：任务完成了，但步骤效率低、成本效益差。

**这就是为什么需要单独的 Agent 评估体系**。

## 二、Agent 性能量化的 6 大指标维度

### 2.1 维度 1：任务完成度

| 指标 | 定义 | 计算方式 | 目标 |
|------|------|---------|------|
| **任务成功率** | 端到端完成任务的比例 | 成功任务数 / 总任务数 | > 85% |
| **部分完成率** | 任务完成了一部分 | 部分成功数 / 总任务数 | 监控即可 |
| **任务失败率** | 任务完全失败 | 失败任务数 / 总任务数 | < 10% |

**实战案例**：
```
客服 Agent 任务类型分布：
- 查询订单（简单）：成功率 95%
- 退款申请（中等）：成功率 78%
- 投诉处理（复杂）：成功率 62%

整体任务成功率：78%（加权平均）
```

**关键洞察**：
- 简单任务成功率应该 > 95%
- 复杂任务成功率 > 70% 即可接受
- 如果复杂任务成功率 < 50%，说明 Agent 设计有问题

### 2.2 维度 2：步骤效率

| 指标 | 定义 | 计算方式 | 目标 |
|------|------|---------|------|
| **平均步数** | 完成任务平均用了多少步 | 总步数 / 任务数 | 越少越好 |
| **最优步数比** | 实际步数 / 最优步数 | 实际步数 / 理论最优步数 | < 1.5 |
| **冗余步骤率** | 重复 / 无用的步骤比例 | 冗余步数 / 总步数 | < 20% |

**实战案例**：
```
任务：查询订单状态

Agent A（DAG 模式）：
Step 1: 意图识别 → 订单查询
Step 2: 调用 get_order API
Step 3: 格式化输出
总步数：3 步

Agent B（ReAct 模式）：
Step 1: 思考 → 需要查询订单
Step 2: 调用 get_order API
Step 3: 观察 → 订单状态已支付
Step 4: 思考 → 需要返回给用户
Step 5: 返回结果
总步数：5 步

最优步数比：A = 3/3 = 1.0，B = 5/3 = 1.67
```

**关键洞察**：
- DAG 模式通常比 ReAct 模式步骤效率更高
- 如果最优步数比 > 2.0，说明 Agent 在"绕路"
- 冗余步骤率高 → 检查 Prompt 是否清晰

### 2.3 维度 3：工具使用效果

| 指标 | 定义 | 计算方式 | 目标 |
|------|------|---------|------|
| **工具成功率** | 工具调用成功的比例 | 成功调用数 / 总调用数 | > 95% |
| **工具选择准确率** | 选对工具的比例 | 正确选择数 / 总选择数 | > 90% |
| **工具调用延迟** | 工具调用的平均耗时 | 总延迟 / 调用数 | 取决于工具 |

**实战案例**：
```
Agent 有 10 个工具，统计一个月数据：

工具                调用次数  成功率  选择准确率
───────────────  ────────  ──────  ──────────
search_orders       500      98%     95%
get_order_detail    450      99%     97%
refund_request      120      85%     88%  ← 成功率低
send_email          80       92%     90%
...

整体工具成功率：94%
整体工具选择准确率：93%
```

**关键洞察**：
- 工具成功率 < 90% → 检查工具 API 是否稳定
- 工具选择准确率 < 85% → 检查工具描述是否清晰
- 某个工具成功率特别低 → 优先修复

### 2.4 维度 4：成本效益

| 指标 | 定义 | 计算方式 | 目标 |
|------|------|---------|------|
| **单任务成本** | 完成一个任务的平均成本 | 总成本 / 任务数 | < $0.5 |
| **Token 效率** | 每任务的 Token 消耗 | 总 Token / 任务数 | 越少越好 |
| **成本效益比** | 任务价值 / 成本 | 任务价值（业务指标） / 成本 | > 10 |

**实战案例**：
```
客服 Agent 月度成本分析：

总任务数：10,000
总 Token 消耗：50,000,000
总成本：$5,000

单任务成本：$0.5
Token 效率：5000 Token / 任务

任务价值估算：
- 每个任务节省人工客服 10 分钟
- 人工客服成本：$0.5 / 分钟
- 每任务价值：$5

成本效益比：$5 / $0.5 = 10
```

**关键洞察**：
- 单任务成本 > $1 → 需要优化（Prompt 压缩、缓存、小模型）
- 成本效益比 < 5 → Agent 不划算（不如人工）
- Token 效率低 → 检查是否有冗余上下文

### 2.5 维度 5：用户满意度

| 指标 | 定义 | 计算方式 | 目标 |
|------|------|---------|------|
| **点赞率** | 用户点赞的比例 | 点赞数 / 总反馈数 | > 80% |
| **任务完成时长** | 用户等待的总时长 | 从提问到任务完成 | < 30s |
| **重试率** | 用户重新提问的比例 | 重试次数 / 总任务数 | < 10% |

**实战案例**：
```
客服 Agent 用户反馈统计：

总任务数：10,000
用户反馈数：3,000（30% 反馈率）
点赞数：2,400
点踩数：600

点赞率：80%
平均任务完成时长：18s
重试率：8%
```

**关键洞察**：
- 点赞率 < 70% → 需要优化 Agent 输出质量
- 任务完成时长 > 60s → 用户流失风险高
- 重试率 > 15% → Agent 理解能力有问题

### 2.6 维度 6：稳定性

| 指标 | 定义 | 计算方式 | 目标 |
|------|------|---------|------|
| **重复执行一致率** | 同一任务跑多次，结果一致的比例 | 一致次数 / 总次数 | > 90% |
| **失败恢复率** | 失败后能自动恢复的比例 | 恢复次数 / 失败次数 | > 50% |
| **异常率** | 触发异常的比例 | 异常次数 / 总任务数 | < 1% |

**实战案例**：
```
同一任务跑 10 次：

任务：查询订单 20260706 的状态

第 1 次：成功，返回"已支付"
第 2 次：成功，返回"已支付"
第 3 次：成功，返回"已支付"
第 4 次：失败，API 超时
第 5 次：成功，返回"已支付"
第 6 次：成功，返回"已支付"
第 7 次：成功，返回"已支付"
第 8 次：成功，返回"已支付"
第 9 次：成功，返回"已支付"
第 10 次：成功，返回"已支付"

一致率：9/10 = 90%
失败恢复率：Agent 没有重试逻辑，恢复率 0%
```

**关键洞察**：
- 一致率 < 80% → LLM 输出不稳定，需要降低 temperature
- 失败恢复率 < 30% → 需要加重试机制
- 异常率 > 1% → 需要加异常处理

## 三、5 种评估方法详解

### 3.1 方法对比

| 方法 | 准确性 | 成本 | 速度 | 适用阶段 |
|------|--------|------|------|---------|
| **黄金集评估** | 中 | 极低 | 极快 | 开发期 / CI |
| **LLM-as-Judge** | 中高 | 低 | 快 | 开发期 / 生产 |
| **用户模拟** | 中 | 中 | 中 | 上线前 |
| **人工抽检** | 最高 | 高 | 慢 | 生产期 |
| **A/B Test** | 真实 | 中 | 中 | 生产期 |

### 3.2 方法 1：黄金集评估

**核心思想**：构建一个"标准答案集"，每次跑 Agent 对比答案。

**构建流程**：
```
1. 从真实用户日志抽样 50-100 个任务
2. 人工标注"标准答案"（1-2 人花 1-2 天）
3. 标注"评估维度"（任务完成率 / 步骤效率 / 成本）
4. 存入 JSON / YAML 文件
```

**示例**：
```json
{
  "test_cases": [
    {
      "id": "order-query-001",
      "task": "查询订单 20260706 的状态",
      "expected_output": "订单 20260706 状态：已支付",
      "expected_steps": 3,
      "expected_tools": ["get_order"],
      "tags": ["简单任务", "订单查询"]
    },
    {
      "id": "refund-001",
      "task": "帮我申请退款，订单号 20260705",
      "expected_output": "退款申请已提交，预计 3 个工作日到账",
      "expected_steps": 5,
      "expected_tools": ["get_order", "check_refund_eligibility", "submit_refund"],
      "tags": ["中等任务", "退款"]
    }
  ]
}
```

**评估流程**：
```
黄金集（50 条）→ 跑 Agent → 对比输出 vs 标准答案
   ↓
计算指标：
- 任务成功率
- 步骤效率（实际步数 / 期望步数）
- 工具选择准确率
   ↓
报告失败案例 → 改进 Agent
```

**CI 集成**：
```yaml
# .github/workflows/agent-eval.yml
name: Agent Evaluation

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  evaluate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Run Agent Evaluation
        run: |
          python eval/run_eval.py \
            --goldset=eval/goldset.json \
            --output=eval/results.json
      - name: Check Thresholds
        run: |
          python eval/check_thresholds.py \
            --results=eval/results.json \
            --min-success-rate=0.85 \
            --min-tool-accuracy=0.90
```

### 3.3 方法 2：LLM-as-Judge

**核心思想**：用 GPT-4（或更强的模型）当"裁判"，评估 Agent 的输出。

**适用场景**：
- 任务输出复杂（无法用 Exact Match 对比）
- 需要评估"质量"（流畅性、有用性、安全性）
- 人工评估成本太高

**实施流程**：
```
1. 准备评估 Prompt
2. 准备 Agent 输出 + 标准答案（可选）
3. 调用 GPT-4 评分（0-10 分）
4. 聚合分数 → 评估报告
```

**评估 Prompt 示例**：
```python
JUDGE_PROMPT = """
你是一个严格的评估员。请给 Agent 的表现打分（0-10 分）：

用户任务：{task}
Agent 输出：{agent_output}
标准答案（参考）：{expected_output}

评分维度：
1. 任务完成度（40%）：是否完成了用户任务？
2. 输出质量（30%）：是否准确、有用、流畅？
3. 步骤效率（20%）：是否用了合理的步数？
4. 成本控制（10%）：是否避免了不必要的调用？

请输出：
- 总分：0-10
- 各维度分数
- 理由：[详细说明]
"""
```

**校准流程**：
```
1. 准备 50 条样本（Agent 输出 + 人工评分）
2. 用 LLM Judge 评分
3. 计算 LLM Judge 与人工评分的一致性（Cohen's Kappa）
4. Kappa > 0.6 → 可信
5. Kappa < 0.6 → 调整 Prompt 或放弃 LLM Judge
```

**实战案例**：
```
人工评分 vs LLM Judge 评分（50 条样本）：

人工评分：8.2 / 10（平均）
LLM Judge：7.9 / 10（平均）

Cohen's Kappa：0.68 → 可信
```

### 3.4 方法 3：用户模拟

**核心思想**：用 LLM 模拟真实用户，跑 100 个场景。

**适用场景**：
- 上线前的最终验证
- 无法人工跑 100 个场景
- 需要覆盖长尾场景

**实施流程**：
```
1. 从真实日志抽样 100 个用户 query
2. 用 GPT-4 模拟用户（给 query + 人设）
3. Agent 处理用户 query
4. 评估 Agent 输出（LLM Judge / 人工抽检）
5. 报告失败案例
```

**用户模拟 Prompt**：
```python
USER_SIMULATION_PROMPT = """
你是一个真实的用户。你的任务是：

用户背景：{user_persona}
用户目标：{user_goal}

请根据背景和目标，向 Agent 提问。
如果 Agent 的回答不满意，你可以追问或重新提问。
最多交互 5 轮。

开始：
"""
```

**实战案例**：
```
用户模拟：100 个场景

结果：
- 任务成功率：82%
- 平均步数：4.3 步
- 平均成本：$0.35 / 任务
- 失败案例：18 个

失败案例分析：
- 8 个是复杂任务（退款 + 投诉）
- 5 个是工具调用失败
- 5 个是 Agent 理解错误

改进方向：
- 优化复杂任务的 Prompt
- 加工具调用重试机制
```

### 3.5 方法 4：人工抽检

**核心思想**：每月随机抽 100-200 条，人工评估。

**适用场景**：
- 生产环境的持续监控
- 验证自动化评估的准确性
- 发现新的失败模式

**实施流程**：
```
1. 随机抽样 100-200 条（按月 / 按周）
2. 3 位评估员独立评分（0-5 分）
3. 计算评分一致性（Kappa）
4. 聚合分数 → 评估报告
5. 发现失败模式 → 改进 Agent
```

**评估维度**：
| 维度 | 1 分 | 3 分 | 5 分 |
|------|------|------|------|
| 任务完成度 | 完全没完成 | 部分完成 | 完全完成 |
| 输出质量 | 错误 / 有害 | 一般 | 准确 / 有用 / 流畅 |
| 步骤效率 | 冗余步骤 > 50% | 冗余步骤 20-50% | 冗余步骤 < 20% |
| 成本控制 | 成本超标 > 100% | 成本超标 20-100% | 成本超标 < 20% |

### 3.6 方法 5：A/B Test

**核心思想**：新版本 vs 旧版本，用真实用户数据对比。

**适用场景**：
- 重大版本更新（换模型 / 改架构）
- 验证优化效果
- 决策是否全量上线

**实施流程**：
```
1. 分流：50% 用户用 v1，50% 用 v2
2. 运行 1-2 周（收集足够数据）
3. 对比指标：
   - 任务成功率
   - 用户满意度（点赞率）
   - 成本效益比
4. 统计显著性检验（p < 0.05）
5. 决策：v2 全面优于 v1 → 全量上线
```

**工具**：
- **LaunchDarkly**（Feature Flag + A/B）
- **Statsig**（A/B Test + 指标）
- **Eppo**（企业 A/B Test）

## 四、评估流水线设计

### 4.1 完整流水线

```
开发期：
  ├─ 黄金集评估（每次 commit 跑）
  ├─ LLM-as-Judge（每周跑 1 次）
  └─ 人工抽检（每 2 周 1 次）

上线前：
  ├─ 用户模拟（100 场景）
  ├─ 人工抽检（50 条）
  └─ 性能压测（QPS / 延迟）

生产期：
  ├─ 持续监控（任务成功率 / 成本 / 延迟）
  ├─ 人工抽检（每月 100 条）
  └─ A/B Test（新版本上线前）
```

### 4.2 CI/CD 集成

```yaml
# .gitlab-ci.yml
stages:
  - eval
  - test
  - deploy

agent-eval:
  stage: eval
  script:
    - python eval/run_goldset.py --goldset=eval/goldset.json
    - python eval/run_llm_judge.py --samples=50
    - python eval/check_thresholds.py \
        --min-success-rate=0.85 \
        --min-tool-accuracy=0.90 \
        --max-cost-per-task=0.5
  artifacts:
    paths:
      - eval/results/

load-test:
  stage: test
  script:
    - locust -f eval/load_test.py -u 100 -r 10 --run-time 5m
  only:
    - main

deploy:
  stage: deploy
  script:
    - ./deploy.sh
  only:
    - main
  when: on_success
```

### 4.3 评估报告模板

```
Agent 评估报告（2026-07-06）

黄金集评估（50 条）：

  - 平均步数：4.2 步
  - 平均成本：$0.42 / 任务（阈值 $0.5）✅

LLM-as-Judge（50 条抽样）：
  - 任务完成度：8.5 / 10
  - 输出质量：8.2 / 10
  - 步骤效率：7.8 / 10
```

## 六、最佳实践

### 6.1 评估体系搭建

1. **先建黄金集**：50 条起步，持续扩充
2. **6 维度同时看**：单维度优化会掉入陷阱
3. **LLM-as-Judge 需校准**：Cohen's Kappa > 0.6 才可信
4. **A/B Test 是金标准**：用真实用户数据，别只信离线评估
5. **持续监控**：线上指标变化（数据漂移）

### 6.2 常见陷阱

| 陷阱 | 症状 | 修复 |
|------|------|------|
| **混淆 LLM 评估 vs Agent 评估** | 只报"LLM 准确率 92%" | 明确分层：LLM 层 + Agent 层 |
| **单维度优化** | 任务完成率 95% 但成本爆炸 | 6 维度同时看，设约束 |
| **无评估集 = 盲飞** | "感觉挺准的"但没量化 | 建 50+ 条黄金集，CI 集成 |
| **LLM Judge 未校准** | LLM Judge 与人工一致性 < 0.5 | 先用 50 条校准 |
| **忽视成本维度** | 任务完成率 95% 但每任务 $10 | 设成本约束（< $1 / 任务） |
