# 千亿级向量检索架构 —— 分布式 + 分层 + GPU + 联邦

← 返回 [技术栈](../README.md)

> 千亿级（100B / 1000B）向量检索是 10B 的**质变**。本文从**5 个架构转变**（单跳→2 跳、单一→分层、单机→分布式、CPU→GPU、通用→路由）+ **分层索引架构** + **业界算法**（ScANN / Faiss / SPANN / Falcon）+ **真实案例**系统讲清楚千亿级向量检索。

> **面试场景**：这是架构师级面试题——很多人学完 10B 选型后以为"千亿 = 加机器"。面试版（30/60/90 秒话术）见 [咬文嚼字·11.ai/vector-search-at-scale](../../../13.split-hairs/11.ai/vector-search-at-scale/README.md)。

> **兄弟章节**：[10亿级向量检索选型](../vector-search-algorithms/README.md)（HNSW/IVF/DiskANN 详解）

## 一、规模 vs 架构本质

### 1.1 数据规模量化

| 规模 | 数据（FP16, 1024 维）| 内存（图）| 磁盘 | 单机可行 |
|------|---------------------|----------|------|---------|
| 10M | 20 GB | 200 MB | 100 GB | ✅ |
| 100M | 200 GB | 2 GB | 1 TB | ✅ |
| 10B | 20 TB | 200 GB | 4 TB | ✅ 多盘 |
| **100B** | 200 TB | 2 TB | 40 TB | ❌ 必须分布式 |
| **1000B** | 2 PB | 20 TB | 400 TB | ❌ 多集群 |

### 1.2 关键洞察：质变不是量变

```
错误认知：
千亿级 = 10 亿级 × 100 = 加机器

正确认知：
千亿级 = 完全不同的技术栈（5 个架构转变）
```

## 二、5 个架构转变（详细）

### 2.1 转变 1：单跳 → 2 跳检索

**10B 单跳**：
```
Query → DiskANN 图搜索 → top-k（< 5ms）
```

**100B+ 2 跳**：
```
Query → 分片路由
      ↓
  ┌─ 分片 1: 粗排 top-1000（< 5ms）
  ├─ 分片 2: 粗排 top-1000
  ├─ ... (并行)
  └─ 分片 N: 粗排 top-1000
      ↓
  全局精排：合并 top-10000 → top-10（< 10ms）
      ↓
  返回

总延迟：10-20ms
```

**精排的关键**：
- 粗排只用压缩向量（PQ 量化）
- 精排需要读全量原始向量（cache + SSD）
- 精排数据量：top-1000 × 1024 维 = 4 MB（可缓存）

### 2.2 转变 2：单一索引 → 分层索引（Hot/Cold Tiering）

```
┌─────────────────────────────────┐
│ Hot tier（5%，500M）             │
│ - HNSW 全内存                   │
│ - 延迟 < 1ms                    │
│ - 命中率 60%                    │
├─────────────────────────────────┤
│ Warm tier（20%，2B）              │
│ - DiskANN + NVMe SSD            │
│ - 延迟 5-10ms                   │
│ - 命中率 30%                    │
├─────────────────────────────────┤
│ Cold tier（75%，7.5B）            │
│ - IVF-PQ + 对象存储（S3）       │
│ - 延迟 50-100ms                 │
│ - 命中率 10%                    │
└─────────────────────────────────┘

加权平均延迟：0.6 × 1 + 0.3 × 10 + 0.1 × 100 = 13ms
```

**业界案例**：
- **Bing 搜索**：3 层分层
- **Google 搜索**：4 层（含 TPU 加速层）
- **Meta 推荐**：2 层（内存 HNSW + SSD DiskANN）

### 2.3 转变 3：单机 → 分布式分片 + 副本

**100B 分片架构**：
```
                  路由层（Consistent Hashing）
                            │
        ┌───────────────────┼───────────────────┐
        ↓                   ↓                   ↓
    分片 1               分片 2               分片 N
    5B 向量              5B 向量              5B 向量
    DiskANN              DiskANN              DiskANN
    + 3 副本             + 3 副本             + 3 副本
    NVMe SSD 8TB         NVMe SSD 8TB         NVMe SSD 8TB
```

**关键设计**：
- 分片数 = √N（100B → 10-20 分片）
- 副本数 = 2-3（高可用）
- 路由策略：主键 hash（一致性 hash）
- 跨分片查询：并行 + 合并 top-k

**1000B 多集群架构**：
```
              全局路由层（独立集群）
                            │
        ┌───────────────────┼───────────────────┐
        ↓                   ↓                   ↓
    集群 A（300B）       集群 B（300B）       集群 C（400B）
    - 60 分片            - 60 分片            - 80 分片
    - 同机房             跨机房               跨数据中心
    - 同步副本           异步副本             异步副本
```

### 2.4 转变 4：CPU → GPU 加速 + 极限压缩

| 算法/方案 | 适用规模 | 关键创新 | 性能 |
|----------|---------|---------|------|
| **ScANN**（Google 2020）| 10B-100B | 各向异性向量量化（AVQ） | 4x 压缩，Recall 99%+ |
| **GPU-DiskANN**（NVIDIA 2024）| 10B-100B | GPU 加速图遍历 | 100x 距离加速 |
| **Faiss 1.x**（Meta）| 10B-100B | IVF-PQ + GPU | 100x 距离加速 |
| **SPANN**（微软 2023）| 100B+ | SSD + 内存混合 + 近似图 | 100B/50ms |
| **Falcon**（Meta 2024）| 100B+ | 极限压缩 + 联邦 | 1-bit 压缩，Recall 90% |

**GPU 加速原理**：
```
CPU 距离计算：512 维 × 1 亿向量 × 8 字节 = 50ms
GPU 距离计算：512 维 × 1 亿向量 × 8 字节 = 0.5ms（100x）
```

**关键挑战**：
- 图遍历仍受 CPU 限制 → GPU-DiskANN 协同
- GPU 内存有限 → 配合 NVMe SSD
- 数据传输瓶颈 → NVLink / PCIe 4.0+

### 2.5 转变 5：通用召回 → 查询路由

```
用户查询 → Query Router → 选 1-3 个相关分片 → 检索
```

**3 种路由策略**：

| 策略 | 适用 | 实现 |
|------|------|------|
| **按主键范围** | 用户/商品 ID 类 | Consistent Hashing |
| **按主题路由** | 文档类（多领域）| 小模型判断查询主题 |
| **按时间路由** | 新闻 / 实时数据 | 时间窗分片 |

**关键设计**：
- 路由器独立部署（不能和向量库耦合）
- 路由器性能：< 1ms（不能用 LLM 决策）
- 路由准确率 > 90%（否则召回损失）

## 四、延迟 vs 召回 vs 成本三元权衡

### 4.1 权衡矩阵

| 优先级 | 延迟目标 | 推荐方案 | 召回损失 | 成本 |
|--------|---------|---------|---------|------|
| **延迟优先** | < 5ms | Hot tier + HNSW 全内存（5% 数据）| 命中率 60% | $50万+ |
| **召回优先** | < 100ms | ScANN + 全数据 + GPU | 0% | $200万+ |
| **成本优先** | < 50ms | DiskANN + 压缩（不分层）| 2-5% | $10万+ |
| **平衡** | < 20ms | 分层 + 2 跳 + GPU | 3-5% | $30万+ |

### 4.2 千亿级还要不要"毫秒级"？

**反直觉结论**：千亿级的瓶颈**不是延迟**，而是**召回率 + 成本 + 一致性**。

| 指标 | 千亿级现实 |
|------|----------|
| 5ms 可达吗 | 能，但召回损失 5-10% |
| 10-20ms 可达吗 | ✅ 平衡点（2 跳 + 分层）|
| 50-100ms 可达吗 | ✅ 推荐场景可接受 |
| 真正 5ms 场景 | 只查 hot tier（5% 数据）|

## 五、关键工程问题

### 5.1 索引更新

```
增量更新（新增向量）：
- 写 hot tier → 异步下沉到 warm → 异步下沉到 cold
- 实时索引 vs 批量索引（平衡一致性 / 性能）

全量重建：
- 频率：周 / 月
- 策略：双写（old + new 并行服务，逐步切流量）
```

### 5.2 一致性 / 副本同步

```
同步副本（强一致）：
- 写入延迟高（数十 ms）
- 适合：金融 / 强一致业务

异步副本（最终一致）：
- 写入延迟低（ms）
- 适合：搜索 / 推荐（容忍短暂不一致）

混合：
- 副本按一致性分级
- 关键副本同步，非关键副本异步
```

### 5.3 故障容灾

```
节点故障：
- 副本自动接管
- 路由层健康检查（5 秒探测）

机房故障：
- 跨机房副本
- DNS 切流

数据中心故障：
- 跨数据中心副本
- 流量降级（部分功能关闭）
```

## 六、实战选型决策

### 6.1 决策矩阵

| 业务规模 | 数据规模 | 推荐方案 |
|---------|---------|---------|
| < 1000 万 | < 100 GB | HNSW 全内存 |
| 1000 万 - 1 亿 | < 1 TB | HNSW + 量化 |
| 1-10 亿 | 1-10 TB | DiskANN（10 亿级专题）|
| **10-100 亿** | **10-100 TB** | **分布式 DiskANN + 分层** ⭐ |
| **100-1000 亿** | **100 TB - 1 PB** | **ScANN / Faiss GPU + 分层 + 副本** ⭐ |
| > 1000 亿 | > 1 PB | **ScANN + TPU + 联邦** |

### 6.2 业界方案选择

| 场景 | 推荐方案 |
|------|---------|
| 搜索（高召回）| ScANN + 分层 + GPU |
| 推荐（高 QPS）| Faiss GPU + 副本 + 缓存 |
| RAG（中规模）| Milvus + DiskANN + 分片 |
| 大规模 RAG | Milvus 集群 + ScANN |
| 内部工具 | Qdrant + HNSW（< 1 亿）|

## 七、与"驾驭演进"主线的关联

向量检索规模演进是 [驾驭演进主线](../../04-architecture/llm-control-evolution/README.md) 在检索领域的体现：

| 驾驭阶段 | 向量检索规模 | 适配 |
|---------|------------|------|
| Prompt | < 1 万 | 人工查 |
| Context | 1-100 万 | 关键词 / 简单向量 |
| Harness | < 1000 万 | HNSW 全内存 |
| **Loop** | **1-10 亿** | **DiskANN** |
| **未来** | **10-1000 亿** | **分布式 + 分层 + GPU** |

**核心范式转移**：从"全内存图"到"分布式 + 分层 + GPU"——和数据库从"集中式"到"分布式 + 分片 + 联邦"是同一个故事。
