# 万亿级向量检索架构 —— 多集群联邦 + 专用硬件 + 边缘计算

← 返回 [技术栈](../README.md)

## 一、规模 vs 架构本质

### 1.1 数据规模量化

| 规模 | 数据（FP16, 1024 维）| 压缩后总 | 节点数 | 硬件成本 |
|------|---------------------|----------|--------|----------|
| 10B | 20 TB | 4 TB | 1 节点 | $1-5万 |
| 100B | 200 TB | 40 TB | 10-100 节点 | $30-200万 |
| **1T** | **2 PB** | **400 TB - 4 PB** | **1000+ 节点** | **$1M - $10M+** |
| 10T+ | 20 PB+ | 4 PB+ | 10000+ 节点 | $10M+（仅巨头）|

### 1.2 关键洞察：万亿级是"另一个世界"

```
千亿级：1 集群（10-100 节点） + GPU 加速 + 分层索引
       ↓
万亿级：多集群（10-100 个集群）+ 跨数据中心 + 联邦 + TPU + 边缘
       ↓
质变点：
- 跨集群延迟成为主要瓶颈
- 一致性 CAP 只能 AP（基本放弃 C）
- 召回率从 95% 降到 90%（要接受 trade-off）
```

## 二、5 个新挑战（详细）

### 2.1 挑战 1：多集群 + 跨数据中心联邦

```
                  全球路由层（独立服务）
                            │
        ┌───────────────────┼───────────────────┐
        ↓                   ↓                   ↓
    美西集群             美东集群            欧洲集群
    300B 向量            300B 向量           400B 向量
    1000 节点            1000 节点            800 节点
    ScANN + GPU          ScANN + GPU         ScANN + TPU
```

**关键问题**：
- **全球路由**：不是 consistent hash，是**地理感知**（用户请求路由到最近的集群）
- **跨集群一致性**：CAP 权衡——基本只能 AP（放弃 C），否则延迟爆炸
- **跨集群延迟**：同集群 < 5ms / 跨集群 **50-100ms**

**联邦检索流程**：
```
用户查询（美西用户）→ 美西路由层
  → 美西集群（命中 80%，5ms）
  → 美东集群（兜底 20%，跨集群 50ms）
  → 欧洲集群（兜底剩余，全局联邦）

最终：80% 5ms + 15% 50ms + 5% 150ms
加权平均：0.8×5 + 0.15×50 + 0.05×150 = 19ms
```

### 2.2 挑战 2：专用硬件（TPU / FPGA / 定制 ASIC）

| 硬件 | 适用规模 | 特点 | 公开程度 |
|------|---------|------|---------|
| **NVIDIA A100/H100** | 10B-100B | GPU 通用，100x CPU 加速 | 完全公开 |
| **Google TPU v5** | 100B-1T | TPU 专用，10x GPU | 公开论文 + 闭源部署 |
| **AWS Trainium2** | 100B+ | AWS 定制，性价比 | 部分公开 |
| **Habana Gaudi** | 100B+ | Intel 收购，性价比 | 部分公开 |
| **定制 ASIC** | 1T+ | 巨头内部 | 完全不公开 |

**为什么万亿级不能只靠 GPU**：
```
NVIDIA A100 单卡：
- 显存：80 GB
- 算力：312 TFLOPS（FP16）
- 1 万亿向量 × 1024 维 × 2 字节 = 2 PB
- 单卡只能放 40K 向量（80GB / 2KB）
- 需要 25M 块 A100 = $25B+（不现实）

vs Google TPU v5：
- HBM：32-96 GB / 芯片
- 单 Pod（4096 芯片）算力 ≈ 100 PFLOPS
- 1 Pod 能放 50M 向量
- 1 万亿向量需要 20K Pod = $200M（仍很贵但可行）
```

**关键洞察**：万亿级需要**专用硬件协同**，不是堆 GPU。

### 2.3 挑战 3：边缘计算 + 端侧缓存（4 层架构）

```
┌─────────────────────────────────┐
│ 端侧缓存（命中 30%）             │
│ - 浏览器 / App 内置 cache        │
│ - 延迟 < 1ms                    │
│ - 适合：用户最近查询、热门结果    │
├─────────────────────────────────┤
│ CDN 边缘缓存（命中 20%）          │
│ - 边缘节点（Vercel / Cloudflare）│
│ - 延迟 5-10ms                   │
│ - 适合：热门 query 的 top-100     │
├─────────────────────────────────┤
│ 区域集群（命中 30%）             │
│ - 同区域数据中心                  │
│ - 延迟 10-20ms                  │
│ - 适合：常规查询                  │
├─────────────────────────────────┤
│ 全球集群（兜底 20%）             │
│ - 跨数据中心                      │
│ - 延迟 50-100ms                 │
│ - 适合：长尾 / 罕见查询           │
└─────────────────────────────────┘

加权平均：0.3×1 + 0.2×10 + 0.3×20 + 0.2×100 = 27ms
```

**业界案例**：
- **Google 搜索**：浏览器预取 + CDN 边缘 + 区域集群 + 全球回源
- **Meta 推荐**：用户设备预训练 cache + 区域数据中心 + 跨区域联邦

### 2.4 挑战 4：极限压缩 + 重新定义召回率

| 压缩级别 | 压缩比 | 内存（1T 向量）| Recall | 适用 |
|---------|--------|---------------|--------|------|
| **FP16** | 1x | 2 PB | 100% | 1 亿-10 亿 |
| **INT8** | 2x | 1 PB | 99% | 10 亿-100 亿 |
| **PQ8** | 8x | 256 TB | 97% | 100 亿+ |
| **PQ4** | 16x | 128 TB | 92% | **1 万亿+** |
| **1-bit** | 32x | 64 TB | 85% | 极限压缩 |

**为什么 Recall 90% 仍可接受**：
- 重排序（rerank）可补回 5-10% Recall
- 多路召回融合（向量 + BM25）可补回
- 业务场景容忍少量召回损失（推荐 / 搜索）

### 2.5 挑战 5：分层实时架构（预计算 + 流式更新）

```
                预计算层（分钟级更新）
                    ↓
                索引层（秒级更新）
                    ↓
                缓存层（毫秒级过期）
                    ↓
                在线查询
```

**预计算的内容**：
- 用户偏好预计算（用户最近 1000 次点击的相似向量）
- 热门 query 预计算 top-100
- 长尾 query 实时检索全量

## 四、延迟 vs 召回 vs 成本三元权衡

### 4.1 万亿级权衡矩阵

| 优先级 | 延迟目标 | 推荐方案 | 召回损失 | 成本 |
|--------|---------|---------|---------|------|
| **延迟极致** | < 10ms | 端侧 cache + CDN + 少量数据 | 命中率 30% | $5M+ |
| **召回极致** | < 200ms | 全集群 + 联邦 + 全数据 | 0% | $10M+ |
| **成本极致** | < 50ms | 极限压缩（1-bit）+ 边缘 | 5-10% | $3M+ |
| **平衡** | < 50ms | 分层 + 联邦 + 极限压缩 | 3-5% | $5M+ |

### 4.2 反常识结论：万亿级不是"毫秒级"问题

```
10B：延迟是核心指标（5ms 是 hard requirement）
100B：延迟 + 成本 + 一致性 三元权衡
1T：召回 + 成本 + 一致性 + 工程复杂度 四元权衡，延迟可放宽到 50-100ms
```

## 五、诚实面对局限：万亿级 ≠ "我也能做"

**反常识结论**：**万亿级不是大多数团队的工程目标**。

| 团队规模 | 真正需要的规模 | 推荐方案 |
|---------|--------------|---------|
| 中小团队 | < 1 亿 | Milvus / Qdrant 单机 |
| 大型团队 | 1-10 亿 | Milvus / Qdrant 集群 |
| 巨头 | 10-100 亿 | Milvus 集群 + ScANN |
| **超巨头** | **> 1000 亿** | **自研 + 联邦 + 专用硬件** |

**大多数团队到千亿级就到顶了**。万亿级是 Google / Meta 这种公司的问题。

## 六、向量检索规模演进全景（10B → 100B → 1T）

| 规模 | 主算法 | 架构 | 延迟 | Recall | 成本 |
|------|--------|------|------|--------|------|
| **10B** | DiskANN（10亿级）| 单机多盘 | < 5ms | > 95% | $1-5万 |
| **100B** | ScANN / Faiss GPU（千亿级）| 集群 + 分层 + GPU | < 20ms | > 95% | $30-200万 |
| **1T+** | ScANN + TPU + 极限压缩（万亿级）| 多集群 + 联邦 + 边缘 | < 50ms | > 90% | $1M - $10M+ |

**演进规律**：
- 规模每升 10x → 架构质变（不是加机器）
- 延迟要求反而下降（多集群 / 联邦延迟天然高）
- Recall 容忍度下降（必须接受 trade-off）
- 成本非线性增长（专用硬件 + 联邦运维）
