# 模型效率优化：让大模型跑得更快更省

当 GPT-4 级别的模型拥有万亿参数、推理一次消耗数瓦特电力时，"能不能跑起来"和"跑得快不快"就成了 AI 落地的生死线。从手机端的离线助手到云端的高并发服务，模型效率优化技术决定了 AI 能力能否从实验室走向真实世界。

本文系统梳理近年（2024 起）主流的模型效率优化技术族：量化、剪枝、蒸馏、推测解码和模型合并；快照截至 2026-07。

---

## 一、效率优化的核心挑战

大模型推理的三大瓶颈：

| 瓶颈 | 表现 | 影响 |
|------|------|------|
| **显存墙** | 7B 模型 FP16 需 14GB 显存，70B 需 140GB | 消费级 GPU 无法运行 |
| **计算墙** | 自回归逐 token 生成，延迟线性累积 | 实时交互体验差 |
| **带宽墙** | 大模型推理是 memory-bound，受限于显存带宽 | GPU 算力利用率低 |

效率优化的目标是在 **尽量不损失模型能力** 的前提下，突破这三个瓶颈。

---

## 二、量化（Quantization）— 压缩模型参数

### 2.1 基本原理

量化的核心是将模型权重从高精度浮点数（FP32/FP16/BF16）转换为低精度整数或浮点格式（INT8/INT4/NF4），从而：

- **减少显存占用**：INT8 比 FP16 省 50%，INT4 省 75%
- **加速计算**：整数运算比浮点运算更快（尤其在支持 INT8 的硬件上）
- **降低带宽需求**：每次从显存加载的参数量减少

### 2.2 主流量化方案对比

| 方案 | 精度 | 方法 | 是否需要校准数据 | 特点 |
|------|------|------|----------------|------|
| **GPTQ** | INT4/INT8 | 逐层量化 + OBQ 近似 | 是（少量校准数据） | 速度快，GPU 友好 |
| **AWQ** | INT4 | 保护显著权重通道 | 是（少量校准数据） | 精度损失更小 |
| **GGUF** (llama.cpp) | 2-8 bit | 多种混合量化策略 | 否 | CPU/边缘设备友好 |
| **bitsandbytes** | INT8/INT4 | 运行时动态量化 | 否 | HuggingFace 集成好 |
| **QLoRA** | NF4 | 量化基础模型 + LoRA 微调 | 否 | 微调时省显存 |

### 2.3 量化的工程选择

```text
选型决策树：
├── 目标设备是 GPU？
│   ├── 追求速度 → GPTQ (INT4)
│   ├── 追求精度 → AWQ (INT4)
│   └── 快速实验 → bitsandbytes (INT8/INT4)
├── 目标设备是 CPU/边缘？
│   └── GGUF 格式 + llama.cpp
└── 需要微调？
    └── QLoRA (NF4 + LoRA adapters)
```

### 2.4 量化的精度损失

量化不是免费的午餐。关键观察：

- **INT8 量化**：几乎无精度损失（< 0.5%），适合绝大多数场景
- **INT4 量化**：轻微精度损失（1-3%），在大多数任务上可接受
- **INT3 及以下**：精度损失显著，需要配合量化感知训练（QAT）
- **Outlier 问题**：少量权重值异常大（outlier），直接量化会导致严重精度损失 → AWQ 和 SmoothQuant 通过保护/平滑 outlier 解决

## 三、剪枝（Pruning）— 去除冗余参数

### 3.1 核心思想

大模型中存在大量"冗余"参数——对模型输出贡献极小的权重连接。剪枝就是识别并移除这些冗余参数，使模型变得更稀疏。

### 3.2 两种剪枝范式

| 类型 | 方法 | 优点 | 缺点 |
|------|------|------|------|
| **非结构化剪枝** | 按单个权重的绝对值/梯度大小剪枝 | 剪枝比例高（可达 90%+） | 产生稀疏矩阵，需要特殊硬件/软件支持才能加速 |
| **结构化剪枝** | 按整行/整列/整个注意力头剪枝 | 剪枝后模型可直接在标准硬件加速 | 精度损失更大，需要更精细的策略 |

### 3.3 LLM 时代的剪枝新进展

- **Wanda (Pruning by Weights and Activations)**：无需重训练，通过权重 × 输入激活值的乘积来评估权重重要性，在 LLaMA 上实现了 50-60% 非结构化剪枝且精度损失极小
- **SliceGPT**：对 Transformer 进行结构化剪枝，通过 PCA 降低每层的嵌入维度，可以直接减少模型参数和推理 FLOPs
- **LLM-Pruner**：基于依赖图的自动结构化剪枝，识别 Transformer 中的结构依赖关系，确保剪枝后模型仍然可运行

### 3.4 剪枝 vs 量化的选择

- **量化**：通用性强，几乎所有模型都能量化，工业界首选
- **剪枝**：理论压缩比更高，但工程落地更复杂
- **最佳实践**：先剪枝再量化，叠加使用效果最佳

## 四、知识蒸馏（Knowledge Distillation）

### 4.1 核心思想

知识蒸馏的详细内容已在 [知识蒸馏](../distillation/) 专题中系统展开，这里仅概述其在效率优化中的角色。

蒸馏的核心流程：
1. 用大模型（Teacher）在目标任务上生成高质量输出
2. 用小模型（Student）学习 Teacher 的输出分布（软标签）
3. Student 在保持较小参数量的同时，逼近 Teacher 的性能

### 4.2 蒸馏在效率优化中的定位

| 对比维度 | 量化 | 剪枝 | 蒸馏 |
|---------|------|------|------|
| 模型架构 | 不变 | 微调 | 完全重建 |
| 训练成本 | 低（校准） | 中（可能需要重训练） | 高（需要训练 Student） |
| 精度保持 | 好 | 中 | 好 |
| 端侧部署 | 适合 | 取决于稀疏度 | 非常适合 |

### 4.3 蒸馏 + 量化的组合拳

工业界最常见的组合是：**先蒸馏得到小模型，再对小模型进行量化**。

例如：
1. 用 LLaMA-70B（Teacher）蒸馏出 LLaMA-7B 级别的 Student
2. 对 Student 做 INT4 量化
3. 最终模型在消费级 GPU 上即可运行，性能接近 70B 原始模型

→ 更多关于蒸馏的技术细节，参见 [知识蒸馏](../distillation/)。

## 五、推测解码（Speculative Decoding）

### 5.1 自回归生成的瓶颈

LLM 的自回归生成是 **串行的**：每生成一个 token，都要等前一个 token 生成完毕。这意味着即使 GPU 有海量计算单元，大部分时间也在等待显存读取。

### 5.2 Draft + Verify 加速

推测解码的核心思想：**用一个小而快的 Draft 模型先"猜"多个 token，再用大模型一次性并行验证。**

```text
传统自回归：
大模型: token_1 → token_2 → token_3 → token_4 → token_5
        (100ms)   (100ms)   (100ms)   (100ms)   (100ms)
        总耗时：500ms

推测解码：
小模型: token_1 → token_2 → token_3 → token_4 → token_5  (快速串行)
        (10ms)    (10ms)    (10ms)    (10ms)    (10ms)
大模型: 并行验证 token_1~token_5                            (一次前向)
        (120ms)
结果: token_1~token_3 验证通过，token_4 不匹配，从 token_4 重新 draft
        总耗时：约 170ms（加速 ~3x）
```

### 5.3 关键设计选择

| 选择 | 推荐 | 原因 |
|------|------|------|
| Draft 模型 | 同系列的 0.5B-1.5B 模型 | 词汇表一致，预测准确率高 |
| Draft 长度 | 4-8 个 token | 太长则验证接受率下降 |
| 验证策略 | 基于概率的接受/拒绝 | 保证输出分布与原始大模型一致 |

### 5.4 实际加速比

- 简单任务（翻译、摘要）：2-3x 加速
- 复杂任务（代码生成、数学推理）：1.5-2x 加速
- 配合量化：Draft 模型也可以量化，进一步降低开销

## 六、模型合并（Model Merging）

### 6.1 核心思想

模型合并是一种 **无需训练** 的方法，将多个微调后的模型权重直接合并为一个模型，使合并后的模型同时具备各个源模型的能力。

### 6.2 主要合并方法

| 方法 | 原理 | 特点 |
|------|------|------|
| **Linear Merging** | 加权平均各模型权重 | 最简单，但容易稀释各模型特长 |
| **DARE** (Drop And REscale) | 随机丢弃部分 delta 权重后合并 | 减少参数冲突，保持各模型能力 |
| **TIES** (TrIm, Elect Sign & Merge) | 修剪小权重 + 投票选方向 + 合并 | 解决正负干扰问题 |
| **SLERP** | 球面线性插值 | 在高维空间保持方向一致性 |

### 6.3 实际应用场景

- **多能力融合**：将代码微调模型 + 数学微调模型 + 对话微调模型合并，得到一个全能模型
- **社区模型融合**：HuggingFace 上的 Open LLM Leaderboard 中，很多顶级模型都是通过合并社区贡献的微调版本得到的
- **快速原型**：无需重新训练就能组合出具有多种能力的模型

## 七、效率优化技术全景对比

| 技术 | 压缩比 | 速度提升 | 精度损失 | 是否需训练 | 适用阶段 |
|------|--------|---------|---------|-----------|---------|
| INT8 量化 | 2x | 1.5-2x | < 1% | 否 | 推理 |
| INT4 量化 | 4x | 2-3x | 1-3% | 否 | 推理 |
| 非结构化剪枝 | 2-10x | 需特殊硬件 | 1-5% | 可能 | 推理/训练 |
| 结构化剪枝 | 1.5-3x | 1.5-3x | 2-5% | 可能 | 推理/训练 |
| 知识蒸馏 | 5-100x | 5-100x | 2-10% | 是 | 训练 |
| 推测解码 | 不压缩 | 2-3x | 0（无损） | 否 | 推理 |
| 模型合并 | 不压缩 | 不加速 | 可变 | 否 | 训练后 |

**最佳实践组合**：蒸馏（得到小模型）→ 剪枝（去除冗余）→ 量化（压缩精度）→ 推测解码（加速生成）

## 八、面试话术

### 30 秒版
> "模型效率优化有五大技术方向：量化通过降低参数精度（FP16→INT4）减少显存和加速计算；剪枝去除冗余参数使模型更稀疏；蒸馏用大模型训练出能力相近的小模型；推测解码用小模型先猜多个 token 再让大模型并行验证来加速生成；模型合并无需训练地将多个微调模型融合。工业界通常组合使用，先蒸馏再量化是最常见的方案。"

### 90 秒版
> "模型效率优化解决的是大模型'跑得动、跑得快、跑得省'的问题。量化是最通用的方案，GPTQ 和 AWQ 是 GPU 场景的首选，可以将 70B 模型压缩到消费级 GPU 可运行的程度，INT8 几乎无精度损失，INT4 损失在 1-3%。GGUF 格式配合 llama.cpp 则适合 CPU 和边缘设备。

> 剪枝分为非结构化和结构化两种。Wanda 等方法可以在不重训练的情况下实现 50%+ 的剪枝率，但非结构化剪枝需要特殊硬件支持才能实际加速。结构化剪枝如 SliceGPT 可以直接减少 FLOPs。

> 蒸馏是从根本上改变模型大小的方法，先训练小模型学习大模型的输出分布，再配合量化可以进一步压缩。推测解码是加速生成的创新方案，用小 Draft 模型先猜多个 token，大模型一次验证，实现 2-3 倍加速且无损。模型合并如 DARE/TIES 可以无需训练地融合多个微调模型的能力。

> 工程中的最佳实践是根据部署场景组合使用：端侧部署用蒸馏+量化，云端高并发用量化+推测解码。"

## 九、常见面试题

| 题目 | 核心考点 |
|------|---------|
| 量化的核心原理？ | 高精度浮点 → 低精度整数/浮点，减少显存 + 加速计算 |
| GPTQ 和 AWQ 的区别？ | AWQ 保护显著权重通道，精度损失更小 |
| 结构化剪枝 vs 非结构化剪枝？ | 前者按整行/列/头剪枝可直接加速，后者产生稀疏矩阵需特殊支持 |
| 推测解码为什么能加速？ | 小模型快速串行 draft + 大模型一次并行验证，减少自回归步数 |
| 模型合并是否需要训练？ | 不需要，直接操作微调后的权重差值（delta weights） |
| 量化精度损失的关键因素？ | Outlier 权重处理、量化粒度、是否保护显著通道 |

## 十、最佳实践

| 场景 | 推荐方案 | 实践要点 |
|------|---------|---------|
| 消费级 GPU 跑 70B 模型 | AWQ INT4 | 配合 vLLM 或 TGI 部署 |
| CPU/边缘设备 | GGUF + llama.cpp | 选择 Q4_K_M 或 Q5_K_M 量化方案 |
| 端侧小模型 | 蒸馏 → INT4 量化 | 先蒸馏到 1-3B，再量化 |
| 低延迟在线服务 | INT8 量化 + 推测解码 | 量化减少显存，推测解码加速生成 |
| 多能力融合 | DARE/TIES 合并 | 合并同基础模型的多个 LoRA 微调版本 |
| 快速原型验证 | bitsandbytes INT8 | HuggingFace 一行代码集成 |
