# 推理能力增强：从"直觉回答"到"系统推理"

人类面对复杂问题时，不会脱口而出答案，而是会逐步拆解、反复验证、甚至回头修正——这种"慢思考"的能力，正是大语言模型从"能说话"走向"能解题"的关键跨越。

从 2022 年的 Chain-of-Thought 到 2024-2026 年的 Tree-of-Thought、Graph-of-Thought 和 ReAct，推理增强技术正在让 LLM 从"模式匹配引擎"进化为"系统推理器"。

---
---

## 一、为什么推理能力如此重要？

标准 LLM 的工作模式是 **自回归生成**：给定 prompt，逐 token 预测下一个 token。这种模式在简单问答和文本生成上表现出色，但面对需要 **多步逻辑推理** 的任务（数学证明、代码调试、复杂规划）时，"一次生成"往往力不从心。

核心矛盾在于：**模型的推理深度受限于生成路径的单一性**。一旦在推理早期走错方向，后续 token 只会沿着错误路径越走越远。

推理增强技术的目标，就是让模型具备以下能力：

- **问题拆解**：将复杂问题分解为可管理的子问题
- **多路径探索**：不局限于单一推理链，而是尝试多种推理方向
- **自我验证**：在推理过程中检查中间结果的正确性
- **回溯修正**：发现错误后能回退到正确节点重新推理

## 二、CoT（Chain-of-Thought）— 思维链推理

### 2.1 核心思想

Chain-of-Thought 由 Google 的 Wei et al. (2022) 提出，核心思想极其简洁：**在 prompt 中引导模型"一步一步思考"，而不是直接给出答案。**

传统 Few-shot Prompting：
```
问：一个商店有 23 个苹果，卖了 17 个，又进了 6 个，现在有几个？
答：12
```

CoT Prompting：
```
问：一个商店有 23 个苹果，卖了 17 个，又进了 6 个，现在有几个？
答：让我一步一步来算。
商店最初有 23 个苹果。
卖了 17 个后：23 - 17 = 6 个。
又进了 6 个后：6 + 6 = 12 个。
所以答案是 12。
```

看似简单的改变，却让模型在 GSM8K 数学推理基准上的准确率从 17.7% 跃升至 78.7%——因为推理步骤为模型提供了"计算空间"，让每一步的中间结果都能被后续 token 利用。

### 2.2 三种主要变体

| 变体 | 方法 | 优点 | 缺点 |
|------|------|------|------|
| **Few-shot CoT** | 在 prompt 中提供包含推理步骤的示例 | 效果稳定，适合结构化任务 | 需要人工构造高质量示例 |
| **Zero-shot CoT** | 仅添加 "Let's think step by step" | 零成本，无需示例 | 效果弱于 Few-shot，推理步骤不可控 |
| **Self-Consistency** | 多次采样生成多条推理链，投票选多数答案 | 显著提升准确率 | 推理成本倍增（通常 40-100 次采样） |

### 2.3 Self-Consistency 详解

Self-Consistency（Wang et al., 2022）的核心洞察是：**正确的推理路径可能有很多种，但正确答案通常是同一个。**

流程：
1. 对同一个问题，用 CoT 提示让模型生成 N 条不同的推理链（通过提高 temperature 参数增加多样性）
2. 提取每条推理链的最终答案
3. 对 N 个答案做多数投票（Majority Voting），选择出现次数最多的答案

这个方法在数学推理上效果惊人，能将 CoT 的准确率再提升 5-15 个百分点。代价是需要 N 次推理，适合对准确率要求极高的场景。

## 三、ToT（Tree-of-Thought）— 从链式到树搜索

### 3.1 CoT 的局限性

CoT 的根本限制在于 **线性推理链**：一旦某一步推理出错，后续所有步骤都会被"污染"。更关键的是，CoT 无法回溯——模型不能"后悔"回到之前的某个分叉点重新探索。

### 3.2 ToT 的核心设计

Tree-of-Thought（Yao et al., 2023）将推理从"链"扩展为"树"，引入了搜索算法：

```
         [问题]
        /      \
    [思路A]    [思路B]
    /    \        \
 [A1]   [A2]    [B1]
  ↓      ↓        ↓
评估  评估     评估 → 回溯到思路A
```

核心组件：
- **Thought 分解**：将推理过程拆分为多个"思考步骤"（Thought Step）
- **生成器**：在每个节点生成多个候选思路（BFS/DFS 扩展）
- **评估器**：对每个候选思路打分（"这个方向有前途吗？"）
- **搜索算法**：BFS（广度优先）或 DFS（深度优先）遍历推理树

### 3.3 GoT（Graph-of-Thought）— 进一步泛化

Graph-of-Thought 将树结构进一步泛化为 **有向无环图（DAG）**：

- **合并节点**：两条独立的推理路径可以合并，取各自优点
- **精炼节点**：对某个推理步骤进行迭代优化
- **评分传播**：图上各节点的评分可以相互影响

GoT 的优势在于更接近人类思维的非线性特征——我们在解决问题时，经常将不同思路交叉融合，而非严格遵循树状结构。

## 四、ReAct（Reasoning + Acting）— 推理与工具调用交替

### 4.1 核心思想

ReAct（Yao et al., 2022）的核心创新是让模型在推理过程中 **交替进行思考（Thought）和行动（Action）**：

```
Thought 1: 我需要查找今年的 GDP 数据来回答这个问题
Action 1: search("2026年中国GDP")
Observation 1: 搜索结果显示 2026 年中国 GDP 预计为...
Thought 2: 有了 GDP 数据，我还需要人口数据来计算人均
Action 2: search("2026年中国人口")
Observation 2: 搜索结果显示...
Thought 3: 现在我可以计算了，人均 GDP = ...
Answer: 2026 年中国人均 GDP 约为...
```

### 4.2 ReAct 的优势

| 特性 | 纯推理（CoT） | 纯行动（直接调工具） | ReAct |
|------|-------------|-------------------|-------|
| 可解释性 | 高 | 低 | 高 |
| 事实准确性 | 低（易幻觉） | 高 | 高 |
| 复杂任务 | 可能迷失方向 | 缺乏规划 | 规划+验证 |

### 4.3 与 Agent 架构的关系

ReAct 是构建 [Agent 架构](../../04-architecture/) 的核心推理范式。在实际的 Agent 系统中：

- **ReAct 循环** = Agent 的"思考-行动"主循环
- **工具调用** = ReAct 中的 Action 步骤
- **观察解析** = 将工具返回结果转化为 Observation
- **终止条件** = 模型判断已获得足够信息，输出最终答案

→ 更多关于 Agent 架构的细节，参见 [L4 架构设计](../../04-architecture/)。

## 五、推理增强技术对比

| 技术 | 推理结构 | 是否可回溯 | 工具调用 | 适用场景 | 推理成本 |
|------|---------|-----------|---------|---------|---------|
| Zero-shot CoT | 单链 | 否 | 否 | 简单多步推理 | 1x |
| Few-shot CoT | 单链 | 否 | 否 | 结构化推理任务 | 1x + 示例 |
| Self-Consistency | 多链投票 | 否 | 否 | 高准确率要求 | N x（N=40~100） |
| ToT | 树搜索 | 是 | 否 | 复杂规划/博弈 | 10-50x |
| GoT | 图搜索 | 是 | 否 | 需要融合多思路 | 20-100x |
| ReAct | 思考-行动交替 | 部分 | 是 | 需要外部信息的推理 | 3-10x |

## 六、与沉思模型的关系

推理增强技术与 [沉思模型（Rumination）](../rumination-models/) 有天然的互补关系：

- **CoT/ToT** 提供推理的 **方法论**（怎么想）
- **沉思模型** 提供推理的 **基础设施**（在哪里想、想多久）
- **ReAct** 是沉思模型"感知-思考-行动"闭环的具体实现范式

GLM-Z1-Rumination 等沉思模型本质上是将 ToT/ReAct 等推理范式内化为模型的默认行为模式，而非仅通过 prompt 引导。

## 七、面试话术

### 30 秒版
> "推理增强是让 LLM 从'直觉回答'进化到'系统推理'的技术族。CoT 通过引导模型逐步推理提升准确率；ToT 将推理从链式扩展为树搜索，支持回溯；ReAct 让推理与工具调用交替进行，是 Agent 系统的核心范式。这些技术共同解决了 LLM '一次生成'模式的推理深度不足问题。"

### 90 秒版
> "推理增强解决的核心问题是 LLM 自回归生成的'一次性'特性。CoT 是最基础的方案，通过在 prompt 中引导模型'一步一步思考'，在 GSM8K 数学推理上将准确率从 17.7% 提升到 78.7%。Zero-shot CoT 只需加一句'Let's think step by step'，Few-shot CoT 需要提供推理示例。Self-Consistency 进一步提升，通过多次采样生成多条推理链然后投票选多数答案。
>
> CoT 的局限在于线性推理链无法回溯，ToT 将其扩展为树搜索，在每个节点生成多个候选思路并评分，支持 BFS/DFS 遍历。GoT 进一步泛化为图结构，允许思路合并和精炼。
>
> ReAct 是推理与工具调用的融合，模型交替进行 Thought 和 Action，特别适合需要外部信息的推理任务，也是构建 Agent 系统的核心范式。在实际工程中，选择哪种方案取决于任务复杂度与延迟预算的权衡。"

## 八、常见面试题

| 题目 | 核心考点 |
|------|---------|
| CoT 为什么能提升推理准确率？ | 中间推理步骤为后续 token 提供"计算空间"，减少跳步错误 |
| Zero-shot CoT 和 Few-shot CoT 的区别？ | 前者仅需提示语引导，后者需提供推理示例；效果 Few-shot > Zero-shot |
| Self-Consistency 如何工作？ | 多次采样生成多条推理链，对最终答案做多数投票 |
| ToT 相比 CoT 的核心优势？ | 支持多路径探索 + 回溯 + 评估，不再受限于单链 |
| ReAct 的核心循环是什么？ | Thought → Action → Observation 交替进行 |
| ReAct 与 Agent 的关系？ | ReAct 是 Agent 的推理内核，Action 对应工具调用 |
| 推理增强的成本代价？ | Self-Consistency N 倍、ToT 10-50 倍，需要权衡准确率与延迟/成本 |

## 九、最佳实践

| 场景 | 推荐方案 | 实践要点 |
|------|---------|---------|
| 简单多步推理 | Zero-shot CoT | 添加"请一步一步思考"即可，成本最低 |
| 数学/逻辑推理 | Few-shot CoT + Self-Consistency | 构造 4-8 个高质量推理示例，采样 40 次投票 |
| 复杂规划/博弈 | ToT (BFS) | 定义清晰的 thought 粒度和评估函数 |
| 需要实时信息 | ReAct | 配置搜索引擎/知识库作为 Action 工具集 |
| Agent 系统 | ReAct + CoT | Thought 步骤用 CoT 引导，Action 调用外部工具 |
| 低延迟场景 | 蒸馏 CoT 能力到小模型 | 参见 [知识蒸馏](../distillation/) — 用大模型的 CoT 数据微调小模型 |
