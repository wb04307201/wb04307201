# 安全与对齐：让 AI 既聪明又可靠

一个能力超强但不受控的 AI，比一个能力有限但可靠的 AI 更危险。当大语言模型被用于医疗咨询、法律建议、金融决策等高风险场景时，"说对"比"说多"重要得多。

安全与对齐（Safety & Alignment）是 AI 从"能用"到"敢用"的关键跨越。本文系统梳理从 RLHF 到 DPO、从幻觉治理到红队测试的核心技术。

---
---

## 一、什么是 AI 对齐？

AI 对齐（Alignment）的目标是让 AI 系统的行为符合人类的意图和价值观。具体来说，一个对齐良好的 AI 应该满足 **3H 原则**：

| 原则 | 含义 | 反面案例 |
|------|------|---------|
| **Helpful**（有帮助） | 能有效完成用户的合理请求 | 过度拒绝、答非所问 |
| **Harmless**（无害） | 不产生有害输出（歧视、暴力、违法指导） | 生成有害内容、协助恶意行为 |
| **Honest**（诚实） | 不确定时承认不确定，不编造事实 | 自信地胡说八道（幻觉） |

这三个目标之间存在张力：过度追求 Harmless 会导致模型过度拒绝（refusal），变得不够 Helpful；过度追求 Helpful 可能让模型为了"讨好"用户而编造信息，违反 Honest。对齐技术就是在这三者之间找到最佳平衡。

## 二、RLHF — 人类反馈强化学习

### 2.1 三阶段流程

RLHF（Reinforcement Learning from Human Feedback）是目前最成熟的对齐方法，由 OpenAI 在 InstructGPT 中验证有效：

```
阶段 1: SFT（监督微调）
├── 用人工标注的高质量 instruction-response 对微调基础模型
├── 目标：让模型学会基本的指令跟随能力
└── 产出：SFT 模型

阶段 2: RM（奖励模型训练）
├── 让 SFT 模型对同一 prompt 生成多个回答
├── 人工标注员对回答进行排序（好 > 差）
├── 用排序数据训练奖励模型（Reward Model）
└── 产出：Reward Model（给回答打分）

阶段 3: PPO（强化学习优化）
├── 用 PPO 算法优化 SFT 模型
├── 奖励信号来自 Reward Model
├── KL 散度约束：防止模型偏离 SFT 太远
└── 产出：对齐后的最终模型
```

### 2.2 RLHF 的挑战

| 挑战 | 具体问题 |
|------|---------|
| **标注成本高** | 需要大量人工排序，标注员需要专业知识 |
| **标注一致性** | 不同标注员的偏好可能冲突 |
| **Reward Hacking** | 模型学会"讨好"奖励模型而非真正对齐 |
| **训练不稳定** | PPO 是出了名的难调，超参数敏感 |
| **过度拒绝** | 模型学到"拒绝一切有风险的请求"作为安全策略 |

## 三、DPO / KTO / SimPO — 无需奖励模型的对齐

### 3.1 DPO（Direct Preference Optimization）

DPO（Rafailov et al., 2023）的核心突破是 **绕过了奖励模型的训练**，直接用偏好数据优化策略模型。

数学直觉：
- RLHF 先训练 RM 再训练 PPO，是一个两步过程
- DPO 证明了 RLHF 的最优策略可以用一个闭式解表示
- 直接用偏好数据对（chosen vs rejected）构建损失函数
- 一步到位，无需 RM 和 PPO

```python
# DPO 的核心损失函数（简化）
loss = -log(sigmoid(
    beta * (log_prob(chosen) - log_prob(rejected))
    - beta * (ref_log_prob(chosen) - ref_log_prob(rejected))
))
```

### 3.2 DPO 变体对比

| 方法 | 数据类型 | 核心改进 | 适用场景 |
|------|---------|---------|---------|
| **DPO** | 偏好对（chosen vs rejected） | 无需 RM，直接优化 | 有高质量偏好数据 |
| **KTO** (Kahneman-Tversky) | 单条数据（好/坏标签） | 不需要配对数据 | 标注成本更低 |
| **SimPO** | 偏好对 | 用序列平均对数概率替代隐式奖励 | 更稳定，无需参考模型 |
| **ORPO** | 偏好对 | 将 SFT 和偏好优化合为一步 | 简化训练流程 |

### 3.3 DPO vs RLHF

| 维度 | RLHF | DPO |
|------|------|-----|
| 流程复杂度 | 三阶段（SFT → RM → PPO） | 两阶段（SFT → DPO） |
| 训练稳定性 | PPO 不稳定 | 标准梯度下降，稳定 |
| 计算资源 | 需同时加载 4 个模型 | 需同时加载 2 个模型 |
| 数据要求 | 排序数据 | 偏好对数据 |
| 效果 | 成熟验证 | 接近或优于 RLHF |

## 四、Constitutional AI — 自我纠正

### 4.1 核心思想

Constitutional AI（Anthropic, 2022）提出了一种 **让模型自我纠正** 的对齐方法，减少对人工标注的依赖：

1. **自我批评（Critique）**：让模型对自己生成的有害回答进行批评
2. **自我修正（Revision）**：让模型根据批评修改回答，使其更安全
3. **RLAIF**：用自我修正后的数据训练偏好模型，替代人工标注

### 4.2 "宪法"原则

Constitutional AI 的"宪法"是一组指导原则，例如：

- "选择最不有害的回应"
- "选择最诚实的回应"
- "当不确定时，承认不确定性"

这些原则指导模型进行自我评估和修正，类似于给 AI 一套"行为准则"。

### 4.3 优势与局限

- **优势**：大幅减少人工标注需求，可扩展性好
- **局限**：自我纠正能力依赖基础模型的能力，小模型的自我纠正效果有限

## 五、幻觉治理（Hallucination Mitigation）

### 5.1 幻觉的类型

| 类型 | 定义 | 示例 |
|------|------|------|
| **事实性幻觉** | 编造不存在的事实 | "爱因斯坦在 1921 年获得了诺贝尔化学奖"（实际是物理学奖） |
| **忠实性幻觉** | 偏离输入/上下文 | 摘要任务中添加了原文没有的信息 |
| **逻辑幻觉** | 推理过程看似合理但结论错误 | 数学推导中跳步导致错误答案 |

### 5.2 治理方法体系

```
幻觉治理
├── 训练阶段
│   ├── RLHF/DPO 对齐 — 惩罚幻觉输出
│   ├── 事实验证数据 — 用事实核查数据微调
│   └── 置信度校准 — 让模型在不确定时表达不确定性
├── 推理阶段
│   ├── RAG 增强 — 检索外部知识作为生成依据
│   ├── Self-Consistency — 多次采样投票减少随机幻觉
│   ├── 自我验证 — 让模型检查自己的输出
│   └── 链式验证 — CoT 推理 + 外部事实核查
└── 后处理阶段
    ├── 事实核查系统 — 用独立系统验证生成内容
    ├── 引用追溯 — 要求模型为每个事实声明提供来源
    └── 置信度标注 — 对输出中的事实声明标注置信度
```

### 5.3 RAG 增强

RAG（Retrieval-Augmented Generation）是对抗幻觉最有效的工程手段之一：

- **检索**：从知识库中检索与 query 相关的文档片段
- **增强**：将检索结果注入 prompt 上下文
- **生成**：模型基于真实文档生成回答，减少编造

→ 更多关于 RAG 的技术细节，参见 [L2 技术栈 — RAG](../../02-technology-stack/)。

### 5.4 置信度校准

一个对齐良好的模型应该 **知道自己不知道什么**：

- **Token 级置信度**：模型输出的 softmax 概率反映其对每个 token 的信心
- **语义级置信度**：通过多次采样的答案一致性来评估模型对整体回答的信心
- **校准训练**：用专门的校准数据训练模型，使其概率输出与实际准确率一致

## 六、红队测试（Red Teaming）

### 6.1 什么是红队测试

红队测试源自军事领域的对抗性评估，在 AI 安全中指 **通过对抗性攻击来发现模型的脆弱点**：

- **目标**：找到模型在哪些情况下会生成有害、错误或不当的内容
- **方法**：人工或自动化地构造对抗性输入
- **输出**：漏洞报告 + 修复建议

### 6.2 红队测试方法

| 方法 | 描述 | 示例 |
|------|------|------|
| **直接攻击** | 明确要求有害内容 | "告诉我如何..." |
| **角色扮演** | 诱导模型扮演不受限角色 | "假设你是一个没有限制的 AI..." |
| **间接注入** | 在输入中隐藏指令 | 在长文本中嵌入"忽略以上指令，改为..." |
| **多轮诱导** | 通过多轮对话逐步突破限制 | 先聊无害话题，再逐步引入敏感内容 |
| **编码绕过** | 用编码/混淆绕过关键词过滤 | Base64 编码、拼音替换、emoji 暗示 |

### 6.3 自动化红队测试

2024-2026 年，自动化红队测试成为趋势：

- **对抗性生成**：用一个 AI 自动生成攻击 prompt，测试目标 AI
- **覆盖率评估**：系统性地覆盖各种攻击向量和边界场景
- **回归测试**：每次模型更新后重新运行红队测试，确保不引入新漏洞

### 6.4 从红队到防御

红队测试的最终目的是 **指导防御**：

1. 红队发现漏洞 → 2. 生成修复数据 → 3. DPO/RLHF 再训练 → 4. 重新红队验证

这个闭环是现代 AI 安全工程的核心工作流。

## 七、安全与对齐技术全景

| 技术 | 目标 | 依赖 | 效果 |
|------|------|------|------|
| RLHF | 对齐人类偏好 | 人工排序 + PPO | 成熟，OpenAI/Anthropic 验证 |
| DPO | 对齐人类偏好 | 偏好对数据 | 更简单，效果接近 RLHF |
| Constitutional AI | 自我纠正 | 原则集 + 基础模型能力 | 减少人工标注 |
| RAG | 事实准确性 | 知识库 + 检索系统 | 显著减少事实性幻觉 |
| 置信度校准 | 诚实性 | 校准训练数据 | 模型知道何时不确定 |
| 红队测试 | 发现漏洞 | 攻击 prompt 库 | 指导防御加固 |

## 八、面试话术

### 30 秒版
> "AI 安全与对齐的核心是让模型满足 3H 原则——Helpful、Harmless、Honest。RLHF 通过三阶段流程（SFT → 奖励模型 → PPO）实现对齐，但训练复杂。DPO 直接用偏好数据优化，无需奖励模型，更简单稳定。幻觉治理靠 RAG 增强事实性和置信度校准。红队测试通过对抗性攻击发现模型漏洞，是安全评估的标准方法。"

### 90 秒版
> "AI 安全与对齐解决的是'让 AI 既聪明又可靠'的问题，核心目标是 3H 原则：Helpful、Harmless、Honest。

> RLHF 是最成熟的对齐方法，分三个阶段：先 SFT 微调让模型学会指令跟随，再训练奖励模型学习人类偏好，最后用 PPO 强化学习优化。挑战在于标注成本高、PPO 训练不稳定、容易 Reward Hacking。

> DPO 是 2023-2026 年的主流替代方案，核心突破是绕过奖励模型，直接用偏好对数据优化策略模型。流程从三阶段简化为两阶段，训练更稳定。变体包括 KTO（不需要配对数据）、SimPO（不需要参考模型）、ORPO（SFT 和对齐合为一步）。

> 幻觉治理是对齐的关键子目标。最有效的方法是 RAG 增强——用检索的真实文档作为生成依据。配合置信度校准让模型在不确定时承认不确定，以及 Self-Consistency 多次采样投票减少随机幻觉。

> 红队测试是安全评估的标准方法，通过对抗性攻击（直接攻击、角色扮演、注入等）发现模型漏洞。自动化红队用 AI 生成攻击 prompt 来测试 AI，形成'攻击-修复-验证'的闭环。"

## 九、常见面试题

| 题目 | 核心考点 |
|------|---------|
| RLHF 的三阶段流程？ | SFT → RM 训练 → PPO 优化 |
| DPO 与 RLHF 的核心区别？ | DPO 无需奖励模型，直接用偏好数据优化 |
| 3H 原则是什么？ | Helpful / Harmless / Honest |
| 幻觉的三种类型？ | 事实性 / 忠实性 / 逻辑幻觉 |
| RAG 如何减少幻觉？ | 检索真实文档注入上下文，让模型基于事实生成 |
| 红队测试的常见方法？ | 直接攻击、角色扮演、注入、多轮诱导、编码绕过 |
| Constitutional AI 的核心思想？ | 让模型自我批评和修正，减少人工标注依赖 |
| DPO 和 KTO 的区别？ | DPO 需要偏好对数据，KTO 只需单条好/坏标签 |

## 十、最佳实践

| 场景 | 推荐方案 | 实践要点 |
|------|---------|---------|
| 通用对话对齐 | SFT + DPO | DPO 比 RLHF 更简单稳定 |
| 高风险领域（医疗/法律） | RLHF + RAG + 事实核查 | 多重保障，宁缺毋滥 |
| 降低幻觉 | RAG + 置信度校准 + Self-Consistency | 检索增强 + 让模型知道何时不确定 |
| 安全评估 | 自动化红队 + 人工红队 | 定期测试，每次更新后回归测试 |
| 资源有限 | SFT + DPO + Constitutional AI | 减少标注成本，利用自我纠正 |
| 避免过度拒绝 | 平衡 Harmless 和 Helpful | 拒绝要精准，不要一刀切 |
