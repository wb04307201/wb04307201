# 用 AI 生成后端接口代码，你怎么审核验收？

> AI 编程时代的代码验收方法论。考察的不是"你会不会用 AI"，而是 **能不能对 AI 产出建立分层质量门禁** + **知不知道 AI 在后端代码上最常漏什么** + **能不能识别"绿色测试"陷阱**。

## 引子：一个"看起来完美"的 800 行订单接口 PR

```text
周五下午，小李甩来一个 PR：

- 800 行，AI 一把生成的「订单创建 + 支付回调」接口
- 代码很规整：分层清晰、有注释、命名漂亮
- CI 全绿：18 个单元测试全过 ✅
- 小李："测试都过了，能 merge 吗？"

你盯着屏幕，隐约觉得哪里不对——
但"测试全绿"这四个字，让你几乎想直接点 Approve。
```

**陷阱在哪？** 那 18 个测试，是 **AI 自己写的**——它测的是"AI 对需求的理解"，不是"需求本身"。AI 把 `扣库存` 和 `创建订单` 放在两个事务里，测试却只断言了返回值 200。**并发下会超卖，测试永远绿。**

**核心真相**：AI 生成后端代码的验收，**不能看"能不能跑"，要看"错在哪一层"**。前端错了用户看得见，后端错了——**要么是钱，要么是数据，要么是越权**。

---

## 一、核心原理：6 层审核体系（从外到内）

AI 后端代码的审核，按"离故障有多近"分 6 层，越往下越是 AI 的重灾区：

| 层 | 审什么 | AI 漏检率 | 故障后果 |
|----|--------|----------|---------|
| ① 接口契约 | 参数校验 / 幂等 / 鉴权 / 返回结构 | 中 | 脏数据 / 重复下单 |
| ② 业务正确性 | 边界 / 事务边界 / 并发 / 状态机 | **高** | 超卖 / 资损 |
| ③ 安全 | SQL 注入 / 越权 / 敏感信息 | **极高** | 数据泄露 / 拖库 |
| ④ 性能 | N+1 / 索引 / 大事务 / 分页 | 高 | 慢查询 / 雪崩 |
| ⑤ 可测试性 | 真断言 / 异常处理 / 日志 | 中 | 假测试 / 难排障 |
| ⑥ AI 幻觉专项 | 虚构 API / 依赖版本 / 配置 | 中 | 编译过但语义错 |

> 记忆口诀：**契约、业务、安全、性能、可测、幻觉**——前 3 层出问题会上新闻，必须逐行看。

## 二、代码示例：AI 最爱埋的 3 个雷

### 雷 1：事务边界拆错 → 超卖（第 ② 层）

```java
// ❌ AI 生成：两个方法各开各的事务
public void createOrder(Long skuId, int qty) {
    stockService.deduct(skuId, qty);   // 事务 A：扣库存
    orderService.save(buildOrder(...)); // 事务 B：建订单
}
// 扣库存成功、建订单失败 → 库存凭空消失；并发下还超卖

// ✅ 正确：同一事务 + 乐观锁/行锁兜底
@Transactional
public void createOrder(Long skuId, int qty) {
    int ok = stockMapper.deductIfEnough(skuId, qty); // UPDATE ... WHERE stock >= qty
    if (ok == 0) throw new BizException("库存不足");
    orderMapper.save(buildOrder(...));
}
```

### 雷 2：水平越权 → 谁都能查别人订单（第 ③ 层）

```java
// ❌ AI 生成：只校验"登录了"，没校验"是不是你的"
@GetMapping("/order/{id}")
public Order get(@PathVariable Long id) {
    return orderMapper.findById(id); // 改个 id 就能看别人订单（IDOR）
}

// ✅ 正确：带上当前用户维度过滤
public Order get(@PathVariable Long id) {
    return orderMapper.findByIdAndUserId(id, currentUserId()); // 越权直接查不到
}
```

### 雷 3：SQL 拼接 → 注入（第 ③ 层）

```java
// ❌ AI 在"动态排序/动态条件"时最容易拼接字符串
String sql = "SELECT * FROM orders ORDER BY " + sortField; // sortField 来自前端

// ✅ 白名单校验 + 参数化
if (!ALLOWED_SORT.contains(sortField)) throw new BizException("非法排序字段");
```

## 三、常见陷阱

### 陷阱 1：以为"测试全绿 = 验收通过"

- **真相**：AI 写的测试测的是"AI 对需求的理解"。**测试和被测代码同源同错**，会一起绿。必须人工补"AI 想不到的用例"——并发、越权、空值、超大入参。

### 陷阱 2：以为幂等性、事务、并发 AI 会自动处理

- **真相**：这三样是 AI 的**结构性盲区**。AI 按"单请求、串行、happy path"生成代码，幂等键、事务边界、行锁几乎从不主动加。**回调接口、扣款接口，必查幂等。**

### 陷阱 3：以为"鉴权" = AI 加了 `@PreAuthorize` 就够了

- **真相**：`@PreAuthorize` 是**垂直权限**（角色能不能访问这个接口）。AI 几乎从不加**水平权限**（这条数据是不是你的，IDOR）。后者才是越权重灾区。

### 陷阱 4：把 AI 编造的 API / 配置当成"新知识"

- **真相**：AI 会**自信地**调用不存在的方法、编造依赖版本、写出看似合理的配置项。编译能过不代表存在——**陌生 API 一律去官方文档核对**。

## 四、最佳实践：分级验收门禁

不是所有接口都用同一把尺子。**按风险分级，配不同门禁**：

| 风险级 | 接口类型 | 验收门禁 |
|--------|---------|---------|
| 🟢 低 | 内部只读查询 / 配置读取 | AI 测试 + 静态扫描通过即可 |
| 🟡 中 | 普通 CRUD 增删改 | 人工过 ①②③层 + 补越权/边界用例 |
| 🔴 高 | 支付 / 资金 / 权限 / 对外开放 | 逐行 review + 安全扫描 + 契约测试 + **禁止 AI 独立完成** |

**工具兜底**（人审之前先过机器）：

- 静态扫描：SonarQube / Semgrep（注入、空指针、坏味道）
- 安全扫描：OWASP API Top 10 规则集（越权、鉴权缺失）
- 依赖核对：SCA 工具查 AI 编造/带漏洞的依赖版本

> 一句话原则：**AI 负责产出，人负责为风险背书**。CI 门禁挡机械错误，人审挡"语义对但业务错"。

## 五、面试话术（90 秒版本）

> 审核 AI 生成的后端接口代码，我不看"能不能跑"，看"错在哪一层"。我用 6 层体系：契约、业务、安全、性能、可测试、幻觉。
>
> 重点盯前 3 层，因为后端出错要么是钱、要么是数据、要么是越权：
>
> 1. **业务正确性**——事务边界、幂等、并发。AI 按 happy path 写，超卖和重复下单是它最常漏的。
> 2. **安全**——尤其**水平越权（IDOR）**，AI 几乎从不主动加"这条数据是不是你的"的过滤；还有动态 SQL 拼接注入。
> 3. **AI 幻觉**——编造的 API 和依赖版本，编译过不代表语义对。
>
> 最大的陷阱是"测试全绿就 approve"——AI 写的测试和它的代码**同源同错**，我必须人工补 AI 想不到的用例（并发、越权、空值）。
>
> 落地上按风险分级：低风险接口机器扫过就行，支付/权限这类高风险接口逐行 review + 安全扫描，**禁止 AI 独立完成**。
