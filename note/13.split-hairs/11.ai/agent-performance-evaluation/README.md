# 面试官问：如何量化一个 Agent 的性能？

> 一句话定位：**Agent 性能 ≠ LLM 性能**。面试官追问的不是"模型准确率多少"，而是"**任务完成率、步骤效率、工具使用效果、成本效益比**"——四层指标 + 五种评估方法。完整指标体系见 [主模块深度章节](../../../11.ai/07-llmops/05-agent-evaluation/README.md)。

> **系列定位**：高频 AI 工程面试题（Agent 系统成为主流，评估是核心问题）。考察的不是"LLM 准确率"，而是 **任务完成度** + **步骤效率** + **成本效益比**。

## 引子：面试官的"分层"陷阱

面试官："你们做的 Agent 系统，怎么量化性能？"

阿明答："我们用 GPT-4 做 Judge，准确率 92%。"

面试官追问："**准确率是指什么？LLM 单次调用的准确率，还是整个 Agent 完成任务的准确率？**"

阿明答："呃……是 LLM 的输出和标准答案对比的准确率。"

面试官："**那 Agent 跑 10 步才能完成的任务，第 3 步错了但第 7 步自己纠正了，这种怎么算？**"

阿明愣住。

**这道题的陷阱**：面试官考察的不是 LLM 单次调用的质量（那是 LLM 评估），而是 **整个 Agent 系统的端到端性能**（任务完成率、步骤效率、工具使用效果、成本效益比）。

**核心区分**：
- **LLM 评估**：单次调用质量（准确性、相关性、安全性）
- **Agent 评估**：多步任务的整体表现（完成率、效率、成本、稳定性）

今天讲清楚：
1. Agent 性能量化的 6 大指标维度
2. 5 种评估方法怎么组合用
3. 3 大常见陷阱怎么避
4. 面试怎么答这题

## Q1：Agent 性能量化和 LLM 评估有什么本质区别？

**答**：**层次不同**。

| 维度 | LLM 评估 | Agent 评估 |
|------|---------|-----------|
| **评估对象** | 单次 LLM 调用 | 多步任务执行 |
| **核心问题** | "这个答案对不对？" | "任务完成了没？花了多少步？成本多少？" |
| **指标** | 准确率 / BLEU / ROUGE | 任务完成率 / 步骤效率 / 工具成功率 |
| **时间维度** | 无（单次） | 有（多步循环） |
| **成本维度** | Token 消耗 | 多步累计成本 + 重试成本 |
| **失败模式** | 幻觉 / 不相关 | 卡死 / 循环 / 工具调用失败 / 成本爆炸 |

**关键洞察**：Agent 可能每一步 LLM 调用都"对"，但整体任务失败（比如工具调用顺序错、陷入死循环、成本超标）。

---

## Q2：Agent 性能量化的 6 大指标维度是什么？

**答**：

| 维度 | 核心指标 | 说明 |
|------|---------|------|
| **1. 任务完成度** | 任务成功率 / 部分完成率 | 端到端完成目标的比例 |
| **2. 步骤效率** | 平均步数 / 最优步数比 | 完成同样任务，Agent A 用 5 步，Agent B 用 12 步 |
| **3. 工具使用效果** | 工具成功率 / 工具选择准确率 | 调了 10 次工具，几次成功？选对工具了吗？ |
| **4. 成本效益** | 单任务成本 / Token 效率 | 完成一个任务花了多少钱？多少 Token？ |
| **5. 用户满意度** | 点赞率 / 任务完成时长 | 用户觉得有用吗？等了多久？ |
| **6. 稳定性** | 重复执行一致率 / 失败恢复率 | 跑 10 次，几次成功？失败了能自己恢复吗？ |

**反直觉点**：
- **任务成功率 90% 但成本爆炸** → 不可持续（每任务 $10）
- **步骤效率高但任务完成率低** → 快但错（3 步完成但 50% 失败）
- **工具成功率 100% 但工具选择准确率低** → 执行没问题但选错工具

**最佳实践**：6 个维度必须**同时看**，单维度优化会掉入陷阱。

## Q3：5 种评估方法怎么组合用？

**答**：

| 方法 | 适用阶段 | 成本 | 准确性 | 组合建议 |
|------|---------|------|--------|---------|
| **1. 黄金集评估** | 开发期 / CI | 低 | 中 | 每次发版必跑（回归测试） |
| **2. LLM-as-Judge** | 开发期 / 生产 | 中 | 中高 | 用 GPT-4 评估复杂任务（需校准） |
| **3. 用户模拟** | 上线前 | 中 | 中 | 用 LLM 模拟用户跑 100 个场景 |
| **4. 人工抽检** | 生产期 | 高 | 最高 | 每月 100-200 条抽样 |
| **5. A/B Test** | 生产期 | 中 | 真实 | 新版本上线前必做 |

**推荐组合**：
```
开发期：黄金集（回归）+ LLM-as-Judge（复杂任务）
上线前：用户模拟（100 场景）+ 人工抽检（50 条）
生产期：A/B Test（新版本）+ 持续监控（指标漂移）
```

**关键技巧**：
- **LLM-as-Judge 需校准**：先用 50 条人工标注，计算 LLM Judge 与人类的一致性（Cohen's Kappa > 0.6 才可信）
- **用户模拟要逼真**：不是随机 query，而是从真实日志抽样
- **A/B Test 是金标准**：用真实用户数据，别只信离线评估

---

## Q4：3 大常见陷阱怎么避？

**答**：

| 陷阱 | 症状 | 修复 |
|------|------|------|
| **混淆 LLM 评估 vs Agent 评估** | 只报"LLM 准确率 92%"，不说任务完成率 | 明确分层：LLM 层 + Agent 层 |
| **单维度优化** | 任务完成率 95% 但每任务 $10 | 6 维度同时看，设约束（成本 < $1 / 任务） |
| **无评估集 = 盲飞** | "感觉挺准的"但没量化 | 建 50+ 条黄金集，CI 集成 |

**最严重的陷阱是"无评估集"**——你不知道真实性能，每次改 Prompt / 换模型都赌运气。

---

## Q5：面试怎么答这题？

**30 秒简版**：
> "Agent 性能量化要分 6 个维度：**任务完成率、步骤效率、工具使用效果、成本效益、用户满意度、稳定性**。评估方法上，开发期用**黄金集 + LLM-as-Judge**，上线前用**用户模拟 + 人工抽检**，生产期用 **A/B Test + 持续监控**。关键是**不能只看 LLM 准确率**，要看端到端的任务完成率和成本效益比。"

**60 秒扩展版**：
> "具体来说，我们会建一个 50 条以上的黄金集（从真实用户日志抽样），每次发版跑回归。评估时不仅看任务完成率，还看步骤效率（完成同样任务用了多少步）和成本效益（每任务花了多少钱）。LLM-as-Judge 我们用 GPT-4，但会先用 50 条人工标注校准（Cohen's Kappa > 0.6 才可信）。上线前跑用户模拟（LLM 模拟用户跑 100 场景），上线后做 A/B Test。我们之前有个案例，LLM 准确率 92% 但任务完成率只有 68%——因为 Agent 在第 3 步工具调用时经常选错工具，后来优化工具描述才提到 85%。"
