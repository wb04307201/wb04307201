# Agent Evaluation 评测系统设计 · 阿里一面题

> 一句话定位：Agent 评测 = **6 大指标（任务 40% + 步骤 20% + 工具 10% + 成本 10% + 满意 15% + 稳定 5%）+ 5 种方法组合 + 4 阶段 Pipeline + 7 反模式**。完整深度见 [主模块 agent-evaluation 专题](../../../11.ai/05-agent-evaluation/README.md)。

> **系列定位**：经典阿里 / 字节 / 美团 Agent 岗一面必问。考察的不是"评测是什么"，而是 **6 维量化能力** + **5 方法组合** + **Pipeline 全链路** + **7 反模式避坑**。

## 引子：阿里一面 CTO 阿明的 3 个崩溃现场

```text
场景：2024 Q4 某大厂 Agent 平台 CTO 阿明——
- 痛点 1：Agent 上线时评测跑得很好，但用户投诉"回复慢 / 不准 / 重复问"
- 痛点 2：A/B 实验跑了 2 天显示新版更好，结果第 3 天崩了
- 痛点 3：对手 Agent 升级后没监测，性能悄悄掉了 10% 没人知道
```

**决策现场**：
1. **架构师候选人会问**：「设计 Agent Evaluation 评测系统的 6 大指标是什么？」
2. **资深候选人会问**：「5 种评测方法怎么组合？LLM-as-Judge 怎么避免偏见？」
3. **CTO 候选人会问**：「Pipeline 各阶段怎么串？漂移检测怎么做？自动回滚？」

普通候选人会答"用 GPT-4 当裁判"——踩中"**单一指标、单一 Judge、一次性评测**"3 大雷区。
高分候选人会答：**6 维 + 5 方法 + 4 阶段 Pipeline + 7 反模式 + 多模型投票 + 漂移检测**。

## 一、核心原理（必选）

### 1.1 6 大评测指标

| 权重 | 指标 | 一句话 |
|------|------|--------|
| **40%** | 任务完成率 | Agent 是否真的完成任务 |
| **20%** | 步骤效率 | 最优步骤数 / 实际步骤数 |
| **10%** | 工具使用准确率 | 选对工具 + 参数正确 + 调用顺序 |
| **10%** | 成本 | 单任务 token / 美元 |
| **15%** | 用户满意度 | CSAT / NPS / 行为推断 |
| **5%** | 稳定性 | 同任务多次成功率方差 |

**生产阈值**：综合 ≥ 0.80 且任务完成率 ≥ 0.85

### 1.2 5 种评测方法组合

| 方法 | 适用 | 客观度 |
|------|------|--------|
| **1 自动化指标** | 有 ground truth 任务 | ⭐⭐⭐⭐⭐ |
| **2 黄金集（500 条）** | 离线核心测试 | ⭐⭐⭐⭐⭐ |
| **3 LLM-as-Judge** | 开放式输出 | ⭐⭐⭐ |
| **4 A/B Test** | 上线决策 | ⭐⭐⭐⭐⭐ |
| **5 用户模拟（1000+）** | 并发压测 | ⭐⭐⭐ |

**实战组合**：
- 80% 自动化（核心指标）+ 20% LLM-judge（开放式）
- + A/B Test + 用户模拟（上线前）

### 1.3 4 阶段 Pipeline

```
黄金集（持续维护）
  → 离线 CI（PR 触发，阈值 0.85）
  → A/B Test（5% 流量，7 天）
  → 在线监控（漂移检测 + 自动回滚）
```

### 1.4 LLM-as-Judge 反偏见

```
单一 Judge → 3 偏见（位置 / 长度 / 同源）

正解：3-5 个模型投票 + 截尾均值
```

完整见 [主模块 03-llm-as-judge.md](../../../11.ai/05-agent-evaluation/03-llm-as-judge.md)

## 二、面试话术（90 秒版本 / 6 问）

> ⚠️ **模板不是背答案**——面试现场结合题目微调。

### 题目 A：阿里一面"如何设计 Agent Evaluation"

**高分答案**（4 层递进，60-90 秒）：

```
1. 【场景区分 · 15 秒】
"Agent ≠ LLM：多步 + 工具 + 状态，LLM 评测不够"

2. 【6 大指标 · 30 秒】
"40% 任务完成率 + 20% 步骤效率 + 10% 工具准确率 +
 10% 成本 + 15% 满意度 + 5% 稳定性。综合 ≥ 0.80 通过"

3. 【5 种方法 · 30 秒】
"自动化（核心）+ 黄金集（500 条）+ LLM-as-Judge（多模型投票）
 + A/B Test + 用户模拟"

4. 【4 阶段 Pipeline · 30 秒】
"黄金集持续维护 → CI PR 触发阈值 0.85 → A/B 5% 流量 7 天
 → 在线漂移检测 + 自动回滚"

5. 【7 反模式 · 15 秒】
"单一指标 / 一次性 / A/B 太短 / 黄金集不更新 / 阈值 100% /
 无漂移告警 / 测试集太干净"
```

### 题目 B：单一 GPT-4 judge 不行吗？

**高分答案**（40 秒）：

```
"不行。GPT-4 Judge 有 3 大偏见：
- 位置偏好（喜欢第一个）
- 长度偏好（喜欢长答）
- 同源偏见（GPT-4 判 GPT-4 的输出更松）

正解：3-5 个不同模型投票 + 截尾均值
（GPT-4 + Claude + Gemini，去掉最高最低）

并且每个 Judge 看 trajectory + output 给出 4 维度分数：
任务完成度（40%）+ 输出质量（30%）+ 工具使用（20%）+ 鲁棒性（10%）"
```

### 题目 C：A/B Test 用多少流量？

**高分答案**（35 秒）：

```
"梯度放量：
5% → 24h 监控
25% → 24h
50% → 24h
100%

时长：≥ 7 天（含工作日 + 周末 + 高峰期）

显著性：任务完成率显著提升（p-value < 0.05）+
         无显著恶化（延迟 / 成本 / 满意度）

回滚：一键切回 control（秒级生效）
      关键阈值：完成率下降 ≥ 2% 立即回滚"
```

### 题目 D：漂移检测怎么做？

**高分答案**（30 秒）：

```
"3 步：
1. 每日快测 50 条
2. 与历史 30 天数据 KS 检验（p < 0.05 = 显著漂移）
3. 自动告警 + 可选自动回滚

实战阈值：
- 单日 score 跌破 0.80 → 告警
- 连续 3 天下降 → 触发回滚 review
- 行业事件 / 新法规触发 → 立即重测

工具：阿里云 / Datadog / Langfuse / Helicone
提示：漂移检测是 SRE 的延续，不是评测组的专利"
```

### 题目 E：黄金集怎么构建？

**高分答案**（35 秒）：

```
"3 阶段：
1. 初始（业务专家手工）：500 条
   - 难度分级：easy / medium / hard / adversarial
   - 标注：期望输出 + 期望工具 + 期望步骤数

2. 持续（每周自动）：
   - 用户反馈失败用例 +10 条/周
   - 边缘 case 单独标注

3. 季度 review：
   - 答案是否过时（业务变更）
   - 难度分级是否准确
   - 是否有重复

反模式：黄金集永远 1 年前标的数据 → 业务早就变了"
```

### 题目 F：评测和业务脱钩怎么办？

**高分答案**（30 秒）：

```
"5 大对齐方法：
1. A/B Test 看业务指标（CSAT / 留存 / GMV）
2. 黄金集包含业务关键场景（不是技术指标）
3. 评测分数与线上投诉关联分析（每月 review）
4. 用户访谈补充黄金集（季度）
5. 业务方参与黄金集标注（不是工程师单方面）

反模式：工程师自嗨评测，业务方觉得'分数高了但用户走了'"
```

## 三、常见陷阱（必选，7 个核心反模式）

### 陷阱 1：只看任务完成率

- **错误**：完成率 100% 但每任务 50 步 / $5
- **真相**：用户根本没耐心等完
- **修复**：6 维加权 + 阈值 0.80

### 陷阱 2：上线前跑一次就发布

- **错误**：评测一次高 → 发布 → 出事故
- **真相**：评测数据 ≠ 生产真实数据
- **修复**：每次 PR CI + A/B + 在线监控

### 陷阱 3：A/B Test 时间太短

- **错误**：跑 1-2 天下结论
- **真相**：工作日 / 周末流量差异
- **修复**：≥ 7 天 + 显著性检验

### 陷阱 4：黄金集不更新

- **错误**：1 年前标的数据
- **真相**：业务早已变化
- **修复**：每周补充 + 月度 review

### 陷阱 5：阈值不合理

- **错误**：100% 完成率 = 永不发版
- **真相**：过于严格阻塞迭代
- **修复**：阈值 80-85%（按业务调整）

### 陷阱 6：漂移无告警

- **错误**：在线运行无监控
- **真相**：事故 3 天才发现
- **修复**：每日漂移检测 + 自动告警 + 回滚

### 陷阱 7：测试集太干净

- **错误**：黄金集全 happy path
- **真相**：生产遇到对抗 query 就崩
- **修复**：注入对抗样本（乱序 / 拼写错 / 模糊）

## 四、最佳实践（4 大场景方案）

### 方案 A：MVP（团队 1-3 人）

```
- 50 条黄金集
- 自动化指标 + GPT-4 Judge（单模型先够用）
- PR 触发 CI
- 阈值 0.80 阻塞
```

### 方案 B：增长（团队 5-10 人）

```
- 200 条黄金集（每周补 10）
- 多模型 Judge（GPT-4 + Claude + Gemini + 截尾均值）
- A/B Test 5% 流量
- 在线监控大盘
```

### 方案 C：成熟（团队 > 10 人）

```
- 500+ 条黄金集（持续优化）
- 用户模拟 1000+ 并发
- 漂移检测 + KS 检验
- 自动回滚机制
- LLM-judge 多模型投票
```

### 方案 D：金融 / 高合规

```
- 100% 黄金集覆盖（业务关键）
- 人工审核兜底（每天抽样 10%）
- 审计日志完整留存
- 阈值 ≥ 0.99（任务完成率）
```
