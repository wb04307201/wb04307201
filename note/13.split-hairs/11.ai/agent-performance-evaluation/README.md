# Agent Evaluation 评测系统设计 · 阿里一面题

> 一句话定位：Agent 评测 = **6 大指标（任务 40% + 步骤 20% + 工具 10% + 成本 10% + 满意 15% + 稳定 5%）+ 5 种方法组合 + 4 阶段 Pipeline + 7 反模式**。完整深度见 [主模块 agent-evaluation 专题](../../../11.ai/06-agent-evaluation/README.md)。

> **系列定位**：经典阿里 / 字节 / 美团 Agent 岗一面必问。考察的不是"评测是什么"，而是 **6 维量化能力** + **5 方法组合** + **Pipeline 全链路** + **7 反模式避坑**。

---

## 引子：面试官的"分层"陷阱

```text
场景：2024 Q4 某大厂 Agent 平台 CTO 阿明面试——
```

面试官："你们做的 Agent 系统，怎么量化性能？"

阿明答："我们用 GPT-4 做 Judge，准确率 92%。"

面试官追问："**准确率是指什么？LLM 单次调用的准确率，还是整个 Agent 完成任务的准确率？**"

阿明答："呃……是 LLM 的输出和标准答案对比的准确率。"

面试官："**那 Agent 跑 10 步才能完成的任务，第 3 步错了但第 7 步自己纠正了，这种怎么算？**"

阿明愣住。

**这道题的陷阱**：面试官考察的不是 LLM 单次调用的质量（那是 LLM 评估），而是 **整个 Agent 系统的端到端性能**（任务完成率、步骤效率、工具使用效果、成本效益比）。

**核心区分**：

| 维度 | LLM 评估 | Agent 评估 |
|------|---------|-----------|
| **评估对象** | 单次 LLM 调用 | 多步任务执行 |
| **核心问题** | "这个答案对不对？" | "任务完成了没？花了多少步？成本多少？" |
| **指标** | 准确率 / BLEU / ROUGE | 任务完成率 / 步骤效率 / 工具成功率 |
| **时间维度** | 无（单次） | 有（多步循环） |
| **成本维度** | Token 消耗 | 多步累计成本 + 重试成本 |
| **失败模式** | 幻觉 / 不相关 | 卡死 / 循环 / 工具调用失败 / 成本爆炸 |

**关键洞察**：Agent 可能每一步 LLM 调用都"对"，但整体任务失败（比如工具调用顺序错、陷入死循环、成本超标）。

普通候选人会答"用 GPT-4 当裁判"——踩中"**单一指标、单一 Judge、一次性评测**"3 大雷区。
高分候选人会答：**6 维 + 5 方法 + 4 阶段 Pipeline + 7 反模式 + 多模型投票 + 漂移检测**。

## 一、核心原理（必选）

### 1.1 6 大评测指标

| 权重 | 指标 | 一句话 |
|------|------|--------|
| **40%** | 任务完成率 | Agent 是否真的完成任务 |
| **20%** | 步骤效率 | 最优步骤数 / 实际步骤数 |
| **10%** | 工具使用准确率 | 选对工具 + 参数正确 + 调用顺序 |
| **10%** | 成本 | 单任务 token / 美元 |
| **15%** | 用户满意度 | CSAT / NPS / 行为推断 |
| **5%** | 稳定性 | 同任务多次成功率方差 |

**生产阈值**：综合 ≥ 0.80 且任务完成率 ≥ 0.85

**反直觉点**：
- **任务成功率 90% 但成本爆炸** → 不可持续（每任务 $10）
- **步骤效率高但任务完成率低** → 快但错（3 步完成但 50% 失败）
- **工具成功率 100% 但工具选择准确率低** → 执行没问题但选错工具

### 1.2 5 种评测方法组合

| 方法 | 适用 | 客观度 |
|------|------|--------|
| **1 自动化指标** | 有 ground truth 任务 | ⭐⭐⭐⭐⭐ |
| **2 黄金集（500 条）** | 离线核心测试 | ⭐⭐⭐⭐⭐ |
| **3 LLM-as-Judge** | 开放式输出 | ⭐⭐⭐ |
| **4 A/B Test** | 上线决策 | ⭐⭐⭐⭐⭐ |
| **5 用户模拟（1000+）** | 并发压测 | ⭐⭐⭐ |

**实战组合**：
- 80% 自动化（核心指标）+ 20% LLM-judge（开放式）
- + A/B Test + 用户模拟（上线前）

**关键技巧**：
- **LLM-as-Judge 需校准**：先用 50 条人工标注，计算 LLM Judge 与人类的一致性（Cohen's Kappa > 0.6 才可信）
- **用户模拟要逼真**：不是随机 query，而是从真实日志抽样
- **A/B Test 是金标准**：用真实用户数据，别只信离线评估

### 1.3 4 阶段 Pipeline

```text
黄金集（持续维护）
  → 离线 CI（PR 触发，阈值 0.85）
  → A/B Test（5% 流量，7 天）
  → 在线监控（漂移检测 + 自动回滚）
```

### 1.4 LLM-as-Judge 反偏见

```text
单一 Judge → 3 偏见（位置 / 长度 / 同源）

正解：3-5 个模型投票 + 截尾均值
（GPT-4 + Claude + Gemini，去掉最高最低）
```

完整见 [主模块 03-llm-as-judge.md](../../../11.ai/06-agent-evaluation/03-llm-as-judge.md)

## 二、面试话术（6 问精选）

> ⚠️ **模板不是背答案**——面试现场结合题目微调。

### 题目 A：如何量化一个 Agent 的性能？（30 秒简版 + 90 秒详版）

**30 秒简版**：
> "Agent 性能量化要分 6 个维度：**任务完成率 40%、步骤效率 20%、工具使用效果 10%、成本效益 10%、用户满意度 15%、稳定性 5%**。评估方法上，开发期用**黄金集 + LLM-as-Judge**，上线前用**用户模拟 + 人工抽检**，生产期用 **A/B Test + 持续监控**。关键是**不能只看 LLM 准确率**，要看端到端的任务完成率和成本效益比。"

**90 秒详版**（4 层递进）：

```text
1. 【场景区分 · 15 秒】
"Agent ≠ LLM：多步 + 工具 + 状态，LLM 评测不够"

2. 【6 大指标 · 30 秒】
"40% 任务完成率 + 20% 步骤效率 + 10% 工具准确率 +
 10% 成本 + 15% 满意度 + 5% 稳定性。综合 ≥ 0.80 通过"

3. 【5 种方法 · 30 秒】
"自动化（核心）+ 黄金集（500 条）+ LLM-as-Judge（多模型投票）
 + A/B Test + 用户模拟"

4. 【4 阶段 Pipeline · 30 秒】
"黄金集持续维护 → CI PR 触发阈值 0.85 → A/B 5% 流量 7 天
 → 在线漂移检测 + 自动回滚"
```

> "具体来说，我们会建一个 50 条以上的黄金集（从真实用户日志抽样），每次发版跑回归。评估时不仅看任务完成率，还看步骤效率（完成同样任务用了多少步）和成本效益（每任务花了多少钱）。LLM-as-Judge 我们用 GPT-4，但会先用 50 条人工标注校准（Cohen's Kappa > 0.6 才可信）。我们之前有个案例，LLM 准确率 92% 但任务完成率只有 68%——因为 Agent 在第 3 步工具调用时经常选错工具，后来优化工具描述才提到 85%。"

### 题目 B：单一 GPT-4 judge 不行吗？

**高分答案**（40 秒）：

```text
"不行。GPT-4 Judge 有 3 大偏见：
- 位置偏好（喜欢第一个）
- 长度偏好（喜欢长答）
- 同源偏见（GPT-4 判 GPT-4 的输出更松）

正解：3-5 个不同模型投票 + 截尾均值
（GPT-4 + Claude + Gemini，去掉最高最低）

并且每个 Judge 看 trajectory + output 给出 4 维度分数：
任务完成度（40%）+ 输出质量（30%）+ 工具使用（20%）+ 鲁棒性（10%）"
```

### 题目 C：A/B Test 用多少流量？

**高分答案**（35 秒）：

```text
"梯度放量：
5% → 24h 监控
25% → 24h
50% → 24h
100%

时长：≥ 7 天（含工作日 + 周末 + 高峰期）

显著性：任务完成率显著提升（p-value < 0.05）+
         无显著恶化（延迟 / 成本 / 满意度）

回滚：一键切回 control（秒级生效）
      关键阈值：完成率下降 ≥ 2% 立即回滚"
```

### 题目 D：漂移检测怎么做？

**高分答案**（30 秒）：

```text
"3 步：
1. 每日快测 50 条
2. 与历史 30 天数据 KS 检验（p < 0.05 = 显著漂移）
3. 自动告警 + 可选自动回滚

实战阈值：
- 单日 score 跌破 0.80 → 告警
- 连续 3 天下降 → 触发回滚 review
- 行业事件 / 新法规触发 → 立即重测

工具：阿里云 / Datadog / Langfuse / Helicone"
```

### 题目 E：黄金集怎么构建？

**高分答案**（35 秒）：

```text
"3 阶段：
1. 初始（业务专家手工）：500 条
   - 难度分级：easy / medium / hard / adversarial
   - 标注：期望输出 + 期望工具 + 期望步骤数

2. 持续（每周自动）：
   - 用户反馈失败用例 +10 条/周
   - 边缘 case 单独标注

3. 季度 review：
   - 答案是否过时（业务变更）
   - 难度分级是否准确
   - 是否有重复

反模式：黄金集永远 1 年前标的数据 → 业务早就变了"
```

### 题目 F：评测和业务脱钩怎么办？

**高分答案**（30 秒）：

```text
"5 大对齐方法：
1. A/B Test 看业务指标（CSAT / 留存 / GMV）
2. 黄金集包含业务关键场景（不是技术指标）
3. 评测分数与线上投诉关联分析（每月 review）
4. 用户访谈补充黄金集（季度）
5. 业务方参与黄金集标注（不是工程师单方面）

反模式：工程师自嗨评测，业务方觉得'分数高了但用户走了'"
```

## 三、常见陷阱（7 个核心反模式）

### 陷阱 1：只看任务完成率

- **错误**：完成率 100% 但每任务 50 步 / $5
- **真相**：用户根本没耐心等完
- **修复**：6 维加权 + 阈值 0.80

```yaml
# ❌ 错误：只看任务完成率
eval:
  metrics: [task_success_rate]
  threshold: 0.95  # 只看这一个数

# ✅ 正确：6 维加权评分
eval:
  score: 0.4 * task_completion + 0.2 * step_efficiency
       + 0.1 * tool_accuracy + 0.1 * cost
       + 0.15 * satisfaction + 0.05 * stability
  threshold: 0.80  # 综合分 ≥ 0.80 才通过
```

### 陷阱 2：上线前跑一次就发布

- **错误**：评测一次高 → 发布 → 出事故
- **真相**：评测数据 ≠ 生产真实数据
- **修复**：每次 PR CI + A/B + 在线监控

```yaml
# ❌ 错误：手动跑一次评测就发布
deploy:
  steps:
    - run: python eval.py  # 跑一次
    - run: ./deploy.sh     # 直接发

# ✅ 正确：CI 自动评测 + 阈值门禁 + 灰度发布
deploy:
  stages: [eval, canary, production]
  eval:
    - run: python eval.py --threshold 0.85
    - fail_if: score < 0.85  # 不达标就阻塞
  canary:
    - traffic: 5%            # 先放 5% 流量
    - duration: 7d           # 观察 7 天
```

### 陷阱 3：A/B Test 时间太短

- **错误**：跑 1-2 天下结论
- **真相**：工作日 / 周末流量差异
- **修复**：≥ 7 天 + 显著性检验

### 陷阱 4：黄金集不更新

- **错误**：1 年前标的数据
- **真相**：业务早已变化
- **修复**：每周补充 + 月度 review

### 陷阱 5：阈值不合理

- **错误**：100% 完成率 = 永不发版
- **真相**：过于严格阻塞迭代
- **修复**：阈值 80-85%（按业务调整）

### 陷阱 6：漂移无告警

- **错误**：在线运行无监控
- **真相**：事故 3 天才发现
- **修复**：每日漂移检测 + 自动告警 + 回滚

### 陷阱 7：测试集太干净

- **错误**：黄金集全 happy path
- **真相**：生产遇到对抗 query 就崩
- **修复**：注入对抗样本（乱序 / 拼写错 / 模糊）

## 四、最佳实践（4 大场景方案）

### 方案 A：MVP（团队 1-3 人）

```text
- 50 条黄金集
- 自动化指标 + GPT-4 Judge（单模型先够用）
- PR 触发 CI
- 阈值 0.80 阻塞
```

### 方案 B：增长（团队 5-10 人）

```text
- 200 条黄金集（每周补 10）
- 多模型 Judge（GPT-4 + Claude + Gemini + 截尾均值）
- A/B Test 5% 流量
- 在线监控大盘
```

### 方案 C：成熟（团队 > 10 人）

```text
- 500+ 条黄金集（持续优化）
- 用户模拟 1000+ 并发
- 漂移检测 + KS 检验
- 自动回滚机制
- LLM-judge 多模型投票
```

### 方案 D：金融 / 高合规

```text
- 100% 黄金集覆盖（业务关键）
- 人工审核兜底（每天抽样 10%）
- 审计日志完整留存
- 阈值 ≥ 0.99（任务完成率）
```
