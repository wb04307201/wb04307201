# 10亿 vs 千亿级向量检索：架构演进 + 5 个关键转变

> 一句话定位：10B（十亿）级用 DiskANN 单机多盘可解；100B-1000B（千亿）级必须**分布式 + 分层索引 + 2 跳检索 + GPU 加速**——是**完全不同的技术栈**。深度原理见 [主模块千亿级专题](../../../11.ai/02-technology-stack/vector-search-at-scale/README.md) + [10亿级专题（兄弟）](../../../11.ai/02-technology-stack/vector-search-algorithms/README.md)。

> **系列定位**：经典 AI 架构面试题（中高频、架构师级）。考察的不是"千亿级能不能做"，而是 **5 个关键架构转变** + **业界真实案例** + **延迟 vs 召回 vs 成本三元权衡**。

## 引子：面试官的"规模陷阱"

阿明去面试某搜索引擎公司，面试官问：

> "10亿级向量我们刚做完了。现在业务要扩到 1000 亿，怎么做？"

阿明答："加机器？分片？"

面试官追问："分 100 个分片，每个分片查 top-100，全局排序需要合并多少数据？"

阿明答："……10000 个候选？"

面试官："但 1000 亿级，单分片 10 亿向量，单跳检索 < 5ms 已经极限了。多分片并行查询再合并 → 100ms 起步。你如何保证 P99 < 20ms？"

阿明答不上来。

**这道题的陷阱在于**：很多人以为"千亿级 = 10 亿级 × 100 = 加机器"。但实际上：

- 10B → 单机 DiskANN ✅
- 100B → 分布式 DiskANN + 分片路由 ⚠️
- 1000B → ScANN + GPU + 分层 + 联邦 ⚠️⚠️

**是质变不是量变**。

今天讲清楚：**5 个关键架构转变 + 业界真实案例 + 延迟-召回-成本权衡**。

## 一、核心原理：规模差异的本质

### 1.1 数据规模 vs 硬件约束

| 规模 | 数据（FP16, 1024 维）| 内存（DiskANN 图）| 磁盘 | 单机可行？ |
|------|---------------------|-------------------|------|----------|
| 10M | 20 GB | 200 MB | 100 GB | ✅ |
| 100M | 200 GB | 2 GB | 1 TB | ✅ |
| 10B | 20 TB | 200 GB | 4 TB（压缩） | ✅ 单机多盘 |
| **100B** | **200 TB** | **2 TB** | **40 TB** | ❌ 必须分布式 |
| **1000B** | **2 PB** | **20 TB** | **400 TB** | ❌ 多集群 |

### 1.2 关键洞察：千亿级不是 10 亿级 × 100

**错误的认知**：
```
千亿级 = 10 亿级 × 100 = 加 100 台机器 + 分片
```

**正确的认知**：
```
千亿级 = 完全不同的技术栈：
- 单跳 → 2 跳检索
- 单一索引 → 分层索引（hot/warm/cold）
- CPU → CPU + GPU 混合
- 整数索引 → 极限压缩（1-bit / 2-bit）
- 单机房 → 跨数据中心联邦
```

## 二、5 个关键架构转变

### 2.1 转变 1：从"单跳"到"2 跳"检索

**10B 单跳**：
```
Query → DiskANN 图搜索 → top-k（< 5ms）
```

**100B+ 2 跳**：
```
Query → 分片路由
      ↓
  ┌─ 分片 1: 粗排 top-1000（< 5ms）
  ├─ 分片 2: 粗排 top-1000
  ├─ ... (并行)
  └─ 分片 N: 粗排 top-1000
      ↓
  全局精排：合并 top-10000 → top-10（< 10ms）
      ↓
  返回
```

**关键点**：
- 粗排只用压缩向量（DiskANN 图 + PQ 量化）
- 精排需要读全量原始向量（缓存 / SSD）
- 2 跳总延迟 10-20ms，100B 可达

### 2.2 转变 2：从"单一索引"到"分层索引"（Hot/Cold Tiering）

```
100B+ 数据分层：
┌─────────────────────────────────┐
│ Hot tier（5%，500M）             │
│ - HNSW 全内存                   │
│ - 延迟 < 1ms                    │
│ - 适合：热门文档 / 频繁查询       │
├─────────────────────────────────┤
│ Warm tier（20%，2B）              │
│ - DiskANN + NVMe SSD            │
│ - 延迟 5-10ms                   │
│ - 适合：常规文档                  │
├─────────────────────────────────┤
│ Cold tier（75%，7.5B）            │
│ - IVF-PQ + 对象存储（S3）       │
│ - 延迟 50-100ms                 │
│ - 适合：冷门文档 / 归档           │
└─────────────────────────────────┘
```

**业界案例**：Bing / Google 搜索都采用 hot/warm/cold 三层。

### 2.3 转变 3：从"单机副本"到"分布式分片 + 副本"

**100B 分片架构**：
```
                  路由层（Consistent Hashing）
                            │
        ┌───────────────────┼───────────────────┐
        ↓                   ↓                   ↓
    分片 1               分片 2               分片 N
    5B 向量              5B 向量              5B 向量
    DiskANN              DiskANN              DiskANN
    + 3 副本             + 3 副本             + 3 副本
```

**关键设计**：
- 分片数 = √N（100B → 10-20 分片）
- 副本数 = 2-3（高可用）
- 路由策略：主键 hash（一致性 hash）
- 跨分片查询：并行 + 合并

**1000B 分片架构**：
- 100+ 节点
- 跨数据中心副本（异地容灾）
- 路由层独立部署（独立服务集群）
- 一致性 hash + 虚拟节点

### 2.4 转变 4：从"CPU"到"GPU 加速 + 极限压缩"

| 算法/方案 | 适用规模 | 关键创新 |
|----------|---------|---------|
| **ScANN**（Google 2020）| 10B-100B | 各向异性向量量化（AVQ），压缩比 4x |
| **GPU-DiskANN**（NVIDIA 2024）| 10B-100B | GPU 加速图遍历 + DiskANN |
| **Faiss 1.x**（Meta）| 10B-100B | IVF-PQ + GPU（100x 加速）|
| **SPANN**（微软 2023）| 100B+ | SSD + 内存混合 + 近似图 |
| **Falcon**（Meta 2024）| 100B+ | 极限压缩 + 联邦检索 |

**GPU 加速原理**：
```
CPU 距离计算：512 维 × 1 亿向量 × 8 字节 = 50ms
GPU 距离计算：512 维 × 1 亿向量 × 8 字节 = 0.5ms（100x）
```

### 2.5 转变 5：从"通用召回"到"查询路由"

100B+ 不可能一次查全量，必须按业务分流：

```
用户查询 → Query Router → 选 1-3 个相关分片 → 检索
```

**3 种路由策略**：

| 策略 | 适用 | 实现 |
|------|------|------|
| **按主键范围** | 用户/商品 ID 类 | Consistent Hashing |
| **按主题路由** | 文档类（多领域）| 小模型判断查询主题 |
| **按时间路由** | 新闻 / 实时数据 | 时间窗分片 |

## 三、千亿级业界真实案例

### 3.1 Bing 搜索（千亿级网页向量）

- 规模：1000B+ 网页
- 方案：分层（hot/warm/cold）+ ScANN + GPU + 跨数据中心
- 延迟：P99 < 100ms

### 3.2 Google 搜索（万亿级 +）

- 规模：万亿级
- 方案：ScANN + 自研 TPU + 分片 + 联邦
- 延迟：P99 < 200ms（搜索场景可接受）

### 3.3 Meta 推荐（千亿级 user/item 向量）

- 规模：1000B+ user × item
- 方案：Faiss GPU + 分片 + 联邦
- 延迟：P99 < 50ms（推荐场景）

### 3.4 Pinterest 视觉搜索（百亿级图像）

- 规模：10B+ 图像
- 方案：DiskANN + 分片（部分自研）
- 延迟：P99 < 30ms

### 3.5 Spotify 音乐推荐（百亿级 audio 向量）

- 规模：10B+ 音频
- 方案：Faiss + HNSW 全内存
- 延迟：P99 < 20ms

## 四、延迟 vs 召回 vs 成本三元权衡

### 4.1 权衡矩阵

| 优先级 | 延迟目标 | 推荐方案 | 召回损失 | 成本 |
|--------|---------|---------|---------|------|
| **延迟优先** | < 5ms | Hot tier + HNSW 全内存 | < 5% 数据 | $50万+ |
| **召回优先** | < 100ms | ScANN + 全数据 + GPU | 0% | $200万+ |
| **成本优先** | < 50ms | DiskANN + 压缩 | 2-5% | $10万+ |
| **平衡** | < 20ms | 分层索引 + 2 跳 | 3-5% | $30万+ |

### 4.2 千亿级还要不要"毫秒级"？

**反直觉结论**：千亿级的瓶颈**不是延迟**，而是**召回率 + 成本 + 一致性**。

- 能做到 5ms：极限压缩 + GPU → 召回损失 5-10%
- 不必做到 5ms：很多场景（推荐 / 搜索）P99 100ms 已够用
- 真正 5ms 场景：通常只查 hot tier（5% 数据），不是全量

## 五、常见陷阱（面试必踩）

### 陷阱 1：把千亿级当 10 亿级 × 100

错。千亿级是**质变**，需要完全不同的技术栈。

### 陷阱 2：以为单跳检索可行

错。100B+ 单跳极限 ~5ms / 分片，2 跳检索是常态。

### 陷阱 3：忽视分层索引

错。没有分层（hot/warm/cold），千亿级成本会爆炸。

### 陷阱 4：以为 GPU 加速 = 解决一切

半对。GPU 加速距离计算，但**图遍历受 CPU 限制**——需要 GPU-DiskANN 等算法协同。

### 陷阱 5：把延迟当唯一指标

错。千亿级要权衡延迟 / 召回 / 成本，**优先级看业务场景**。

### 陷阱 6：忽视一致性 / 副本同步

错。分布式场景下，**索引更新 + 副本同步**是工程难题。

## 六、面试话术（90 秒版本）

面试官："千亿级向量检索怎么做？"

**30 秒简版**：

> "千亿级（100B+）是 10 亿级（10B）的**质变**，不是简单加机器。需要 **5 个架构转变**：
> 1. 单跳 → **2 跳**（粗排 + 精排）
> 2. 单一索引 → **分层索引**（hot/warm/cold tiering）
> 3. 单机 → **分布式分片 + 副本**（100+ 节点）
> 4. CPU → **GPU 加速**（ScANN / Faiss / GPU-DiskANN）
> 5. 通用召回 → **查询路由**（按主键 / 主题 / 时间分流）
>
> 业界案例：Bing 千亿级用 ScANN + GPU + 分层；Google 万亿级用 TPU + 联邦。千亿级瓶颈**不是延迟，是召回 + 成本 + 一致性**。"

**60 秒扩展版**（如果面试官追问细节）：

> "具体来说，千亿级必须做**分布式 + 分层**：
>
> **分层架构**（3 层）：
> - Hot tier（5% 数据）：HNSW 全内存，< 1ms 命中
> - Warm tier（20%）：DiskANN + NVMe SSD，< 10ms 命中
> - Cold tier（75%）：IVF-PQ + S3，< 100ms 命中
>
> **查询流程**（2 跳）：
> 1. Query Router 选 1-3 个相关分片
> 2. 各分片用压缩向量粗排 top-1000（5-10ms）
> 3. 全局精排合并 top-10000 → top-10（10-20ms）
> 4. 精排需要读全量原始向量（cache + SSD）
>
> **业界方案**：ScANN（Google 2020，各向异性量化 4x）、Faiss GPU（100x 距离加速）、SPANN（微软 2023，SSD + 内存混合）。
>
> 关键是：**不要追求千亿级毫秒级**，应该权衡延迟 / 召回 / 成本——很多场景 P99 100ms 已够用。"
