# LLM Benchmark 深度剖析：分数是怎么测出来的？

> 大模型排行榜上的分数背后藏着什么？考察的不是"哪个模型分高"，而是 **Benchmark 的设计原理** + **计分方法** + **数据污染** + **刷分手段** + **如何正确看榜**。完整评估方法见 [LLM 评估体系](../../../11.ai/07-llmops/04-llm-evaluation/README.md)。

> **系列定位**：AI 工程面试题（模型选型 / 评估必考）。覆盖从经典 Benchmark 到 Chatbot Arena 的完整评测生态。

---

## 引子：GPT-4 的 MMLU 90% 意味着什么？

```text
新闻标题："GPT-4 在 MMLU 上达到 90.3%，超越人类平均水平！"

你的第一反应：GPT-4 比大多数人聪明？

真相：
  MMLU = 57 个学科的 15,000 道多选题
  90% = 做对了 90% 的选择题
  人类平均 = 80%（本科生水平）
  
但问题是：
  1. 多选题 ≠ 开放式推理
  2. 训练数据可能包含 MMLU 原题（数据污染）
  3. 90% 的 MMLU ≠ 90% 的实际工作能力
```

面试官问："Benchmark 分数可信吗？怎么判断一个模型的真实能力？"

如果你只能回答"看排行榜"——面试就结束了。

## 一、5 大主流 Benchmark 分类

### 1.1 知识理解类

| Benchmark | 测什么 | 格式 | 题量 | 计分 |
|-----------|--------|------|------|------|
| **MMLU** | 57 学科知识 | 4 选 1 多选 | ~15,000 题 | 正确率 % |
| **ARC** | 科学推理 | 4 选 1 | ~8,000 题 | 正确率 % |
| **HellaSwag** | 常识推理 | 4 选 1 完形 | ~10,000 题 | 正确率 % |
| **TruthfulQA** | 真实性 | 开放式 | 817 题 | LLM Judge % |

### 1.2 代码能力类

| Benchmark | 测什么 | 格式 | 题量 | 计分 |
|-----------|--------|------|------|------|
| **HumanEval** | 函数生成 | 给 docstring 写代码 | 164 题 | Pass@1 % |
| **MBPP** | Python 编程 | 给描述写代码 | 974 题 | Pass@1 % |
| **SWE-bench** | 真实 GitHub Issue | 给 Issue 写 Patch | 2,294 题 | Resolve % |

**HumanEval Pass@1 的含义**：
```text
给模型一个函数的 docstring + 测试用例
模型生成 1 次代码
如果通过了所有测试用例 → Pass
Pass@1 = 164 题中 1 次就通过的比例

例：Pass@1 = 80% → 164 题中 131 题一次通过
```

### 1.3 数学推理类

| Benchmark | 测什么 | 格式 | 题量 | 计分 |
|-----------|--------|------|------|------|
| **GSM8K** | 小学数学应用题 | 开放式数值 | ~8,500 题 | 正确率 % |
| **MATH** | 高中/竞赛数学 | 开放式 | ~12,500 题 | 正确率 % |
| **AIME** | 美国数学邀请赛 | 整数答案 | 每年 30 题 | 正确率 % |

### 1.4 对话质量类

| Benchmark | 测什么 | 格式 | 计分 |
|-----------|--------|------|------|
| **MT-Bench** | 多轮对话 | 80 题，2 轮对话 | GPT-4 打分（1-10） |
| **AlpacaEval** | 指令跟随 | 805 题 | GPT-4 胜率 % |
| **WildBench** | 真实用户查询 | 1,024 题 | LLM Judge |

### 1.5 竞技场（Arena）类

| 平台 | 原理 | 计分 |
|------|------|------|
| **Chatbot Arena**（LMSYS） | 盲测对比：两个匿名模型回答同一问题，用户投票 | ELO Rating |
| **Open LLM Leaderboard**（HuggingFace） | 自动跑多个 Benchmark 取平均 | 综合分 |

## 二、Chatbot Arena ELO 评分原理

### 2.1 ELO 是什么？

ELO 源自国际象棋排名，核心思想：**赢了强对手加分多，赢了弱对手加分少**。

```text
每场比赛：
  模型 A vs 模型 B（匿名，用户不知道谁是谁）
  用户投票：A 胜 / B 胜 / 平局
  
  ELO 更新公式：
    Ra' = Ra + K × (Sa - Ea)
    
    Ra = A 当前分数
    K  = 调整系数（通常 32）
    Sa = 实际结果（1=胜, 0.5=平, 0=负）
    Ea = 预期胜率 = 1 / (1 + 10^((Rb-Ra)/400))
```

### 2.2 为什么 Arena 被认为最可信？

| 优势 | 说明 |
|------|------|
| **盲测** | 用户不知道哪个是 GPT-4 哪个是 Claude → 无品牌偏见 |
| **真实场景** | 用户提自己的问题 → 不是固定题库 |
| **抗刷分** | 没有固定题目 → 无法在训练集中"作弊" |
| **统计显著** | 50 万+ 投票 → 结果可靠 |

### 2.3 Arena 的局限

| 局限 | 说明 |
|------|------|
| **用户偏好 ≠ 真实质量** | 用户偏爱"长回答"和"有格式的回答" |
| **英语偏重** | 80%+ 投票来自英语用户 |
| **新模型样本不足** | 新上线模型投票数少，分数不稳定 |
| **不测安全** | 不评估有害内容、幻觉率 |

## 三、Benchmark 的 5 大已知问题

### 3.1 数据污染（Contamination）

```text
问题：Benchmark 的题目可能已经出现在模型的训练数据中

证据：
  - GPT-4 在 MMLU 上 90%，但在"去污染"版本上降到 82%
  - 某些模型在 GSM8K 上分数异常高，但在自出题上表现一般

检测方法：
  - N-gram 重叠：检查 Benchmark 题目与训练数据的文本重叠
  - 变体测试：修改题目中的数字/名字，看模型是否仍然正确
  - 时间切割：只用模型训练截止日期之后发布的 Benchmark
```

### 3.2 Benchmark 刷分（Benchmark Gaming）

```text
手段 1：过度优化特定 Benchmark
  → 训练时加入大量 Benchmark 风格数据
  → 分数涨了，但泛化能力没涨

手段 2：选择性报告
  → 只报告分数高的 Benchmark，忽略分数低的
  → "我们的模型在 X 上 SOTA！"（但 Y、Z 上很差）

手段 3：多次采样取最高
  → Pass@1 = 80%，但 Pass@100 = 95%
  → 报告 Pass@100 但标注为"准确率"
```

### 3.3 指标与真实能力脱节

| Benchmark 高分 | 不代表 |
|---------------|--------|
| MMLU 90% | 能做 90% 的实际工作 |
| HumanEval 90% | 能写 90% 的生产代码 |
| GSM8K 95% | 能做 95% 的数学建模 |

**原因**：Benchmark 是标准化、简化的测试，真实场景更复杂、更模糊。

### 3.4 评估偏差

```text
LLM-as-Judge 偏差：
  - 位置偏差：倾向选第一个/最后一个答案
  - 长度偏差：倾向选更长的回答
  - 自我偏差：倾向选自己生成的回答
  
人工评估偏差：
  - 锚定效应：先看的答案影响判断
  - 疲劳效应：评估越多越不仔细
```

### 3.5 复现性差

```text
同一模型跑两次 Benchmark，分数可能不同：
  - 温度 > 0 → 输出随机
  - 不同硬件 → 浮点精度差异
  - 不同 prompt 格式 → 结果差异
  
"Exact match" 尤其脆弱：
  "42" vs "42." vs "The answer is 42" → 全部算错
```

## 四、如何正确看排行榜

### 4.1 看榜 checklist

| 检查项 | 为什么重要 |
|--------|-----------|
| **谁测的？** | 模型厂商自测 vs 第三方独立测试 |
| **测的哪个版本？** | 同一模型不同版本分数不同 |
| **有去污染吗？** | 是否排除了训练数据中的 Benchmark 题目 |
| **多次采样还是单次？** | Pass@1 vs Pass@k 差距可能很大 |
| **prompt 格式统一吗？** | 不同 prompt 格式分数差 5-10% |
| **有置信区间吗？** | 分数 ±2% 以内可能无统计显著差异 |

### 4.2 最可信的评测排序

```text
可信度从高到低：

1. Chatbot Arena ELO（盲测 + 真实场景 + 大样本）
2. 第三方独立评测（LMSYS、Open LLM Leaderboard）
3. 多 Benchmark 综合分（不只看一个 Benchmark）
4. 模型厂商自报（可能有选择性报告）
5. 单一 Benchmark 分数（最不可靠）
```

## 五、面试话术（90 秒版本）

> "Benchmark 分数怎么来的，我从 3 个层面回答：
>
> **分类**：5 大类——知识（MMLU/ARC）、代码（HumanEval/SWE-bench）、数学（GSM8K/MATH）、对话（MT-Bench）、竞技场（Chatbot Arena）。每类测的维度不同，不能跨类比较。
>
> **计分**：选择题直接算正确率；代码题用 Pass@k（生成 k 次至少 1 次通过）；对话题用 LLM-as-Judge 打分；Arena 用 ELO 评分（盲测对比 + 用户投票）。
>
> **可信度**：Benchmark 有 5 大已知问题——数据污染（训练集包含原题）、刷分（过度优化特定 Benchmark）、指标脱节（高分 ≠ 高能力）、评估偏差（LLM Judge 偏爱长回答）、复现性差。最可信的是 Chatbot Arena（盲测 + 真实场景 + 50 万投票），最不可信的是单一 Benchmark 自报分数。
>
> 看榜 checklist：谁测的、哪个版本、有去污染吗、多次采样还是单次、prompt 格式统一吗。"
