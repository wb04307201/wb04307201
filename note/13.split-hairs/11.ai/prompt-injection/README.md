# Prompt Injection 面试深挖（4 大核心）

> ⬅️ [返回 AI 咬文嚼字](../README.md) | [主模块深度专题](../../../11.ai/08-llmops/05-llm-security/README.md)

> **一句话定位**：**4 大核心 prompt injection 题面试深挖版**：直接注入 / 间接注入 / 6 层防御 / 红队测试。

---

## 🎯 4 大核心题

### Q1：直接注入与间接注入的本质区别？

**陷阱**：
- ❌ 只答“用户 vs 系统输入”（不够深）
- ❌ 不知道攻击链中的“信任边界”

**30 秒话术**：
> “直接注入 = **用户输入**携带恶意指令（当前对话）；间接注入 = **外部数据源**（邮件 / 网页 / 文档）藏指令（影响所有用此数据源的对话）。”

**90 秒话术**：
> “**核心是信任边界没划清**。直接注入是用户在 prompt 里写‘忽略以上规则’；间接注入则由模型读取不可信外部内容时触发，攻击者甚至不必直接接触受害者的对话。”
>
> “例如 EchoLeak（CVE-2025-32711）：攻击者发一封邮件 → Microsoft 365 Copilot 解析邮件正文 → 被隐式指令劫持 → 越权调工具 → 敏感数据外泄，**零点击**完成。公开行业材料给出的风险量级是：**83% 已上线 Agent 应用存在至少一种注入漏洞（IBM 2024）**，平均损失 $1.2M。”
>
> “所以不能把注入理解成关键词绕过。它利用的是 LLM 同时处理‘指令’与‘数据’、却无法可靠区分二者的结构性缺陷。**注入 = LLM 时代 OWASP LLM01:2025 头号风险**。”

#### 面试追问：两类注入的爆炸半径为何不同？

- **直接注入**通常绑定当前用户、当前会话，入口显式，较容易记录与回放。
- **间接注入**潜伏在共享邮件、网页、知识库或 RAG 文档中，可跨用户、跨会话复用。
- Agent 一旦同时拥有外部内容读取、敏感数据访问、外部通信三种能力，注入就可能升级为完整外泄链。
- 防守时要把“谁提交内容”与“谁最终受影响”分开建模，而不是只检查聊天输入框。

### Q2：6 层防御栈是什么？为什么不能靠单一防线？

**陷阱**：
- ❌ 只答“做输入审核”（远不够）
- ❌ 不知道“纵深防御”理念

**30 秒话术**：
> “6 层 = **网络 / 输入 / 模型 / 输出 / 工具 / 审计**。任何一层被攻破，其他层兜底。”

**90 秒话术**：
> “参考主模块 [LLM 安全攻防实战](../../../11.ai/08-llmops/05-llm-security/README.md) 第七节。网络层限制来源与出口；输入层识别越狱、编码与外部恶意内容；模型层强化指令层级和安全对齐；输出层检测敏感信息与危险内容；工具层落实最小权限、白名单与二次确认；审计层记录 prompt、检索、模型响应和工具调用。”
>
> “**关键原则：① 输入 / 指令严格分离**（用 messages 而非字符串拼接）；**② 外部内容当数据**（不进 system prompt）；**③ 工具白名单 + 权限门禁**（敏感操作需二次确认）；**④ 输入 + 输出双侧审核**（OpenAI Moderation / 自建分类器）；**⑤ 完整审计日志**（事后 5 分钟定位）。”
>
> “**不要把防御寄托在‘prompt 写得好’**——攻击者总能通过编码、角色扮演、多轮上下文或外部内容诱导出漏洞，必须做纵深防御。单层追求 100% 拦截既不现实，也容易制造大量误报。”

#### 6 层职责速记

| 层 | 核心控制 | 典型失败后的兜底 |
|---|---------|-----------------|
| 网络 | 来源校验、域名与出站限制 | 阻断向攻击者域名回传 |
| 输入 | 注入分类、编码还原、内容隔离 | 模型层继续遵守指令优先级 |
| 模型 | system / developer / user 分层、安全对齐 | 输出层识别泄密结果 |
| 输出 | PII / 密钥 / 危险内容检测 | 工具层拒绝实际执行 |
| 工具 | 白名单、最小权限、二次确认 | 限制越权动作爆炸半径 |
| 审计 | 全链路日志、告警、回放 | 快速定位并沉淀回归样例 |

#### 为什么不是“加一个注入检测器”就结束？

- 检测器本身也存在假阴性；编码混淆和多语言变体会持续演进。
- 合法业务文本可能包含“忽略”“执行”等词，关键词封禁会产生假阳性。
- 即使模型被诱导，只要工具无敏感权限、出口受控，攻击仍难落地。
- 即使发生泄露，完整审计也能回答“读了什么、调用了什么、发向哪里”。

### Q3：OWASP LLM Top 10 (2025) 怎么记？常见漏洞定位？

**陷阱**：
- ❌ 答 10 个具体漏洞但不知道顺序
- ❌ 混淆 2023 版与 2025 版

**30 秒话术**：
> “2025 版本里**最严重的 3 个：LLM01 提示注入 / LLM06 过度自主（Agent 越权）/ LLM02 敏感信息泄露**。OWASP 每年更新。”

**90 秒话术**：
> “完整 10 项：LLM01 Prompt Injection / LLM02 Sensitive Information Disclosure / LLM03 Supply Chain / LLM04 Data and Model Poisoning / LLM05 Improper Output Handling / LLM06 Excessive Agency（Agent 越权）/ LLM07 System Prompt Leakage / LLM08 Vector and Embedding Weaknesses / LLM09 Misinformation / LLM10 Unbounded Consumption。”
>
> “**OWASP Gen AI Security Project** 是权威来源（genai.owasp.org）。面试不应只背清单，而要能把真实攻击链映射到多个条目：外部文档藏指令是 LLM01；Agent 不经确认调用邮件或数据库工具是 LLM06；敏感结果进入响应或外传是 LLM02；应用未经转义执行模型生成的 SQL、HTML 或 shell 则是 LLM05。”
>
> “**写作时引用具体年份**（如 LLM01:2025）避免混淆。2025 版不是 2023 版的简单重排，名称与边界都有变化；回答时先声明版本，再列编号和定位。”

#### 攻击链定位示例

```text
恶意网页进入 RAG
  → LLM01 Prompt Injection
Agent 自动读取内部资料并调用外发工具
  → LLM06 Excessive Agency
机密出现在响应或攻击者可访问的链接中
  → LLM02 Sensitive Information Disclosure
下游应用直接执行模型生成内容
  → LLM05 Improper Output Handling
```

#### 记忆方法

- **入口**：01 注入、03 供应链、04 投毒、08 向量弱点。
- **处理**：05 输出处理、06 过度自主、10 无界消耗。
- **结果**：02 敏感泄露、07 系统提示泄露、09 错误信息。
- 编号用于沟通和审计，控制措施仍要按资产、信任边界与攻击链设计。

### Q4：如何设计 1 套红队测试方案？

**陷阱**：
- ❌ 答“找漏洞测试”（太笼统）
- ❌ 不知道自动化工具链

**30 秒话术**：
> “红队 = **用对抗性 prompt 库主动攻击**，分自动 + 人工。自动工具：Garak / PyRIT / Prompt Armor；人工：定期场景演练 + 编码绕过。”

**90 秒话术**：
> “**核心 5 步**：① 准备攻击 prompt 库（直接 / 间接 / 越权 / 泄露 4 类）；**② 用 AI 自动化生成攻击变种**（Garak 支持 NVIDIA + DeepEval，PyRIT 是 Microsoft 出品）；**③ 跑在 CI/CD 里**（每次模型 / 系统更新自动回归）；**④ 失败案例转 DPO 训练数据**（闭环修复）；**⑤ 严重漏洞走 RLHF 再对齐**。”
>
> “**实战工具清单**：`Garak`（NVIDIA 开源红队，约 50 类攻击）/ `PyRIT`（Microsoft 开源红队 + 风险识别）/ `Prompt Armor`（云端 API 防护）/ `Lakera Guard`（商业 API 输入审核）。人工测试补足自动扫描盲区，重点覆盖 Base64、Unicode、分词扰动、多语言、角色扮演、多轮诱导与间接注入。”
>
> “**关键**：红队不是找 bug，是找**训练数据缺口**。每个失败样例都要进入可复现用例、风险分级、修复措施和回归基线；只有从发现到数据再到对齐形成闭环，安全能力才会随版本提升。”

#### 5 步红队闭环

1. **建库**：按直接注入、间接注入、越权工具、敏感泄露四类维护基准集。
2. **变异**：生成编码、语言、上下文位置、多轮组合和语义等价变种。
3. **执行**：CI/CD 比较攻击成功率、误拒率、泄露率和工具越权率。
4. **修复**：优先修权限与架构，再补规则、分类器和 prompt；失败样例转 DPO 数据。
5. **再对齐**：系统性严重缺口进入 RLHF / RLAIF，并保留固定回归集防止复发。

#### 通过标准不能只看“模型拒答”

- **攻击成功率**：恶意指令实际改变任务或系统行为的比例。
- **敏感泄露率**：密钥、PII、系统提示或内部文档是否进入可见输出。
- **工具越权率**：是否未经授权触发读、写、发信、转账等操作。
- **误拒率**：正常含安全术语或代码的请求是否被错误拦截。
- **可追溯性**：失败后能否在 5 分钟内还原输入、检索、推理与工具链路。

失败案例转成 DPO 偏好对时，可继续阅读 [LLM 对齐面试深挖](../llm-alignment/README.md)，理解 SFT、RLHF、DPO 与 RLAIF 在安全闭环中的分工。
