# 设计一个大文件上传系统 —— 分片 + 断点续传 + 秒传

> 一句话定位：**后端 / 架构面试经典系统设计题**。考察的不是"文件怎么上传"，而是 **分片协议设计** + **断点续传状态管理** + **秒传去重原理** + **存储选型**。深度实战见 [主模块深度章节](../../../../04.system-design/04-high-performance/file-upload/README.md)。

## 引子：面试经典开场

面试官："设计一个大文件上传系统，支持 10GB 文件、断点续传、秒传，QPS 1000。"

大多数人答："前端分片上传。"

面试官追问：
1. "分片大小怎么定？太大太小有什么问题？"
2. "断点续传的 uploadId 怎么管理？过期分片怎么清理？"
3. "秒传用 MD5 去重，MD5 碰撞了怎么办？"
4. "用户删除文件后，物理文件能立即删吗？"

大多数人卡在追问上。**这道题考察的不是"知道分片"，而是"协议设计 + 状态管理 + 存储去重"。**

## 一、核心原理

### 1.1 三大机制

| 机制 | 原理 | 核心挑战 |
|------|------|---------|
| **分片上传** | 文件切成 5MB chunk，并发上传 | 分片大小选择 + 并发控制 |
| **断点续传** | uploadId + 已上传分片记录 | 状态管理 + 过期清理 |
| **秒传** | 上传前查 MD5，存在则引用 | MD5 碰撞 + 引用计数 |

### 1.2 完整流程

```
1. 前端计算文件 MD5
2. POST /init {md5, size} → 秒传检查
3. 秒传命中 → 完成（O(1)）
4. 未命中 → 前端分片（5MB/chunk）
5. 并发 PUT /chunk（3-6 片）
6. 所有分片完成 → POST /merge
7. 后端合并 + 校验 MD5 → 返回文件 URL
```

## 二、7 道精选面试题

### Q1：分片大小怎么定？太大或太小有什么问题？

**答**：推荐 **5MB**，是请求数和单次传输量的平衡点。

| 分片大小 | 10GB 文件分片数 | 问题 |
|----------|---------------|------|
| 1MB | 10,000 片 | 请求太多，HTTP 开销大 |
| **5MB** | **2,000 片** | **平衡点**（推荐） |
| 100MB | 100 片 | 单片失败重传代价高，前端内存大 |

**动态调整**：根据网速自动调整——网速快时分片大（减少请求数），网速慢时分片小（降低单片失败代价）。

### Q2：断点续传怎么实现？uploadId 怎么管理？

**答**：3 层设计——

1. **uploadId 生成**：`POST /init` 时生成 UUID，作为本次上传会话标识
2. **分片状态**：Redis Set 记录已完成分片（`upload:{id}:chunks = {0,1,...,99}`），TTL 24h
3. **续传流程**：断网重连后重新 `POST /init`，后端返回 `uploadedChunks`，前端从断点继续

**过期清理**：定时任务扫描 24h 未活动的 session，删除 OSS 临时分片 + Redis 状态。

### Q3：秒传的原理？MD5 碰撞怎么办？

**答**：
- **秒传原理**：上传前计算文件 MD5，查询 DB 是否已有相同 MD5 的文件。已有 → 创建引用记录（ref_count + 1），不传输任何数据
- **碰撞防护**：MD5 + 文件大小双重校验（两个维度同时碰撞概率趋近于 0）。生产环境可用 SHA-256 替代

### Q4：用户删除文件后，物理文件能立即删吗？

**答**：不能。多个用户可能引用同一物理文件（秒传共享）。

**方案**：引用计数（ref_count）
- 删除时 `ref_count - 1`
- `ref_count == 0` 时才删除物理文件
- 定时 GC 扫描无引用的 orphan object

### Q5：前端上传进度条怎么实现？

**答**：

```javascript
// 基于已上传分片数 / 总分片数
const progress = uploadedChunks.size / totalChunks;
progressBar.style.width = (progress * 100) + '%';

// 精确到字节：XMLHttpRequest.upload.onprogress
xhr.upload.onprogress = (e) => {
    const chunkProgress = e.loaded / e.total;
    // 总分片进度 + 当前分片进度
};
```

### Q6：前端直传 OSS（预签名 URL）有什么好处？

**答**：前端直传 = 后端生成预签名 URL → 前端直接 PUT 到 OSS，不经过后端服务器。

| 维度 | 经过后端 | 预签名直传 |
|------|---------|-----------|
| 后端带宽 | 所有数据经过后端 | 仅签名请求 |
| 延迟 | 2 跳（前端→后端→OSS） | 1 跳（前端→OSS） |
| 安全 | 后端控制 | URL 有效期 + 权限范围 |

### Q7：高并发场景下上传系统怎么优化？

**答**：4 层优化——

1. **CDN 上传加速**：就近边缘节点接收，回源 OSS
2. **预签名直传**：前端直传 OSS，后端只做签名和元数据
3. **Redis 缓存状态**：uploadId 和分片状态放 Redis（避免 DB 频繁写入）
4. **异步合并**：合并任务放 MQ，不阻塞上传 API

## 三、常见陷阱

**陷阱 1：分片不校验 MD5**

```
// ❌ 分片传输损坏但没发现 → 合并后文件损坏
// ✅ 每片上传时校验 chunk MD5，不匹配则重传
```

**陷阱 2：uploadId 不设过期**

```
// ❌ 用户上传一半放弃 → 临时分片永久占用存储
// ✅ Redis TTL 24h + 定时清理 orphan chunks
```

**陷阱 3：秒传只查 MD5 不查文件大小**

```
// ❌ MD5 碰撞（极低概率）→ 返回错误文件
// ✅ WHERE md5 = ? AND file_size = ?
```

## 四、面试话术（30 秒版）

> "大文件上传系统核心三个机制：分片、断点续传、秒传。
>
> 分片：前端把文件切成 5MB chunk，3-6 片并发上传到 OSS。后端逐片接收，Redis 记录已完成分片。
>
> 断点续传：每个上传会话有 uploadId，Redis Set 记录已完成的分片索引（TTL 24h）。断网重连后查询已上传分片，从断点继续。
>
> 秒传：上传前算文件 MD5 发给后端查询，已存在则创建引用记录（ref_count + 1），不传输任何数据。物理文件用内容寻址（object_key = md5），多用户共享。
>
> 删除时引用计数 -1，ref_count == 0 才删物理文件。高并发用预签名直传（前端→OSS）减少后端带宽压力。"
