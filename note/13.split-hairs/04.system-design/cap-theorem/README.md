# CAP 定理与实际选型

## 引子：为什么不能"既要又要还要"？

想象你有两台服务器做数据同步：

```text
        ┌─ Server A ─┐
写请求 ─→│  数据: 100  │──复制──→│  数据: 100  │─ Server B ─┐
        └────────────┘        └────────────┘
```

网络突然断了（分区发生 P）：

- **选择一致性（CP）**：拒绝写入，直到网络恢复 → 数据一致，但部分请求失败
- **选择可用性（AP）**：两边都接受写入 → 都能响应，但数据不一致了

CAP 定理告诉你：**分布式系统中，三者不可兼得**。你必须做出取舍。

---

> 📚 **前置知识**：[分布式系统](../../04.system-design/README.md)

## 一、核心原理

CAP 定理由 Eric Brewer 于 2000 年提出，是分布式系统设计的基石理论。它指出一个分布式系统最多只能同时满足以下三个特性中的两个：

| 特性 | 定义 | 典型表现 |
|------|------|----------|
| **Consistency（一致性）** | 所有节点在同一时刻看到的数据完全相同 | 写入后，任何后续读取都能读到最新值 |
| **Availability（可用性）** | 每个请求都能在合理时间内收到非错误的响应 | 不保证返回最新数据，但保证有响应 |
| **Partition Tolerance（分区容错性）** | 网络分区（部分节点间通信中断）时系统仍能继续运行 | 即使部分消息丢失，系统整体不崩溃 |

**关键结论**：三者中最多选其二。由于分布式系统的网络本质决定了 P 是不可避免的，因此实际工程中真正面临的选择是 **CP（强一致）** 还是 **AP（高可用）**。

---

## 二、为什么 P 是必须的

在单机系统中，不存在网络分区问题，可以兼顾 C 和 A。但在分布式系统中：

1. **网络故障是常态而非异常**：交换机宕机、网线断裂、DNS 故障、防火墙策略变更等都会导致节点间通信中断
2. **规模越大，分区概率越高**：随着节点数量增加，至少一对节点失联的概率趋近于 1
3. **云环境加剧了不确定性**：虚拟化层、容器编排、跨可用区部署都引入了额外的网络故障点

因此，**P 是分布式系统的底线要求**。放弃 P 意味着当网络出现轻微抖动时整个系统瘫痪，这在实际生产中是不可接受的。

于是，CAP 的实际博弈简化为：**在网络分区发生时，优先保证一致性（CP）还是可用性（AP）？**

---

## 三、CP 系统

CP 系统在检测到网络分区时，会选择牺牲部分节点的可用性，确保数据强一致。

### 3.1 ZooKeeper（ZAB 协议）

ZooKeeper 使用 ZAB（ZooKeeper Atomic Broadcast）协议实现原子广播：

- **过半写成功机制**：写入操作需要获得超过半数节点的确认才算成功
- **Leader-Follower 架构**：只有 Leader 能处理写请求，Follower 同步 Leader 的状态
- **分区行为**：当集群分裂时，只有包含多数派（Quorum）的子集群能继续提供服务；少数派节点自动进入只读或不可用状态

**典型场景**：分布式锁、服务注册发现（对一致性要求高的场景）、配置管理

### 3.2 etcd（Raft 协议）

etcd 基于 Raft 共识算法，其核心机制包括：

- **Leader 选举**：节点通过选举产生唯一 Leader，任期（Term）机制防止脑裂
- **日志复制**：Leader 将日志条目复制到 Follower，多数派确认后提交
- **安全性保证**：已提交的日志不会丢失，新 Leader 必须包含所有已提交的日志

**典型场景**：Kubernetes 集群状态存储、分布式协调服务

### 3.3 Redis Cluster

Redis Cluster 采用去中心化的分片方案：

- **16384 个哈希槽（Slot）**：键通过 CRC16 算法映射到槽，槽分配到不同主节点
- **多数派投票**：节点故障检测通过 gossip 协议传播，超过半数主节点标记某节点为 fail 才生效
- **分区行为**：如果某个槽的所有副本都不可达，该槽对应的写操作失败；如果集群中超过半数主节点失联，整个集群停止写服务

**典型场景**：高性能缓存、会话存储、计数器

### 3.4 HBase

HBase 依赖 HDFS 和 ZooKeeper 实现强一致：

- **RegionServer + HMaster 架构**：HMaster 负责元数据管理，RegionServer 处理读写
- **WAL（Write-Ahead Log）**：所有写操作先写日志再落盘，确保崩溃恢复时数据不丢失
- **分区行为**：ZooKeeper 检测 RegionServer 心跳，超时后触发区域迁移，期间对应数据短暂不可用

## 四、AP 系统

AP 系统在检测到网络分区时，选择保持所有节点的可用性，接受数据的临时不一致。

### 4.1 Eureka（Peer-to-Peer 复制）

Eureka 采用对等复制架构：

- **无中心节点**：所有 Eureka 节点地位平等，相互注册并同步数据
- **异步复制**：节点间的注册信息通过异步方式传播，存在短暂延迟
- **分区行为**：网络分区时，各子集群独立运行，各自接受注册和查询；分区恢复后通过增量同步收敛

**权衡**：客户端可能读到过期的服务列表，但保证了服务发现功能始终可用

### 4.2 Nacos（默认 AP + 可选 CP）

Nacos 的独特之处在于支持模式切换：

- **AP 模式（默认）**：使用 Distro 协议，节点间异步同步服务信息，适合服务发现场景
- **CP 模式**：切换到 Raft 协议，保证配置数据的强一致性，适合配置管理场景
- **动态切换**：同一集群中，临时实例走 AP，持久化实例走 CP

### 4.3 Cassandra（可调一致性）

Cassandra 提供灵活的一致性级别：

- **Tunable Consistency**：通过 `read_consistency` 和 `write_consistency` 参数调节
- **一致性级别**：ANY（最低，写任意节点即成功）、ONE、QUORUM、ALL（最高，等待所有节点）
- **Hinted Handoff**：目标节点不可用时，协调者暂存数据，待恢复后补发

**公式**：当 `W + R > N` 时（W 为写入副本数，R 为读取副本数，N 为总副本数），可以保证强一致性

## 五、BASE 理论

BASE 理论是对 CAP 中 AP 方案的延伸和工程化指导，由 eBay 架构师 Dan Pritchett 提出：

| 原则 | 含义 | 实践示例 |
|------|------|----------|
| **Basically Available（基本可用）** | 系统保证核心功能可用，允许部分功能降级 | 电商大促时关闭评论功能，保证下单可用 |
| **Soft State（软状态）** | 系统状态可以在一段时间内不同步，允许中间态 | 缓存与数据库之间的短暂不一致 |
| **Eventually Consistent（最终一致性）** | 经过一段时间后，所有副本会达到一致状态 | DNS 记录传播、CDN 内容刷新 |

**最终一致性的常见实现**：

1. **因果一致性**：有因果关系的数据保证顺序一致
2. **读己之所写**：用户能读到自己刚写入的数据
3. **会话一致性**：同一会话内的请求保证一致
4. **单调读一致性**：用户不会读到比之前更旧的数据
5. **单调写一致性**：用户的写操作按顺序执行

## 六、实际选型

### 6.1 选型决策矩阵

| 业务场景 | 推荐类型 | 核心理由 | 代表技术 |
|----------|----------|----------|----------|
| **服务注册中心** | AP | 服务发现允许短暂不一致，但不能不可用 | Eureka、Nacos（AP） |
| **分布式锁** | CP | 锁的正确性依赖强一致，否则出现双写 | ZooKeeper、etcd |
| **配置中心** | CP | 配置错误可能导致大面积故障，需强一致 | etcd、ZooKeeper |
| **分布式 ID 生成** | CP | ID 不能重复，需全局有序 | Snowflake（本地生成）、Leaf（号段模式） |
| **消息队列** | 视业务而定 | 金融交易需 CP，日志收集可 AP | Kafka（ISR 机制）、RocketMQ |
| **数据同步/ETL** | 视业务而定 | 实时报表可容忍延迟，账务结算需强一致 | Canal（AP）、Debezium |
| **社交 Feed 流** | AP | 用户能接受动态延迟加载 | Cassandra、MongoDB |

### 6.2 混合架构实践

现代系统往往不是单一的 CP 或 AP，而是分层设计：

```text
┌─────────────────────────────────────────┐
│           API Gateway (AP)              │  ← 高可用接入层
├─────────────────────────────────────────┤
│     业务服务层（无状态，水平扩展）        │  ← 通过幂等性+重试补偿
├─────────────────────────────────────────┤
│   缓存层 Redis (AP)  │  DB MySQL (CP)   │  ← 读写分离，最终一致
├─────────────────────────────────────────┤
│    协调层 ZooKeeper/etcd (CP)           │  ← 分布式锁/选主
└─────────────────────────────────────────┘
```

**关键设计原则**：

- **核心链路 CP，非核心链路 AP**：支付、扣库存走强一致；浏览历史、推荐走最终一致
- **写路径 CP，读路径 AP**：写入走共识协议保证正确，读取走副本提升吞吐
- **本地事务 + 消息队列实现最终一致**：通过事务消息、Saga、TCC 等模式补偿

## 七、面试话术（30 秒版）

> "CAP 定理说分布式系统中一致性、可用性、分区容错性三者最多满足两个。由于网络分区不可避免，P 是底线，所以实际是在 CP 和 AP 之间做选择。
>
> 像 ZooKeeper、etcd 这类系统走 CP 路线，通过 Raft/ZAB 等共识协议保证强一致，代价是分区时少数派不可用；而 Eureka、Cassandra 走 AP 路线，保证任何时候都能读写，但数据可能暂时不一致。
>
> 实际选型要看业务：服务注册中心选 AP，因为短暂读到过期服务列表影响不大，但注册功能不能挂；分布式锁和配置中心必须选 CP，因为锁失效或配置不一致会导致严重问题。BASE 理论就是对 AP 方案的工程化指导，通过最终一致性来平衡体验和复杂度。"
