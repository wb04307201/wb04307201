# 分库分表策略选型：什么时候分？怎么分？分完怎么查？

> 一句话定位：分库分表是数据库层面的"水平扩展终极手段" —— 但代价是查询复杂度指数级上升，过早分库分表是最常见的过度设计之一。

> **系列定位**：经典系统设计面试题（数据库、分布式、高并发高频）。考察的不是"分片算法怎么写"，而是 **何时该分** + **分片键选择** + **跨分片查询难题** + **平滑迁移能力**。

---

## 引子：2000 万行的订单表，查一次要 8 秒

```text
你的电商平台上线两年，订单表已经积累到 2000 万行。

慢查询日志：
  SELECT * FROM orders WHERE user_id = 12345 ORDER BY create_time DESC LIMIT 20
  → 耗时 8.3 秒

产品经理："用户投诉'我的订单'页面加载太慢。"
DBA："索引已经加满了，单表 2000 万行是 MySQL 的舒适区上限。"
你："那就分库分表？"
```

**但——你真的需要分库分表吗？** 还是先试试读写分离？或者垂直拆分？

分库分表不是万能药。一旦分了，**分布式 JOIN、跨分片分页、全局排序、分布式事务** 会接踵而至。很多团队分完之后的维护成本比不分还高。决策需要回答三个问题：什么时候该分？怎么分？分完怎么查？

## 一、核心原理

### 1.1 渐进路线（不要一步到位）

```text
单库单表 → 索引优化 → 读写分离 → 垂直拆分（按业务拆库） → 水平分片（按行拆分）
```

### 1.2 垂直拆分 vs 水平拆分

| 维度 | 垂直拆分 | 水平拆分 |
|------|---------|---------|
| **拆什么** | 按列/按业务拆到不同表/库 | 按行拆到多个结构相同的表 |
| **适用场景** | 表中有些列很少被查询 | 单表行数过大（> 1000 万） |
| **难度** | 低 | 高（跨分片查询难题） |

### 1.3 分片键选择（最关键决策）

选择原则：(1) **高频查询条件** —— 80% 查询都带这个字段；(2) **数据分布均匀**；(3) **业务关联数据在同一分片**。

```text
✅ 订单表按 user_id 分片 → 用户查自己的订单 = 单分片查询
❌ 订单表按 order_id 分片 → 用户查自己的订单 = 全分片扫描
```

## 二、代码示例

### 2.1 ShardingSphere-JDBC vs ShardingSphere-Proxy

| 维度 | ShardingSphere-JDBC | ShardingSphere-Proxy |
|------|---------------------|---------------------|
| **部署** | 嵌入应用 JAR | 独立进程 |
| **语言** | 仅 Java | 任意语言 |
| **性能** | 无网络开销 | 多一跳延迟 |
| **推荐** | Java 应用首选 | 多语言 / 遗留系统 |

### 2.2 ShardingSphere-JDBC 配置

```yaml
spring:
  shardingsphere:
    datasource:
      names: ds0,ds1
      ds0:
        jdbc-url: jdbc:mysql://10.0.1.1:3306/order_db_0
      ds1:
        jdbc-url: jdbc:mysql://10.0.1.2:3306/order_db_1
    rules:
      sharding:
        tables:
          orders:
            actual-data-nodes: ds${0..1}.orders_${0..1}
            table-strategy:
              standard:
                sharding-column: user_id
                sharding-algorithm-name: hash-mod
            key-generate-strategy:
              column: order_id
              key-generator-name: snowflake
        sharding-algorithms:
          hash-mod:
            type: HASH_MOD
            props:
              sharding-count: 4
```

### 2.3 四种分片算法

| 算法 | 优点 | 缺点 |
|------|------|------|
| **Hash 取模** | 数据均匀 | 扩缩容困难 |
| **范围分片** | 扩容易 | 可能数据倾斜 |
| **时间分片** | 历史归档方便 | 热点集中在最新分片 |
| **一致性哈希** | 扩缩容只迁移少量数据 | 实现复杂 |

## 三、常见陷阱

### 陷阱 1：数据倾斜
- **真相**：分片键选择不当（如按 `region` 分片），一线城市数据量是三线城市的 10 倍。**必须提前做数据分布分析**。

### 陷阱 2：跨分片 JOIN 无法执行
- **真相**：两表不在同一分片。解决方案：冗余字段、应用层组装、ShardingSphere 绑定表（Binding Tables）。

### 陷阱 3：跨分片分页性能灾难
- **真相**：`LIMIT 10 OFFSET 100000` 在 4 个分片上执行，每个分片扫 100010 行再合并。实际扫描量 = 400040 行。**禁止深分页**，改用游标分页（`WHERE create_time < ? LIMIT 20`）。

### 陷阱 4：分布式事务
- **真相**：分片后跨库操作变成分布式事务。详见 [分布式事务](../distributed-transaction/README.md)。

## 四、最佳实践

**全局 ID**：分库分表后不能用自增 ID，推荐 Snowflake 雪花算法（64 bit = 1 符号 + 41 时间戳 + 10 机器 ID + 12 序列号）。详见 [分布式 ID](../distributed-id/README.md)。

**平滑迁移（双写 + 对账）**：双写新旧库 → 历史数据迁移 → 对账验证 → 灰度切读 → 停写旧库。绝对不能一步到位直接切。

**决策清单**：

| 检查项 | 结论 |
|--------|------|
| 单表 < 500 万行 | ❌ 不需要分，优化索引 |
| 500 万 ~ 2000 万行 | ⚠️ 先读写分离 + 垂直拆分 |
| > 2000 万行 + 索引无效 | ✅ 考虑水平分片 |
| 有跨分片查询需求 | ⚠️ 重新评估分片键或引入 ES |

## 五、面试话术（90 秒版本）

> "分库分表的核心不是'怎么分'，而是'该不该分'。正确路线是 **渐进式**：先优化索引 → 读写分离 → 垂直拆分 → 最后水平分片。
>
> 水平分片的关键是 **分片键选择** —— 选高频查询条件且数据均匀的字段，比如订单表按 `user_id`。分片算法有四种：Hash 取模（均匀但扩缩容难）、范围分片（扩容容易但可能倾斜）、时间分片（适合日志）、一致性哈希（扩缩容最优）。推荐 ShardingSphere-JDBC 做中间件，全局 ID 用 Snowflake。
>
> 分片后三大难题：跨分片 JOIN（冗余字段或应用层组装）、跨分片分页（禁止深分页，改用游标）、分布式事务（Seata/TCC）。迁移方案用 **双写 + 对账 + 灰度切读**，绝对不能一步到位。"
