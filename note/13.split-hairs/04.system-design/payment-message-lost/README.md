# 支付成功没收到消息：全链路排查框架

> 经典系统设计面试题（支付回调、消息可靠性、分布式排查高频）。考察的不是"用哪个 MQ"，而是 **全链路排查思维** + **每段的根因分析** + **短期止血 vs 长期方案** + **面试结构化回答**。

---

## 引子：凌晨 2 点的客诉

```text
🚨 客诉：用户支付宝显示"支付成功"，但订单状态仍是"待支付"
```

支付网关确实扣了用户的钱，但订单系统没收到"支付成功"的消息。

用户截图、客服工单、财务对账全涌过来——你只有 10 分钟定位问题。

面试官问："你怎么排查？从哪开始？可能的根因有哪些？怎么预防？"

如果你只能回答"查日志"——面试基本结束了。

---

## 一、全链路消息流转模型

先建立排查地图。一条支付成功消息从产生到消费，经过 **5 个环节**，每个环节都可能丢消息：

```
支付网关（第三方）
  │
  ① 生产端：支付系统发送"支付成功"消息
  │   ├── 同步发送失败（网络超时、参数校验）
  │   └── 异步回调丢失（第三方没调、调了但失败）
  │
  ② 网络层：消息从支付系统到 MQ / 从第三方到你的服务器
  │   ├── HTTP 回调超时（DNS、防火墙、SLB 健康检查）
  │   └── TCP 连接断开 / 代理层吞请求
  │
  ③ MQ 层：消息进入 Broker 并持久化
  │   ├── 异步刷盘丢失（Broker 宕机时未落盘）
  │   ├── 主从切换丢失（主节点挂了，从节点没同步到）
  │   └── Topic / Queue 配置错误（消息发到错误的 Topic）
  │
  ④ 消费端：消费者拉取并处理消息
  │   ├── 消费者假死 / 宕机
  │   ├── 消费异常未重试（直接 ACK 丢弃）
  │   └── 消费超时（处理太慢被 MQ 认为失败，重新投递到死信队列）
  │
  ⑤ 业务层：消费逻辑正确执行
      ├── 幂等拦截（重复消息被误判为已处理）
      ├── 状态机卡住（订单状态不允许从"已取消"回到"已支付"）
      └── 事务未提交（消费成功但 DB 事务回滚）
```

## 二、逐段排查（5 步定位法）

### Step 1：确认消息是否发出（生产端）

```bash
# 查支付系统的发送日志
grep "支付成功" payment-service.log | grep "orderId=12345"

# 如果是第三方回调（支付宝/微信），查回调接收日志
grep "/api/payment/notify" access.log | grep "12345"
```

**可能发现**：
- 发送日志没有 → 支付系统根本没发消息（业务 bug / 配置错误）
- 回调日志没有 → 第三方没回调（回调 URL 配错 / 域名过期 / HTTPS 证书过期）
- 发送成功但返回超时 → 消息可能已发，但发送端超时重试导致重复

### Step 2：确认消息是否到达 MQ

```bash
# RocketMQ：查看消息轨迹
mqadmin QueryMsgById -n nameserver:9876 -i <msgId>

# Kafka：查看 offset 和 lag
kafka-consumer-groups.sh --describe --group payment-group --bootstrap-server localhost:9092
```

**可能发现**：
- 消息不在 MQ → Broker 丢了（刷盘配置问题 / 主从切换）
- 消息在 MQ 但消费者 lag 很大 → 消费积压（见 [MQ 消息积压](../mq-backlog/)）
- 消息在死信队列 → 消费失败多次被丢弃

### Step 3：确认消费者是否收到

```bash
# 查消费者的拉取日志 / 消费日志
grep "payment-success" consumer.log | grep "orderId=12345"
```

**可能发现**：
- 消费者没有拉取记录 → 消费者挂了 / 未订阅 / Group 配置错误
- 拉取了但处理异常 → 看异常堆栈
- 拉取了但超时 → 消费逻辑太慢（DB 慢查询 / 外部接口超时）

### Step 4：确认消费逻辑是否执行成功

```bash
# 查订单更新日志
grep "updateOrderStatus" order-service.log | grep "orderId=12345"

# 查数据库
SELECT order_id, status, update_time FROM orders WHERE order_id = 12345;
```

**可能发现**：
- 日志显示处理成功但 DB 没更新 → 事务未提交 / 连接池耗尽
- 日志显示"已处理，跳过" → 幂等拦截（可能是重复消息，也可能是误判）
- 日志显示状态机异常 → 订单已经是终态（如"已取消"），无法回到"已支付"

### Step 5：确认是否被幂等误拦截

```bash
# 查幂等表 / Redis 去重记录
SELECT * FROM idempotent_record WHERE request_id = 'pay-notify-12345';

# Redis
GET idempotent:pay-notify:12345
```

**可能发现**：
- 有记录但订单未更新 → 第一次消费失败了，但幂等记录已写入 → 后续重试全被拦截
- 这是最常见的**隐性丢消息**根因

## 三、6 种常见根因

| 排名 | 根因 | 出现频率 | 排查难度 |
|------|------|---------|---------|
| 1 | **第三方回调 URL 配置错误 / 证书过期** | ⭐⭐⭐⭐⭐ | 低（查 access.log） |
| 2 | **消费异常后直接 ACK，消息被丢弃** | ⭐⭐⭐⭐ | 中（查消费日志 + 异常堆栈） |
| 3 | **幂等误拦截：第一次失败但幂等记录已写入** | ⭐⭐⭐⭐ | 高（需对比幂等表 + 订单表） |
| 4 | **消费超时被重新投递到死信队列** | ⭐⭐⭐ | 中（查死信队列） |
| 5 | **Broker 异步刷盘 + 宕机丢消息** | ⭐⭐ | 低（查 Broker 日志） |
| 6 | **网络分区 / 主从切换丢消息** | ⭐ | 高（查 Broker 集群监控） |

## 四、短期止血 + 长期方案

### 短期止血（10 分钟内）

```
1. 确认是单条还是批量
   ├── 单条 → 手动触发补偿（调用支付查询接口确认状态，手动更新订单）
   └── 批量 → 切降级：临时关闭支付回调，改为主动轮询支付网关查状态

2. 通知客服团队：已知问题 + 预计恢复时间

3. 对账脚本紧急跑一轮：
   SELECT o.order_id, o.status, p.pay_status
   FROM orders o
   LEFT JOIN payment_records p ON o.order_id = p.order_id
   WHERE o.status = 'PENDING' AND p.pay_status = 'SUCCESS';
```

### 长期方案

| 层面 | 方案 | 说明 |
|------|------|------|
| **生产端** | 发送确认 + 重试 | 同步发送 + 失败重试 3 次 + 发送端日志告警 |
| **生产端** | 第三方回调验签 + 主动查询 | 不仅依赖回调，定时轮询支付网关确认状态（兜底） |
| **MQ** | 同步刷盘 + 多副本 | RocketMQ: `flushDiskType=SYNC_FLUSH` + 3 副本 |
| **消费端** | 消费失败重试 + 死信队列告警 | 重试 5 次后进死信，死信队列接告警 + 自动补偿 |
| **消费端** | 幂等先查后写 | 先查幂等表 → 不存在才处理 → 处理成功后写幂等（事务内） |
| **业务层** | 定时对账任务 | 每 5 分钟对比支付网关 + 订单表，发现不一致自动补偿 |
| **监控** | 全链路 Trace | 从支付回调 → MQ → 消费 → DB 更新，全链路 traceId 贯穿 |

## 五、面试话术（90 秒版本）

> "这道题我会从全链路排查的角度来回答。
>
> 支付成功但没收到消息，消息经过 5 个环节：第三方支付回调、生产端发送、MQ 存储、消费端拉取、业务层处理。每个环节都可能丢消息。
>
> **排查思路**是 5 步定位法：先查生产端是否发出消息 → 再查 MQ 是否有这条消息 → 然后查消费者是否拉取 → 再查消费逻辑是否执行成功 → 最后查是否被幂等误拦截。
>
> **最常见的 3 个根因**是：回调 URL 配置错误或 HTTPS 证书过期、消费异常后直接 ACK 丢消息、幂等第一次失败但记录已写入导致后续重试全被拦截。
>
> **短期止血**：手动补偿 + 对账脚本。**长期方案**：发送端重试 + MQ 同步刷盘 + 消费端重试 + 死信告警 + 定时对账 + 全链路 Trace。
>
> 另外，不能只依赖回调，要做**主动轮询兜底** — 定时查支付网关确认状态，这是支付宝和微信支付官方文档都推荐的做法。"
