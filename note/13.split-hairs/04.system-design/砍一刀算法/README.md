## Q1：病毒系数 K-Factor 是什么？怎么算？

### 陷阱

- ❌ 只背公式 `K = I × Conv`（不够深）
- ❌ 不知道 `K = 1` / `K > 1` / `K < 1` 的阈值意义

### 30s 话术

> **K-Factor = I × Conv**（每人发出邀请数 × 每邀请转化率）。**K > 1 → 病毒传播 / 指数增长**；**K = 1 → 自维持**；**K < 1 → 衰减**。

### 90s 话术

> K-Factor 的思想起源于流行病学的 **R₀（基本传染数）**，Hotmail 1996 年的邮件签名档是商业病毒营销的经典早期案例。
>
> 放到**拼多多砍一刀**场景：A 发起活动 → 分享到微信 → B 帮忙砍价、进度看似大增 → 倒计时继续逼单。平台并不一定追求长期 `K > 1`，而可能把真实 K 控制在 **0.85～0.95**：传播会衰减，但每轮仍能持续拉新。后期砍价力度从“砍 10 元”骤降为“砍 0.01 元”，让用户感觉逼近完成，却需要继续扩散。
>
> 关键设计原则有四个：**① 胜负欲 + 进度可视 ② 限时紧迫 ③ 福利激励 ④ 信任展现（好友助力记录）**。实战中可用 `Custs(t) = Custs(0) × K^t / (1 + ...)` 表达带衰减或饱和项的增长趋势；此外还要关注**病毒循环时间 CT**，CT 越短，同样 K 值下传播越快。

## Q2：砍一刀的“业务建模”4 步是什么？为什么后期砍不动？

### 陷阱

- ❌ 只答“前后期金额不同”（太浅）
- ❌ 不知道衰减函数如何设计

### 30s 话术

> 4 步：**① 库存锁定（生成订单 / 份额）② 衰减函数分配砍价金额（前大后小）③ 社交关系链验证（谁帮砍）④ 兜底回收（超 24h 完不成 → 退还库存）**。

### 90s 话术

> **核心是衰减函数设计**。线性分配，例如每个人都砍 5 分，很容易被用户算出剩余人数；指数衰减则让前期砍 30 / 20 / 10 元，后期只砍 0.01 / 0.01 元，看似无限接近目标，实际剩余 0.01 元也可能还需邀请 100 人。
>
> 公式可选 `price(t) = base × e^(-k·t)`，也可采用更容易运营调参的分段函数：**0～50% 进度砍 70% 金额，50～90% 砍 25%，90～99% 砍 5%**。完整业务链路还要覆盖库存或权益锁定、助力幂等、关系链真实性验证、活动超时后的库存回收。
>
> 业务侧还要关注三个约束：**① 兜底回收率 ≤ 30%，否则平台亏损 ② 社交验证强度，新用户与老用户的砍价金额不同 ③ 反作弊校验，同 IP / 设备多次帮砍失效**。最终用“总曝光 → 助力转化 → 裂变系数 → 新客留存 → GMV 转化”的全链路指标判断活动价值，而不能只看分享次数。

## Q3：风控引擎 5 层架构是什么？数据怎么流？（重点风控）

### 陷阱

- ❌ 只答“规则引擎”（远不够）
- ❌ 不知道 5 层之间的数据流与反馈闭环

### 30s 话术

> 5 层 = **数据 / 特征 / 规则 / 模型 / 处置**。数据层采集 → 特征层计算 velocity / list → 规则层匹配 Groovy 脚本 → 模型层 XGBoost 评分 → 处置层拒绝 / 验证 / 审核。

### 90s 话术

> **数据层**接入用户行为事件流、黑 / 灰名单和设备指纹；浏览器侧可组合 Canvas、WebGL、Audio 等维度，提高设备识别能力。**特征层**用 RiskFactor 模式把历史行为加工为 velocity（速度类，如“同设备 24h 注册数”）与 list（名单类，如“是否命中黑产设备库”）。
>
> **规则层**用 Drools / Groovy 动态规则承载业务专家经验，既支持评分卡累加，也支持“最坏匹配”——命中最高风险等级就短路。**模型层**用 XGBoost / LightGBM 离线训练、在线毫秒级推理，补足规则难以识别的未知模式。**处置层**根据综合风险执行拒绝、通过、二次验证或人工审核，并把黑名单、审核结果和最终标签写回数据层，形成训练集回流闭环。
>
> 工程重点是同步决策链路的延迟预算。若目标是 **30ms 内**完成判定，就要让数据读取、特征计算、规则匹配、模型推理分别受控，并通过特征预计算、本地缓存、规则编译和模型压缩削减长尾；不能把“五层”理解成五次串行远程调用。

## Q4：黑产 7 类对抗怎么做？风险评分卡如何设计？（重点风控）

### 陷阱

- ❌ 只列举攻击但不回答对应防御（偏科）
- ❌ 不知道评分卡的特征、分值与处置阈值如何设计

### 30s 话术

> 黑产 7 类 = **猫池 / 虚拟号 / 代理 IP / 设备伪造 / 打码 / 群控 / AI 脚本**。评分卡按设备、行为、IP、名单等维度加权，例如 `score = 设备权重 0.4 + 行为 0.3 + IP 0.2 + 名单 0.1`，累计风险分达到 80 分就拒绝。

### 90s 话术

> 黑产 7 类及其防御要成对回答：
>
> 1. **猫池**（64 / 128 卡槽批量收码）→ 号码画像 + 注册速度特征；
> 2. **虚拟手机号**（一卡多收验证码）→ 号段信誉 + 运营商校验；
> 3. **代理 IP 池**（机房 IP / 秒拨 IP）→ IP 画像 + ASN / 地理跳变；
> 4. **设备伪造工具**（改机 + Xposed + Hook）→ 多维设备指纹 + 完整性检测；
> 5. **打码平台**（OCR + AI 解图形验证码）→ 动态挑战 + 行为验证；
> 6. **群控**（百台手机批量操作）→ 设备关联图 + 同步行为聚类；
> 7. **AI 自动化脚本**（基于 LLM 的真实模拟）→ 长序列行为建模 + 持续对抗样本更新。
>
> 评分卡可这样落地：设备指纹命中黑名单库 **+40 分**；同一设备 24h 注册 ≥ 3 次 **+25 分**；IP 命中 IDC 机房 **+20 分**；行为序列异常，例如无滑动只有点击，**+15 分**。总分 **≥ 80 拒绝，≥ 60 二次验证，< 60 通过**。
>
> 关键洞见有三个：**① 单维度必败**，定制 ROM 能绕过单纯设备指纹；**② 必须多维交叉**，设备 + 行为 + 名单 + IP 同时异常才判高风险；**③ 模型 + 规则双轨**，规则可解释、便于运营调参，模型精度更高、擅长发现未知攻击。评分卡权重必须基于历史样本校准，而不是永久写死。
