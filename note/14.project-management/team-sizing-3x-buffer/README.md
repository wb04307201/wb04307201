# 人力配比 + 排期估算：3 倍缓冲原则

> PM 和创业公司 CTO 都常踩的坑：**拍脑袋排 2 周，实际做 6 周；招 3 个工程师，实际产出当 1 个用**。本文用阿里 P5/P6/P7 "2-8-2" 黄金比例 + 排期 3 倍缓冲原则，给出可落地的实战框架。

> **前置知识**：[app-quote-breakdown](../app-quote-breakdown/README.md)（12 大成本拆解）、[self-vs-saas-vs-outsourcing](../self-vs-saas-vs-outsourcing/README.md)（自研 vs 外包）、[outsourcing-pitfalls](../outsourcing-pitfalls/README.md)（外包隐性成本）。

---

## 引言：拍脑袋拍 2 周，实际做 6 周

```text
PM 拍板排期："这个功能，2 周搞定。"
老板追问："确定？"
PM 回："确定，我已经算过了。"

实际开发：
  第 1 周：需求理解，跟用户对齐
  第 2 周：技术方案讨论，2 个分歧，最后改架构
  第 3 周：核心模块写完，开始联调
  第 4 周：测试，发现 5 个 bug，修复
  第 5 周：技术债爆发，被迫重构
  第 6 周：上线，发现遗漏，新一轮修复

老板困惑："你不是说 2 周？"

PM："计划没问题，执行出了偏差。"
```

**真相**：**估算 2 周 = 实际 6 周** 不是计划失误，是 **3 倍缓冲** 没在排期里。

> 公式：**真实工期 = 乐观估算 × 2-3**（项目经理不该用乐观估算，而该做"中位数 × 1.5"或"乐观 × 2.5"）。

**这件事的本质**：不是 PM 不会排，是 **大型软件工程里"精确估算"几乎不可能**。

PM 的核心能力不是"估准"，是 **"留 buffer + 风险预警 + 实时同步"**。

## 一、人力配比：阿里 "2-8-2" 黄金模型

### 模型来源

阿里 P5（初级）/ P6（中级）/ P7（高级）通常按 2-8-2 比例配比。
- **20% 高级 P7+**：能扛方向，能做架构决策，能 cover 复杂 bug
- **80% 中级 P6**：主力写代码，能独立完成模块，**最重要的一群人**
- **20% 初级 P5**：能辅助、能学，但需要有人带

### 为什么不是"3-5-2"或"4-4-2"？

| P:X:Y | 适用阶段 | 问题 |
|------|---------|------|
| 5-3-2（高级过多）| **早期创业公司** | 高级带不动，业务不清晰 → 高级很快走 |
| 3-5-2 | **成长期 30 人** | 中级过少，新人成长慢 |
| **2-8-2** | **稳定期 50-200 人** | 阿里的"黄金比例"，最常见 |
| 1-6-3 | **后期转型 / 维护期** | 主导维护，新人磨合成本低 |
| 0-5-5 | **毕业生训练营** | 中级带新人，全员学习 |

### 看 P6 的级别分布更细

```text
P5 → P6 跳转率：60%（能独立完成模块）
P6 → P7 跳转率：30%（能做架构决策）

P6 是最关键的"主力 + 跳板"。
```

### 团队不健康的表现

| 现象 | 原因 | 修法 |
|------|------|------|
| **P5 太多（>40%）** | 招不到人 / 留不住人 / 项目压低薪资 | 调薪 + 给 P6 明确晋升标准 |
| **P7 太多（>30%）** | **架构蔓延**：每个 P7 都想造一套"自己的轮子" | 架构治理 / 标准化 |
| **P6 太少（<50%）** | 招不到 P6 / P7 抢活 / 上升通道窄 | 给 P6 明确成长路径 |

## 二、排期 3 倍缓冲原则

### 经验公式

```text
真实工期 = 乐观估算 × 2.5（中等项目）
真实工期 = 乐观估算 × 3.0（首次 / 复杂 / AI 时代 / 跨团队）
真实工期 = 乐观估算 × 1.5（成熟项目 / 个人独立）
```

### 为什么是 2-3 倍？（3 个底色）

1. **新手 / 跨团队 / 第一次的需求**：3 倍
2. **有经验 + 内部团队 + 反复做过**：2 倍
3. **AI Coding 时代**：**估算 1 倍速度，但 review + 测试 + 改架构 = 3 倍总时间** —— 与传统项目类似

### 排期表的真实格式

```text
❌ 错误格式：
  功能 X 实现：2 周（5/1 - 5/12）

✅ 正确格式：
  功能 X 实现：
    - 需求对齐：1 周
    - 设计评审：0.5 周
    - 编码（含 review + 测试）：2 周
    - 测试 + 修复：1 周
    - 部署 + 监控：0.5 周
    总计 5 周 = 1 周（清晰需求后）× 5 倍 buffer

  开发 buffer（不要写死）：
    - 风险预警：开个"风险登记册"，每发现一个风险 +0.5 周
    - 中期复盘：第 3 周复盘，发现偏差立即调整
```

### 5 大反例：把估算做错

| 反例 | 失败模式 | 正确做法 |
|------|---------|---------|
| ① **"2 周搞定"** | 老板高兴，3 个月后崩 | 中位数估算 + 加 buffer |
| ② **"我们组员很牛，1 周行"** | 牛人也补不上 4 个会议、3 个 on-call、1 个家人生病 | buffer 不看"个人效率"，看"团队平均效率" |
| ③ **"用 AI Coding 能快 3 倍"** | 写入时间 -50%，但 review 时间 +200% | 速度 ÷ 3 ≠ 省时 ÷ 3 |
| ④ **"我之前做过类似的，应该很快"** | 类似 ≠ 完全相同，**需求 + 环境总有差异** | 不盲目 2 倍，3 倍更稳 |
| ⑤ **"凡客一座"**：老板说"再给我加一个需求"，PM 不改排期 | 加需求 + 加时间，**不可省** | 改排期 / 砍需求 / 延后 |

## 三、AI 时代的 3 个修正（2026+）

### 修正 1：估算 ≠ 真实产能

| 指标 | 传统项目 | AI Coding 时代 |
|------|---------|--------------|
| 写代码速度 | 1x | **2-3x** |
| **review 时间** | 0.2x | **0.8x**（AI 写得多，bug 可能多） |
| **测试时间** | 0.3x | 0.5x（边界条件更多） |
| **架构讨论** | 0.2x | 0.3x（AI 不懂业务） |
| **净速度** | 1.7x | **~2x**（review + 测试 + 改架构拉低净收益） |

→ **AI 加速度 ≠ 净加速度**。写代码 3 倍快，但 review + 测试 + 改架构耗费 2 倍时间。**净提升 ~2 倍，绝不是 3 倍**。

### 修正 2：Cursor 写完 ≠ 工程师交付

```text
常被忽视的"AI Coding 配套设施"：
  - Harness（规范）—— 让 AI 不写烂代码
  - Code Review —— 发现 AI 写的边界条件 bug
  - 单元测试 —— AI 写代码不会主动跑测试
  - ADR / 设计文档 —— AI 不懂业务决策
```

→ 即使 1 人 + Cursor 写完所有代码，仍需要 **2-3 个工程师 review / 测试 / 部署**。
→ "团队人数 × AI 效率" 不等于 "产能"。[12.story/43](../../12.story/43-ai-productivity-paradox.md) 数据：AI Coding 工具当年渗透率 80%，但留存率 30%。

### 修正 3：人才评估不再是"loc / 人 / 月"

- 传统：loc / 月 / dev（代码行）
- AI 时代：**PR 数（被合并的）+ review 数 + 架构贡献 + 业务影响**
- 重构：Harness + Cursor + Claude Code 这套工具能力，是基本素养；**不是"加分"**

## 四、人力估算 3 步法（PM 必背）

### 步骤 1：拆解任务为故事点

```text
每个故事点 = 1 个独立可交付的功能。
复杂任务 = 多个故事点的合计。

例：用户登录模块
  SP-1：账号密码登录
  SP-2：手机号验证码登录
  SP-3：忘记密码流程
  SP-4：第三方 OAuth（Google/GitHub）
  SP-5：单元测试 + 集成测试
  SP-6：UI 适配 + a11y
```

### 步骤 2：每个故事点估时

```text
口径：中位数估计（约 50% 把握能完成）
工具："T恤码"（S/M/L/XL）或数字（1/2/3/5/8）
  S / 1：< 0.5 天
  M / 2：0.5-1 天
  L / 3：1-3 天
  XL / 5：3-5 天
  XXL / 8：> 5 天 → 必须拆分
```

### 步骤 3：加 buffer 得真实工期

```text
单点估算：中位数时间
总工期 = Σ(各点中位数) × 2.5（中等项目）/ × 3（首次 / 复杂）
```

### 实例：登录模块

```text
SP-1（中）：1 天
SP-2（中）：1 天
SP-3（小）：0.5 天
SP-4（大）：3 天
SP-5（中）：1 天
SP-6（小）：0.5 天
Σ = 7 天

真实工期（中位数 × 2.5） = 17.5 天 = 4 周
≈ 1 个月（按月度排期单元给老板看）
```

## 五、实战 checklist（PM 排期前 7 项）

- [ ] **任务拆解**：拆成独立可交付的故事点
- [ ] **估算方式**：用"中位数估算"，不要用"乐观估算"
- [ ] **buffer 系数**：中等项目 × 2.5，首次 / AI 时代 × 3
- [ ] **风险清单**：先列 3-5 个潜在风险，每个 +0.5 周 buffer
- [ ] **中期复盘**：第 3 周检查进度，及时调整后续工时
- [ ] **owner 分配**：每个故事点明确"由谁负责"，避免多人协调拖慢
- [ ] **buffer 可见**：buffer 不是"secret reserve"，**写在进度表里**，让全员可见避免"延后释放"

## 六、避坑指南（6 大坑）

| 坑 | 后果 | 避坑 |
|----|------|------|
| ① **拍 2 周，按 2 周交付** | 延期 3 倍 | 总 buffer ≥ 2.5 倍 |
| ② **招 3 个新人组 8 人团队** | 4 周只能产 1 人份 | 阿里 2-8-2 配比 |
| ③ **老板让加需求，不动排期** | 排期永远不准 | 加需求 + 加时间 |
| ④ **"凡客一座"**：PM 压 dev 完成 | dev 离职 / 跳槽 | buffer 是底线 |
| ⑤ **跨团队协作按"自己团队节奏"排** | A 等 B，B 等 C | 跨团队 × 3 倍 |
| ⑥ **"AI Coding 万能"**：缩减 dev 数 | AI 加速部分，但 review 没减 | **总人数 -20%，但 buffer 不动** |
