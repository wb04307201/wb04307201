## 二、控制平面（Control Plane）

控制平面是 K8s 的"大脑"，负责管理集群状态和调度决策。

### 2.1 API Server

- **作用**：K8s API 的唯一入口，所有组件都通过 API Server 通信
- **特性**：RESTful API + 认证 + 授权 + 准入控制
- **类比**：相当于公司的"前台"，所有请求都要经过它

```bash
# 通过 API Server 访问集群
kubectl get pods  # 实际是 kubectl → API Server → etcd
```

### 2.2 etcd

- **作用**：K8s 的分布式 KV 存储，保存所有集群状态
- **特性**：Raft 共识算法，强一致性
- **重要性**：丢了 etcd = 丢了整个集群

```bash
# etcd 数据示例
/registry/pods/default/nginx-abc123
/registry/services/default/nginx
```

### 2.3 Scheduler（调度器）

- **作用**：决定 Pod 调度到哪个 Node
- **调度策略**：
  - 资源（CPU / 内存）
  - 亲和性（affinity / anti-affinity）
  - 污点容忍（toleration）
  - 拓扑分布（zone / region）

### 2.4 Controller Manager（控制器集合）

- **作用**：通过"控制循环"调谐集群状态（声明式 → 实际状态）
- **核心控制器**：
  - Deployment Controller：保证副本数
  - ReplicaSet Controller：管理 Pod 副本
  - Node Controller：监控 Node 健康
  - Endpoints Controller：维护 Service Endpoints

## 三、工作节点（Worker Node）

工作节点是运行 Pod 的"工人"，每个 Node 上运行 3 个核心组件。

### 3.1 kubelet（节点代理）

- **作用**：API Server 与 Node 之间的"通信兵"
- **职责**：
  - 接收 Pod 规格（PodSpec）
  - 启动 / 停止容器
  - 健康检查（liveness / readiness probe）
  - 上报 Node 和 Pod 状态

### 3.2 kube-proxy（网络代理）

- **作用**：维护 Node 上的网络规则，实现 Service 负载均衡
- **实现方式**：
  - iptables（默认）
  - IPVS（高性能）
  - eBPF（最新）

### 3.3 容器运行时（Container Runtime）

- **作用**：真正运行容器的软件
- **主流选择**：
  - containerd（K8s 默认）
  - CRI-O
  - Docker（已弃用为 K8s 运行时，但仍可作为构建工具）

## 四、Addons（核心插件）

| 插件 | 作用 |
|------|------|
| **kube-proxy** | 网络代理（前面讲过）|
| **CoreDNS** | 集群 DNS（Service 发现）|
| **CNI 插件** | 容器网络（Flannel / Calico / Cilium）|
| **Ingress Controller** | 7 层路由（Nginx / Traefik）|
| **Metrics Server** | 资源指标（CPU/内存）|
| **Dashboard** | Web UI |

## 五、核心工作流程（Pod 创建示例）

```
① kubectl apply -f pod.yaml
   ↓
② kubectl → API Server 提交 Pod 规格
   ↓
③ API Server → etcd 持久化 Pod 规格
   ↓
④ Scheduler 监听 Pod 列表（新 Pod 待调度）
   ↓
⑤ Scheduler → API Server 更新 Pod.Spec.NodeName
   ↓
⑥ kubelet 监听本节点 Pod 变化
   ↓
⑦ kubelet 调用 Container Runtime 启动容器
   ↓
⑧ kubelet → API Server 上报 Pod 状态
   ↓
⑨ etcd 更新 Pod 状态为 Running
```

## 六、高可用（HA）部署模式

| 模式 | 适用 | 成本 |
|------|------|------|
| **单 Master** | 开发 / 测试 | 低 |
| **多 Master（3 节点）** | 生产标准 | 中 |
| **多 Master（5 节点）** | 大型生产 | 高 |
| **托管 K8s（EKS/AKS/GKE）** | 云用户 | 按用量 |

## 七、运维关键点

1. **etcd 备份**：每天至少一次全量备份 + 实时增量
2. **证书管理**：K8s 集群证书默认 1 年，到期前必须续签
3. **升级策略**：次版本最多落后 2 个（如 1.29 / 1.30 / 1.31）
4. **监控告警**：控制平面组件 + 工作节点 + Pod 都要监控
5. **灾难恢复**：备份 + 多可用区 + 跨区域复制
